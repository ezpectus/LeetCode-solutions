# 2957. Remove Adjacent Almost-Equal Characters
## O(n) — Greedy Skip-One Scan

---

## Problem Statement

- Given string `word`, in one operation pick any index and change its character to any lowercase letter.
- Two characters `a` and `b` are **almost-equal** if `a == b` or they are adjacent in the alphabet (`|a - b| ≤ 1`).
- Return the **minimum number of operations** to remove all adjacent almost-equal pairs.

---

## Core Idea — Greedy: Fix a Conflict, Skip the Next Index

**Key insight**: whenever we find two adjacent almost-equal characters at positions `i` and `i+1`, we must change one of them. The optimal greedy choice is to **change `word[i+1]`** (the right one) and then **skip to `i+2`**.

Why change the right character and skip?
- Changing `word[i+1]` resolves the conflict at `(i, i+1)`.
- We can always choose a new value for `word[i+1]` that is not almost-equal to either `word[i]` or `word[i+2]` (26 letters − at most 6 neighbors → always room).
- By skipping to `i+2`, we don't re-examine `i+1` as a left partner — we've already handled it optimally.
- This greedy is correct because changing the right element never worsens future conflicts compared to changing the left.

**Algorithm**:
- Walk `i` from `0` to `n-2`.
- If `|word[i] - word[i+1]| ≤ 1`: increment count, skip to `i+2` (advance `i` by 2 total).
- Otherwise: advance `i` by 1.

---

## Full Optimal Implementation (C#)

```csharp
public class Solution {
    public int RemoveAlmostEqualCharacters(string word) {
        int ops = 0;
        int i = 0;
        while (i < word.Length - 1) {
            if (Math.Abs(word[i] - word[i + 1]) <= 1) {
                ops++;
                i += 2; // skip i+1, it's been "fixed"
            } else {
                i++;
            }
        }
        return ops;
    }
}
```

| **Metric** | **Value** | **Notes** |
|---|---|---|
| **Time** | **O(n)** | Single left-to-right pass, each index visited at most once |
| **Space** | **O(1)** | Only scalar counter and index |

---

## Why This Works — The Skip Argument

**Why is skipping `i+1` always safe?**

When we detect a conflict at `(i, i+1)` and decide to "change" `word[i+1]`, we conceptually set it to some character that is:
- Not almost-equal to `word[i]` (resolves current conflict)
- Not almost-equal to `word[i+2]` (avoids creating a new conflict)

Such a character always exists: `word[i]` and `word[i+2]` together exclude at most 6 letters (`word[i] ± 1` and `word[i+2] ± 1`), leaving at least 20 choices. So we can always find a valid replacement, meaning we never need to touch `i+1` again → safe to skip to `i+2`.

**Why not change the left character `word[i]` instead?**

Changing `word[i]` would require us to re-examine the pair `(i-1, i)` which we already passed — potentially invalidating prior work. Changing the right character and moving forward is always at least as good.

**Why is this greedy globally optimal?**

Any two conflicting pairs `(i, i+1)` and `(i+1, i+2)` that share index `i+1` can be resolved with **one operation** (change `word[i+1]`). The greedy skip ensures we always resolve overlapping conflicts with the minimum single operation, never "wasting" operations on non-overlapping pairs.

## Pitfalls & Edge Cases

- **`|word[i] - word[i+1]| ≤ 1`**, not `== 0`: almost-equal includes adjacent letters, not just equal ones — don't forget the ±1 case
- **Skip by 2, not 1**: after a fix, advance `i += 2` to avoid revisiting the patched character as a new left partner
- **Single character string**: `word.Length == 1` → loop never runs → answer 0, handled naturally
- **All distinct non-adjacent characters** like `"ace"`: no pairs conflict → answer 0

---

## Key Takeaway

"Remove all adjacent almost-equal pairs with minimum changes" → **greedy: scan left to right, on conflict fix the right character and skip it**:

```
i = 0
while i < n - 1:
    if |word[i] - word[i+1]| <= 1:
        ops++
        i += 2   // skip fixed character
    else:
        i += 1
return ops
```

Fixing the right character of each conflict and skipping guarantees no re-work and always leaves room for a valid replacement. O(n) time, O(1) space.
