# 2571. Minimum Operations to Reduce an Integer to 0
## O(log n) — Greedy Bit Manipulation on Consecutive 1-Runs

---

## Problem Statement

- Given a positive integer `n`. In one operation you can **add or subtract any power of 2** from `n`.
- Return the **minimum number of operations** to make `n` equal to `0`.

---

## Core Idea — Greedy on Binary Representation

- Each operation adds or subtracts exactly one power of 2 — meaning it **flips a single bit** (possibly with carry propagation). The goal is to clear all bits.
- Key insight: an **isolated 1-bit** costs 1 operation (just subtract it). A **run of consecutive 1-bits** also costs only 2 operations — add the lowest power to carry them all into a single higher bit, then subtract that bit.
- So the problem reduces to: **count the number of isolated bits and consecutive 1-runs** in the binary representation of `n`.
- We simulate this greedily: scan bits low to high, and whenever we hit a run of consecutive 1s, merge them upward via carry rather than clearing each one individually.

---

## Clean Implementation (C#)

```csharp
public class Solution {
    public int MinOperations(int n) {
        int ops = 0;

        while (n != 0) {
            if ((n & 1) == 0) {
                // Trailing zero — just shift it away
                n >>= 1;
            } else if ((n & 3) == 3) {
                // Two consecutive 1-bits — add 1 to merge the run upward (carry)
                n += 1;
                ops++;
            } else {
                // Isolated 1-bit — subtract it (clear the lowest bit)
                n -= 1;
                ops++;
            }
        }

        return ops;
    }
}
```

---

## Complexity

| Metric | Value | Notes |
|--------|-------|-------|
| **Time Complexity** | O(log n) | At most O(log n) bits; each bit is processed at most twice (once as 1, once as carry) |
| **Space Complexity** | O(1) | No extra memory — operates on `n` in place |

Practical — `n ≤ 10⁵ < 2¹⁷`, so the loop runs at most ~18 iterations.

---

## Why This Works — Example Walkthrough

**Example 1:** `n = 39 = 0b100111`

| Step | n (binary) | action | ops |
|------|-----------|--------|-----|
| start | `100111` | — | 0 |
| `n & 3 == 3` | `100111` | add 1 → `101000` | 1 |
| `n & 3 == 3` | `101000` | shift zeros away → `101` | — |
| `n & 1 == 1`, `n & 3 != 3` | `101` | subtract 1 → `100` | 2 |
| shift | `100` → `1` | — | — |
| isolated | `1` | subtract 1 → `0` | 3 |

Output: `3` ✓

**Example 2:** `n = 54 = 0b110110`

The two separate runs of `11` each cost 2 ops, but the carry from the first merge creates a third `1` that merges again — total 3 ops.

**Key insight on carry propagation:**

`0b0111` (three consecutive 1s) — adding 1 gives `0b1000`. That's one carry-merge turning 3 individual bits into 1 bit, costing just 1 operation. Without merging, clearing three isolated 1s would cost 3 operations. The greedy always checks if merging is cheaper.

---

## Pitfalls & Edge Cases

- **`n & 3 == 3` check** — this catches *any* two consecutive 1s starting at the lowest bit; after shifting away trailing zeros, this is always the correct merge condition
- **Carry creates new runs** — after `n += 1`, the result may have new consecutive 1s at higher bits; the loop handles this naturally since we recheck from the new lowest bit
- **Don't check `n & 3 == 1` separately** — `(n & 1) == 1` and `(n & 3) != 3` together already mean an isolated 1-bit; order of the `if/else` branches matters
- **n = 1** — single bit, one subtract, returns 1 ✓
- **n = power of 2** — already a single bit, one subtract, returns 1 ✓
- **n = 2^k - 1** (all 1s) — carry merges propagate the entire number up to the next power, then one subtract clears it; cost is 2 ✓

---

## Key Takeaway

Think in **binary** — each operation is a single-bit flip with possible carry. The greedy rule is:

- **Isolated 1-bit** → subtract it (1 op to clear)
- **Consecutive 1-run** → add 1 to carry-merge the whole run into one higher bit (1 op now + 1 op later = 2 ops for the entire run, regardless of length)

This is strictly better than clearing each bit in a run individually. The loop naturally handles carry propagation by rechecking the updated `n` on each iteration.
One pass, no recursion, no DP — pure greedy bit manipulation.
