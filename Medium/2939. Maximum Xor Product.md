# 2939. Maximum Xor Product
## O(n) — Greedy Bit-by-Bit

---

## Problem Statement

- Choose `x` with `0 ≤ x < 2ⁿ` to maximize `(a XOR x) * (b XOR x)`.
- Return result mod 10⁹+7.

---

## Core Idea — Greedy from MSB, Balance the Two Values

XOR with `x` lets us **independently set each bit** of the result values (within the first `n` bits). For each bit position from MSB to LSB, decide whether to set `x`'s bit to 0 or 1.

**Case 1: Both `a` and `b` have the same bit (both 0 or both 1)**
- Both 0: set `x_bit = 1` → both values get this bit set → both increase → product increases ✓
- Both 1: setting `x_bit = 1` makes both 0 → worse. Keep `x_bit = 0`.

**Case 2: `a` and `b` have different bits (one has 0, one has 1)**
- One value will gain the bit, the other will lose it — no matter what we choose.
- By AM-GM: `(p+d)*(q-d) ≤ (p+q)²/4` — product is maximized when values are as close as possible.
- **Give the bit to the smaller value** to balance them.

After processing all `n` bits, `a` and `b` hold the final `(a XOR x)` and `(b XOR x)` values.

---

## Full Optimal Implementation (C#)

```csharp
public class Solution {
    public int MaximumXorProduct(long a, long b, int n) {
        const long MOD = 1_000_000_007;

        for (int bit = n - 1; bit >= 0; bit--) {
            long ba = (a >> bit) & 1;
            long bb = (b >> bit) & 1;

            if (ba == bb) {
                // Both 0: set both to 1 (gain this bit)
                // Both 1: already set, can't improve
                if (ba == 0) {
                    a |= (1L << bit);
                    b |= (1L << bit);
                }
            } else {
                // Give the bit to the smaller value
                if (a < b) {
                    a |= (1L << bit);
                    b &= ~(1L << bit);
                } else if (b < a) {
                    b |= (1L << bit);
                    a &= ~(1L << bit);
                }
                // Equal: either choice gives same product
            }
        }

        return (int)((a % MOD) * (b % MOD) % MOD);
    }
}
```

## Why "Give Bit to Smaller" is Optimal

For two values `p < q` and a choice to give a bit `d = 2^k`:
- Give to `p`: result = `(p+d) * (q-d) = p*q + d*(q-p) - d²`
- Give to `q`: result = `(p-d)... wait p<q and p doesn't have this bit, so give to p:
  (p+d)*(q-d) vs. keep: p*(q) — but this bit must go to one of them.

Between `(p+d)*(q-d)` and `(p)*(q)` — can't compare directly since `q` loses `d`.

Key: between `(p+d)*(q-d)` and `(p)*(q+d-d)=p*q`: the values move closer together when we give `d` to the smaller one → by AM-GM, closer values give larger product for the same sum. ✓

## Example Walkthrough

**`a=12 (1100), b=5 (0101), n=4`**

| bit | ba | bb | action | a | b |
|-----|----|----|--------|---|---|
| 3 | 1 | 0 | differ, a=12>b=5 → give to b | 12→4(0100) | 5→13(1101) |
| 2 | 1 | 1 | same 1, skip | 4→4 | 13→13 |
| 1 | 0 | 0 | same 0, set both | 4→6(0110) | 13→15(1111) |
| 0 | 0 | 1 | differ, a=6<b=15 → give to a | 6→7(0111) | 15→14(1110) |

Final: a=7, b=14. Product = 7×14 = **98** ✓

## Pitfalls & Edge Cases

- **`long` throughout**: `a, b < 2⁵⁰` → bit shifts and masks need `long` (64-bit)
- **`~(1L << bit)` for clearing a bit**: use `long` literal `1L` to avoid truncation
- **Bits above n**: bits in `a` and `b` that are ≥ n are never touched by `x` (x < 2ⁿ) → they contribute fixed amounts to the product; the greedy only optimizes bits 0..n-1

---

## Key Takeaway

"Maximize `(a XOR x) * (b XOR x)` over x < 2ⁿ" → **greedy bit-by-bit: both 0 → set both 1; differ → give to smaller to balance**:

```
if ba == bb == 0: set both to 1
if ba != bb: give the 1 to whichever is currently smaller
```

AM-GM drives the "give to smaller" choice — product maximized when values are as equal as possible.
