# 3947. Maximum Number of Items From Sale II
## O(n log n) — Bounded Knapsack Reduction via Price-Floor Greedy Selection

### Problem Statement
You are given a 2D integer array `items`, where `items[i] = [factor_i, price_i]` represents the i-th item type available in unlimited quantities. You are also given an integer `budget`. 

Each copy of item i purchased can grant at most one free copy of another item j if:
- i != j
- `factor_i` cleanly divides `factor_j` (`factor_j % factor_i == 0`)

For each ordered pair (i, j), you can receive a free copy of item j from purchases of item i at most once. However, item j can be received multiple times if triggered by different item types. Return the maximum total number of items (purchased + free) you can obtain within the budget.

### Core Idea — Yield Decomposition and Greedy Baseline
Let C_i be the number of copies we buy of item i, and let S_i be the total number of other items j (j != i) whose factors are multiples of `factor_i`. 

Because each purchase can clear at most one unique valid target item type, buying C_i copies of item i yields a total item count of:
- 0 items if C_i = 0
- C_i + min(C_i, S_i) items if C_i > 0

This splits the purchasing behavior of any item i into two phases:
1. **Yield-2 Phase:** The first S_i copies each give 2 total items (1 bought + 1 free) at the cost of `price_i`.
2. **Yield-1 Phase:** Any copies beyond S_i give only 1 item (1 bought + 0 free) at the cost of `price_i`.

Let `min_price` be the global minimum price among all items. The absolute best way to acquire items with a yield of 1 is to buy the item costing `min_price`. Therefore, we should only ever consider the Yield-2 phase of an item i if its cost-to-benefit ratio beats the baseline:

price_i < 2 * min_price

Any item satisfying this inequality gives a net positive contribution compared to the baseline. Since their individual selections are independent, we can sort all available Yield-2 packages greedily by `price_i` ascending and buy as many as the budget permits, dumping the remaining change into the `min_price` baseline item.

### Algorithm
1. **Frequency Count:** Count the occurrences of each factor using a frequency array. Find the global minimum price `min_price`.
2. **Sieve Multiplication:** Use a harmonic sieve-like loop to calculate `S_factor[f]`, which stores the count of all numbers in the array that are multiples of f. For each item i, S_i = S_factor[factor_i] - 1.
3. **Filter and Group:** Filter out items where S_i > 0 and price_i < 2 * min_price. Group them as available bundles of capacity S_i at unit cost `price_i`.
4. **Greedy Execution:** Sort these bundles by `price_i` ascending. Consume the bundles sequentially. If the budget cannot fully cover a bundle, buy the maximum possible integer copies from it.
5. **Baseline Clean-up:** Spend any remaining budget on the item with `min_price` to capture the final Yield-1 items.

   
### Full Optimal Implementation (C++)
```cpp
class Solution {
public:
    int maximumSaleItems(vector<vector<int>>& items, int budget) {
        int n = items.size();
        
        vector<int> count(n + 1, 0);
        long long min_price = 2e9;
        
        for (const auto& item : items) {
            count[item[0]]++;
            if (item[1] < min_price) {
                min_price = item[1];
            }
        }
        
        vector<int> S_factor(n + 1, 0);
        for (int f = 1; f <= n; ++f) {
            for (int multiple = f; multiple <= n; multiple += f) {
                S_factor[f] += count[multiple];
            }
        }
        
        struct YieldBundle {
            long long price;
            long long quantity;
        };
        vector<YieldBundle> bundles;
        long long threshold = 2 * min_price;
        
        for (const auto& item : items) {
            int f = item[0];
            long long price = item[1];
            int S_i = S_factor[f] - 1;
            
            if (S_i > 0 && price < threshold) {
                bundles.push_back({price, S_i});
            }
        }
        
        sort(bundles.begin(), bundles.end(), [](const YieldBundle& a, const YieldBundle& b) {
            return a.price < b.price;
        });
        
        long long total_items = 0;
        long long remaining_budget = budget;
        
        for (const auto& b : bundles) {
            if (remaining_budget <= 0) break;
            long long take = min(b.quantity, remaining_budget / b.price);
            total_items += 2 * take;
            remaining_budget -= take * b.price;
        }
        
        total_items += remaining_budget / min_price;
        
        return total_items;
    }
};
```

### Complexity
Metric | Value | Notes
Time Complexity | O(n log n) | Sieve takes O(n log n) steps. Sorting the filtered bundles takes O(n log n).
Space Complexity | O(n) | Frequency and bundle vectors scale linearly with the input size.

### Why "Greedy Sorting" Always Converges Correctly
Every Yield-2 item we purchase costs price_i and gives 2 items. If we instead saved that money to buy the baseline item, we would get floor(price_i / min_price) items. Since we filtered out items where price_i >= 2 * min_price, the baseline would give at most floor(price_i / min_price) = 1 item for that amount of money.

Thus, every conversion of budget into a Yield-2 choice guarantees a net marginal item profit. To maximize total items, we must maximize the number of distinct Yield-2 selections. Sorting by cost naturally maximizes the count of transactions we can complete before exhausting our fixed budget.

### Example Walkthrough
#### Example 1: items = [[1,6],[2,4],[3,5]], budget = 19
- **Initialization:**
  - min_price = 4 (from item 1)
  - threshold = 2 * 4 = 8
- **Sieve Factor Counts:**
  - Multiples of 1: item 0, item 1, item 2 -> S_0 = 3 - 1 = 2
  - Multiples of 2: item 1 -> S_1 = 1 - 1 = 0
  - Multiples of 3: item 2 -> S_2 = 1 - 1 = 0
- **Bundle Filtering:**
  - Item 0: price 6 < 8, S_0 = 2. Valid bundle: {price: 6, quantity: 2}
- **Greedy Strategy Pass:**
  - Process bundle {price: 6, quantity: 2}: take = min(2, 19/6) = 2.
  - total_items = 2 * 2 = 4. remaining_budget = 19 - 12 = 7.
- **Baseline Remainder Pass:**
  - Baseline purchases: 7 / min_price = 7 / 4 = 1.
  - total_items = 4 + 1 = 5.
- **Output:** 5 ✓ Matches expected.

### Pitfalls & Edge Cases
- **The O(n^2) Flattening Trap:** It's tempting to expand the bundles by pushing each copy individually into a flat array. However, if all items have factor = 1, then S_i = n - 1. Flattening would create O(n^2) entries, causing memory and time limit overflows. Keeping items grouped as quantity blocks keeps it O(n). ✓
- **Self-Exclusion Adjustment:** When computing S_i, you must subtract 1 from the raw sieve count because an item cannot target its own index to trigger a free copy. ✓
- **64-bit Integer Budgets:** Accumulating total items or performing multiplication checks with values up to 10^9 can easily result in integer overflow. Utilizing long long for cost and quantity arithmetic prevents truncation errors. ✓

### Key Takeaway
When an unbounded item problem changes profit efficiency at a specific item threshold, analyze the baseline value of the lower efficiency state. By treating the baseline as a fallback currency converter, the problem transforms from a scary unbounded knapsack problem into a straightforward greedy sorting procedure.
- **Pattern recognition:** "Multiple copies with bounded bonus yields + unbounded baseline" -> Sieve-driven capability tracking followed by unit cost greedy collection.
