# 2860. Happy Students
## O(n log n) — Sort + Enumerate Group Sizes

---

## Problem Statement

- Student `i` is happy if:
  - Selected AND total selected > `nums[i]`, OR
  - Not selected AND total selected < `nums[i]`
- Return the number of valid group sizes where **all** students are happy.

---

## Core Idea — Sort and Check Boundaries

If we select `k` students, to minimize the condition checks, **greedily select the `k` students with the smallest `nums[i]`** — this gives selected students the best chance of having `nums[i] < k`.

After sorting `nums`:
- **Selected students** `nums[0..k-1]`: each needs `nums[i] < k` → only need to check the **last one**: `nums[k-1] < k`
- **Not-selected students** `nums[k..n-1]`: each needs `nums[i] > k` → only need to check the **first one**: `nums[k] > k`

(Since sorted, if boundary elements satisfy the conditions, all others do too.)

Try every `k` from `0` to `n` — O(n) checks after O(n log n) sort.

---

## Full Optimal Implementation (C#)

```csharp
public class Solution {
    public int CountWays(IList<int> nums) {
        int n = nums.Count;
        int[] sorted = nums.OrderBy(x => x).ToArray();
        int res = 0;

        for (int k = 0; k <= n; k++) {
            bool selectedOk    = (k == 0 || sorted[k-1] < k);
            bool notSelectedOk = (k == n || sorted[k] > k);
            if (selectedOk && notSelectedOk) res++;
        }

        return res;
    }
}
```

## Why Selecting the Smallest k Elements is Optimal

For a group of size `k`, we need all selected students to have `nums[i] < k`. The students most likely to satisfy this are those with the **smallest** `nums[i]`. Any other selection of `k` students would have a larger maximum `nums[i]`, making it harder or impossible to satisfy `nums[i] < k` for all selected.

Similarly, not-selected students need `nums[i] > k` — the students most likely to satisfy this are those with the **largest** `nums[i]`, which are exactly the ones left over after choosing the smallest `k`.

## Example Walkthrough

**`nums=[6,0,3,3,6,7,2,7]`** → sorted: `[0,2,3,3,6,6,7,7]`, n=8

| k | sorted[k-1] < k? | sorted[k] > k? | valid? |
|---|-----------------|----------------|--------|
| 0 | — (k==0) ✓ | 0 > 0? ✗ | ✗ |
| 1 | 0 < 1 ✓ | 2 > 1 ✓ | **✓** |
| 2 | 2 < 2? ✗ | — | ✗ |
| 3 | 3 < 3? ✗ | — | ✗ |
| 4 | 3 < 4 ✓ | 6 > 4 ✓ | **✓** |
| 5 | 6 < 5? ✗ | — | ✗ |
| 6 | 6 < 6? ✗ | — | ✗ |
| 7 | 6 < 7 ✓ | 7 > 7? ✗ | ✗ |
| 8 | 7 < 8 ✓ | — (k==n) ✓ | **✓** |

Count = **3** ✓

## Pitfalls & Edge Cases

- **`k=0`**: no selected students, all must have `nums[i] > 0`. Check `sorted[0] > 0`.
- **`k=n`**: all selected, all must have `nums[i] < n`. Check `sorted[n-1] < n`.
- **Boundary checks**: `k==0` skips the selected check; `k==n` skips the not-selected check.
- **`nums[i] < n`** guaranteed by constraints → `k=n` is always worth checking.

---

## Key Takeaway

"Count valid group sizes for happiness constraints" → **sort + O(n) boundary check per group size**:

```
valid(k) = (k==0 OR sorted[k-1] < k) AND (k==n OR sorted[k] > k)
```

Selecting the smallest `k` elements is optimal — reduces all `k` selected conditions to a single boundary check. O(n log n) total.
