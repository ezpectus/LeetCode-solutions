# 2637. Promise Time Limit
## O(1) — Promise.race with Timeout Rejection

---

## Problem Statement

- Given async function `fn` and time limit `t` ms, return a time-limited version.
- If `fn` resolves within `t` ms → resolve with its result.
- If `fn` takes longer than `t` ms → reject with `"Time Limit Exceeded"`.
- If `fn` throws → reject with that error immediately.

---

## Core Idea — `Promise.race`

**Key insight**: `Promise.race([p1, p2])` settles with whichever promise **settles first** — resolved or rejected. Race `fn(...args)` against a timeout promise that rejects after `t` ms. Whoever wins determines the outcome.

```
Promise.race([
    fn(...args),                                    // resolves if fn finishes in time
    new Promise((_, rej) => setTimeout(() => rej("Time Limit Exceeded"), t))
])
```

- `fn` resolves first → `Promise.race` resolves with `fn`'s value ✓
- Timeout fires first → `Promise.race` rejects with `"Time Limit Exceeded"` ✓
- `fn` throws immediately → `Promise.race` rejects with that error ✓

---

## Full Optimal Implementation (JavaScript)

```javascript
var timeLimit = function(fn, t) {
    return async function(...args) {
        const timeout = new Promise((_, rej) =>
            setTimeout(() => rej("Time Limit Exceeded"), t)
        );
        return Promise.race([fn(...args), timeout]);
    };
};
```

## Complexity

| **Metric** | **Value** | **Notes** |
|---|---|---|
| **Time Complexity** | **O(1)** | Just sets up two promises and races them |
| **Space Complexity** | **O(1)** | Two promise references |

## How `Promise.race` Works

```
Time →  0ms ─────────── t ms ─────────── fn resolves
                          ↑
                    timeout rejects → race rejects immediately

OR:

Time →  0ms ─── fn resolves ─────────── t ms
                    ↑
              race resolves immediately, timeout still pending (but ignored)
```

The losing promise doesn't get cancelled — it still runs to completion in the background. But since `Promise.race` already settled, its outcome is ignored.

---

## Example Walkthrough

**Example 1**: `fn` takes 100ms, `t = 50ms`

```
t=0:   race starts — fn promise (resolves at 100ms) vs timeout (rejects at 50ms)
t=50:  timeout rejects first → race rejects "Time Limit Exceeded" ✓
t=100: fn resolves (ignored — race already settled)
```

**Example 4**: `fn` throws immediately

```
t=0: fn() throws "Error" → fn promise rejects instantly
     race rejects with "Error" before timeout ever fires ✓
```

---

## Pitfalls & Edge Cases

- **`t = 0`** — `setTimeout(..., 0)` fires async (next tick), so `fn` gets one event loop cycle to resolve synchronously before the timeout; in practice at `t=0` the timeout usually wins for any async `fn`
- **Timeout promise leaks** — after the race settles, the losing `setTimeout` still exists in memory until it fires. For `t=1000` this means a 1s dangling timer. In production you'd call `clearTimeout` on the loser, but for this problem it's acceptable
- **`fn` errors vs timeout** — both are rejections; `Promise.race` propagates whichever comes first regardless of whether it's a timeout or a `fn` error ✓
- **`async` wrapper** — the returned function is `async`, so it always returns a Promise; `return Promise.race(...)` inside `async` wraps the result in another Promise, but since `Promise.race` itself returns a Promise, JS unwraps it correctly (Promises don't double-wrap)

---

## Key Takeaway

Timeout wrapper for any async function = **`Promise.race` with a rejection timer**:

```javascript
const timeout = new Promise((_, rej) => setTimeout(() => rej("Time Limit Exceeded"), t));
return Promise.race([fn(...args), timeout]);
```

- `Promise.race` = first to settle wins
- Timeout promise = always rejects after `t` ms
- No additional logic needed — the race handles all cases: fast fn, slow fn, fn that throws

Pattern: any "do X but give up after Y ms" async operation → `Promise.race` with a timeout promise.
