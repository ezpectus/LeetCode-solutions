# 3804. Number of Centered Subarrays
## O(n²) — Brute Force with HashSet Membership Check

---

## Problem Statement

- You are given an integer array `nums` (1 ≤ n ≤ 500, -10⁵ ≤ nums[i] ≤ 10⁵).
- A subarray is **centered** if its sum equals at least one element within that same subarray.
- Return the total number of centered subarrays.

---

## Core Idea — Enumerate Subarrays, Track Elements and Running Sum

Key insight: For each subarray `[l, r]`, we need two things simultaneously:
- The set of elements in the subarray (to check if sum is present)
- The running sum

Approach: Fix left endpoint `l`, expand right endpoint `r`. For each expansion:
- Maintain a `HashSet` of elements seen so far
- Maintain a running `sum`
- Check if `sum` exists in the set

With n ≤ 500, O(n²) with O(n) HashSet operations per subarray gives O(n²) total — fast enough.

---

## Full Optimal Implementation (C#)

```csharp
public class Solution {
    public int CenteredSubarrays(int[] nums) {
        int n = nums.Length;
        int res = 0;

        for (int l = 0; l < n; l++) {
            var elements = new HashSet<int>();
            long sum = 0;

            for (int r = l; r < n; r++) {
                elements.Add(nums[r]);
                sum += nums[r];

                // Centered if sum equals any element in [l, r]
                if (elements.Contains((int)sum)) {
                    res++;
                }
            }
        }

        return res;
    }
}
```

---

## Complexity

| Metric | Value | Notes |
|---|---|---|
| Time Complexity | O(n²) | n² subarrays, O(1) HashSet lookup per expansion |
| Space Complexity | O(n) | HashSet holds at most n elements per left endpoint |

Optimal for n ≤ 500 — total operations ≈ 125,000.

## Why This Works — Example Walkthrough

**Example 1:** `nums = [-1, 1, 0]`

**l = 0:**
| r | Added | Set        | Sum | Sum in Set? |
|---|-------|------------|-----|-------------|
| 0 | -1    | {-1}       | -1  | ✓ res=1     |
| 1 | 1     | {-1, 1}    | 0   | ✗           |
| 2 | 0     | {-1, 1, 0} | 0   | ✓ res=2     |

**l = 1:**
| r | Added | Set     | Sum | Sum in Set? |
|---|-------|---------|-----|-------------|
| 1 | 1     | {1}     | 1   | ✓ res=3     |
| 2 | 0     | {1, 0}  | 1   | ✓ res=4     |

**l = 2:**
| r | Added | Set | Sum | Sum in Set? |
|---|-------|-----|-----|-------------|
| 2 | 0     | {0} | 0   | ✓ res=5     |

**Output:** 5 ✓

---

**Example 2:** `nums = [2, -3]`

**l = 0:**
| r | Added | Set      | Sum | Sum in Set? |
|---|-------|----------|-----|-------------|
| 0 | 2     | {2}      | 2   | ✓ res=1     |
| 1 | -3    | {2, -3}  | -1  | ✗           |

**l = 1:**
| r | Added | Set  | Sum | Sum in Set? |
|---|-------|------|-----|-------------|
| 1 | -3    | {-3} | -3  | ✓ res=2     |

**Output:** 2 ✓

## Pitfalls & Edge Cases

- **Single-element subarrays always centered** — For `[x]`, sum = x, and x is in the set → always counted. Minimum answer is always n.

- **Integer overflow** — With n = 500 and values up to 10⁵, max sum = 500 × 10⁵ = 5 × 10⁷, which fits in `int`. But with negative values mixed in, it's safe to use `long` for `sum` and cast to `int` before the HashSet lookup (since elements themselves are `int`-range).

- **Sum out of element range** — If `sum` overflows `int` range it can never equal an element, so `Contains((int)sum)` correctly returns false after the cast (the cast may wrap, but the original value can't match any element in the valid range).

- **Duplicate elements** — HashSet deduplicates, but that's fine — we only need to know if the sum value exists as an element, not how many times.

- **Negative sums** — Fully handled since both the HashSet and sum support negative integers.

---

## Key Takeaway

Straightforward **brute force enumeration** problem:
- Fix left, expand right, maintain set + running sum
- Single HashSet O(1) lookup per step → O(n²) total

**Pattern recognition:** "Subarray property involving both elements and their sum" → fix one endpoint, incrementally maintain data structure as you expand. No need for anything fancier when n ≤ 500.

The clever observation is that single-element subarrays are always valid (sum = the element itself = element in subarray), guaranteeing minimum answer of n.
