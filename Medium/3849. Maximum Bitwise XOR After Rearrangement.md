# 3849. Maximum Bitwise XOR After Rearrangement
## O(n) — Greedy: Maximize Each Bit Left to Right

---

## Problem Statement

- Binary strings `s` and `t`, both length n. `s` is fixed, `t` can be rearranged.
- Return the binary string of maximum value obtained by XOR of `s` and rearranged `t`.

---

## Core Idea — Greedy Bit Assignment from MSB to LSB

To maximize a binary number, prioritize setting the highest bits to 1.

XOR at position i is 1 iff `s[i] ≠ t[rearranged][i]`. So:
- If `s[i] = '0'`: need a `'1'` from t's pool to get XOR = 1.
- If `s[i] = '1'`: need a `'0'` from t's pool to get XOR = 1.

Greedy left to right: at each position, try to produce `'1'` by using the opposite bit from t's pool. If unavailable, produce `'0'` using whatever remains.

---

## Full Optimal Implementation (C#)

```csharp
public class Solution {
    public string MaximumXor(string s, string t) {
        int n = s.Length;
        int ones = 0, zeros = 0;

        foreach (char c in t) {
            if (c == '1') ones++;
            else zeros++;
        }

        var res = new char[n];

        for (int i = 0; i < n; i++) {
            if (s[i] == '0') {
                // Want '1' from t to get XOR=1
                if (ones > 0) { res[i] = '1'; ones--; }
                else          { res[i] = '0'; zeros--; }
            } else {
                // Want '0' from t to get XOR=1
                if (zeros > 0) { res[i] = '1'; zeros--; }
                else           { res[i] = '0'; ones--; }
            }
        }

        return new string(res);
    }
}
```

## Why Greedy Is Optimal

Binary values are compared lexicographically (highest bit wins). Setting bit i to 1 is always better than setting a lower bit, regardless of what happens to bits i+1 to n−1.

**Exchange argument:** Suppose the greedy sets bit i to 0 (because the required complementary bit isn't available), but an optimal solution sets bit i to 1 by "saving" that bit from an earlier position. But the greedy already maximized the earlier position — any swap that gains bit i would lose an earlier bit, which is a worse trade. So greedy is optimal. ✓

## Example Walkthrough

**Example 2:** `s = "0110"`, `t = "1110"`

Pool: ones=3, zeros=1.

| i | s[i] | Wants | Available | res[i] | Pool after |
|---|------|-------|-----------|--------|------------|
| 0 | '0'  | '1'   | ones=3    | **'1'** | ones=2 |
| 1 | '1'  | '0'   | zeros=1   | **'1'** | zeros=0 |
| 2 | '1'  | '0'   | zeros=0! use '1' | **'0'** | ones=1 |
| 3 | '0'  | '1'   | ones=1    | **'1'** | ones=0 |

**Output:** "1101" ✓

**Example 3:** `s = "0101"`, `t = "1001"`

Pool: ones=2, zeros=2.

Every position gets its desired bit → "1111" ✓

## Pitfalls & Edge Cases

- **Pool exhaustion:** when the desired bit type runs out, use the other type and accept XOR=0. ✓
- **All same bits in t:** e.g. t="111". zeros=0. All s[i]='1' positions get XOR=0; all s[i]='0' positions get XOR=1. Still greedy handles correctly. ✓
- **All 0s in s:** always want '1's from t, use them all up on leftmost positions. ✓

---

## Key Takeaway

"Maximize binary value by rearranging one operand of XOR" → greedy from MSB, assigning each position the complementary bit (if available in pool) to produce 1.

**Pattern:** Maximize binary number greedily = process bits left to right, make each bit 1 if possible without violating constraints.
