# 3961. Maximize Sum of Device Ratings
## O(m * n log n) — Bottleneck Isolation via Universal Sink Placement

### Problem Statement
You are given a 2D integer array `units` of size `m x n`, where `units[i][j]` represents the capacity of the j-th unit in the i-th device. Each device contains exactly n units. The rating of a device is the minimum capacity among all its current units. An empty device has a rating of 0.

You can perform the following operation any number of times:
- Choose a device i that has not been used as a source before.
- Remove exactly one unit from device i and add it to any different device.
- Mark device i as used so it cannot donate another unit.

Return the maximum possible sum of the ratings of all devices after any number of operations.

### Core Idea — The Universal Sink Strategy
Each device can donate at most one unit. To maximize a device's rating after a donation, it should always give away its smallest unit, causing its rating to rise to its second-smallest unit. However, every donated unit must be placed in a different device, potentially dragging that destination's minimum rating down.

If we sort all devices by their initial minimum unit, the device with the absolute smallest unit in the entire system (`u[0][0]`) becomes our key leverage point:
- **Case 1 (Device 0 is NOT a source):** It retains its initial rating of `u[0][0]`. Because `u[0][0]` is the global minimum, Device 0 can accept *every single donation* from all other devices without its rating dropping any lower than it already is. This allows all other devices to safely donate their minimums and upgrade to their second-smallest units for free.
- **Case 2 (Device 0 IS a source):** Device 0 donates `u[0][0]` and upgrades to its second-smallest unit `u[0][1]`. However, since `u[0][0]` is the global minimum, whichever device receives it will inevitably have its rating forced down to `u[0][0]`. To minimize the damage, we greedily pick a single device `j` to act as the "sink" to receive `u[0][0]`, sacrificing its potential upgrade to absorb the blow.

By evaluating both scenarios, the problem reduces to a simple linear check.

### Algorithm
1. **Edge Case Handling:** If `m = 1`, no transfers are possible because there is no other device. Return the minimum element of the single device.
2. **Identify Key Units:** For each device, sort its units to easily find its minimum (`u[i][0]`) and second-smallest unit (`u[i][1]`). If `n = 1`, the second-smallest unit is treated as 0.
3. **Sort Devices:** Sort the list of devices based on their initial minimum unit (`u[i][0]`) in ascending order. Device 0 now holds the global minimum.
4. **Calculate Best Upgrades:** For all devices from `1` to `m - 1`, calculate their best possible rating `V[i] = max(u[i][0], u[i][1])`.
5. **Evaluate Case 1:** Sum up `u[0][0]` and all `V[i]` values.
6. **Evaluate Case 2:** Calculate the sum if Device 0 upgrades to `u[0][1]`, Device `j` is sacrificed to drop to `u[0][0]`, and all other devices take their best upgrades. We pick `j` to minimize the loss.
7. **Return Result:** Return the maximum of Case 1 and Case 2.

### Full Optimal Implementation (C#)
```csharp
public class Solution {
    public long MaxRatings(int[][] units) {
        int m = units.Length;
        int n = units[0].Length;
        
        // If there's only one device, no transfers can be made
        if (m == 1) {
            long minVal = units[0][0];
            for (int j = 1; j < n; j++) {
                if (units[0][j] < minVal) {
                    minVal = units[0][j];
                }
            }
            return minVal;
        }
        
        // Extract the two smallest units for each device
        var devices = new List<(long u0, long u1)>();
        for (int i = 0; i < m; i++) {
            Array.Sort(units[i]);
            long u0 = units[i][0];
            long u1 = n > 1 ? units[i][1] : 0;
            devices.Add((u0, u1));
        }
        
        // Sort devices by their primary minimum unit ascending
        devices.Sort((a, b) => a.u0.CompareTo(b.u0));
        
        long globalMin = devices[0].u0;
        long globalMinNext = devices[0].u1;
        
        long totalV = 0;
        long minLossV = long.MaxValue;
        
        // Process all devices except the one with the global minimum
        for (int i = 1; i < m; i++) {
            long v = Math.Max(devices[i].u0, devices[i].u1);
            totalV += v;
            if (v < minLossV) {
                minLossV = v;
            }
        }
        
        // Case 1: Global minimum device is NOT a source (acts as the universal sink)
        long case1 = globalMin + totalV;
        
        // Case 2: Global minimum device IS a source (another device is sacrificed to be the sink)
        long case2 = 0;
        if (n > 1) {
            case2 = globalMinNext + globalMin + totalV - minLossV;
        }
        
        return Math.Max(case1, case2);
    }
}
```

### Complexity

| Metric | Value | Notes |
| :--- | :--- | :--- |
| **Time Complexity** | O(m * n log n) | Sorting each row takes O(n log n). Sorting the main device array takes O(m log m). Since m * n <= 2 * 10^5, this runs efficiently well within limits. |
| **Space Complexity** | O(m) | Storing pairs of the two smallest elements for each of the m devices scales linearly. |

---

### Why "One Universal Sink" Always Suffices
It is never optimal to create complex cyclic shifting networks (e.g., Device A gives to B, B gives to C, C gives to A) to avoid dropping ratings. In any group of devices that mutually donate, the absolute smallest unit among them will inevitably dictate the upper bound of the next device in the chain.

By dedicating exactly one device to absorb the lowest values, we isolate the negative constraint to a single spot. Any secondary tracking becomes redundant because a single sink can accept an infinite number of incoming units without changing its minimum rating boundary.

### Example Walkthrough

#### Example 1: units = [[1,3],[2,2]]
* **Initialization:**
  * Device 0: [1, 3] -> u0 = 1, u1 = 3, V = 3
  * Device 1: [2, 2] -> u0 = 2, u1 = 2, V = 2
* **State Evaluation:**
  * globalMin = 1, globalMinNext = 3
  * totalV = 2 (from Device 1), minLossV = 2
* **Calculations:**
  * Case 1: 1 + 2 = 3
  * Case 2: 3 + 1 + 2 - 2 = 4
* **Output:** 4 ✓ Matches expected.

#### Example 2: units = [[1,2,3],[4,5,6]]
* **Initialization:**

  * Device 1: [4, 5, 6] -> u0 = 4, u1 = 5, V = 5
* **State Evaluation:**
  * globalMin = 1, globalMinNext = 2
  * totalV = 5 (from Device 1), minLossV = 5
* **Calculations:**
  * Case 1: 1 + 5 = 6
  * Case 2: 2 + 1 + 5 - 5 = 3
* **Output:** 6 ✓ Matches expected.

### Pitfalls & Edge Cases
* **The Single Device Restriction:** If m = 1, attempting to look for an alternative destination device causes an out-of-bounds evaluation or illegal move logic. This must be caught immediately to return the baseline minimum. ✓
* **Single Unit Array Constraints:** When n = 1, moving a unit completely empties the device, reducing its rating to 0. Accessing `units[i][1]` will crash the program. A safety check must force u1 = 0 when n = 1. ✓
* **64-bit Integer Overflow:** With m up to 10^5 and unit values up to 10^5, the total combined sum can reach up to 10^10. Using standard 32-bit integers will cause numeric wrapping; all sum variables must utilize long. ✓

---

### Key Takeaway
> When an optimization problem allows unlimited targeting but tracks properties based entirely on minimums, look for the global absolute minimum. It will either serve as a free shield to absorb all updates or force a single strategic sacrifice elsewhere.

* **Pattern recognition:** "Maximize sum of minimums with single-use donations" -> Sort by individual minimum boundaries, isolate the lowest global constraint, and split logic into a binary choice.
