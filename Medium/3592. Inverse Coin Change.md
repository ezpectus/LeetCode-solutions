# 3592. Inverse Coin Change
## $O(n^2)$ — Reconstruct Denominations with Final Consistency Check

---

## Problem Statement

You are given a **1‑indexed** integer array `numWays`, where:
* `numWays[i]` = number of ways to make total amount `i`
* Using an **infinite supply** of some unknown coin denominations
* Each denomination is a positive integer $\le$ `numWays.length`, and all denominations are unique

**Your task:**
> Recover a sorted list of denominations that could produce exactly this `numWays` array.  
> If no such set exists, return an empty array.

---

## Core Idea — DP Inversion + Mandatory Final Equality Check

Standard unbounded coin change DP:
`ways[0] = 1`  
`ways[i] = Σ ways[i - coin]` over all coins $\le i$.

We want to invert this process: from `numWays` back to the set of coin values. We simulate building the coin set from small to large:

1. **Maintain `dp[i]`**: number of ways to form `i` using the coins we have already chosen.
2. **Initially**: `dp[0] = 1`, `dp[i] = 0` for `i > 0`.
3. **For each amount `i` from 1 to `n`**:
   * If `dp[i] > numWays[i-1]` → **impossible**.
   * If `dp[i] == numWays[i-1]` → no new coin of value `i` is needed.
   * If `dp[i] < numWays[i-1]` → the only way to increase `dp[i]` is to introduce a coin of value `i`.

### Critical detail (bug fix)
At the end, it is not enough that `dp[i] <= numWays[i-1]` for all `i`. We must have **exact equality**:
`dp[i] == numWays[i-1]` for all `i`.

If for some `i` we have `dp[i] < numWays[i-1]` after processing all possible coin values (1..n), we can no longer add any new coin, so the configuration is impossible.

---

## Algorithm

1. Let `n = numWays.Length`.
2. Create `dp[0..n]` as `long`:
   * `dp[0] = 1`
   * `dp[i] = 0` for `i > 0`
3. Create an empty list `coins`.
4. **For `i` from 1 to `n`**:
   * Let `target = numWays[i - 1]`.
   * If `dp[i] > target` → return `[]`.
   * If `dp[i] < target`:
     * Add coin `i` to `coins`.
     * **Update DP**: For `j` from `i` to `n`:
       * `dp[j] += dp[j - i]`
       * If `dp[j] > numWays[j - 1]` → return `[]`.
5. **Final consistency check**:
   * For `i` from 1 to `n`:
     * If `dp[i] != numWays[i - 1]` → return `[]`.
6. Return `coins`.

---

## Full Implementation (C#)

```csharp
public class Solution {
    public IList<int> FindCoins(int[] numWays) {
        int n = numWays.Length;
        long[] dp = new long[n + 1];
        dp[0] = 1;

        var coins = new List<int>();

        for (int i = 1; i <= n; i++) {
            long target = numWays[i - 1];

            if (dp[i] > target)
                return new List<int>(); // impossible

            if (dp[i] == target)
                continue;

            // dp[i] < target -> coin i must exist
            coins.Add(i);

            for (int j = i; j <= n; j++) {
                dp[j] += dp[j - i];
                // Early exit if we exceed the target at any point
                if (dp[j] > numWays[j - 1])
                    return new List<int>(); 
            }
        }

        // Final consistency check: dp must match numWays exactly
        for (int i = 1; i <= n; i++) {
            if (dp[i] != numWays[i - 1])
                return new List<int>();
        }

        return coins;
    }
}
```

## Pitfalls & Edge Cases

* **Indexing:** `numWays` is conceptually 1-indexed, but arrays in code are 0-indexed → always use `numWays[i - 1]`.
* **Exceeding ways:** If at any point `dp[i] > numWays[i-1]`, it is `impossible`, as adding more coins will only increase the number of ways.
* **Final Check:** If after iterating through all `i` we still find `dp[i] < numWays[i-1]`, it is `impossible` (since denominations greater than `n` cannot be used).
* **Data Types:** Be sure to use `long` for `dp`, as values in `numWays` can reach up to $2 \times 10^8$.

---

## Example Trace (Example 3)

**Input:** `numWays = [1, 2, 3, 4, 15]`

1.  The reconstruction process will suggest coins (e.g., `[1, 2, 3, 5]`).
2.  After performing all updates, the final value of `dp[5]` will be calculated.
3.  If the final `dp[5]` is strictly less than 15, the **Final consistency check** fails:
    `dp[5] != numWays[4]` (15).
4.  The algorithm correctly returns `[]`.

---

## Key Takeaway

This is a **DP inversion** problem. We simulate the Coin Change process "forward," and as soon as the current number of ways is less than required, we are **forced** to add a coin of exactly that denomination.

**Critical step:** The final consistency check for all frequencies is mandatory, because after passing all $i$, we can no longer add new coins to correct any remaining discrepancies.
