# 3604. Minimum Time to Reach Destination in Directed Graph  
## $O((n + m) \log n)$ — Dijkstra with Time Windows on Edges

---

## Problem Statement

You are given a directed graph with `n` nodes (`0 … n−1`). Each directed edge is defined as:  
`[u, v, start, end]`

You start at node `0` at time `0`. At each integer time `t` you may:
* **Wait** at your current node (cost: +1 time unit)
* **Traverse an outgoing edge** `(u → v)` **only if**:  
  `start <= t <= end`

Traversal takes **exactly 1 time unit** (you arrive at time `t + 1`).

**Your task:**
> Find the minimum time to reach node `n−1`, or return `-1` if it is impossible.

---

## Core Idea — Dijkstra with Modified Relaxation

This is a shortest‑path problem where:
* **Nodes:** Graph vertices
* **Distance:** Earliest possible time you can reach each node
* **Edges:** Constrained by availability windows `[start, end]`

### Relaxation Rule

Suppose we are at node `u` at time `t`. For an outgoing edge `(u → v, start, end)`:
1. If `t > end`, the edge is permanently closed and unusable.
2. If `t < start`, we must **wait** at node `u` until time `start`.
3. Therefore, the actual departure time is:  
   `depart = max(t, start)`
4. If `depart > end`, the edge is unusable.
5. The arrival time at node `v` will be:  
   `arrive = depart + 1`

We can run a standard **Dijkstra's algorithm** using a min‑priority queue keyed by the earliest arrival time.

---

## Algorithm

1. **Build Adjacency List:** `graph[u] = list of (v, start, end)`
2. **Initialize Distances:** Create a `dist[]` array initialized to $\infty$, except for the starting node: `dist[0] = 0`.
3. **Priority Queue:** Use a min-heap storing pairs of `(time, node)`. Initialize it with `(0, 0)`.
4. **Process Dijkstra:** While the priority queue is not empty:
   * Pop the state `(t, u)`.
   * If `t > dist[u]`, skip this stale state.
   * If `u == n−1`, return `t` (earliest destination reach time).
   * For each outgoing edge `(u → v, start, end)`:
     * If `t > end`, continue.
     * `depart = max(t, start)`
     * If `depart > end`, continue.
     * `arrive = depart + 1`
     * If `arrive < dist[v]`:
       * Update `dist[v] = arrive`.
       * Push `(arrive, v)` into the priority queue.
5. If `dist[n−1]` is still $\infty$, return `-1`.

## Full Implementation (C#)

```csharp
public class Solution {
    public int MinTime(int n, int[][] edges) {
        var graph = new List<(int v, int s, int e)>[n];
        for (int i = 0; i < n; i++)
            graph[i] = new List<(int, int, int)>();

        foreach (var ed in edges) {
            int u = ed[0], v = ed[1], s = ed[2], e = ed[3];
            graph[u].Add((v, s, e));
        }

        const int INF = int.MaxValue;
        int[] dist = new int[n];
        Array.Fill(dist, INF);
        dist[0] = 0;

        var pq = new PriorityQueue<(int t, int u), int>();
        pq.Enqueue((0, 0), 0);

        while (pq.Count > 0) {
            var (t, u) = pq.Dequeue();
            
            if (t > dist[u]) continue;
            if (u == n - 1) return t;

            foreach (var (v, s, e) in graph[u]) {
                if (t > e) continue;

                int depart = Math.Max(t, s);
                if (depart > e) continue;

                int arrive = depart + 1;

                if (arrive < dist[v]) {
                    dist[v] = arrive;
                    pq.Enqueue((arrive, v), arrive);
                }
            }
        }

        return -1;
    }
}
```

## Complexity

| Metric | Value |
| :--- | :--- |
| **Time Complexity** | $O((n + m) \log n)$ where $m$ is the number of edges |
| **Space Complexity** | $O(n + m)$ for the adjacency graph list |

*This efficiently supports larger constraints (up to 100k edges) because Dijkstra processes based on events rather than simulating every discrete time step.*

---

## Pitfalls & Edge Cases

* **Implicit Waiting:** Waiting at a node is completely free except for the time progression, which the Priority Queue organically optimizes.
* **Double Bounds Checking:** Always ensure `t <= end` and `max(t, start) <= end` before stepping onto an edge.
* **Time Increments:** Arrival time is strictly `depart + 1`, not `t + 1` if waiting was required.
* **Disconnected Graphs:** If node `0` has no available outgoing routes, the algorithm gracefully returns `-1`.
* **Large Windows:** Time coordinates can scale up to $10^9$. Because Dijkstra logs instantaneous timeline events rather than iterating frame-by-frame, huge time gaps do not degrade performance.

### Example 2
* **Edges:**
  * `0 → 1` with window `[0..3]`
  * `1 → 3` with window `[7..8]`
  * `0 → 2` with window `[1..5]`
  * `2 → 3` with window `[4..7]`

**Steps Matrix:**
1. **At $t = 0$:** We cannot take edge `0 → 2` because it doesn't open until $t = 1$. We are forced to wait.
2. **At $t = 1$:** We traverse edge `0 → 2`, arriving at node `2` at time $t = 2$.
3. **At $t = 2$:** We evaluate edge `2 → 3`. It doesn't open until $t = 4$. We wait at node `2` until $t = 4$.
4. **At $t = 4$:** We traverse edge `2 → 3`, arriving at destination node `3` at time $t = 5$.

**Answer:** `5`

---

## Key Takeaway

This is a **time-windowed shortest path** problem. Instead of standard graph traversal weights, edge usability relies heavily on dynamic availability timelines. 
Incorporating custom edge-relaxation evaluations (`max(t, start) + 1`) into Dijkstra's algorithm allows the pathfinder 
to naturally handle implicit waiting states efficiently in $O((n + m) \log n)$ time.
