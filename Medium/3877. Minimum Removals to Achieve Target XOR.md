# 🏔️ 3877. Minimum Removals to Achieve Target XOR — Architectural Solution

## Problem Statement

You are given an integer array `nums` and an integer `target`. You may remove any number of elements from `nums` (possibly zero).

Return the **minimum number of removals** required so that the bitwise XOR of the remaining elements equals `target`. If it is impossible to achieve `target`, return `-1`.

The bitwise XOR of an empty array is `0`.

### Constraints
- 1 <= nums.length <= 40
- 0 <= nums[i] <= 10^4
- 0 <= target <= 10^4

---

## 💡 Core Idea

The problem asks us to minimize the number of removals. This is exactly equivalent to **maximizing the number of elements we keep** such that their bitwise XOR sum equals `target`. Once we find the maximum number of elements we can keep (let's call it `max_kept`), the answer will simply be `n - max_kept`.

Although `n` can be up to 40 (which makes an exhaustive subset search of 2^40 impossible), the values of `nums[i]` are quite small (<= 10^4). 
- 10^4 in binary is `10011100010000`, which requires 14 bits.
- The maximum possible XOR sum using 14-bit numbers will never exceed 2^14 = 16384.

This allows us to use **0/1 Knapsack Dynamic Programming**. We define `dp[j]` as the maximum number of elements we can choose to get a XOR sum of `j`.

---

## 💻 Implementation (C++)

```cpp
#include <vector>
#include <algorithm>

using namespace std;

class Solution {
public:
    int minRemovals(vector<int>& nums, int target) {
        int n = nums.size();
        
        // Since nums[i] <= 10000, any XOR sum fits within 14 bits (2^14 = 16384)
        const int MAX_XOR = 16384; 
        
        // dp[j] stores the maximum elements kept to achieve an XOR sum of j
        // Initialize with -1 to represent unreachable XOR states
        vector<int> dp(MAX_XOR, -1);
        dp[0] = 0; // Base case: keeping 0 elements yields an XOR sum of 0
        
        for (int num : nums) {
            vector<int> next_dp = dp;
            for (int j = 0; j < MAX_XOR; j++) {
                if (dp[j] != -1) {
                    next_dp[j ^ num] = max(next_dp[j ^ num], dp[j] + 1);
                }
            }
            dp = next_dp;
        }
        
        // If the target state was never reached, return -1
        if (dp[target] == -1) {
            return -1;
        }
        
        // Minimum removals = Total elements minus maximum elements kept
        return n - dp[target];
    }
};
```

## 📐 Complexity Analysis

| Metric | Value | Justification |
| :--- | :--- | :--- |
| **Time Complexity** | O(n * MAX_XOR) | We iterate through all `n` elements. For each element, we scan the DP state array of a fixed size `MAX_XOR` = 16384. Total operations roughly equal 40 * 16384 = 655,360, which executes instantly. |
| **Space Complexity** | O(MAX_XOR) | The DP vector size is hardcoded to 16384 integers. Since it does not scale with `n`, it can be considered O(1) auxiliary space. |

## ⚠️ Pitfalls & Edge Cases

- **Target Unreachable**: If no combination of elements can XOR to `target`, `dp[target]` remains `-1`. The code explicitly flags this and returns `-1`.
- **Target is 0**: If `target = 0`, the base case `dp[0] = 0` guarantees that if we remove all elements, we get a valid subset of size 0 with XOR sum 0, yielding `n` removals. However, if there's a non-empty subset that XORs to 0, the DP will find it and maximize `dp[0]`, resulting in fewer removals.
- **State Pollution**: When updating `dp` in place, we must make sure an element isn't used multiple times in the same step. Using a temporary `next_dp` array isolates updates correctly for a 0/1 knapsack approach.

## 🔄 Alternative Approaches

- **Meet-in-the-Middle**: Since `n <= 40`, we could split the array into two halves of size 20. Generating all subset XORs for 2^20 states takes around 1 million operations. We could then use a hash map to match pairs from both halves. However, this is significantly more complex to code and uses more memory than the O(1) space DP approach.
- **Pure Backtracking (DFS)**: Exploring all 2^n subsets would take O(2^40) time. This will result in a Time Limit Exceeded (TLE) error since 2^40 is roughly 1.1 * 10^12 operations.
