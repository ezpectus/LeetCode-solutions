# 🏔️ 3890. Integers With Multiple Sum of Two Cubes — Architectural Solution

## Problem Statement

You are given an integer `n`. An integer `x` is considered **good** if there exist at least two distinct pairs `(a, b)` of positive integers such that:
- `a <= b`
- `x = a^3 + b^3`
- `x <= n`

Return an array containing all good integers less than or equal to `n`, sorted in ascending order.

### Constraints
- 1 <= n <= 10^9

---

## 💡 Core Idea

An upper bound of `n = 10^9` might initially seem too large for an exhaustive search. However, because we are dealing with **cubes**, the maximum possible value for either `a` or `b` is strictly bounded by the cube root of `n`.

Since `1000^3 = 10^9`, both `a` and `b` will never exceed 1000.

The total number of valid pairs `(a, b)` where `a <= b` and `a^3 + b^3 <= 10^9` is quite small (at most around 500,000 pairs). This allows us to use a highly efficient generative approach:
1. Generate all possible sums of two cubes `a^3 + b^3` that are less than or equal to `n`.
2. Store these sums in a list.
3. Sort the list to bring identical sums next to each other.
4. Scan the sorted list to identify numbers that appear two or more times (Ramanujan-Hardy "Taxicab" numbers).

---

## 💻 Implementation (C++)

```cpp
#include <vector>
#include <algorithm>

using namespace std;

class Solution {
public:
    vector<int> findGoodIntegers(int n) {
        vector<int> sums;
        
        // Loop through all possible values of 'a'
        for (long long a = 1; a * a * a <= n; ++a) {
            long long a3 = a * a * a;
            
            // Loop through all possible values of 'b' starting from 'a'
            for (long long b = a; ; ++b) {
                long long sum = a3 + b * b * b;
                
                // If the sum exceeds n, any larger 'b' will also exceed n
                if (sum > n) {
                    break;
                }
                
                sums.push_back(static_cast<int>(sum));
            }
        }
        
        // Sort all generated sums to group duplicates together
        sort(sums.begin(), sums.end());
        
        vector<int> result;
        // Identify values that appear at least twice
        for (size_t i = 1; i < sums.size(); ++i) {
            if (sums[i] == sums[i - 1]) {
                // Ensure we don't add the same good integer twice 
                // if it can be represented in 3 or more ways
                if (result.empty() || result.back() != sums[i]) {
                    result.push_back(sums[i]);
                }
            }
        }
        
        return result;
    }
};
```
## 📐 Complexity Analysis

| Metric | Value | Justification |
| :--- | :--- | :--- |
| **Time Complexity** | O(M log M) | Let `M` be the number of valid pairs `(a, b)` such that `a^3 + b^3 <= n`. For `n = 10^9`, `M` is approximately 500,000. Generating the pairs takes O(M) time, and sorting them takes O(M log M) operations, which finishes in less than 20 milliseconds. |
| **Space Complexity** | O(M) | The auxiliary `sums` vector stores all valid combinations under `n`. At most, it holds around 500,000 integers, requiring roughly 2 MB of memory. |

## ⚠️ Pitfalls & Edge Cases

- **Integer Overflow**: Even though `n` fits in a standard 32-bit signed integer, evaluating `b * b * b` can momentarily overflow if `b` goes slightly past 1000 before the boundary check. Declaring `a` and `b` as `long long` completely eliminates this risk.
- **Higher Taxicab Multiplicities**: Some numbers can be expressed as the sum of two cubes in three or more distinct ways (e.g., Ta(3) = 87,539,319). The condition `result.back() != sums[i]` ensures that such numbers are only added to our final answer once.
- **Small Inputs (n < 1729)**: The smallest taxicab number is 1729 (`1^3 + 12^3` and `9^3 + 10^3`). For any input `n < 1729`, the loops will collect single-occurrence sums, the duplicate scanner will find nothing, and an empty vector will be correctly returned.

## 🔄 Alternative Approaches

- **Hash Map Frequency Counter**: Instead of sorting all sums at the end, we could use an `unordered_map<int, int>` to count frequencies on the fly. While this achieves O(M) average time complexity theoretically, the constant-factor overhead of hash bucket allocations and a final sorting pass on the keys makes the flat `std::vector` + `std::sort` approach faster and cleaner in practice.
- **Priority Queue (Min-Heap Expansion)**: We could initialize a min-heap with pairs `(a, a)` and dynamically push `(a, b+1)` as elements are popped. This is similar to Dijkstra's or a merge-sort pointer approach. While it avoids generating all sums up front, it introduces O(log K) heap overhead per step, making it over-engineered for the small search space of 10^9.
