# 2971. Find Polygon With the Largest Perimeter
## O(n log n) — Sort + Prefix Sum Scan

---

## Problem Statement

- Choose any subset of `nums` (at least 3 elements) to form a polygon.
- A valid polygon: all sides except the largest sum to more than the largest.
- Maximize the perimeter.

---

## Core Idea — Sort, then Greedily Include All Previous Elements

**Key insight**: for a sorted array `a₁ ≤ a₂ ≤ ... ≤ aₖ`, the polygon condition is `a₁+...+aₖ₋₁ > aₖ` (sum of all smaller sides > largest side). Including more smaller sides only helps — it never hurts to include all elements up to index `i` when testing `nums[i]` as the longest side.

**Algorithm**:
- Sort `nums`.
- Scan left to right, tracking `prefix = sum of all elements seen so far`.
- At each element `x`: if `prefix > x`, valid polygon exists with perimeter `prefix + x`. Update `res`.
- After scanning, `res` holds the maximum perimeter.

**Why scan left to right (not right to left)?** We want the largest perimeter, which tends to come from larger elements. But we need the prefix (sum of all smaller elements) to exceed the current element. Scanning left to right builds the prefix naturally — by the time we reach a large `x`, `prefix` includes all smaller elements.

---

## Full Optimal Implementation (C#)

```csharp
public class Solution {
    public long LargestPerimeter(int[] nums) {
        Array.Sort(nums);
        long prefix = 0, res = -1;

        foreach (int x in nums) {
            if (prefix > x)
                res = prefix + x;
            prefix += x;
        }

        return res;
    }
}
```

## Example Walkthrough

**`nums=[1,12,1,2,5,50,3]`** → sorted: `[1,1,2,3,5,12,50]`

| x | prefix (before) | prefix > x? | res | prefix (after) |
|---|----------------|-------------|-----|----------------|
| 1 | 0 | No | -1 | 1 |
| 1 | 1 | No | -1 | 2 |
| 2 | 2 | No | -1 | 4 |
| 3 | 4 | Yes | 4+3=**7** | 7 |
| 5 | 7 | Yes | 7+5=**12** | 12 |
| 12 | 12 | No | 12 | 24 |
| 50 | 24 | No | 12 | 74 |

Answer: **12** ✓ (polygon with sides [1,1,2,3,5])

---

## Pitfalls & Edge Cases

- **`long` for prefix and res**: `n=10⁵`, `nums[i]=10⁹` → max prefix ≈ `10¹⁴`, overflows `int`
- **Update `res` before adding `x` to prefix**: check if `prefix > x` first, then update `prefix += x`
- **`res` initialized to `-1`**: returned if no valid polygon found (as required)
- **Why include ALL previous elements?** More elements = larger prefix = easier to satisfy `prefix > x`. Never optimal to skip a smaller element when trying to form a valid polygon.

---

## Key Takeaway

"Largest polygon perimeter from array" → **sort + scan with prefix sum**:

```
for each x in sorted nums:
    if prefix > x: res = prefix + x   // valid polygon, update best
    prefix += x
```

O(n log n) total. The greedy works because including all smaller elements maximizes the chance of `prefix > x` and maximizes the perimeter when valid.
