# 3976. Maximum Subarray Sum After Multiplier
## O(n) — Three-State DP (Before / During / After) Run Twice for Multiply and Divide

---

## Problem Statement

Array `nums`, positive integer `k`. Choose exactly one subarray and apply exactly one operation to it:
- **Multiply**: each element ×k.
- **Divide**: each element ÷k, using **floor** for positive values, **ceiling** for negative values.

Then return the maximum sum of any (possibly different) subarray of the resulting array.

---

## Core Insight 1 — Localize: Only the Intersection With the Answer Subarray Matters

Let `[l, r]` be the subarray ultimately chosen for the final sum. The operation subarray `[i, j]` may extend outside `[l, r]` — but elements outside `[l, r]` never contribute to the sum, so their transformation is irrelevant. Since the intersection of two intervals is itself an interval, **the effective transformed region within `[l, r]` is some contiguous sub-subarray `T ⊆ [l, r]`** (possibly empty, if the operation is applied entirely outside `[l, r]`).

**This reduces the problem to:** choose `S = [l, r]` and an optional contiguous `T ⊆ S`, apply one operation type to `T`, maximize `sum(S)` after transformation.

---

## Core Insight 2 — C# Integer Division Already Implements Floor/Ceil Correctly

C# integer division truncates toward zero. For `v ≥ 0`: truncation toward zero = floor. For `v < 0`: truncation toward zero = ceiling. This is **exactly** the problem's required rounding rule:

```
v = 7,  k = 2:  7 / 2  =  3   (floor(3.5) = 3) ✓
v = -7, k = 2: -7 / 2  = -3   (ceil(-3.5) = -3) ✓
```

So `divide_transform[i] = nums[i] / k` (plain C# integer division) requires **no branching on sign** — a major implementation simplification.

---

## Core Insight 3 — The Per-Element Gain Determines Which Operation Helps

Define `gainMultiply(v) = v×k − v = v(k−1)` and `gainDivide(v) = v/k − v` (integer division).

| Sign of v | gainMultiply | gainDivide |
|---|---|---|
| v > 0 | ≥ 0 (grows) | ≤ 0 (shrinks toward 0) |
| v < 0 | ≤ 0 (grows more negative) | ≥ 0 (shrinks toward 0) |
| v = 0 | 0 | 0 |

**Multiply helps positive runs; divide helps negative runs.** Since the operation is chosen once for the whole subarray `S`, we compute the best achievable gain under each operation independently and take the overall max.

---

## Reducing to a Three-State DP

For a fixed transform array `alt[]` (either the multiply array `m[i] = nums[i]×k` or the divide array `d[i] = nums[i]/k`), we want the maximum sum subarray of `nums` where an optional contiguous sub-segment is replaced by `alt[]` values. Track three states as a single left-to-right scan, each representing the max sum of a subarray **ending at index i**:

- **`before[i]`** — no transformation used yet (plain Kadane on `nums`).
- **`during[i]`** — currently inside the transformed segment `T` (which ends exactly at i).
- **`after[i]`** — the transformed segment has already ended; now back to original values.

**Transitions:**
```
before[i] = max(nums[i], before[i-1] + nums[i])                         // standard Kadane

during[i] = max( alt[i],                    // start T fresh at i
                  during[i-1] + alt[i],      // extend T
                  before[i-1] + alt[i] )     // T starts at i, preceded by untransformed prefix

after[i]  = max( after[i-1] + nums[i],       // continue post-T segment
                  during[i-1] + nums[i] )     // T ends at i-1, resume original values at i
```

The answer for this `alt[]` pass = `max over all i of max(before[i], during[i], after[i])`.

**Why `before[i]` is included in the candidate answer:** it represents choosing `S` with the operation applied somewhere entirely outside `S` (T empty within S) — i.e., "don't bother transforming inside this particular S." This correctly captures the case where transforming hurts more than it helps.

**Final answer** = `max(pass with alt = multiply array, pass with alt = divide array)`. Both passes share the same `before[]` sequence, so they can run in a single combined O(n) scan.

---

## Algorithm

```
1. For i = 0 to n-1:
     m = nums[i] * k
     d = nums[i] / k          // C# truncation = correct floor/ceil

     before[i] = max(nums[i], before[i-1] + nums[i])

     duringM[i] = max(m, duringM[i-1] + m, before[i-1] + m)
     afterM[i]  = max(afterM[i-1] + nums[i], duringM[i-1] + nums[i])

     duringD[i] = max(d, duringD[i-1] + d, before[i-1] + d)
     afterD[i]  = max(afterD[i-1] + nums[i], duringD[i-1] + nums[i])

     ans = max(ans, before[i], duringM[i], afterM[i], duringD[i], afterD[i])
2. Return ans.
```

## Full Optimal Implementation (C#)

```csharp
public class Solution {
    public long MaxSubarraySum(int[] nums, int k) {
        int n = nums.Length;
        const long NEG = long.MinValue / 4;

        long before = 0;
        long duringM = NEG, afterM = NEG;
        long duringD = NEG, afterD = NEG;
        long ans = long.MinValue;

        for (int i = 0; i < n; i++) {
            long v = nums[i];
            long m = v * k;
            long d = v / k;   // truncation toward zero = floor(v>=0), ceil(v<0)

            long newBefore, newDuringM, newAfterM, newDuringD, newAfterD;

            if (i == 0) {
                newBefore  = v;
                newDuringM = m;
                newAfterM  = NEG;
                newDuringD = d;
                newAfterD  = NEG;
            } else {
                newBefore  = Math.Max(v, before + v);

                newDuringM = Math.Max(m, Math.Max(duringM + m, before + m));
                newAfterM  = Math.Max(afterM + v, duringM + v);

                newDuringD = Math.Max(d, Math.Max(duringD + d, before + d));
                newAfterD  = Math.Max(afterD + v, duringD + v);
            }

            ans = Math.Max(ans, newBefore);
            ans = Math.Max(ans, Math.Max(newDuringM, newAfterM));
            ans = Math.Max(ans, Math.Max(newDuringD, newAfterD));

            before = newBefore;
            duringM = newDuringM; afterM = newAfterM;
            duringD = newDuringD; afterD = newAfterD;
        }

        return ans;
    }
}
```

| 3 | 4 | max(4,3+4)=7 | max(8,6+8,3+8)=**14** | max(3+4,6+4)=10 | max(2,1+2,3+2)=5 | max(2+4,1+4)=6 |
| 4 | -5 | max(−5,7−5)=2 | max(−10,14−10,7−10)=4 | max(10−5,14−5)=9 | max(−2,5−2,7−2)=5 | max(6−5,5−5)=1 |

Maximum across all cells = **14** (at `duringM[3]`) ✓

**Interpretation:** `duringM[3] = 14` means the transformed (multiplied) segment is `T = [2,3]` (values 3,4 → 6,8), reached via `before[1] + m[2] + m[3]`-style chaining: `before[1]=−1` is *not* the path taken — actually the optimal path is `before[1]=-1` plus m[2]+m[3]? Let's check: 6+8=14, and before[1]=-1 would give -1+6+8=13, not 14. The actual winning transition is `duringM[2]=6` (which itself came from `m[2]=6` alone, i.e. T starts fresh at i=2) `+ m[3]=8` = 14, meaning **S = T = [2,3]** exactly (no untransformed prefix). This matches the example: multiply subarray [3,4] (values at indices 2,3), sum = 6+8 = 14. ✓

## Why Three States (Not Two) Are Necessary

A naive two-state DP (before/during only) would miss configurations where the transformed segment T is a strict interior sub-subarray of S — i.e., S continues with original values **after** T ends. The `after` state is what allows: original prefix → transformed middle → original suffix, all within one chosen S. Without it, the DP could only find T as a *suffix* of S (T ending exactly at S's right edge), missing cases like Example 1 if the optimal answer had needed elements after the multiplied region (it didn't here, but the structure is necessary in general — e.g., `nums=[5, -100, 3]`, multiplying `[-100]` won't help, but consider a case where transforming a middle dip and keeping both positive ends matters).

## Pitfalls & Edge Cases

- **C# division truncation must not be re-derived with branching.** Writing `(v < 0) ? -((-v)/k) : v/k` for "floor" is actually wrong for negative v (that formula gives floor, not the required ceiling). The correct simplification is simply `v / k` in C#, relying on truncation-toward-zero behavior. Double-check this matches the problem's explicit floor (positive) / ceiling (negative) rule — verified above. ✓

- **`NEG` sentinel must not overflow when added.** Using `long.MinValue` directly and adding `v` (up to 10⁵) would overflow. `long.MinValue / 4` leaves enormous headroom (still effectively "very negative" but safe under repeated additions across n=10⁵ steps). ✓

- **`afterM`/`afterD` start invalid (`NEG`) at i=0:** there is no "after" state before any transformation has occurred. The transition `during[i-1] + v` only produces a valid after-state once `during` has been populated (from i=1 onward). ✓

- **`k = 1`:** `m[i] = v`, `d[i] = v` — both transforms degenerate to identity. All DP states collapse to the same value as plain Kadane, and the answer correctly equals the standard maximum subarray sum (operation is effectively a no-op, as expected). ✓

- **All-negative array, both operations strictly harmful (multiply) or only mildly helpful (divide):** the `before[i]` state always remains a valid fallback candidate, guaranteeing the answer is never worse than plain Kadane's max subarray sum. ✓

- **Operation must be applied to a non-empty subarray, but answer need not use it within S:** Handled by including `before[i]` in the final answer — the *global* operation is still mandatorily applied somewhere in the full array, just not necessarily overlapping the chosen S. This is implicit in the problem's structure (operation is on the whole array; only what lands within S matters for the sum). ✓

- **Long overflow on the sum itself:** worst case sum magnitude ≈ n × max(|v|×k) = 10⁵ × (10⁵×10⁵) = 10¹⁵, comfortably within `long` range.

---

## Key Takeaway

**"Choose subarray S for the answer, optionally transform a sub-subarray T ⊆ S with one global operation"** reduces to a rolling three-state DP: before (untransformed prefix / pure Kadane), during (actively inside the transformed run), after (transformed run has ended, back to original values). Run the DP once per candidate transform array (here: multiply array and divide array), sharing the `before` state, and take the global max.

**Pattern recognition:** Any "subarray sum, optionally apply a transformation to one contiguous interior sub-subarray" problem (e.g., "flip one subarray's signs", "double one subarray", "delete one subarray") is solved by this same three-state machine: before → during → after, each a simple O(1) transition per index, giving O(n) total regardless of how many transform variants need to be tried (just run the machine once per variant, sharing shared states where possible).
