# 2622. Cache With Time Limit
## O(1) per operation ‚Äî Map + setTimeout Auto-Cleanup

---

## Problem Statement

- Implement a `TimeLimitedCache` class with three methods:
  - `set(key, value, duration)` ‚Äî stores a key-value pair that expires after `duration` ms. Returns `true` if a non-expired entry for that key already existed, `false` otherwise.
  - `get(key)` ‚Äî returns value if key exists and hasn't expired, `-1` otherwise.
  - `count()` ‚Äî returns number of non-expired keys.

---

## Core Idea ‚Äî Map + setTimeout for Auto-Cleanup

**Key insight**: Store entries in a `Map`. On each `set`, schedule a `setTimeout` to auto-delete the key after `duration` ms. This means `get` and `count` never need to manually check expiry ‚Äî the Map only ever contains live entries.

**On `set`**:
- Check if an un-expired entry already exists ‚Üí determines the return value
- `clearTimeout` on the old timer if key existed (prevent premature deletion of the new entry)
- Store `{ value, timer }` and schedule `setTimeout(() => cache.delete(key), duration)`

**Why clear old timer?**
If you `set(1, 42, 50)` then `set(1, 99, 200)`, the first timer would delete key `1` after 50ms ‚Äî but the second `set` extended the lifetime to 200ms. Without `clearTimeout`, the old timer fires and wrongly removes the new entry.

---

## Full Optimal Implementation (JavaScript)

```javascript
var TimeLimitedCache = function() {
    this.cache = new Map(); // key ‚Üí { value, timer }
};

TimeLimitedCache.prototype.set = function(key, value, duration) {
    const exists = this.cache.has(key); // un-expired entry (auto-cleaned by setTimeout)

    if (exists) clearTimeout(this.cache.get(key).timer); // cancel old expiry

    const timer = setTimeout(() => this.cache.delete(key), duration);
    this.cache.set(key, { value, timer });

    return exists;
};

TimeLimitedCache.prototype.get = function(key) {
    return this.cache.has(key) ? this.cache.get(key).value : -1;
};

TimeLimitedCache.prototype.count = function() {
    return this.cache.size;
};
```

---

## Complexity

| **Metric** | **Value** | **Notes** |
|---|---|---|
| **Time Complexity** | **O(1)** per operation | Map get/set/delete + setTimeout scheduling are all O(1) |
| **Space Complexity** | **O(n)** | n = number of currently live (non-expired) entries |

---

## Why This Design is Clean

**Alternative**: store `expireAt = Date.now() + duration` and check on every `get`/`count`. This works but `count()` becomes O(n) since you'd need to filter. With `setTimeout` auto-cleanup:

| | setTimeout cleanup | Manual expiry check |
|---|---|---|
| `get` | O(1) ‚Äî just check Map | O(1) ‚Äî check timestamp |
| `count` | O(1) ‚Äî `Map.size` | O(n) ‚Äî filter expired |
| `set` | O(1) + timer overhead | O(1) |

`setTimeout` approach wins on `count()` and is semantically cleaner.

## Example Walkthrough

**Example 2**: `set(1,42,50)` at t=0, `set(1,50,100)` at t=40

```
t=0:  set(1, 42, 50)  ‚Üí exists=false, timer A fires at t=50
t=40: set(1, 50, 100) ‚Üí exists=true (key 1 live), clearTimeout(A), timer B fires at t=140
t=50: get(1) ‚Üí 50  ‚úì  (timer A was cancelled, key still alive)
t=120: get(1) ‚Üí 50 ‚úì  (timer B fires at t=140)
t=140: timer B fires ‚Üí cache.delete(1)
t=200: get(1) ‚Üí -1 ‚úì
t=250: count() ‚Üí 0  ‚úì
```

## Pitfalls & Edge Cases

- **Must `clearTimeout` on overwrite** ‚Äî forgetting this causes the old timer to delete a freshly set key mid-life
- **`exists` check before overwrite** ‚Äî check `this.cache.has(key)` *before* calling `cache.set()`; otherwise you'd always see the new entry
- **`duration = 0`** ‚Äî `setTimeout(..., 0)` fires asynchronously (next event loop tick), so `get` right after `set` still returns the value; matches expected JS behavior
- **No need to store `expireAt`** ‚Äî since `setTimeout` handles deletion automatically, `get` and `count` don't need timestamps at all

---

## Key Takeaway

Time-expiring cache in JS ‚Üí **Map + `setTimeout` for auto-cleanup**:

- `setTimeout(() => map.delete(key), duration)` = self-managing TTL
- Always `clearTimeout` old timer on overwrite
- `get` and `count` become trivial O(1) Map operations ‚Äî no manual expiry logic needed

Pattern: whenever entries have a TTL in JS, `setTimeout` + cleanup is cleaner than polling or manual timestamp checks.
