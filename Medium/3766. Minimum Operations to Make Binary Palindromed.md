# 3766. Minimum Operations to Make Binary Palindrome
## O(n · D) — Bidirectional Brute Force Search per Element

---

## Problem Statement

- You are given `nums` (1 ≤ n ≤ 5000, 1 ≤ nums[i] ≤ 5000).
- Each operation: increment or decrement `nums[i]` by 1.
- A **binary palindrome** has a binary representation (no leading zeros) that reads the same forward and backward.
- Return `ans[i]` = minimum operations to convert `nums[i]` into a binary palindrome.

---

## Core Idea — Expand Outward from Each Number

For each number, check if it's already a binary palindrome. If not, expand symmetrically — check `n-d` and `n+d` for increasing `d` — until a palindrome is found. Take the first (minimum) distance.

**Why brute force works:** nums[i] ≤ 5000 means at most 13 binary bits. Among k-bit numbers, palindromes appear roughly every 2^(k/2) ≈ 64 steps. So the worst-case search per element is tiny — well within constraints.

**Why bidirectional:** The nearest binary palindrome could be either above or below the current value. Check both at each distance `d` and return as soon as either is found.

---

## Full Optimal Implementation (C#)

```csharp
public class Solution {
    public int[] MinOperations(int[] nums) {
        int[] res = new int[nums.Length];

        for (int i = 0; i < nums.Length; i++) {
            res[i] = MinOps(nums[i]);
        }

        return res;
    }

    private int MinOps(int n) {
        if (IsBinaryPalindrome(n)) return 0;

        for (int d = 1; ; d++) {
            // Check below first (decrements), then above
            if (n - d >= 1 && IsBinaryPalindrome(n - d)) return d;
            if (IsBinaryPalindrome(n + d)) return d;
        }
    }

    private bool IsBinaryPalindrome(int n) {
        string bin = Convert.ToString(n, 2);
        int l = 0, r = bin.Length - 1;

        while (l < r) {
            if (bin[l] != bin[r]) return false;
            l++;
            r--;
        }

        return true;
    }
}
```

| Metric | Value | Notes |
|---|---|---|
| Time Complexity | O(n · D) | D ≈ 64 max steps per element for nums ≤ 5000 |
| Space Complexity | O(log n) | Binary string for palindrome check |

Effectively O(n) for this constraint — total ops ≈ 5000 × 64 = 320,000.

## Why This Works — Example Walkthrough

**Example 1:** `nums = [1, 2, 4]`

| n | Binary | Palindrome? | d=1 down | d=1 up | ans |
|---|--------|-------------|----------|--------|-----|
| 1 | `1`    | ✓           | —        | —      | 0   |
| 2 | `10`   | ✗           | 1 = `1` ✓ | 3 = `11` ✓ | 1 |
| 4 | `100`  | ✗           | 3 = `11` ✓ | 5 = `101` ✓ | 1 |

**Output:** [0, 1, 1] ✓

---

**Example 2:** `nums = [6, 7, 12]`

| n  | Binary | Palindrome? | Nearest Search | ans |
|----|--------|-------------|----------------|-----|
| 6  | `110`  | ✗           | d=1: 5 = `101` ✓ | 1 |
| 7  | `111`  | ✓           | —              | 0   |
| 12 | `1100` | ✗           | d=1: 11=`1011`✗, 13=`1101`✗ | — |
|    |        |             | d=2: 10=`1010`✗, 14=`1110`✗ | — |
|    |        |             | d=3: 9=`1001`✓, 15=`1111`✓  | 3 |

**Output:** [1, 0, 3] ✓

## Binary Palindrome Examples

| Number | Binary | Palindrome? |
|--------|--------|-------------|
| 1      | `1`    | ✓           |
| 3      | `11`   | ✓           |
| 5      | `101`  | ✓           |
| 7      | `111`  | ✓           |
| 9      | `1001` | ✓           |
| 15     | `1111` | ✓           |
| 21     | `10101`| ✓           |

All binary palindromes start and end with `1` (no leading zeros), so `0` is never a binary palindrome.

## Pitfalls & Edge Cases

- **`n = 1`** — `"1"` is a palindrome, 0 ops.

- **`n - d >= 1` guard** — Never go below 1 since nums[i] ≥ 1 and 0 in binary is `"0"` which is trivially a palindrome but invalid.

- **Changing bit length** — When crossing a power of 2 (e.g., 4=`100` → 3=`11`), the number of bits changes. This is handled naturally since `Convert.ToString(n, 2)` always produces the minimal binary rep without leading zeros.

- **Symmetric discovery** — At the same distance `d`, both `n-d` and `n+d` might be palindromes. We check down first (smaller = lexicographically preferred in case of equal ops), but since we only care about the count, it doesn't matter which we return.

- **Small numbers are dense with palindromes** — Single-bit and two-bit numbers are always palindromes (1=`1`, 3=`11`), so small values converge quickly.

---

## Key Takeaway

Constraints drive the approach. With nums[i] ≤ 5000:
- At most 13 binary bits
- Binary palindromes spaced ~2^(k/2) apart for k-bit numbers
- Max search depth ≈ 64 → brute force is trivially O(n)

**Pattern recognition:** When the value range is small enough, "find nearest valid number" is always a bidirectional linear scan from the current value. No need for generating all palindromes or binary search — expand outward and stop at the first hit.

The palindrome check itself is O(log n) per call (binary string length), making each element's total work O(D · log n) which is still O(1) for fixed constraint bounds.
