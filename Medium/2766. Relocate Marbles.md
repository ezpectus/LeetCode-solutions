# 2766. Relocate Marbles
## O((n + m) log n) — HashSet of Occupied Positions

---

## Problem Statement

- Move all marbles from `moveFrom[i]` to `moveTo[i]` at each step.
- Return sorted list of occupied positions after all moves.

---

## Core Idea — Track Occupied Positions as a Set

We don't care how many marbles are at each position — only **whether a position is occupied**. A `HashSet` of occupied positions handles this:

- `remove(moveFrom[i])` — position becomes empty (all marbles left)
- `add(moveTo[i])` — position becomes occupied
- Skip if `moveFrom[i] == moveTo[i]` (no-op)

---

## Full Optimal Implementation (C#)

```csharp
public class Solution {
    public IList<int> RelocateMarbles(int[] nums, int[] moveFrom, int[] moveTo) {
        var pos = new HashSet<int>(nums);

        for (int i = 0; i < moveFrom.Length; i++) {
            if (moveFrom[i] != moveTo[i]) {
                pos.Remove(moveFrom[i]);
                pos.Add(moveTo[i]);
            }
        }

        var res = pos.ToList();
        res.Sort();
        return res;
    }
}
```

---

## Complexity

| **Metric** | **Value** |
|---|---|
| **Time** | **O((n+m) + k log k)** — k = final distinct positions |
| **Space** | **O(n)** — HashSet |

---

## Pitfalls & Edge Cases

- **`moveFrom[i] == moveTo[i]`** — skip; removing and re-adding would work too but unnecessary
- **Multiple marbles at same position** — irrelevant; HashSet tracks occupancy not count
- **`moveTo[i]` already occupied** — `Add` on an existing element is a no-op in HashSet ✓

---

## Key Takeaway

"Track occupied positions under bulk moves" → **HashSet**: only presence matters, not count. Remove source, add destination, sort at end.
