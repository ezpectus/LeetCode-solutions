# 3828. Final Element After Subarray Deletions
## O(1) — Game Theory: Alice Always Takes max(first, last)

---

## Problem Statement

- Alice and Bob alternate (Alice first). Each turn, remove any **contiguous** subarray of size < current length.
- Alice maximizes the final element; Bob minimizes it.
- Return the optimal final element.

---

## Core Insight — Bob Can End the Game on His First Turn

**Key observation:** If Alice doesn't end the game immediately, Bob can respond by removing all but one element — ending the game on his turn. Bob will pick the **minimum endpoint** of whatever Alice left.

After Alice's move, the remaining array has some first element `f'` and last element `l'`. Bob can always eliminate everything except `min(f', l')`.

**What can Alice's move leave?**

She removes a contiguous subarray from the original array, leaving a "prefix + suffix" concatenation. The new endpoints `(f', l')` must be:

| Alice's removal | f' | l' |
|-----------------|----|----|
| Prefix [0..b]   | nums[b+1] | nums[n-1] |
| Suffix [a..n-1] | nums[0]   | nums[a-1] |
| Middle [a..b]   | nums[0]   | nums[n-1] |

In every case: `min(f', l') ≤ max(nums[0], nums[n-1])`.

So Bob always limits the result to ≤ `max(nums[0], nums[n-1])`.

**But Alice CAN achieve exactly `max(nums[0], nums[n-1])`** by removing all n-1 elements in one shot — keeping either the first or last element and ending the game before Bob plays.

**Optimal strategy:** Alice immediately removes n-1 elements, keeping whichever of `nums[0]` or `nums[n-1]` is larger. Bob never gets to play.

---

## Full Optimal Implementation (C#)

```csharp
public class Solution {
    public int FinalElement(int[] nums) {
        return Math.Max(nums[0], nums[^1]);
    }
}
```

| Metric | Value | Notes |

| Space Complexity | O(1) | No extra space |

## Formal Proof of Optimality

**Alice achieves ≥ max(nums[0], nums[n-1]):**
- Remove [1..n-2] (middle n-2 elements): keeps [nums[0], nums[n-1]], Bob picks min. Not optimal.
- Remove [0..n-2] (first n-1): keeps nums[n-1]. Game over.
- Remove [1..n-1] (last n-1): keeps nums[0]. Game over.
- Alice picks the larger → achieves max(nums[0], nums[n-1]). ✓

**Bob ensures ≤ max(nums[0], nums[n-1]):**
After any Alice move leaving array of size ≥ 2 with endpoints (f', l'):
- `min(f', l') ≤ max(nums[0], nums[n-1])` as shown in the table above.
- Bob removes all but `min(f', l')` → result ≤ max(nums[0], nums[n-1]). ✓

Combined: optimal value = exactly max(nums[0], nums[n-1]).

## Example Walkthrough

**Example 1:** `nums = [1, 5, 2]`

max(nums[0], nums[2]) = max(1, 2) = **2**.

Optimal: Alice removes [0..1] = {1,5} → keeps {2}. Game over. ✓

Note: Alice cannot keep 5 (interior) because removing both 1 and 2 requires a non-contiguous removal.

**Example 2:** `nums = [3, 7]`

max(3, 7) = **7**.

Alice removes [0..0] = {3} → keeps {7}. ✓

---

## Pitfalls & Edge Cases

- **n = 1:** nums[0] = nums[^1], max = nums[0]. No moves possible. ✓

- **Interior max:** If the largest element is not at an endpoint (like [1,5,2] with 5 in the middle), Alice cannot get it. Bob can always "protect" against interior elements — any attempt by Alice to expose an interior element puts Bob in control, who then eliminates it.

- **Why not multi-turn play?** Even if Alice tries a multi-turn strategy, Bob always ends the game on his first turn by removing all but one element. Alice's only "safe" play is ending the game herself on turn 1.

---

## Key Takeaway

The constraint "remove a **contiguous** subarray" is the key restriction. It means:
- The only elements Alice can directly "save" by removing n-1 elements in one move are the **first and last** elements.
- Any other element Alice tries to keep requires Bob to play, and Bob immediately ends the game with the minimum endpoint.

**Answer = max(nums[0], nums[n-1]).**

**Pattern recognition:** Game theory problems where one player can end the game on their first turn often collapse to a single evaluation. Check if either player can "shortcut" to victory.
