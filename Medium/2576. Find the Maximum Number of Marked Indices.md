# 2576. Find the Maximum Number of Marked Indices
## O(n log n) — Sort + Binary Search on Answer

---

## Problem Statement

- You are given a **0-indexed** integer array `nums` (1 ≤ n ≤ 10⁵), where each `nums[i]` (1 ≤ nums[i] ≤ 10⁹).
- You can repeatedly pick two **unmarked** indices `i` and `j` such that `2 * nums[i] <= nums[j]`, then mark both.
- Return the **maximum possible number of marked indices**.

---

## Core Idea — Binary Search on Answer

**Key insight**: Sort the array, then **binary search on the number of pairs `k`**. If we can form `k` pairs, we pair the `k` smallest elements with the `k` largest elements. Specifically, check whether `2 * nums[i] <= nums[n-k+i]` holds for all `i in [0, k)`.

**Why pair smallest with largest (shifted by k)?**
- After sorting, the optimal `k` pairs are always: `nums[0..k-1]` as the "small" sides and `nums[n-k..n-1]` as the "large" sides
- Pairing `nums[i]` with `nums[n-k+i]` is the tightest possible check — if even this alignment fails, no other assignment of `k` pairs can work

**Binary search setup**:
- Search space: `k` in `[0, n/2]` (at most `n/2` pairs possible)
- Monotone property: if `k` pairs are achievable, then `k-1` pairs are also achievable → binary search for the maximum valid `k`

---

## Full Optimal Implementation (C#)

```csharp
public class Solution {
    public int MaxNumOfMarkedIndices(int[] nums) {
        Array.Sort(nums);
        int n = nums.Length;

        int lo = 0, hi = n / 2;
        while (lo < hi) {
            int mid = (lo + hi + 1) / 2;
            if (CanMark(nums, mid)) lo = mid;
            else hi = mid - 1;
        }
        return lo * 2;
    }

    bool CanMark(int[] nums, int k) {
        int n = nums.Length;
        // Pair nums[0..k-1] (small) with nums[n-k..n-1] (large)
        for (int i = 0; i < k; i++) {
            if (2 * nums[i] > nums[n - k + i]) return false;
        }
        return true;
    }
}
```

## Why This Works — Example Walkthrough

**Failing case (old two-pointer)**: `nums = [57,40,...,92]` → gave `60`, expected `58`

The old two-pointer approach over-counted because `left` advanced in cases where the global pairing was actually invalid. Binary search avoids this by explicitly verifying the **exact pairing** for every candidate `k`.

**Example 2**: `nums = [9,2,5,4]` → sorted: `[2,4,5,9]`, `n=4`

- `k=2`: `2*nums[0]=4 <= nums[2]=5` ✅, `2*nums[1]=8 <= nums[3]=9` ✅ → valid
- `lo=2` → return `4` ✓

**Example 1**: `nums = [3,5,2,4]` → sorted: `[2,3,4,5]`

- `k=2`: `2*2=4 <= 4` ✅, `2*3=6 <= 5` ❌ → invalid
- `k=1`: `2*2=4 <= 5` ✅ → valid
- `lo=1` → return `2` ✓

**Example 3**: `nums = [7,6,8]` → sorted: `[6,7,8]`

- `k=1`: `2*6=12 <= 8` ❌ → invalid
- `lo=0` → return `0` ✓

## Pitfalls & Edge Cases

- **Two-pointer greedy is wrong** — it can advance `left` in cases where the global optimal pairing doesn't support it, leading to overcounting
- **Binary search upper bound is `n/2`** — integer division floors automatically for odd-length arrays
- **Pairing must be positional** — `nums[i]` pairs with `nums[n-k+i]`, not arbitrarily; this is the tightest valid alignment after sorting
- **Monotone property required** — if `k` pairs work, `k-1` also works (trivially true by dropping the hardest pair)
- **Duplicate values** — sorting handles them correctly; no special casing needed

---

## Key Takeaway

When asked for the **maximum number of pairs** satisfying a condition, think **binary search on the answer**:

- **Sort** so that the optimal pairing structure becomes positional
- **Binary search on `k`** — the number of pairs
- **Check in O(n)**: pair the `k` smallest with the `k` largest and verify all constraints

The two-pointer approach seems intuitive but breaks on subtle cases — binary search on the answer is the correct and robust solution here.
