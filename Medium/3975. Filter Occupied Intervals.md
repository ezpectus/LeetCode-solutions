# 3975. Filter Occupied Intervals
## O(n log n) — Sort & Merge Touching Intervals, Then Subtract One Interval

---

## Problem Statement

2D array `occupiedIntervals`, each `[start, end]` inclusive, possibly overlapping or touching. Given `[freeStart, freeEnd]` (inclusive), produce the result of:
1. **Merging** all occupied intervals that overlap **or touch** (e.g., `[1,1]` and `[2,2]` touch → merge to `[1,2]`).
2. **Subtracting** the free interval `[freeStart, freeEnd]` from the merged result.

Return remaining intervals, sorted, non-overlapping, minimal count.

---

## Core Insight — Two Independent Classical Operations in Sequence

This problem is exactly **"Merge Intervals"** followed by **"Remove Interval"** (LeetCode 56 + LeetCode 1272), chained together. No interaction between the two phases beyond using the merged output as input to the subtraction.

**Phase 1 — Merge with touching:** standard interval merge, but the overlap condition is relaxed: merge if `next.start ≤ current.end + 1` (not just `next.start ≤ current.end`). This captures both true overlaps and the "touching" case.

**Phase 2 — Subtract a single interval:** for each merged interval `[s, e]`, compute its remainder after removing `[freeStart, freeEnd]`:
- **No overlap** (`e < freeStart` or `s > freeEnd`): keep `[s, e]` unchanged.
- **Fully covered** (`s ≥ freeStart` and `e ≤ freeEnd`): interval vanishes entirely.
- **Partial overlap on the left** (`s < freeStart ≤ e ≤ freeEnd`): keep `[s, freeStart−1]`.
- **Partial overlap on the right** (`freeStart ≤ s ≤ freeEnd < e`): keep `[freeEnd+1, e]`.
- **Free interval strictly inside** (`s < freeStart` and `freeEnd < e`): split into two pieces `[s, freeStart−1]` and `[freeEnd+1, e]`.

---

## Why "Touching" Requires `next.start ≤ current.end + 1`

Standard merge-intervals uses `next.start ≤ current.end` for overlap. Here intervals like `[1,1]` and `[2,2]` don't overlap by that rule (2 > 1) but must still merge because they're **consecutive integers** with no gap between them. Extending the merge condition by +1 captures adjacency: `2 ≤ 1 + 1` ✓ triggers a merge.

This matters specifically because the problem deals with **integer points** (not continuous time) — `[1,1]∪[2,2]` covers `{1,2}`, which is indistinguishable from the single block `[1,2]` once non-overlap is the goal.

---

## Algorithm

```
1. Sort occupiedIntervals by start.
2. Merge pass: maintain (curStart, curEnd).
   For each interval [s, e] in sorted order:
     if s <= curEnd + 1: curEnd = max(curEnd, e)   // merge (overlap or touch)
     else: push (curStart, curEnd); curStart = s; curEnd = e
   Push final (curStart, curEnd).
3. Subtract pass: for each merged [s, e]:
     if e < freeStart or s > freeEnd: result.Add([s, e])           // no overlap
     else:
       if s < freeStart: result.Add([s, freeStart - 1])             // left remainder
       if e > freeEnd:   result.Add([freeEnd + 1, e])               // right remainder
4. Return result.
```

## Full Optimal Implementation (C#)

```csharp
public class Solution {
    public IList<IList<int>> FilterOccupiedIntervals(int[][] occupiedIntervals, int freeStart, int freeEnd) {
        // Phase 1: sort and merge intervals that overlap OR touch
        Array.Sort(occupiedIntervals, (a, b) => a[0] - b[0]);

        var merged = new List<int[]>();
        int curStart = occupiedIntervals[0][0];
        int curEnd   = occupiedIntervals[0][1];

        for (int i = 1; i < occupiedIntervals.Length; i++) {
            int s = occupiedIntervals[i][0], e = occupiedIntervals[i][1];
            if (s <= curEnd + 1) {
                curEnd = Math.Max(curEnd, e);   // overlap or touch -> merge
            } else {
                merged.Add(new int[] { curStart, curEnd });
                curStart = s;
                curEnd = e;
            }
        }
        merged.Add(new int[] { curStart, curEnd });

        // Phase 2: subtract [freeStart, freeEnd] from each merged interval
        var result = new List<IList<int>>();
        foreach (var iv in merged) {
            int s = iv[0], e = iv[1];

            if (e < freeStart || s > freeEnd) {
                result.Add(new List<int> { s, e });        // no overlap at all
                continue;
            }
            if (s < freeStart) {
                result.Add(new List<int> { s, freeStart - 1 });  // left remainder
            }
            if (e > freeEnd) {
                result.Add(new List<int> { freeEnd + 1, e });    // right remainder
            }
            // fully covered case: neither branch fires, interval vanishes
        }

        return result;
    }
}
```

**Merge pass:**

| 3 | (10,10) | [10,12] | 10 ≤ 11 ✓ | curEnd=max(10,12)=12 |
| 4 | (10,12) | [14,16] | 14 ≤ 13? ✗ | push (10,12); curStart=14,curEnd=16 |
| end | | | | push (14,16) |

**Merged = [[2,8], [10,12], [14,16]]** — matches the example's stated merge result ✓

| Interval | e < 7 or s > 11? | s < 7? | e > 11? | Output |

## Pitfalls & Edge Cases

- **Touching merge condition off-by-one:** Using `s <= curEnd` (standard merge) instead of `s <= curEnd + 1` would leave `[10,10]` and `[10,12]`... wait these literally overlap at 10, so that case merges either way. But `[1,1]` and `[2,2]` would NOT merge under the standard condition (2 > 1), producing the wrong merged output `[[1,1],[2,2]]` instead of `[[1,2]]`. Always use `+1` for "touch or overlap" semantics. ✓

- **Integer overflow on `curEnd + 1`:** With `endi ≤ 10⁹`, `curEnd + 1` can reach `10⁹+1`, well within `int` range (max ≈ 2.1×10⁹). No overflow risk here, but worth noting if constraints were tighter to `int.MaxValue`.

- **Empty `occupiedIntervals`:** Constraint guarantees `occupiedIntervals.length ≥ 1`, so `occupiedIntervals[0]` is always safe to access without a null/bounds check.

- **Free interval doesn't touch any merged interval:** Handled by the `e < freeStart || s > freeEnd` guard — interval passes through unchanged. ✓

- **Free interval exactly matches a merged interval:** `s == freeStart` and `e == freeEnd`. Then `s < freeStart` is false and `e > freeEnd` is false — interval vanishes entirely (correct, since `[freeStart, freeEnd]` fully covers `[s, e]`). ✓

- **Result must remain sorted:** Since merged intervals are processed in increasing order of start, and each interval's subtraction produces remainder(s) that stay in the same relative position (left remainder < right remainder, both within the original interval's bounds), the result list is automatically sorted without an extra sort step. ✓

- **Minimum interval count guarantee:** Because Phase 1 fully merges touching/overlapping intervals before subtraction, no two output intervals can be adjacent or overlapping after Phase 2 (the only way two merged intervals could end up adjacent post-subtraction would require them to already be very close pre-subtraction, but they're guaranteed non-touching after Phase 1's merge — gap ≥ 2 between merged intervals — and subtraction only shrinks intervals further, preserving that gap). ✓

## Key Takeaway

**Chain two classical interval algorithms:** "merge overlapping/touching intervals" (LC 56 variant) feeds directly into "remove a single interval from a list of disjoint intervals" (LC 1272 single-interval case). Each phase is independently O(n log n) / O(n), and there's no need to interleave them — compute the full merged set first, then subtract.

**Pattern recognition:** "Merge intervals with custom touch/overlap rule, then exclude a fixed region" → solve as two sequential standard interval operations: (1) sort + merge with the appropriate boundary condition (`≤` vs `≤ +1`), (2) for each resulting interval, classify against the excluded region into one of five cases (no overlap, fully covered, left remainder, right remainder, split into two).
