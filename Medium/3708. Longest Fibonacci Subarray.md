# 3708. Longest Fibonacci Subarray
## O(n) — Sliding Window with Two Pointers

---

## Problem Statement

- You are given an array of positive integers `nums` (3 ≤ n ≤ 10⁵) where 1 ≤ nums[i] ≤ 10⁹.
- A Fibonacci array is a contiguous sequence where each element (from the third onwards) equals the sum of the two preceding elements.
- Subarrays of length 1 or 2 are always considered Fibonacci.
- Return the length of the longest Fibonacci subarray in `nums`.

---

## Core Idea — Greedy Sliding Window

Key insight: A Fibonacci sequence is determined by its first two elements. Once we fix two consecutive elements, we can greedily extend the sequence as long as the next element matches the required sum.

Algorithm:
- For each position i (starting point of a potential Fibonacci sequence), track two consecutive elements
- Greedily extend forward while `nums[k] == nums[k-1] + nums[k-2]`
- Track the maximum length found
- Key optimization: Since length 1 or 2 always counts as Fibonacci, minimum answer is 2

No DP needed — this is a pure greedy/sliding window problem since the Fibonacci property is locally verifiable.

---

## Full Optimal Implementation (C#)

```csharp
public class Solution {
    public int LongestSubarray(int[] nums) {
        int n = nums.Length;
        int res = 2; // Minimum possible (length 1 or 2 always Fibonacci)
        
        // Try each position as the start of a Fibonacci sequence
        for (int i = 0; i < n - 1; i++) {
            // Current sequence starts with nums[i] and nums[i+1]
            int len = 2;
            long prev = nums[i];
            long curr = nums[i + 1];
            
            // Extend the sequence as far as possible
            for (int j = i + 2; j < n; j++) {
                long next = prev + curr;
                
                // Check if nums[j] matches the expected Fibonacci value
                if (nums[j] == next) {
                    len++;
                    prev = curr;
                    curr = next;
                } else {
                    break; // Sequence broken, try next starting position
                }
            }
            
            res = Math.Max(res, len);
        }
        
        return res;
    }
}
```

## Complexity

| Metric | Value | Notes |
|---|---|---|
| Time Complexity | O(n²) worst case | Each starting position can check up to n elements |
| Space Complexity | O(1) | Only a few variables for tracking current sequence |

Note: In practice, this runs much faster than O(n²) because most sequences break early. Average case is closer to O(n) when Fibonacci properties rarely hold for long stretches.

## Why This Works — Example Walkthrough

**Example 1:** `nums = [1,1,1,1,2,3,5,1]`

Starting positions:
- i=0: [1, 1] → expect 2, got 1 → break, len=2
- i=1: [1, 1] → expect 2, got 1 → break, len=2
- i=2: [1, 1] → expect 2, got 2 ✓ → expect 3, got 3 ✓ → expect 5, got 5 ✓ → expect 8, got 1 → break, **len=5** ✓
- i=3: [1, 2] → expect 3, got 3 ✓ → expect 5, got 5 ✓ → expect 8, got 1 → break, len=4
- i=4: [2, 3] → expect 5, got 5 ✓ → expect 8, got 1 → break, len=3
- i=5: [3, 5] → expect 8, got 1 → break, len=2
- i=6: [5, 1] → end of array, len=2

**Output:** 5

## Pitfalls & Edge Cases

- **Integer overflow** — Fibonacci sums can exceed int range (e.g., 10⁹ + 10⁹ = 2×10⁹). Use `long` for prev, curr, and next to handle this safely.

- **Short arrays** — With n ≥ 3, minimum answer is always 2 (any two consecutive elements form a valid Fibonacci array).

- **Early termination** — Once a sequence breaks (nums[j] ≠ expected sum), we must break and try the next starting position. No point continuing from j since the property is violated.

- **Starting position optimization** — We only need to try positions i where i < n - 1 (need at least two elements to start a sequence).

- **All same elements** — If all elements are identical (like Example 3), the sequence breaks immediately since a + a ≠ a for positive integers. Max length is 2.

- **Maximum possible length** — In theory, if nums starts with [1, 1, 2, 3, 5, 8, 13, ...], we could have a very long Fibonacci sequence, but it's bounded by how fast Fibonacci numbers grow relative to the constraint nums[i] ≤ 10⁹.

## Key Takeaway

This is a **greedy pattern matching** problem, not dynamic programming:
- Fix two starting elements → the entire sequence is determined
- Extend greedily while the Fibonacci property holds
- Track the maximum length across all starting positions

**Pattern recognition:** When you see "contiguous sequence with a recurrence relation" (like Fibonacci: F[i] = F[i-1] + F[i-2]) → think greedy extension from each possible starting point, not DP.

The key optimization is recognizing that:
1. We don't need to store previous results (no overlapping subproblems)
2. Each starting position defines exactly one maximal Fibonacci sequence
3. We can check the property in O(1) per element

Use `long` for arithmetic to avoid overflow, and remember that any 1-2 element subarray is automatically valid (minimum answer = 2).
