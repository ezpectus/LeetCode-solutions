# 2772. Apply Operations to Make All Array Elements Equal to Zero
## O(n) — Greedy with Difference Array

---

## Problem Statement

- Choose any subarray of size exactly `k` and decrement all elements by 1, any number of times.
- Return `true` if we can make all elements 0.

---

## Core Idea — Greedy Left-to-Right + Difference Array

**Key insight**: process positions left to right. At position `i`, after accounting for all operations started before `i`, if `nums[i]` is still positive we **must** start exactly `nums[i] - running` new operations at `i` (can't start them earlier — those positions are already decided; can't start them later — `nums[i]` would remain positive).

**Difference array trick**: instead of tracking which positions are covered by each operation, maintain `dec[i]` = net change in active operation count at position `i`. `running` = sum of `dec[0..i]` = how many operations are currently active.

**At each position `i`**:
1. Update `running += dec[i]` (operations ending/starting here)
2. `need = nums[i] - running` = additional operations needed
3. If `need < 0` → over-decremented → impossible
4. If `need > 0`:
   - If `i + k > n` → subarray of size k doesn't fit → impossible
   - Start `need` new operations: `dec[i] += need`, `dec[i+k] -= need`, `running += need`

---

## Full Optimal Implementation (C#)

```csharp
public class Solution {
    public bool CheckArray(int[] nums, int k) {
        int n = nums.Length;
        int[] dec = new int[n + 1]; // difference array
        long running = 0;

        for (int i = 0; i < n; i++) {
            running += dec[i];
            long need = nums[i] - running;
            if (need < 0) return false;       // too many operations applied
            if (need > 0) {
                if (i + k > n) return false;  // subarray doesn't fit
                dec[i] += (int)need;
                dec[i + k] -= (int)need;
                running += need;
            }
        }
        return true;
    }
}
```

| **Metric** | **Value** |
|---|---|
| **Time** | **O(n)** — single pass |
| **Space** | **O(n)** — difference array |

---

## Why Greedy is Correct

At position `i`, the number of operations that cover it is exactly `running`. We need `nums[i]` operations total to zero it out. Since we process left-to-right and can only start new operations at or after `i`, any deficit at `i` **must** be covered by starting operations exactly at `i`. Starting them later would miss position `i`. Starting them earlier was already decided.

This is a forced choice at every step — there's no flexibility in how many operations start at `i`.

## Example Walkthrough

**`nums=[2,2,3,1,1,0], k=3`**

| i | nums[i] | running | need | dec update | running after |
|---|---------|---------|------|------------|---------------|
| 0 | 2 | 0 | 2 | dec[0]+=2, dec[3]-=2 | 2 |
| 1 | 2 | 2 | 0 | — | 2 |
| 2 | 3 | 2 | 1 | dec[2]+=1, dec[5]-=1 | 3 |
| 3 | 1 | 3+dec[3]=-2+3=1 | 0 | — | 1 |
| 4 | 1 | 1 | 0 | — | 1 |
| 5 | 0 | 1+dec[5]=-1+1=0 | 0 | — | 0 |

All positions handled, no negatives → `true` ✓

---

## Pitfalls & Edge Cases

- **`need < 0`** — more operations cover this position than needed; impossible to undo
- **`i + k > n`** — subarray of size k starting at `i` would go out of bounds; impossible
- **`nums[i] = 0`** — `need = 0 - running`; if `running > 0` → negative need → false (can't un-decrement a zero)
- **`long` for running** — `nums[i]` up to 10⁶, n up to 10⁵ → running can reach 10⁶, fits in int but safer with long

---

## Key Takeaway

"Apply k-length subarray decrements to zero out array" → **greedy left-to-right + difference array**:

- At each position, the number of new operations to start is **forced** (exactly the deficit)
- Difference array tracks active operation count in O(1) per position
- Two failure conditions: negative deficit (over-decremented) or subarray out of bounds
