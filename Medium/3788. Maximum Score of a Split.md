# 3788. Maximum Score of a Split
## O(n) — Prefix Sum + Suffix Min Arrays

---

## Problem Statement

- Given `nums` of length n (2 ≤ n ≤ 10⁵, -10⁹ ≤ nums[i] ≤ 10⁹).
- Split at index `i` (0 ≤ i < n−1): score = `sum(nums[0..i]) − min(nums[i+1..n-1])`.
- Return the maximum score over all valid split indices.

---

## Core Idea — Precompute Both Sides, Scan Once

`score(i) = prefixSum[i] − suffixMin[i]`

Both components can be precomputed in O(n), then one scan over all valid i gives the answer.

- `prefixSum[i]` = running sum from left through index i
- `suffixMin[i]` = minimum of the suffix starting at i+1 (precomputed right-to-left)

---

## Full Optimal Implementation (C#)

```csharp
public class Solution {
    public long MaximumScore(int[] nums) {
        int n = nums.Length;

        // Suffix min: suffixMin[i] = min(nums[i], nums[i+1], ..., nums[n-1])
        int[] suffixMin = new int[n];
        suffixMin[n - 1] = nums[n - 1];
        for (int i = n - 2; i >= 0; i--) {
            suffixMin[i] = Math.Min(nums[i], suffixMin[i + 1]);
        }

        // Scan splits, accumulating prefix sum
        long prefixSum = 0;
        long res = long.MinValue;

        for (int i = 0; i < n - 1; i++) {
            prefixSum += nums[i];
            // suffixMin[i+1] = min of nums[i+1..n-1]
            long score = prefixSum - suffixMin[i + 1];
            res = Math.Max(res, score);
        }

        return res;
    }
}
```

## Complexity

| Metric | Value | Notes |
|---|---|---|
| Time Complexity | O(n) | One right-to-left pass + one left-to-right pass |
| Space Complexity | O(n) | Suffix min array |

## Example Walkthrough

**Example 1:** `nums = [10, -1, 3, -4, -5]`

Suffix min (right to left):
| i | nums[i] | suffixMin[i] |
|---|---------|--------------|
| 4 | −5      | −5           |
| 3 | −4      | −5           |
| 2 | 3       | −5           |
| 1 | −1      | −5           |
| 0 | 10      | −5           |

Scan splits:
| i | prefixSum | suffixMin[i+1] | score    |
|---|-----------|----------------|----------|
| 0 | 10        | −5             | 15       |
| 1 | 9         | −5             | 14       |
| **2** | **12**| **−5**         | **17** ← |
| 3 | 8         | −5             | 13       |

**Output:** 17 ✓

---

**Example 2:** `nums = [-7, -5, 3]`

suffixMin = [−7, −5, 3]

| i | prefixSum | suffixMin[i+1] | score |
|---|-----------|----------------|-------|
| 0 | −7        | −5             | **−2** |
| 1 | −12       | 3              | −15  |

**Output:** −2 ✓

---

**Example 3:** `nums = [1, 1]`

suffixMin = [1, 1]

| i | prefixSum | suffixMin[i+1] | score |
|---|-----------|----------------|-------|
| 0 | 1         | 1              | **0** |

**Output:** 0 ✓

## Pitfalls & Edge Cases

- **Overflow** — prefixSum can reach 10⁵ × 10⁹ = 10¹⁴, suffixMin as low as −10⁹. Score up to ~10¹⁴ + 10⁹ → use `long`. ✓

- **Valid split range** — `i` goes from 0 to n−2 inclusive. The suffix `[i+1..n-1]` is always non-empty. ✓

- **All negative nums** — Both prefixSum and suffixMin are negative. The best score might still be very negative (as in Example 2). `long.MinValue` initialisation ensures correctness. ✓

---

## Key Takeaway

Classic two-pass O(n) pattern: precompute suffix aggregates, then accumulate prefix aggregates in a single forward scan. Each split checks in O(1).

**Pattern recognition:** `score(i) = f(prefix) + g(suffix)` where f and g are independently computable → precompute one side, accumulate the other inline.
