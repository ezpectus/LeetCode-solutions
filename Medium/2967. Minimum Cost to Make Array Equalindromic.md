# 2967. Minimum Cost to Make Array Equalindromic
## O(n log n) — Median + Nearest Palindrome Search

---

## Problem Statement

- Given array `nums` of length `n`.
- A **special move**: pick index `i` and value `x`, pay `|nums[i] - x|`, set `nums[i] = x`.
- An array is **equalindromic** if all elements equal some palindromic number `y < 10^9`.
- Return the **minimum total cost** to make `nums` equalindromic.

---

## Core Idea — Median is Optimal, Then Snap to Nearest Palindrome

**Step 1 — Why the median?**

The cost function `sum(|nums[i] - y|)` is minimized at the **median** of `nums`. This is the classic L1 minimization result — the median minimizes total absolute deviation.

**Step 2 — `y` must be palindromic.**

The true optimum `y` might not be a palindrome. We must find the nearest palindrome(s) to the median and take the cheapest.

**Strategy**:
1. Sort `nums`, find the median `m`.
2. Find the nearest palindrome ≤ `m` and nearest palindrome ≥ `m`.
3. Evaluate total cost for both candidates, return the minimum.

**Why check both neighbors?** The cost function is convex — the palindrome just below or just above the median could be cheaper than one further away.

---

## Full Optimal Implementation (C#)

```csharp
public class Solution {
    public long MinimumCost(int[] nums) {
        Array.Sort(nums);
        int n = nums.Length;
        long median = nums[n / 2];

        long palLeft  = NearestPalindrome(median, -1);
        long palRight = NearestPalindrome(median, +1);

        return Math.Min(Cost(nums, palLeft), Cost(nums, palRight));
    }

    // Walk in direction (-1 = down, +1 = up) from start until palindrome found
    private long NearestPalindrome(long start, int dir) {
        long x = start;
        while (!IsPalindrome(x)) x += dir;
        return x;
    }

    private bool IsPalindrome(long x) {
        string s = x.ToString();
        int l = 0, r = s.Length - 1;
        while (l < r) if (s[l++] != s[r--]) return false;
        return true;
    }

    private long Cost(int[] nums, long y) {
        long total = 0;
        foreach (int v in nums) total += Math.Abs(v - y);
        return total;
    }
}
```

| **Metric** | **Value** | **Notes** |

| **Space** | **O(1)** | In-place sort, scalar variables |

## Why This Works

**Why median minimizes `sum|nums[i] - y|`?**

For any target `y`, moving `y` by 1 changes the cost by `(# elements > y) − (# elements < y)`. At the median, these counts balance → cost is flat or minimized. For even `n`, any value between the two middle elements is optimal — we pick `nums[n/2]` as the upper-middle median.

**Why check only the two nearest palindromes?**

The L1 cost function is convex (V-shaped) with minimum at the median. Moving away from the median in either direction strictly increases cost. So among all palindromes, the two closest to the median (one on each side) are the best candidates.

**Why walk step by step to find nearest palindrome?**

The maximum gap between consecutive palindromes near `10^9` is small (at most a few hundred for 9-digit numbers), so walking is fast in practice. Alternatively you can generate the palindrome analytically by mirroring digits — but the walk is simpler and correct within constraints.

**Why `nums[n/2]` and not `nums[(n-1)/2]`?**

For odd `n`, both give the same middle element. For even `n`, both middle candidates yield the same optimal palindrome search since the entire interval between them is equally optimal — checking from `nums[n/2]` (upper middle) is sufficient because we search both directions.

## Example Walkthrough

**`nums = [1,2,3,4,5], target = 19`** → sorted: `[1,2,3,4,5]`

Median = `nums[2] = 3`

Nearest palindromes:
- Walk down from 3: `3` is already a palindrome → `palLeft = 3`
- Walk up from 3: `3` is already a palindrome → `palRight = 3`

Cost(3) = |1-3|+|2-3|+|3-3|+|4-3|+|5-3| = 2+1+0+1+2 = **6** ✓

---

**`nums = [10,12,13,14,15]`** → sorted: `[10,12,13,14,15]`

Median = `nums[2] = 13`

Nearest palindromes:
- Walk down: 13→12→11 → `11` is palindrome → `palLeft = 11`
- Walk up: 13→14→...→22 → wait, 13→14→15→...→ `palRight`? Actually: 14, 15, 16, 17, 18, 19, 20, 21, 22 → `22` is palindrome → `palRight = 22`

Cost(11) = |10-11|+|12-11|+|13-11|+|14-11|+|15-11| = 1+1+2+3+4 = **11**
Cost(22) = |10-22|+|12-22|+|13-22|+|14-22|+|15-22| = 12+10+9+8+7 = **46**

Answer: `min(11, 46)` = **11** ✓

---

**`nums = [22,33,22,33,22]`** → sorted: `[22,22,22,33,33]`

Median = `nums[2] = 22`

`22` is a palindrome → `palLeft = palRight = 22`

Cost(22) = 0+0+0+11+11 = **22** ✓

## Pitfalls & Edge Cases

- **Check both directions**: the nearest palindrome below the median could be much closer than the one above (or vice versa) — never assume one dominates
- **`y` must be positive**: constraints say `1 ≤ nums[i] ≤ 10^9`, so median ≥ 1; walk down won't go below 1 in practice
- **Use `long` for costs**: sum of `|nums[i] - y|` with `n = 10^5` elements each up to `10^9` → up to `10^14`, overflows `int`
- **Single element array**: median is `nums[0]`; nearest palindrome to it is the answer — handled naturally

---

## Key Takeaway

"Minimize `sum|nums[i] - y|` where `y` is palindromic" → **find median, snap to nearest palindrome on each side**:

```
sort(nums)
median = nums[n/2]
palLeft  = nearest palindrome ≤ median
palRight = nearest palindrome ≥ median
return min(cost(palLeft), cost(palRight))
```

Median minimizes L1 cost → nearest palindromes are the only candidates → O(n log n) sort + O(n) cost evaluation.
