# 2952. Minimum Number of Coins to be Added
## O(n log n) — Greedy Reachability Extension

---

## Problem Statement

- Given array `coins` and integer `target`.
- An integer `x` is **obtainable** if some subsequence of coins sums to `x`.
- Return the **minimum number of coins** to add so that every integer in `[1, target]` is obtainable.

---

## Core Idea — Greedy "Reach" Expansion

**Key insight**: maintain a variable `reach` = the maximum value currently obtainable using all coins processed so far (i.e., every integer in `[1, reach]` is obtainable).

Initially `reach = 0` (nothing obtainable). Sort coins ascending and process one by one:

**At each step**, if the next coin `c ≤ reach + 1`:
- It extends coverage: every value in `[1, reach + c]` is now obtainable → `reach += c`

If `c > reach + 1`:
- There is a **gap**: value `reach + 1` is not obtainable and `c` is too large to help.
- We must add a new coin. The **optimal coin to add** is exactly `reach + 1` — it fills the gap and doubles our coverage.
- After adding `reach + 1`: `reach = reach + (reach + 1) = 2*reach + 1`, increment answer, and **re-examine the same coin `c`**.

Repeat until `reach ≥ target`.

**Why add `reach + 1` specifically?**
Adding any value `v ≤ reach + 1` covers the gap at `reach + 1`. Using `v = reach + 1` extends coverage the furthest in one step (greedy maximum gain), minimizing total additions.

---

## Full Optimal Implementation (C#)

```csharp
public class Solution {
    public int MinimumAddedCoins(int[] coins, int target) {
        Array.Sort(coins);
        long reach = 0;
        int added = 0, i = 0;

        while (reach < target) {
            if (i < coins.Length && coins[i] <= reach + 1) {
                reach += coins[i++];
            } else {
                // Gap at reach+1: add coin of value reach+1
                reach += reach + 1;
                added++;
            }
        }

        return added;
    }
}
```

## Why This Works — The Reachability Invariant

**Invariant**: after processing some prefix of sorted coins plus any added coins, every integer in `[1, reach]` is obtainable.

**Base case**: `reach = 0` — empty range, trivially true.

**When `coins[i] ≤ reach + 1`**: since every value in `[1, reach]` is obtainable, adding coin `coins[i]` means every value in `[coins[i], reach + coins[i]]` is now also obtainable (take any value in `[1, reach]` and add `coins[i]`). Combined with `[1, reach]`, we get `[1, reach + coins[i]]` → `reach += coins[i]`. ✓

**When `coins[i] > reach + 1`**: value `reach + 1` is not expressible — no subset of existing coins can sum to it (all remaining coins are ≥ `coins[i] > reach + 1`, too large to bridge the gap). We must add a coin. Best choice: add `reach + 1` itself, extending coverage to `[1, 2*reach + 1]` in one step.

**Why does the loop terminate?** Each iteration either advances `i` (consuming a coin) or doubles `reach` approximately (since `reach → 2*reach + 1`). Doubling from 0 to `target` takes at most O(log target) additions, so the total loop runs O(n + log target) times.

## Pitfalls & Edge Cases

- **Sort first**: the greedy relies on consuming coins in ascending order — without sorting, small coins processed late won't fill early gaps
- **Use `long` for `reach`**: `reach` can grow up to `2*target` in one step; with `target = 10^5` this is fine for `int`, but using `long` avoids surprises if constraints expand
- **Re-examine same coin after adding**: after adding a coin and updating `reach`, the current `coins[i]` may now be ≤ `reach + 1` — do NOT advance `i` when adding a coin
- **Loop condition `reach < target`**: stop as soon as `[1, target]` is fully covered; no need to process remaining coins

---

## Key Takeaway

"Cover all integers `[1, target]` with minimum added coins" → **greedy reachability extension**:

```
sort coins
reach = 0, added = 0, i = 0

while reach < target:
    if coins[i] <= reach + 1:
        reach += coins[i++]       // extend coverage with existing coin
    else:
        reach += reach + 1        // add optimal coin to fill gap
        added++

return added
```

If next sorted coin fits within current reach → extend. If gap exists → greedily add `reach+1`, doubling coverage. O(n log n) sort + O(n + log target) greedy loop.
