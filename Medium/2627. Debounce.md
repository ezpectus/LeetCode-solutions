# 2627. Debounce
## O(1) — Closure Timer with clearTimeout Reset

---

## Problem Statement

- Given a function `fn` and delay `t` ms, return a **debounced** version.
- A debounced function delays execution by `t` ms after the last call.
- If called again within `t` ms, the previous pending execution is **cancelled** and the timer restarts.
- Do **not** use `_.debounce()`.

---

## Core Idea — Single Timer Reference in Closure

**Key insight**: Store one `timer` variable in the closure. On every call:
1. `clearTimeout(timer)` — cancel any previously scheduled execution
2. `setTimeout(() => fn(...args), t)` — schedule a new execution `t` ms from now
3. Save the new timer ID back to `timer`

This way only the **last call** within any `t`-ms window actually fires — all earlier calls get cancelled by the next one.

**Why closure?**
`timer` must persist between calls but remain private to this debounced function. A closure variable is exactly that — lives as long as the returned function exists, shared across all its invocations.

---

## Full Optimal Implementation (JavaScript)

```javascript
var debounce = function(fn, t) {
    let timer = null;

    return function(...args) {
        clearTimeout(timer);                        // cancel previous pending call
        timer = setTimeout(() => fn(...args), t);   // schedule new call
    };
};
```

---

## Complexity

| **Metric** | **Value** | **Notes** |
|---|---|---|
| **Time Complexity** | **O(1)** per call | `clearTimeout` + `setTimeout` are both O(1) |
| **Space Complexity** | **O(1)** | Only one `timer` ID stored |

## Timeline Visualization

**Example 1**: `t=50ms`, calls at `t=50` and `t=75`

```
t=50:  call(1) → clearTimeout(null), set timer A (fires at t=100)
t=75:  call(2) → clearTimeout(A) ← cancelled!, set timer B (fires at t=125)
t=125: timer B fires → fn(2) ✓
```

Output: `[{t: 125, inputs: [2]}]` ✓

**Example 2**: `t=20ms`, calls at `t=50` and `t=100`

```
t=50:  call(1) → clearTimeout(null), set timer A (fires at t=70)
t=70:  timer A fires → fn(1) ✓  (t=100 call hasn't come yet, no cancellation)
t=100: call(2) → clearTimeout(null), set timer B (fires at t=120)
t=120: timer B fires → fn(2) ✓
```

Output: `[{t:70, inputs:[1]}, {t:120, inputs:[2]}]` ✓

---

## Debounce vs Throttle

| | **Debounce** | **Throttle** |
|---|---|---|
| Fires | After last call + delay | At most once per interval |
| Use case | Search input, resize handler | Scroll listener, rate limiting |
| Implementation | Reset timer on each call | Block calls until interval passes |

---

## Pitfalls & Edge Cases

- **`clearTimeout(null)`** — safe in JS; `clearTimeout` with `null`/`undefined` is a no-op, so the `let timer = null` initialization is fine
- **Must save new timer ID** — `timer = setTimeout(...)` overwrites the old ID so the next call can cancel this one
- **Arrow function in `setTimeout`** — `() => fn(...args)` captures `args` from the current call's scope; each invocation creates a new closure over its own `args`
- **`t = 0`** — still debounces: `setTimeout(..., 0)` fires asynchronously (next event loop tick), so rapid synchronous calls still get collapsed

---

## Key Takeaway

Debounce = **one timer variable in a closure, reset on every call**:

```javascript
let timer = null;
return (...args) => {
    clearTimeout(timer);
    timer = setTimeout(() => fn(...args), t);
};
```

- `clearTimeout` cancels the pending call
- `setTimeout` schedules a new one
- Only the last call in any burst actually executes

Pattern: "execute only after activity stops for X ms" → debounce. "Execute at most once per X ms regardless of activity" → throttle.
