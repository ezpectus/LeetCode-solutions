# 2568. Minimum Impossible OR  
*O(n + 32) — Greedy Power-of-Two Check*

---

## Problem Statement

- You are given a **0-indexed** integer array `nums`.
- An integer `x` is **expressible** if it can be obtained as the **bitwise OR** of **some non-empty subsequence** of `nums`.
- Return the **smallest positive integer** (greater than 0) that is **not expressible**.

**Examples**:

**Example 1**  
Input: nums = [2,1]  
Expressible: 1, 2, 3 (1|2)  
Smallest missing: **4**

**Example 2**  
Input: nums = [5,3,2]  
Cannot make 1 → **1**

**Constraints**:
- 1 ≤ nums.length ≤ 10⁵
- 1 ≤ nums[i] ≤ 10⁹

---

## Core Idea — Greedy: Smallest Power of 2 Not Covered

**Beautiful observation**:
- All possible ORs of subsequences form a set that is **upward closed** in terms of bits.
- The smallest missing positive integer must be a **power of 2** (because if a number with multiple bits is missing, some smaller power-of-2 subset would be missing earlier).
- So we only need to check powers of 2: 1, 2, 4, 8, 16, ..., up to ~2³¹
- If a power of 2 (say 2^i) is **not present** in `nums`, then **no subsequence can produce exactly 2^i** (because ORing numbers cannot "turn off" bits — it can only keep or add them).
- Therefore, the smallest 2^i that does **not** appear in `nums` is the answer (or if all small powers appear, then the next one after the maximum OR).

**Algorithm** (very clean & fast):
1. Check for each power of 2 starting from 1 (2^0) up to 2^31:
   - If this power of 2 is **not** in `nums` → return it
2. If all powers up to 2^31 are present → impossible in constraints → but won't happen

**Time**: O(n + 32) — Contains check is O(n) per call, but 32 calls → O(32n) worst-case, but in practice fast  
**Better**: use HashSet for O(1) lookups → total O(n + 32)

---

## Clean Implementation (C#)

```csharp
public class Solution {
    public int MinImpossibleOR(int[] nums) {
        HashSet<int> present = new HashSet<int>(nums);
        
        for (int i = 0; i < 32; i++) {
            int val = 1 << i;
            if (!present.Contains(val)) {
                return val;
            }
        }
        
        // Theoretically unreachable under constraints
        return -1;
    }
}
```

## Complexity

| **Metric**            | **Value**              | **Notes**                                                                 |
|-----------------------|------------------------|---------------------------------------------------------------------------|
| **Time Complexity**   | **O(n + 32)**          | Building HashSet from nums: O(n)<br>32 checks for powers of 2: each lookup O(1) → O(32) |
| **Space Complexity**  | **O(n)**               | HashSet stores up to n elements (in worst case all distinct)              |

**Optimal** — very fast in practice (n ≤ 10⁵), constant-time checks after preprocessing.

## Why This Works — Example Walkthrough

**Example 1**: nums = [2,1]

**Present values**: 1 (2⁰), 2 (2¹)  
**Expressible**: 1, 2, 3 (1|2)  
**Check powers of 2**:
- 2⁰ = 1 → present
- 2¹ = 2 → present
- 2² = 4 → **not present** → cannot form exactly 4 (any OR including higher bits will be ≥ 4 but with extra bits set or larger)  
→ **4** → correct

**Example 2**: nums = [5,3,2]

**Present values**: 2 (2¹), 3 (2⁰ + 2¹), 5 (2⁰ + 2²)  
**Check powers of 2**:
- 2⁰ = 1 → **not present** → cannot form exactly 1 (no way to get only the 0-th bit set)  
→ **1** → correct

**Correct** —  
Any expressible number is the OR of some subsequence → its set bits are the **union** of the bits of the chosen numbers.  
To form **exactly** 2^i (only the i-th bit set), you **must** have at least one number in nums that is **exactly** 2^i (or multiple numbers whose OR is exactly 2^i, but if no number has only that bit, higher bits would be set).  
Since OR cannot "turn off" bits, the only way to get a pure power of 2 is if that power of 2 **exists** in the array.  
Thus, the smallest power of 2 missing from nums is the smallest non-expressible positive integer.

---

## Key Takeaway

This is a **beautiful greedy bit observation** problem:

* The smallest positive integer not expressible as the OR of any subsequence **must be a power of 2**.
* To form exactly 2^i (only the i-th bit set), the array **must contain** the value 2^i itself (otherwise any OR would either miss the bit or include extra bits).
* Simply check powers of 2 starting from 1 (2⁰) upwards:
  - The first 2^i **not present** in nums is the answer

**Pure, clean, optimal** — O(n) preprocessing (HashSet) + O(32) checks, extremely fast, mathematically elegant, and correctly identifies the minimal impossible OR value without computing any actual ORs.
