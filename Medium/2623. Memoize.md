# 2623. Memoize
## O(1) avg per call — Closure Cache with JSON Key

---

## Problem Statement

- Given a function `fn`, return a **memoized** version that never calls `fn` twice with the same arguments.
- On repeated calls with identical arguments, return the cached result directly.
- Works for any number of arguments (sum, fib, factorial tested).

---

## Core Idea — Closure + Map with Serialized Key

**Key insight**: Use a `Map` stored in a closure. The cache key = `JSON.stringify(args)` — serializes the full argument list into a unique string regardless of argument count.

**Why `JSON.stringify(args)`?**
- `args` is an array, so `[2, 3]` → `"[2,3]"` and `[3, 2]` → `"[3,2]"` — **order matters**, matching the problem requirement that `(a,b)` and `(b,a)` are separate cache entries
- Works for any number of args: 0, 1, 2, ...
- Simple and reliable for the given constraint (integer inputs only)

---

## Full Optimal Implementation (JavaScript)

```javascript
function memoize(fn) {
    const cache = new Map();

    return function(...args) {
        const key = JSON.stringify(args);
        if (cache.has(key)) return cache.get(key);
        const res = fn(...args);
        cache.set(key, res);
        return res;
    };
}
```

---

## Complexity

| **Metric** | **Value** | **Notes** |
|---|---|---|
| **Time Complexity** | **O(k)** per call | k = number of args; `JSON.stringify` is O(k); Map lookup O(1) avg |
| **Space Complexity** | **O(n × k)** | n = unique call combinations; k = serialized key length |

For the given constraints (integers, ≤ 2 args), effectively **O(1)** per call.

---

## How Closure Works Here

```javascript
function memoize(fn) {
    const cache = new Map();   // ← created once, lives in closure
    return function(...args) { // ← every call shares the same cache
        // ...
    };
}
```

`cache` is created once when `memoize(fn)` is called. The returned function **closes over** `cache` — every invocation of the memoized function reads/writes the same Map. This is the classic closure-as-private-state pattern.

## Example Walkthrough

**Example 1**: `memoize(sum)`, calls `(2,2)`, `(2,2)`, `(1,2)`

| call | key | cache hit? | fn called? | res |
|------|-----|-----------|------------|-----|
| (2,2) | `"[2,2]"` | ❌ | ✅ | 4 |
| (2,2) | `"[2,2]"` | ✅ | ❌ | 4 |
| (1,2) | `"[1,2]"` | ❌ | ✅ | 3 |

`callCount = 2` ✓

**Example 3**: `memoize(fib)`, call `fib(5)` once

- `memoize` wraps only the **outer** call — `fib(5)` is computed once, result `8` cached
- Second call `memoFib(5)` → cache hit, `fn` never called again
- `callCount = 1` ✓

## Pitfalls & Edge Cases

- **`(a,b)` vs `(b,a)` are different keys** — `JSON.stringify([2,3]) = "[2,3]"` ≠ `"[3,2]"` → correct per problem spec
- **`JSON.stringify` for integers only** — works perfectly here; for general memoization with objects/functions as args, a more robust key strategy (e.g. `Map` of `Map`) would be needed
- **Inner recursive calls not memoized** — if `fib` calls itself internally, those recursive calls bypass the memoized wrapper; the problem's `fib`/`factorial` are defined to call the *original* function recursively, so only the top-level calls benefit from the cache
- **`undefined` return values** — `cache.has(key)` check (not `cache.get(key) !== undefined`) ensures functions that return `undefined` are also cached correctly

---

## Key Takeaway

Memoization in JS = **closure + Map + serialized key**:

```javascript
const cache = new Map();
const key = JSON.stringify(args);
if (cache.has(key)) return cache.get(key);
const res = fn(...args);
cache.set(key, res);
return res;
```

- Closure holds the cache privately across all calls
- `JSON.stringify(args)` turns any argument list into a unique, order-sensitive string key
- `cache.has()` (not `cache.get() !== undefined`) is the correct existence check

Pattern: any pure function with expensive computation + repeated calls → wrap with this memoize shell.
