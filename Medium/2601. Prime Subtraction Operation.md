# 2601. Prime Subtraction Operation
## O(n · √M + n · log P) — Sieve + Greedy + Binary Search

---

## Problem Statement

- You are given a **0-indexed** integer array `nums` (1 ≤ n ≤ 1000), where each `nums[i]` (1 ≤ nums[i] ≤ 1000).
- Operation (each index at most once): pick index `i` and prime `p < nums[i]`, subtract `p` from `nums[i]`.
- Return `true` if you can make `nums` strictly increasing, `false` otherwise.

---

## Core Idea — Greedy + Binary Search on Primes

**Key insight**: Process left to right. For each element, greedily subtract the **largest possible prime** that keeps the element strictly greater than the previous one. Maximally reducing each element gives the most room for future elements.

**For element `nums[i]` with previous value `prev`:**
- We need the result to be `> prev`, so we need `nums[i] - p > prev` → `p < nums[i] - prev`
- To leave the smallest possible value (maximizing future room), use the **largest prime strictly less than `nums[i] - prev`**
- If even without any subtraction `nums[i] <= prev` → impossible → return `false`

**Binary search**: primes up to 1000 are precomputed via sieve. Use binary search to find the largest prime `< limit` in O(log P).

---

## Full Optimal Implementation (C#)

```csharp
public class Solution {
    public bool PrimeSubOperation(int[] nums) {
        // Sieve of Eratosthenes up to 1000
        bool[] sieve = new bool[1001];
        Array.Fill(sieve, true);
        sieve[0] = sieve[1] = false;
        for (int i = 2; i * i <= 1000; i++)
            if (sieve[i])
                for (int j = i * i; j <= 1000; j += i)
                    sieve[j] = false;

        var primes = new List<int>();
        for (int i = 2; i <= 1000; i++)
            if (sieve[i]) primes.Add(i);

        int prev = 0;
        foreach (int num in nums) {
            int curr = num;
            // largest prime strictly less than (curr - prev)
            int limit = curr - prev; // prime must be < limit
            if (limit <= 0) return false;

            // binary search: last prime < limit
            int lo = 0, hi = primes.Count - 1, best = -1;
            while (lo <= hi) {
                int mid = (lo + hi) / 2;
                if (primes[mid] < limit) { best = primes[mid]; lo = mid + 1; }
                else hi = mid - 1;
            }

            if (best != -1) curr -= best;
            if (curr <= prev) return false;
            prev = curr;
        }

        return true;
    }
}
```

## Complexity

| **Metric** | **Value** | **Notes** |
|---|---|---|
| **Time Complexity** | **O(M log M + n log P)** | M=1000 for sieve; n elements × O(log P) binary search per element |
| **Space Complexity** | **O(P)** | P ≈ 168 primes up to 1000 |

**Optimal** — sieve is one-time O(M log M); per-element work is O(log P) ≈ O(log 168) ≈ 7 ops.

## Why This Works — Example Walkthrough

**Example 1**: `nums = [4, 9, 6, 10]`

| curr | prev | limit = curr-prev | largest prime < limit | after sub | valid? |
|------|------|-------------------|-----------------------|-----------|--------|
| 4 | 0 | 4 | 3 | 1 | 1 > 0 ✅ |
| 9 | 1 | 8 | 7 | 2 | 2 > 1 ✅ |
| 6 | 2 | 4 | 3 | 3 | 3 > 2 ✅ |
| 10 | 3 | 7 | 5 | 5 | 5 > 3 ✅ |

Returns `true` ✓

**Example 3**: `nums = [5, 8, 3]`

| curr | prev | limit | largest prime < limit | after sub | valid? |
|------|------|-------|-----------------------|-----------|--------|
| 5 | 0 | 5 | 3 | 2 | 2 > 0 ✅ |
| 8 | 2 | 6 | 5 | 3 | 3 > 2 ✅ |
| 3 | 3 | 0 | limit ≤ 0 | — | ❌ |

Returns `false` ✓

## Pitfalls & Edge Cases

- **`limit = curr - prev`**, not `curr` — the prime must be small enough so `curr - prime > prev`; subtracting too large a prime makes the element too small
- **`limit <= 0` early return** — if `curr <= prev` before any subtraction, it's impossible regardless of primes
- **No valid prime found (`best == -1`)** — when `limit <= 2` (no prime fits), skip subtraction and check if raw `curr > prev`
- **Greedy is optimal** — reducing each element as much as possible (largest valid prime) never hurts future elements; a smaller current value only helps the next element have more room
- **Each index used at most once** — the problem allows one operation per index; we use at most one prime per element, which matches

---

## Key Takeaway

"Make array strictly increasing with bounded decrements" → **greedy left-to-right, minimize each element as much as legally possible**:

- For each element, the constraint is `result > prev` → `prime < curr - prev`
- Maximize the subtracted prime (= minimize the result) using **binary search on precomputed primes**
- If `curr <= prev` at any point → impossible

Pattern: greedy + precomputed sorted structure + binary search for optimal value within constraint.
