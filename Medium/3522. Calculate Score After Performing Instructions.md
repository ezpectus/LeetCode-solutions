# 3522. Calculate Score After Performing Instructions  
## O(n) — Direct Simulation with Visited Array

---

## Problem Statement

You are given:

- `instructions`: an array of strings of length `n`
- `values`: an integer array of length `n`

Each index `i` represents an instruction:

- `"add"` → add `values[i]` to the score, then move to `i + 1`
- `"jump"` → move to `i + values[i]`
- If the next index goes out of bounds → stop
- If you ever revisit an index → stop (infinite loop prevention)

Return the **total score** accumulated before termination.

Constraints:

- `1 ≤ n ≤ 10⁵`
- `instructions[i] ∈ {"add", "jump"}`
- `|values[i]| ≤ 10⁵`
- Score may exceed 32‑bit → must use `long`

---

## Core Idea — Direct Simulation with Cycle Detection

This problem is a **deterministic state machine**:

- You start at index `0`
- Each instruction deterministically moves you to the next index
- There are only `n` states
- If you ever revisit a state → infinite loop → terminate

This is identical to:

- simulating a pointer walking through an array,
- with a `visited[]` array to detect cycles.

### Why `visited[]` is enough?

Because:

- Each index is processed at most once
- Once visited, revisiting means a loop
- No branching, no recursion, no backtracking

Thus the entire simulation is **O(n)**.

### Why `long`?

Worst case:

- 100k `"add"` instructions
- each with value up to 100k

Max score ≈ `10⁵ × 10⁵ = 10¹⁰` → fits in `long`, not in `int`.

---

## Full Optimal Implementation (C#)

```csharp
public class Solution {
    public long CalculateScore(string[] instructions, int[] values) {
        int n = instructions.Length;
        bool[] visited = new bool[n];
        long score = 0;
        int i = 0;

        while (i >= 0 && i < n && !visited[i]) {
            visited[i] = true;

            if (instructions[i] == "add") {
                score += values[i];
                i++;
            }
            else { // "jump"
                i += values[i];
            }
        }

        return score;
    }
}
```

## Complexity

| Metric | Value | Notes |
|---|---|---|
| Time Complexity | **O(n)** | Each index visited at most once |
| Space Complexity | **O(n)** | `visited[]` boolean array |
| Score Range | up to **1e10** | must use `long` |

## Why This Works — Example Walkthrough

### Example 1

```
instructions = ["add", "jump", "add", "add"]
values       = [  5 ,   2   ,  7 ,  3 ]
```

| Step | i | Instruction | Value | Action | Score | Next i |
|---|---|---|---|---|---|---|
| 1 | 0 | add  | 5 | score += 5 | 5 | 1 |
| 2 | 1 | jump | 2 | i += 2 | 5 | 3 |
| 3 | 3 | add  | 3 | score += 3 | 8 | 4 |
| 4 | 4 | — | — | out of bounds → stop | 8 | — |

Final score: **8**

---

### Example 2 — Infinite Loop Prevention

```
instructions = ["add", "jump", "jump"]
values       = [  1 ,   1   ,  -1 ]
```

| Step | i | Instruction | Value | Next i | Notes |
|---|---|---|---|---|---|
| 1 | 0 | add  | 1 | 1 | score = 1 |
| 2 | 1 | jump | 1 | 2 | |
| 3 | 2 | jump | -1 | 1 | revisiting index 1 → stop |

Final score: **1**

---

## Pitfalls & Edge Cases

- **Revisiting an index** must immediately stop — otherwise infinite loop.
- **Negative jumps** are allowed.
- **Jumping out of bounds** ends the simulation.
- **Score overflow** → must use `long`.
- **Comparing strings**: `"add"` vs `"jump"` — no typos.
- **Visited array** is faster than `HashSet<int>` and uses less memory.

---

## Alternative Approaches (Not Needed but Good to Know)

### 1. HashSet for visited  
Slower, unnecessary. `bool[]` is optimal.

### 2. Graph cycle detection (Floyd)  
Overkill — this is a linear chain, not a general graph.

### 3. Recursion  
Impossible — depth may reach 100k → stack overflow.

Direct simulation is the cleanest and fastest.

---

## Key Takeaway

This problem is a **pure deterministic simulation**:

- Walk through the array
- Track visited indices to avoid loops
- Add to score or jump based on instruction
- Stop on revisit or out-of-bounds

A classic example where **simple logic beats complex algorithms**.
