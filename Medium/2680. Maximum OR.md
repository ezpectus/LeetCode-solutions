# 2680. Maximum OR
## O(n) ‚ÄĒ Greedy: All k Shifts to One Element + Prefix/Suffix OR

---

## Problem Statement

- Multiply any element by 2 up to `k` times (each √ó2 = left shift by 1 bit).
- Return the maximum possible bitwise OR of the entire array.

---

## Core Idea ‚ÄĒ Why Concentrate All Shifts on One Element

**Key insight**: multiplying by 2 is a left bit-shift. Each shift pushes bits into higher positions. Applying all `k` shifts to one element gives `nums[i] << k` ‚ÄĒ the highest possible bits from that element.

**Why not split shifts across elements?**

OR is monotone ‚ÄĒ bits only turn on, never off. If we split `k` shifts between two elements, we gain bits from both but at lower positions than concentrating on one. Formally:

```
(a << j) | (b << (k-j))  ‚Č§  (a << k) | b    when a has high bits
```

Concentrating on the element that benefits most (highest value element, or one that "fills gaps" in OR) always wins. Since we try all `n` candidates, we don't need to figure out which is best analytically.

**Algorithm**:
1. Build `pref[i]` = OR of `nums[0..i-1]` and `suff[i]` = OR of `nums[i+1..n-1]`
2. For each candidate `i`: compute `pref[i] | (nums[i] << k) | suff[i+1]`
3. Return max over all candidates

---

## Full Optimal Implementation (C#)

```csharp
public class Solution {
    public long MaximumOr(int[] nums, int k) {
        int n = nums.Length;
        long[] pref = new long[n + 1];
        long[] suff = new long[n + 1];

        for (int i = 0; i < n; i++)
            pref[i+1] = pref[i] | nums[i];
        for (int i = n-1; i >= 0; i--)
            suff[i] = suff[i+1] | nums[i];

        long res = 0;
        for (int i = 0; i < n; i++) {
            long val = pref[i] | ((long)nums[i] << k) | suff[i+1];
            res = Math.Max(res, val);
        }
        return res;
    }
}
```

## Complexity

| **Metric** | **Value** | **Notes** |
|---|---|---|
| **Time Complexity** | **O(n)** | Two passes to build prefix/suffix, one pass to find max |
| **Space Complexity** | **O(n)** | Two arrays of size n+1 |

---

## Why Prefix/Suffix OR

To evaluate "apply all shifts to `nums[i]`, OR with everything else", we need OR of all elements **except** `nums[i]`. Naively this is O(n) per element ‚Üí O(n¬≤) total. Prefix/suffix OR precomputes it in O(n):

```
OR of all except i = pref[i] | suff[i+1]
```

This is the standard "product of all except i" pattern but with OR instead of multiplication.

## Example Walkthrough

**Example 2**: `nums = [8,1,2], k = 2`

```
pref = [0, 8, 9, 11]     (0, 8, 8|1=9, 9|2=11)
suff = [11, 3, 2, 0]     (8|1|2=11, 1|2=3, 2, 0)
```

| i | nums[i]<<k | pref[i] | suff[i+1] | total |
|---|-----------|---------|-----------|-------|
| 0 | 8<<2=32 | 0 | 3 | 32\|0\|3 = **35** |
| 1 | 1<<2=4 | 8 | 2 | 4\|8\|2 = 14 |
| 2 | 2<<2=8 | 9 | 0 | 8\|9\|0 = 9 |

Max = **35** ‚úď

## Pitfalls & Edge Cases

- **`long` for shifted values** ‚ÄĒ `nums[i] ‚Č§ 10‚ĀĻ`, `k ‚Č§ 15` ‚Üí `nums[i] << k` can reach ~`3√ó10¬Ļ¬≥`, overflows `int`; use `(long)nums[i] << k`
- **n = 1** ‚ÄĒ `pref[0] = 0`, `suff[1] = 0`, result = `nums[0] << k` ‚úď
- **All shifts to one element is always optimal** ‚ÄĒ OR of shifted element dominates splitting; proved by monotonicity of OR and bit position argument

---

## Key Takeaway

"Maximize OR after k multiplications by 2" ‚Üí **greedy: all shifts to one element** + **prefix/suffix OR for O(n) evaluation**:

```
res = max over i of: pref[i] | (nums[i] << k) | suff[i+1]
```

Splitting shifts across elements can never beat concentrating them ‚ÄĒ higher bit positions from one large shifted value always dominate lower bits from two moderately shifted values.
