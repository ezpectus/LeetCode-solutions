# 3824. Minimum K to Reduce Array Within Limit
## O(n log V) — Binary Search on k + O(n) Feasibility Check

---

## Problem Statement

- `nonPositive(nums, k)` = minimum operations to make all elements ≤ 0, where each operation reduces one element by k.
- For element `x`, ceil(x / k) operations are needed.
- Find minimum `k` such that `Σ ceil(nums[i] / k) ≤ k²`.

---

## Core Idea — Monotone Condition → Binary Search

Key observations:
1. `ceil(x / k)` is non-increasing in k → `nonPositive(nums, k)` is non-increasing in k.
2. `k²` is strictly increasing in k.

So if the condition holds for k₁, it holds for all k > k₁ → **condition is monotone** → binary search.

**Feasibility check for a given k:** compute `Σ ceil(nums[i] / k)` in O(n) and compare with k². Use early exit when sum exceeds k².

**Upper bound for k:** at most 10⁵ (since at k = max(nums), all ceil values equal 1, so sum = n ≤ 10⁵ ≤ k² whenever k ≥ 317; and all nums[i] ≤ 10⁵).

---

## Full Optimal Implementation (C#)

```csharp
public class Solution {
    public int MinimumK(int[] nums) {
        int lo = 1, hi = 100001; // k ≤ 10^5 is always sufficient

        while (lo < hi) {
            int mid = (lo + hi) / 2;
            if (IsValid(nums, mid)) hi = mid;
            else lo = mid + 1;
        }

        return lo;
    }

    private bool IsValid(int[] nums, int k) {
        long limit = (long)k * k;
        long sum = 0;

        foreach (int x in nums) {
            sum += (x + k - 1) / k; // ceil(x / k)
            if (sum > limit) return false; // early exit
        }

        return true;
    }
}
```

## Why the Condition Is Monotone

If `Σ ceil(nums[i] / k₁) ≤ k₁²`:
- For k₂ > k₁: `Σ ceil(nums[i] / k₂) ≤ Σ ceil(nums[i] / k₁) ≤ k₁² < k₂²`.
- So the condition still holds. ✓

This guarantees binary search correctness — the feasible region is a suffix `[k_min, ∞)`.

## Example Walkthrough

**Example 1:** `nums = [3, 7, 5]`

| k | Σ ceil(nums[i]/k)   | k² | Valid? |
|---|---------------------|-----|--------|
| 1 | 3+7+5 = 15          | 1   | ✗      |
| 2 | 2+4+3 = 9           | 4   | ✗      |
| **3** | **1+3+2 = 6**   | **9** | **✓** |

Binary search converges to k = **3** ✓

---

**Example 2:** `nums = [1]`

k=1: sum = ceil(1/1) = 1 ≤ 1 = k². **Valid immediately.**

**Output:** 1 ✓

## Pitfalls & Edge Cases

- **Overflow** — sum can reach n × max_nums = 10¹⁰; k² can reach 10¹⁰. Use `long`. ✓

- **Early exit** — once `sum > k²`, stop accumulating. Significant speedup when k is small (sum grows large quickly). ✓

- **Upper bound justification** — at k = 10⁵: ceil(nums[i] / 10⁵) = 1 for all i (since nums[i] ≤ 10⁵). Sum = n ≤ 10⁵ ≤ (10⁵)² = 10¹⁰. Always valid. So answer ≤ 10⁵. ✓

- **Edge: all nums = 1, n = 10⁵** — answer = ⌈√n⌉ = 317. Binary search finds this correctly since at k = 316: sum = 10⁵, k² = 99856 < 10⁵. At k = 317: k² = 100489 ≥ 10⁵. ✓

---

## Key Takeaway

**"Find minimum k satisfying a monotone condition"** → binary search on k.

The two components move in opposite directions as k increases:
- Left side (sum of operations): non-increasing
- Right side (k²): strictly increasing

Their crossing point is the answer, found in O(log V) iterations.

**Pattern recognition:** Any time you need to minimize a parameter k where the feasibility condition is monotone, binary search is the right tool. Check monotonicity first (here: both halves move to make the condition easier as k grows).
