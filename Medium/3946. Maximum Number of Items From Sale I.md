# 3946. Maximum Number of Items From Sale I
## O(n² + n·budget) — Knapsack with "First-Copy Bonus + Unbounded Filler"

---

## Problem Statement

- `items[i] = [factor_i, price_i]`. Unlimited copies of each available.
- Buying ≥1 copy of item `i` grants **1 free copy of every other item `j`** where `factor_i` divides `factor_j`.
- Maximize total copies (purchased + free) with total purchase cost ≤ budget.

---

## Core Insight #1 — Free Grants Are Item-Intrinsic, Not Subset-Dependent

For item `i`, define:
```
freeGrant(i) = number of j ≠ i (over ALL items) such that factor_i divides factor_j
```

This count depends only on `i` and the **fixed** item list — not on which other items are also activated. If we activate a set `S` (buy ≥1 copy of each item in `S`), the total purchased+free count (ignoring extra copies beyond the first) is:

```
Σ_{i ∈ S} (1 + freeGrant(i))
```

Define `value(i) = 1 + freeGrant(i)`. This turns the "activation" decision into a **0/1 knapsack**: weight = `price(i)`, value = `value(i)`.

---

## Core Insight #2 — Leftover Budget Goes Entirely to the Cheapest Activated Item

Once a set `S` is activated (paying `price(i)` once per item), buying **extra** copies of an already-activated item gives `+1` per copy at cost `price(i)` — no additional `freeGrant`. To maximize extra copies from leftover budget, dump **all** leftover into the **cheapest-priced** item in `S` (best rate `1/price`).

```
leftover = budget - Σ_{i∈S} price(i)
bonus = floor(leftover / min_{i∈S} price(i))
total(S) = Σ_{i∈S} value(i) + bonus
```

We must choose `S` to maximize `total(S)`.

---

## Core Insight #3 — Process by Price Descending, Anchor = Cheapest in S

For the optimal `S`, let `a` = the cheapest-priced item in `S` (the "anchor"). All **other** items in `S` have `price ≥ price(a)`.

**Algorithm:** Sort items by price **descending**. Maintain a running 0/1 knapsack DP array `dp[]` over items processed so far (all have price ≥ current item's price, since descending). For each item `i` (price `p`, value `v`), **before** inserting it into `dp[]`:

- Treat `i` as the anchor. Reserve budget `p` for its first copy.
- With `remaining = budget - p`, the best total using `dp[]` (other items, all priced ≥ p) plus filler bonus at rate `1/p`:
  ```
  candidate = v + max over b in [0, remaining] of ( dp[b] + floor((remaining - b) / p) )
  ```
- Update the global answer with `candidate`.

Then insert item `i` into `dp[]` (standard 0/1 knapsack update) so future (cheaper) items can use it as a "more expensive teammate."

Since we try **every** item as a potential anchor, and ties are naturally handled (same-price items processed earlier are still valid teammates with `price ≥ p`), this covers the true optimum.

## Full Optimal Implementation (C#)

```csharp
public class Solution {
    public int MaximumSaleItems(int[][] items, int budget) {
        int n = items.Length;
        int[] factor = new int[n];
        int[] price = new int[n];
        for (int i = 0; i < n; i++) {
            factor[i] = items[i][0];
            price[i] = items[i][1];
        }

        // freeGrant(i) = count of j != i with factor[i] dividing factor[j]
        int[] value = new int[n];
        for (int i = 0; i < n; i++) {
            int cnt = 0;
            for (int j = 0; j < n; j++) {
                if (i != j && factor[j] % factor[i] == 0) cnt++;
            }
            value[i] = 1 + cnt;
        }

        int[] order = new int[n];
        for (int i = 0; i < n; i++) order[i] = i;
        Array.Sort(order, (a, b) => price[b].CompareTo(price[a])); // descending

        int[] dp = new int[budget + 1]; // dp[b] = best 0/1 knapsack value using items processed so far
        int answer = 0;

        foreach (int idx in order) {
            int p = price[idx], v = value[idx];

            if (p <= budget) {
                int remaining = budget - p;
                int bestExtra = 0;
                for (int b = 0; b <= remaining; b++) {
                    int extra = dp[b] + (remaining - b) / p;
                    if (extra > bestExtra) bestExtra = extra;
                }
                answer = Math.Max(answer, v + bestExtra);
            }

            // Insert item idx into the knapsack DP for future (cheaper) anchors
            for (int b = budget; b >= p; b--) {
                if (dp[b - p] + v > dp[b]) dp[b] = dp[b - p] + v;
            }
        }

        return answer;
    }
}
```

## Example Walkthrough

**Example 1:** `items=[[6,2],[2,6],[3,4]], budget=9`

factors=[6,2,3], prices=[2,6,4].

freeGrant(0): factor6 divides 2? no. divides 3? no. → 0. value0=1.
freeGrant(1): factor2 divides 6? yes. divides 3? no. → 1. value1=2.
freeGrant(2): factor3 divides 6? yes. divides 2? no. → 1. value2=2.

Processing order (descending price): item1(p=6,v=2), item2(p=4,v=2), item0(p=2,v=1).

- **item1 anchor:** remaining=3, dp all 0 → bestExtra=0. candidate=2. answer=2. Insert item1.
- **item2 anchor:** remaining=5. dp[0]=0+5/4=1 best. candidate=2+1=3. answer=3. Insert item2.
- **item0 anchor:** remaining=7. dp now reflects items1,2 combos. Best found: dp[4]=2 (item2 alone) + (7-4)/2=1 → 3; or dp[0]=0+(7)/2=3. bestExtra=3. candidate=1+3=**4**. answer=4.

**Output:** 4 ✓ (matches: 2 copies item0 + 1 copy item2, cost 8, 1 free item0 from item2 → 3 purchased + 1 free = 4)

---

**Example 2:** `items=[[2,4],[3,2],[4,1],[6,4],[12,4]], budget=8`

freeGrant: item0(f2)→3 (divides 4,6,12), item1(f3)→2 (divides6,12), item2(f4)→1 (divides12), item3(f6)→1(divides12), item4(f12)→0.
values=[4,3,2,2,1], prices=[4,2,1,4,4].

Processing descending by price (4,4,4,2,1), by the time item2 (p=1, v=2) is reached as anchor, `dp[]` already reflects combinations of items {0,1,3,4}. The combo item0(p4,v4)+item1(p2,v3) reaches `dp[6]=7` (cost6, value7). With remaining=7 budget for the anchor pass: `extra = dp[6] + (7-6)/1 = 7+1 = 8`. Candidate = `v(item2) + extra = 2 + 8 = `**10**.

Construction: activate item0, item1, item2 (cost 4+2+1=7), leftover=1, bonus=1 extra copy of item2 (cheapest, price1) → total = (4+3+2) + 1 = **10**.

**Output:** 10 ✓

## Pitfalls & Edge Cases

- **Budget < cheapest price:** no item affordable, answer stays 0 (no candidate ever updates it). ✓

- **Ties in price:** processing order among tied items doesn't affect correctness — any tied item processed later still sees the others (processed earlier) in `dp[]` as valid teammates (price ≥ anchor's price holds with equality). ✓

- **freeGrant counts equal factors too:** if `factor[i] == factor[j]`, `factor[j] % factor[i] == 0` is true, correctly counted (divisibility includes equality). ✓

- **Single item (n=1):** freeGrant=0 always (no j≠i exists). Reduces to: if price≤budget, answer=floor(budget/price); else 0. ✓

- **Integer overflow:** values and budget are small (≤1500ish), no overflow risk with `int`. ✓

---

## Key Takeaway

This is a **"0/1 activation bonus + unbounded same-rate filler"** knapsack pattern: the first unit of an item carries bonus value (here, `freeGrant`), while every unit (including the first) contributes a baseline rate. Processing anchors in descending price order while maintaining an incremental knapsack DP elegantly handles the coupling between "which item provides the best filler rate" and "which other items are worth activating," achieving O(n·budget) instead of a naive O(n²·budget) or worse.

**Pattern recognition:** "Pay once for a bonus, then unlimited extra units at a flat rate" → separate the bonus (0/1 knapsack) from the filler (rate-based), and use a sorted-anchor technique to combine them efficiently.
