# 3955. Valid Binary Strings With Cost Limit
## O(2ⁿ) — Backtracking with Early Pruning

---

## Problem Statement

- Generate all binary strings of length n (n ≤ 12) such that:
  1. No two consecutive `'1'` characters.
  2. Cost (sum of indices where `s[i]='1'`) ≤ k.

---

## Core Idea — Backtrack Position by Position, Prune on Cost

Since `n ≤ 12`, the total search space (even without pruning) is at most Fibonacci-bounded (~2⁹⁰₃ ≈ a few hundred for the no-consecutive-1s constraint), trivially small. Build the string character by character:
- At each position, try `'0'` (always valid) and `'1'` (valid only if previous char isn't `'1'`).
- Track running cost; if adding a `'1'` at position i would make cost exceed k, skip that branch (prune — cost only increases as we add more `'1'`s, so this is always safe).
- At the end (position == n), record the completed string.

---

## Full Optimal Implementation (C#)

```csharp
public class Solution {
    public IList<string> GenerateValidStrings(int n, int k) {
        var result = new List<string>();
        var chars = new char[n];

        void Backtrack(int pos, int cost, bool prevWasOne) {
            if (pos == n) {
                result.Add(new string(chars));
                return;
            }

            // Try '0'
            chars[pos] = '0';
            Backtrack(pos + 1, cost, false);

            // Try '1' (only if previous wasn't '1', and cost stays within k)
            if (!prevWasOne && cost + pos <= k) {
                chars[pos] = '1';
                Backtrack(pos + 1, cost + pos, true);
            }
        }

        Backtrack(0, 0, false);
        return result;
    }
}
```

## Complexity

| Metric | Value | Notes |
|---|---|---|
| Time Complexity | O(2ⁿ) worst case | n ≤ 12, trivially fast; pruning + no-consecutive-1s constraint further reduces this |
| Space Complexity | O(n) recursion depth + O(output size · n) for results |

## Example Walkthrough

**Example 1:** `n=3, k=1`

Backtrack explores: position0 try '0' then '1'(cost0,pos0<=1 ok); position1 try '0' then '1' (only if prev not '1'); position2 similarly.

Valid completions found: "000"(cost0), "010"(cost1), "100"(cost0). "001"(cost2>1, pruned at pos2), "101"(cost0+2=2>1, pruned at pos2 since prev at pos0 is '1', pos1 forced '0', pos2 '1' would need cost0+2<=1? no, pruned).

**Output:** ["000","010","100"] ✓ (order may vary)

**Example 2:** `n=1,k=0`

position0: try '0'→"0" added. try '1': prevWasOne=false, cost+0=0<=0 ok →"1" added.

**Output:** ["0","1"] ✓

---

## Pitfalls & Edge Cases

- **k=0 edge case:** only positions with index0 can be '1' (since any '1' at position≥1 adds ≥1 to cost, exceeding k=0). Handled naturally by the `cost+pos<=k` check. ✓

- **Pruning safety:** since cost is monotonically non-decreasing as we add more characters, pruning a branch once `cost+pos>k` for a '1' choice is always safe — no future '0' choices reduce cost, and the '1' branch is the only one that increases it. ✓

- **n=12 max:** total binary strings without consecutive 1s is bounded by Fibonacci(n+2) (a well known identity), here F(14)=377, tiny. ✓

---

## Key Takeaway

Pure backtracking/DFS over small search space (n≤12) with constraint pruning. No need for DP or combinatorial formulas — direct enumeration with early cost-based pruning is both simplest and fast enough.
