# 3603. Minimum Cost Path with Alternating Directions II  
## $O(m \cdot n)$ — Dynamic Programming with Alternating Move/Wait States

---

## Problem Statement

You are given:
* A grid of size `m × n`
* Entry cost of cell `(i, j)` is:  
  `enter(i, j) = (i + 1) * (j + 1)`
* `waitCost[i][j]` — cost to wait on that cell for 1 second

The path starts at `(0, 0)` at **second 1**, paying its entry cost. Then, the rules alternate based on the current second:

### Odd seconds (1, 3, 5, …):
You **must move** to `(i + 1, j)` or `(i, j + 1)` and pay the **entry cost** of the new cell.

### Even seconds (2, 4, 6, …):
You **must wait** in place and pay `waitCost[i][j]`.

**Goal:** Reach `(m − 1, n − 1)` with the **minimum total cost**.

---

## Core Idea — DP with Forced Alternation

Let:
* `moves = (m − 1) + (n − 1)` — total number of moves needed (down + right)
* Total seconds = `2 * moves − 1` *(since the sequence goes: move, wait, move, wait, ..., move)*

We must perform exactly `m − 1` moves down and `n − 1` moves right. The order is arbitrary, but **every move is followed by a forced wait**, except the last move.

Thus, for each cell `(i, j)`:
* To reach it, we must have made exactly `i + j` moves.
* The number of waits before reaching `(i, j)` is:  
  `waits = i + j − 1`

### DP State
Let `dp[i][j]` = minimum cost to reach cell `(i, j)`.

### Transitions
To reach `(i, j)`:
* **From `(i − 1, j)`:** The previous second was odd $\rightarrow$ move into `(i, j)`. After moving, if `(i, j)` is **not** the final cell, we must wait once.
* **From `(i, j − 1)`:** Same logic.

### Transition Formula

If `(i, j)` is **not** the target:
```text
dp[i][j] = min(
    dp[i−1][j] + enter(i, j) + waitCost[i][j],
    dp[i][j−1] + enter(i, j) + waitCost[i][j]
)
```
If (i, j) is the target (m − 1, n − 1):

```text
dp[i][j] = min(
    dp[i−1][j] + enter(i, j),
    dp[i][j−1] + enter(i, j)
)
```
Base Case: ```dp[0][0] = enter(0, 0) = 1```


## Full Implementation (C#)
```csharp
public class Solution {
    public long MinCost(int m, int n, int[][] waitCost) {
        long[,] dp = new long[m, n];
        const long INF = long.MaxValue / 4;

        for (int i = 0; i < m; i++)
            for (int j = 0; j < n; j++)
                dp[i, j] = INF;

        // Entry cost for (0,0): (0+1)*(0+1)
        dp[0, 0] = 1; 

        for (int i = 0; i < m; i++) {
            for (int j = 0; j < n; j++) {

                if (i == 0 && j == 0) continue;

                long enter = (long)(i + 1) * (j + 1);
                bool isLast = (i == m - 1 && j == n - 1);

                // From top
                if (i > 0 && dp[i - 1, j] < INF) {
                    long cost = dp[i - 1, j] + enter;
                    if (!isLast) cost += waitCost[i][j];
                    dp[i, j] = Math.Min(dp[i, j], cost);
                }

                // From left
                if (j > 0 && dp[i, j - 1] < INF) {
                    long cost = dp[i, j - 1] + enter;
                    if (!isLast) cost += waitCost[i][j];
                    dp[i, j] = Math.Min(dp[i, j], cost);
                }
            }
        }

        return dp[m - 1, n - 1];
    }
}
```

## Complexity

| Metric | Value |
| :--- | :--- |
| **Time Complexity** | $O(m \cdot n)$ |
| **Space Complexity** | $O(m \cdot n)$ |

*Constraints: $m \cdot n \le 100,000 \rightarrow$ fits easily within memory and time limits.*

---

## Pitfalls & Edge Cases

* **Forced Waits:** You always wait after every move except the final one.
* **Entry Cost formula:** Entry cost is `(i + 1) * (j + 1)`, which is 1-indexed based on position.
* **Overflow:** Costs add up quickly, so the DP table must use `long`, not `int`.
* **Grid Shapes:** The grid can be extremely tall or wide; loop boundaries and transitions must handle edge rows/columns correctly.
* **Starting Cell:** `(0, 0)` has no wait at the beginning — you only pay its entry cost.

## Example Walkthrough

### Example 3
* **Grid Size:** `m = 2, n = 3`
* **Wait Costs:**
  ```text
  [[6, 1, 4],
   [3, 2, 5]]
  ```

  ## Path Trace:
  
  - Enter (0,0): $\text{cost} = 1$
  - Move right to (0,1): $\text{cost} = +2$
  - Wait at (0,1): $\text{cost} = +1$
  - Move down to (1,1): $\text{cost} = +4$
  - Wait at (1,1): $\text{cost} = +2$
  - Move right to (1,2) (Target): $\text{cost} = +6$(No wait required since it's the final cell)
  - Total Cost: $1 + 2 + 1 + 4 + 2 + 6 = 16$

  ## Key Takeaway:
 - This problem is a pure grid dynamic programming task with a temporal twist: every single move (odd second) is bound to a forced wait (even second), unless it's the final move.
 - This allows us to use a clean, deterministic $O(m \cdot n)$ DP approach without needing complex shortest-path graph algorithms like Dijkstra.
