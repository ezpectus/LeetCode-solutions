# 3969. Valid Subarrays With Matching Sum Digits I
## O(n²) — Prefix Sum Enumeration with Two Independent Digit Checks

---

## Problem Statement

Given `nums` and digit `x`, count subarrays `nums[l..r]` whose sum `S` satisfies **both**:
- Last digit of `S` equals `x` → `S % 10 == x`
- First digit of `S` equals `x` → leading digit of `S` equals `x`

Constraints: `n ≤ 1500`, `nums[i] ≤ 10⁹`, `x ∈ [1..9]`.

---

## Core Insight #1 — The Two Conditions Are Structurally Different

**Last digit** lives in `ℤ/10ℤ` — purely modular, order-independent:
```
S % 10 == x
```

**First digit** lives in a family of magnitude intervals — purely range-based, scale-dependent:
```
S ∈ [x · 10^k, (x+1) · 10^k)  for some integer k ≥ 0
```

For example, `x = 1` → valid sums: `1, 10–19, 100–199, 1000–1999, ...`
These two conditions are **orthogonal** — neither implies nor constrains the other.

---

## Core Insight #2 — O(n²) Is Exactly Tight for n = 1500

There are `n(n+1)/2 ≤ 1,125,750` subarrays to check. For each, the work is:

- **Last digit check:** `sum % 10`, one operation.
- **First digit check:** peel digits by repeatedly dividing by 10. Max sum = `1500 × 10⁹ = 1.5 × 10¹²`, which has at most 13 digits → at most 12 divisions.

Total operations ≈ `1.1M × 13 ≈ 14M` — comfortably within time limits. No clever data structure needed.

---

## Core Insight #3 — Prefix Sums Keep the Inner Loop O(1) Per Pair

Precompute `prefix[i] = nums[0] + ... + nums[i-1]`. Then:
```
sum(l, r) = prefix[r+1] - prefix[l]
```

Max prefix value = `1.5 × 10¹²` — fits in `long long` (max ~`9.2 × 10¹⁸`). Use `long long` throughout to avoid overflow.

---

## First-Digit Extraction

```cpp
int firstDigit(long long s) {
    while (s >= 10) s /= 10;
    return (int)s;
}
```

No floating-point, no `log10` precision issues — just integer division until one digit remains.

---

## Full Implementation (C++)

```cpp
class Solution {
public:
    int countValidSubarrays(vector<int>& nums, int x) {
        int n = nums.size();

        // Prefix sums in long long (max sum = 1500 * 1e9 = 1.5e12)
        vector<long long> prefix(n + 1, 0);
        for (int i = 0; i < n; i++) prefix[i + 1] = prefix[i] + nums[i];

        auto firstDigit = [](long long s) -> int {
            while (s >= 10) s /= 10;
            return (int)s;
        };

        int count = 0;
        for (int l = 0; l < n; l++) {
            for (int r = l; r < n; r++) {
                long long s = prefix[r + 1] - prefix[l];
                if (s % 10 == x && firstDigit(s) == x) count++;
            }
        }
        return count;
    }
};
```

---

## Complexity

| Metric | Value | Notes |
|---|---|---|
| Time Complexity | O(n² · log(maxSum)) | n² pairs × ≤13 divisions per firstDigit call |
| Space Complexity | O(n) | prefix array |

In practice, `log(maxSum)` = constant ≤ 13, so this is effectively O(n²) ≈ 2.25M iterations for n = 1500.

## Could We Do Better?

For completeness — the two conditions pull in different directions for sub-quadratic approaches:

**Last digit** alone is O(n): for each `r`, we need `prefix[l] % 10 == (prefix[r+1] % 10 - x + 10) % 10`. A size-10 count array over `prefix[i] % 10` lets us look this up in O(1).

**First digit** alone is O(n log S): since `nums[i] ≥ 1`, `sum(l, r)` is strictly increasing in `r` for fixed `l`. Binary search finds the endpoints of each valid interval `[x·10^k, (x+1)·10^k)` in O(log n) per interval, and there are O(log S) such intervals.

**Combining both** efficiently requires intersecting a modular congruence class with a set of magnitude intervals — non-trivial, and pointless at n = 1500 where O(n²) runs in milliseconds.

---

## Pitfalls & Edge Cases

- **Overflow:** `1500 × 10⁹ = 1.5 × 10¹²` exceeds `int` (max ~2.1 × 10⁹). Always use `long long` for both prefix sums and the sum variable.

- **Avoid `log10` for first digit:** `(int)log10(s)` has floating-point precision issues near powers of 10 (e.g., `log10(100)` might return `1.9999...`). The integer-division loop is exact and just as fast.

- **`x ∈ [1..9]`:** The problem guarantees no `x = 0`, so `firstDigit(s) == x` is always comparing with a nonzero digit — no edge cases around sums of 0 (impossible anyway, since `nums[i] ≥ 1`).

- **Single-element subarrays:** Naturally handled — `l == r` gives `sum = nums[l]`, checked the same way.

---

## Key Takeaway

This is a **"dual-condition digit check on subarray sums"** pattern. When the two conditions are structurally independent (modular vs. magnitude), no single clever trick handles both — but at small `n`, brute-force prefix-sum enumeration with O(1) per-pair checks is the clean and correct approach. The instinct to optimize should be suppressed until constraints force it.

**Pattern recognition:** "Check two digit-level properties of subarray sums" + `n ≤ 1500` → O(n²) prefix enum. Reserve cleverness for the `n ≤ 10⁵` "II" variant.
