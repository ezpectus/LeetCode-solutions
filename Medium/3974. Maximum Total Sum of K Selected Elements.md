# 3974. Maximum Total Sum of K Selected Elements
## O(n log n) — Per-Element max(1, coef) Weight + Rearrangement Inequality

---

## Problem Statement

Array `nums`, integers `k` and `mul`. Select exactly `k` elements, process them in any chosen order. For each, independently choose:
- **Add**: contribute `value`
- **Multiply**: contribute `value × current_mul`

After each processed element (regardless of choice), `mul` decreases by 1 (can go to 0 or negative). Maximize total sum.

---

## Core Insight — The Multiplier Sequence Is Fixed; Only Assignment Varies

Regardless of which `k` elements are chosen or what order they're processed in, the sequence of multiplier values used is **always**:
```
mul, mul−1, mul−2, …, mul−k+1
```
These are `k` consecutive descending integers — fixed in value, just waiting to be assigned to elements via the processing order. The only real decisions are:
1. **Which `k` elements** to select.
2. **Which multiplier value** (from the fixed sequence) pairs with each selected element.
3. **Add or multiply** for each element — but this choice is actually determined automatically (see below).

---

## Per-Element Choice Collapses to `max(value, value × coef)`

For an element with value `v > 0` (since `1 ≤ nums[i]`) paired with multiplier `coef`:
```
contribution = max(v, v × coef) = v × max(1, coef)
```

**Why:** `v × coef > v ↔ coef > 1` (since v > 0). So:
- If `coef > 1`: multiply is strictly better.
- If `coef == 1`: add and multiply are equal.
- If `coef ≤ 0` or `coef < 1`: add is at least as good (multiply is worse or equal).

Define `f(coef) = max(1, coef)`. The contribution is exactly `v × f(coef)`, and **`f(coef) ≥ 1` always** — confirming that pairing any element with any coefficient never produces negative regret. The add/multiply decision is now fully determined: just compute `f(coef)`, no separate branching needed.

---

## Rearrangement Inequality → Sort-and-Pair

We need to choose `k` values from `nums` and assign each one a distinct coefficient from `{mul, mul−1, …, mul−k+1}` to maximize:
```
Σ value_i × f(coef_i)
```

**Two separate optimizations, both resolved by the same principle:**

**(a) Which k elements to select:** Since `f(coef) ≥ 1 > 0` for every coefficient in the sequence, and `value > 0`, every pairing contributes a strictly positive amount that increases with `value`. For a *fixed* set of coefficients, larger values always contribute more. So **the top k values by magnitude** should be selected — using a smaller value is never beneficial since it can only decrease an already-positive contribution.

**(b) How to pair them with coefficients:** `f` is non-decreasing in `coef` (since `f(c)=max(1,c)` is non-decreasing). By the **rearrangement inequality**, `Σ aᵢbᵢ` is maximized when both sequences `{aᵢ}` and `{bᵢ}` are sorted in the **same order** (both ascending or both descending). Here `a = value` and `b = f(coef)`, both non-negative — so pairing the largest value with the largest coefficient (and so on, descending) maximizes the sum.

**Conclusion:** Sort `nums` descending, take the top `k`, and pair the `i`-th (0-indexed) largest value with coefficient `mul − i`.

---

## Algorithm

```
1. Sort nums in descending order.
2. ans = 0
3. For i = 0 to k-1:
     coef = mul - i
     ans += nums_sorted[i] * max(1, coef)
4. Return ans.
```

---

## Full Optimal Implementation (C#)

```csharp
public class Solution {
    public long MaxSum(int[] nums, int k, int mul) {
        Array.Sort(nums);
        Array.Reverse(nums);   // descending order

        long ans = 0;
        for (int i = 0; i < k; i++) {
            long coef = mul - i;
            long weight = Math.Max(1L, coef);
            ans += (long)nums[i] * weight;
        }
        return ans;
    }
}
```

| Metric | Value | Notes |
|---|---|---|

| Space | O(1) extra | In-place sort (excluding sort's internal overhead) |

For n = 10⁵: sort ≈ 10⁵ × 17 ≈ 1.7M comparisons. Trivially fast.

## Example Walkthroughs

### Example 1 — `nums = [6,1,2,9]`, `k = 3`, `mul = 2`, Expected: 26

Sorted descending: `[9, 6, 2, 1]`. Top 3: `9, 6, 2`.

| i | value | coef = mul−i | f(coef) = max(1,coef) | contribution |
|---|-------|--------------|------------------------|---------------|
| 0 | 9 | 2 | 2 | 18 |
| 1 | 6 | 1 | 1 | 6 |
| 2 | 2 | 0 | 1 | 2 |

Total = 18 + 6 + 2 = **26** ✓ — matches the example's choice (multiply 9, multiply 6, add 2).

---

### Example 2 — `nums = [3,7,5,2]`, `k = 2`, `mul = 4`, Expected: 43

Sorted descending: `[7, 5, 3, 2]`. Top 2: `7, 5`.

| i | value | coef | f(coef) | contribution |
|---|-------|------|---------|---------------|
| 0 | 7 | 4 | 4 | 28 |
| 1 | 5 | 3 | 3 | 15 |

Total = 28 + 15 = **43** ✓

---

### Example 3 — `nums = [4,4]`, `k = 1`, `mul = 1`, Expected: 4

Sorted descending: `[4, 4]`. Top 1: `4`.

| i | value | coef | f(coef) | contribution |
|---|-------|------|---------|---------------|
| 0 | 4 | 1 | 1 | 4 |

Total = **4** ✓

---

### Illustrative Negative-Coefficient Case — `nums = [10, 1]`, `k = 2`, `mul = 1`

Sorted descending: `[10, 1]`.

| i | value | coef = mul−i | f(coef) | contribution |
|---|-------|--------------|---------|---------------|
| 0 | 10 | 1 | 1 | 10 |
| 1 | 1 | 0 | max(1,0)=1 | 1 |

Total = 11. *(Both elements use addition effectively — coef 1 is a tie, coef 0 forces add since 1×0 < 1.)*

If `mul = -3` instead: coefficients are `-3, -4`. Both `f = 1`. Total = 10×1 + 1×1 = 11 — same result, confirming that very negative `mul` simply forces every element into add-mode, contributing its raw value.

## Why Selecting Top-K and Sorting-to-Pair Can Be Done Independently

A subtlety: steps (a) "select top k" and (b) "pair largest-with-largest" are two separate applications of the same monotonic-product principle, but they don't interfere because:
- Step (a) fixes *which* k coefficient values get used (always the full fixed sequence `mul, mul-1, ..., mul-k+1`, since exactly k elements are always processed) — selection only decides *which values*, not which coefficients.
- Step (b) then optimally assigns those fixed k coefficients to the fixed k selected values.

Both steps independently favor "biggest with biggest," so sorting both arrays (or sorting one and using a fixed descending coefficient ramp) is correct without further case analysis.

---

## Pitfalls & Edge Cases

- **`coef` can be ≤ 0:** `mul - i` for large `i` or small `mul` can be zero or negative. `max(1, coef)` correctly forces these elements into "add" mode — multiplying by ≤0 would only reduce or negate the contribution, which is never chosen since add (contributing the raw positive value) dominates.

- **`int` overflow on intermediate products:** `nums[i] ≤ 10⁵` and `|coef| ≤ 10⁵` in extreme cases (though coef ≤ mul ≤ 10⁵; but with k ≤ n ≤ 10⁵, coef can go as low as `mul - k + 1` which could be very negative, e.g. `1 - 10⁵ + 1 ≈ -10⁵`). The product after `max(1, coef)` is bounded by `max(1, mul) ≤ 10⁵`-ish, but to be safe, all arithmetic should use `long`. The reference implementation casts both factors to `long` before multiplying.

- **Sum overflow:** max total ≈ k × (10⁵ × 10⁵) = 10⁵ × 10¹⁰ = 10¹⁵, fits comfortably in `long` (max ≈ 9.2×10¹⁸).

- **`k = n` (select everything):** The algorithm still works — sort all of `nums` descending and pair every element with its coefficient. No special-casing needed.

- **All elements equal:** Sorting is stable/irrelevant since values are identical; pairing order among equal values doesn't affect the sum (any permutation of the top-k set gives the same total since they're indistinguishable). ✓

- **`mul = 1`, `k = 1`:** Single coefficient = 1, `f(1) = 1`. Contribution = the single largest value, regardless of add/multiply choice (they're equal). ✓

## Key Takeaway

**Two-stage greedy reducible to one sort:** when each element's contribution under an optimal local choice becomes `value × f(assigned_parameter)` with `f` monotonic, and parameters are drawn from a *fixed* sequence — (1) the add/multiply branching collapses into a single weight function `f`, (2) selecting elements and assigning parameters are both resolved by the rearrangement inequality (sort both descending, pair index-by-index).

**Pattern recognition:** "Choose between two operations per element, where the choice's benefit depends on a parameter that changes deterministically with processing order" → check if the per-element optimal choice reduces to `value × max(constant, parameter)` or similar monotonic weight. If so, the entire problem collapses to: sort by value, sort/derive the parameter sequence, pair greedily via rearrangement inequality.
