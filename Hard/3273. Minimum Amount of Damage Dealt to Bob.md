# 3273. Minimum Amount of Damage Dealt to Bob
## O(n log n) — Greedy: Minimize Weighted Completion Time, Sort by Damage/Time Ratio

---

## Problem Statement

`n` enemies, each dealing `damage[i]` per second while alive. Bob attacks one enemy per second with `power` damage. Enemies deal damage first each second, then Bob attacks. Minimize total damage Bob receives.

---

## Core Insight #1 — Focus Fire: Kill One Enemy Completely Before Moving On

Should Bob split attacks between enemies or kill them one at a time?

**Claim:** The optimal strategy always kills enemies one at a time (focus fire).

**Proof sketch:** If Bob attacks enemies A and B alternately, both remain alive longer, each dealing more total damage. Any interleaved attack schedule can be rearranged into a focus-fire schedule without increasing total damage. Specifically: rearranging Bob's attack sequence to group all attacks on the same enemy together doesn't change kill times or damage totals — it only changes when each enemy dies (killing one sooner is always better or equal).

So we only need to choose **the order** in which to kill enemies.

---

## Core Insight #2 — Kill Time Formula

To kill enemy `i`, Bob needs exactly:
```
t[i] = ⌈health[i] / power⌉  seconds of focused attacks
```

If Bob kills enemies in order σ[0], σ[1], ..., σ[n-1], the completion time of enemy σ[k] is:
```
T[k] = t[σ[0]] + t[σ[1]] + ... + t[σ[k]]  (cumulative kill time)
```

Enemy σ[k] deals damage every second it's alive — from second 1 through second T[k]. So:
```
damage from enemy σ[k] = damage[σ[k]] × T[k]
total = Σ_k damage[σ[k]] × T[k]
```

---

## Core Insight #3 — Exchange Argument: Sort by damage[i]/t[i] Descending

This is the classical **weighted completion time** problem: minimize `Σ w_k × C_k` where `C_k = Σ_{j≤k} p_j`.

**Exchange argument:** Consider two adjacent enemies A and B in the ordering.

- **A before B:** `w_A × p_A + w_B × (p_A + p_B)` where w=damage, p=t
- **B before A:** `w_B × p_B + w_A × (p_A + p_B)`

A before B is better iff:
```
w_A × p_A + w_B × p_A + w_B × p_B  <  w_B × p_B + w_A × p_B + w_A × p_A
⟺  w_B × p_A  <  w_A × p_B
⟺  w_A / p_A  >  w_B / p_B
```

**Optimal ordering: sort by `damage[i] / t[i]` in decreasing order.**

Tie-breaking: when `damage[A]/t[A] == damage[B]/t[B]`, both orderings give equal total — any tie-breaking is correct.

## Full Implementation (C++)

```cpp
class Solution {
public:
    long long minDamage(int power, vector<int>& damage, vector<int>& health) {
        int n = damage.size();
        vector<int> idx(n);
        iota(idx.begin(), idx.end(), 0);

        // t[i] = ceil(health[i] / power) — time to kill enemy i
        auto t = [&](int i) {
            return (health[i] + power - 1) / power;
        };

        // Sort by damage[i]/t[i] descending, using cross-multiplication to avoid floating point
        // damage[a]/t[a] > damage[b]/t[b]  ⟺  damage[a]*t[b] > damage[b]*t[a]
        sort(idx.begin(), idx.end(), [&](int a, int b) {
            return (long long)damage[a] * t(b) > (long long)damage[b] * t(a);
        });

        long long total = 0, cumTime = 0;
        for (int i : idx) {
            cumTime += t(i);
            total += (long long)damage[i] * cumTime;
        }

        return total;
    }
};
```

## Example Walkthrough

### Example 1 — `power=4, damage=[1,2,3,4], health=[4,5,6,8]`

**Compute t[i] = ⌈health[i]/4⌉:**
| Enemy | health | t[i] | damage[i] | ratio d/t |
|---|---|---|---|---|
| 0 | 4 | 1 | 1 | 1.00 |
| 1 | 5 | 2 | 2 | 1.00 |
| 2 | 6 | 2 | 3 | 1.50 |
| 3 | 8 | 2 | 4 | **2.00** |

**Sort by ratio desc:** 3 → 2 → {0,1} (tie).

**Accumulation (order: 3, 2, 0, 1):**
| Step | Enemy | t[i] | cumTime | damage × cumTime |
|---|---|---|---|---|
| 1 | 3 | 2 | 2 | 4 × 2 = 8 |
| 2 | 2 | 2 | 4 | 3 × 4 = 12 |
| 3 | 0 | 1 | 5 | 1 × 5 = 5 |
| 4 | 1 | 2 | 7 | 2 × 7 = 14 |

**Total = 8+12+5+14 = 39** ✓

---

### Example 2 — `power=1, damage=[1,1,1,1], health=[1,2,3,4]`

t = [1, 2, 3, 4]. Ratios: 1/1=1, 1/2=0.5, 1/3=0.33, 1/4=0.25.

Sort: enemy 0, 1, 2, 3. cumTimes: 1, 3, 6, 10.

Total = 1×1 + 1×3 + 1×6 + 1×10 = **20** ✓

---

### Example 3 — `power=8, damage=[40], health=[59]`

t[0] = ⌈59/8⌉ = 8. Total = 40 × 8 = **320** ✓

## Pitfalls & Edge Cases

- **No floating-point in comparator.** `damage[a]/t[a] > damage[b]/t[b]` should be written as `damage[a]*t(b) > damage[b]*t(a)` to avoid floating-point precision errors. Cast to `long long` before multiplication.

- **⌈health/power⌉ = (health + power - 1) / power** with integer division. Don't use `health/power` (floor).

- **Tie-breaking is irrelevant.** When `damage[a]*t(b) == damage[b]*t(a)`, both orderings give the same total. The comparator can use `>` (strict) — ties resolve arbitrarily.

- **Why not just attack the enemy dealing highest current damage?** This greedy fails because it ignores kill times. An enemy dealing 1000 damage/sec but requiring 10000 seconds to kill might be better to deprioritize versus one dealing 999 damage/sec but dying in 1 second. The ratio `damage/t` captures both factors.

- **Single enemy (n=1):** t[0] seconds, total = damage[0] × t[0]. Trivially correct from the formula.

---

## Key Takeaway

This is the **weighted job scheduling** problem: minimize `Σ w_i × C_i` where `C_i` is the completion time and `w_i = damage[i]`, processing time `p_i = t[i] = ⌈health[i]/power⌉`. The optimal order is by decreasing `w_i/p_i` ratio, proven via the exchange argument (adjacent swap shows higher ratio belongs first).

**Pattern recognition:** "Process items sequentially, each item i takes time p_i, incurs cost w_i × (completion time)" → sort by `w_i/p_i` descending. Cross-multiply to avoid floating-point: sort by `w_a × p_b > w_b × p_a`.
