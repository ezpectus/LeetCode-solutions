# 1815. Maximum Number of Groups Getting Fresh Donuts
## O(batchSize × stateSpace) — Greedy Preprocessing + Bitmask-like Memoized DP

---

## Problem Statement

- Donuts are baked in batches of `batchSize`. A group is **happy** if the leftover from the previous group is 0 (they get fresh donuts).
- You can reorder groups freely.
- Return the **maximum number of happy groups**.
- Constraints: `batchSize ≤ 9`, `groups.length ≤ 30`, `groups[i] ≤ 10⁹`.

---

## Core Idea — Three-Phase Approach

**Key observation**: Only `groups[i] % batchSize` matters — the actual size is irrelevant beyond its remainder. This caps the state space.

**Phase 1 — Free happy groups (remainder = 0)**:
Groups where `groups[i] % batchSize == 0` are **always happy** regardless of position — they consume an exact number of full batches, leaving 0 leftover. Count them directly.

**Phase 2 — Greedy complementary pairing**:
For remainders `r` and `batchSize - r`: if we place a group with remainder `r` then one with remainder `batchSize-r`, the combined leftover is `r + (batchSize-r) = batchSize ≡ 0`. So the **second group** of the pair is always happy. Greedily pair as many as possible — each pair contributes **1 happy group** (the second).

Special case: if `batchSize` is even, remainder `batchSize/2` paired with itself → every 2 such groups give 1 happy.

**Phase 3 — DP on unpaired remainders**:
After pairing, some remainders remain unpaired. Use **memoized DFS** with state = `(counts of each remaining remainder, current leftover)`. Try all orderings of remaining groups, track the running leftover mod batchSize.

## Full Optimal Implementation (C#)

```csharp
public class Solution {
    int batchSize;
    Dictionary<(string, int), int> memo = new();

    public int MaxHappyGroups(int batchSize, int[] groups) {
        this.batchSize = batchSize;
        int[] cnt = new int[batchSize];
        int baseHappy = 0;

        foreach (int g in groups) {
            int r = g % batchSize;
            if (r == 0) baseHappy++;   // Phase 1: always happy
            else cnt[r]++;
        }

        // Phase 2: greedy complementary pairing
        int pairs = 0;
        for (int r = 1; r <= batchSize / 2; r++) {
            int comp = batchSize - r;
            if (r != comp) {
                int p = Math.Min(cnt[r], cnt[comp]);
                pairs += p;
                cnt[r] -= p;
                cnt[comp] -= p;
            } else {
                pairs += cnt[r] / 2;
                cnt[r] %= 2;
            }
        }

        // Phase 3: DP on remaining unpaired groups
        // State: counts array (indices 1..batchSize-1) + current leftover
        int[] remaining = cnt[1..]; // cnt[1], cnt[2], ..., cnt[batchSize-1]
        return baseHappy + pairs + Dp(remaining, 0);
    }

    int Dp(int[] cnt, int leftover) {
        string key = string.Join(",", cnt);
        if (memo.TryGetValue((key, leftover), out int cached)) return cached;

        int best = 0;
        for (int r = 1; r < batchSize; r++) {
            if (cnt[r-1] == 0) continue;
            cnt[r-1]--;
            int happy = leftover == 0 ? 1 : 0;
            best = Math.Max(best, happy + Dp(cnt, (leftover + r) % batchSize));
            cnt[r-1]++;
        }

        memo[(key, leftover)] = best;
        return best;
    }
}
```

## Complexity

| **Metric** | **Value** | **Notes** |
|---|---|---|
| **Time Complexity** | **O(S × batchSize)** | S = number of distinct DP states; bounded by remaining groups × batchSize |
| **Space Complexity** | **O(S)** | Memoization table |

**Why fast in practice**: after greedy pairing, at most 1 unpaired group per remainder → at most `batchSize-1` ≤ 8 remaining groups. States = at most `2^8 × batchSize` = 2304 — tiny.

---

## Why Greedy Pairing is Optimal

For complementary pairs `(r, batchSize-r)`:
- Placed consecutively, the first adds `r` to leftover, the second starts with leftover `prev + r`
- If we arrange: `[...some groups...][r-group][comp-group]` where leftover before `r-group` is 0 → both happy
- But no matter the arrangement, placing `r` and `comp` together always makes the **second one happy** if the first started a fresh batch
- The greedy gives the **maximum 1 happy per pair** — this can't be improved since both consuming full batch resets leftover

## Example Walkthrough

**Example 1**: `batchSize=3, groups=[1,2,3,4,5,6]`

Remainders: `[1,2,0,1,2,0]`

**Phase 1**: rem=0 count = 2 → `baseHappy=2`

**Phase 2**: `cnt=[0,1,1]` (cnt[1]=2→after phase1 removal: cnt[1]=2, cnt[2]=2... let me recount)

Actually cnt[1]=2 (from groups 1,4), cnt[2]=2 (from groups 2,5). Pair: `r=1, comp=2`, `min(2,2)=2` pairs → `pairs=2`, `cnt[1]=0, cnt[2]=0`.

**Phase 3**: all zeroes → `dp([0,0], 0) = 0`

Total: `2 + 2 + 0 = 4` ✓

## Pitfalls & Edge Cases

- **`batchSize=1`**: every group has remainder 0 → all happy → answer = `n`
- **After pairing, middle remainder** (`r = batchSize/2` when even): pairs of same remainder → 1 happy per 2 groups; 1 leftover goes to DP
- **DP state key**: use string join of counts array (or tuple) — distinct states are few after pairing
- **`leftover == 0` check**: the first group in DP always sees leftover=0 from the pairing phase cleanup → it's happy; subsequent groups depend on accumulated leftover

---

## Key Takeaway

Hard constraint problems with small modular structure → **greedy preprocessing + memoized DP on reduced state**:

1. **Remainder 0** → free happy groups, handled trivially
2. **Complementary pairs** → greedy maximizes happy groups from pairs in O(batchSize)
3. **Residual unpaired** → DP on `(counts, leftover)` with tiny state space after greedy reduction

The greedy pairing is critical — without it, the DP state space would be exponential. With it, at most 1 group per remainder remains → ≤ 8 groups → manageable.
