# 2630. Memoize II
## O(n · k) — Trie-Based Argument Caching with Object Identity

---

## Problem Statement

- Given a function `fn`, return a memoized version that never calls `fn` twice with the same inputs.
- Inputs are considered identical if they are `===` to each other (strict equality / object identity).
- `fn` can accept any number of arguments of any type.
- Return the cached result on repeated calls with identical input sequences.

---

## Core Idea — Argument Trie with WeakMap / Map per Level

Key insight: We need to cache based on an **ordered sequence** of arguments where each argument is compared by `===`. A simple `JSON.stringify` key fails for objects (two `{}` are different objects but stringify identically). A `Map` key of the whole args array also fails.

Solution: Build a **Trie** where each level corresponds to one argument position. Each node stores:
- A `Map` for primitive values (string, number, boolean, null, undefined, Symbol) — keyed by value
- A `Map` for object/function references — keyed by reference identity

For each call, traverse the trie level-by-level using each argument as the key. At the leaf node, store the cached result.

This handles:
- Primitives: cached by value
- Objects/functions: cached by reference identity (`===`)

---

## Full Optimal Implementation (JavaScript)

```javascript
function memoize(fn) {
    // Each trie node: { children: Map, result: { val, hasVal } }
    // We use a single Map that works for both primitives and object references
    // since Map keys use SameValueZero which is essentially === for our purposes
    const root = new Map();

    return function(...args) {
        let node = root;

        for (const arg of args) {
            if (!node.has(arg)) {
                node.set(arg, new Map());
            }
            node = node.get(arg);
        }

        // Use a special sentinel key to store the result at this leaf
        const RESULT_KEY = Symbol.for('__result__');
        if (!node.has(RESULT_KEY)) {
            node.set(RESULT_KEY, fn(...args));
        }

        return node.get(RESULT_KEY);
    };
}
```

---

## Why Map Works for Object Identity

`Map` uses the [SameValueZero](https://tc39.es/ecma262/#sec-samevaluezero) algorithm for key comparison, which is essentially `===` for all types except `NaN` (but constraints say no `NaN`). This means:

```javascript
const o1 = {};
const o2 = {};
const map = new Map();

map.set(o1, 'a');
map.set(o2, 'b');

map.get(o1); // 'a' — o1 and o2 are different keys
map.get(o2); // 'b'
```

So the trie correctly distinguishes between two different `{}` objects (different references) while reusing the cache when the exact same object reference is passed again.

---

## Complexity

| Metric | Value | Notes |
|---|---|---|
| Time Complexity | O(k) per call | k = number of arguments; traverse one trie level per arg |
| Space Complexity | O(n · k) | n = unique call signatures, k = args per call |

Cache lookup is O(k) per call, which is effectively O(1) for fixed-arity functions.

---

## Why This Works — Example Walkthrough

**Example 1:** `fn = (a, b) => a + b`, calls: `(2,2), (2,2), (1,2)`

Trie state after call 1 `(2, 2)`:
```
root → Map{ 2 → Map{ 2 → Map{ RESULT → 4 } } }
```
Call 2 `(2, 2)`: traverses same path → hits RESULT → returns 4, no `fn` call.

Call 3 `(1, 2)`: new path → `fn` called → returns 3.

**Output:** calls = 1, 1, 2 ✓

---

**Example 2:** `fn = (a, b) => ({...a, ...b})`, calls: `({},{}), ({},{}), ({},{})`

All 6 `{}` are **different object references**. Each pair is a unique trie path:

```
root → Map{
  obj1 → Map{ obj2 → Map{ RESULT → {} } },
  obj3 → Map{ obj4 → Map{ RESULT → {} } },
  obj5 → Map{ obj6 → Map{ RESULT → {} } }
}
```

Every call is a cache miss → `fn` called 3 times.

**Output:** calls = 1, 2, 3 ✓

---

**Example 3:** `const o = {}`, calls: `(o,o), (o,o), (o,o)`

Same `o` reference reused. All calls traverse the same trie path:

```
root → Map{ o → Map{ o → Map{ RESULT → {} } } }
```

Call 1: cache miss → `fn` called. Calls 2 & 3: cache hit.

**Output:** calls = 1, 1, 1 ✓

## Pitfalls & Edge Cases

- **`JSON.stringify` approach fails** — `JSON.stringify({})` produces the same string for all empty objects, but they are different references. Two calls with different `{}` objects must both call `fn`.

- **Single `Map` with string key fails** — Concatenating args into a string key (`args.join(',')`) also fails for objects.

- **`WeakMap` alone doesn't work** — `WeakMap` only accepts objects as keys, can't handle primitives like numbers or strings.

- **Mixed arg types** — A single `Map` per trie node handles both primitives and object references correctly since `Map` uses `===` semantics for all key types.

- **Zero arguments** — If `fn` is called with no args, the loop doesn't execute, and the result is stored directly at the root node under `RESULT_KEY`. Works correctly.

- **`Symbol.for('__result__')` sentinel** — Using a shared Symbol as the result key ensures no collision with actual argument values, since a Symbol is never `===` to a user-provided argument (unless the user explicitly passes the same Symbol, which is an edge case the problem doesn't test).

- **Memory leaks** — Using a plain `Map` (not `WeakMap`) means object-keyed entries won't be garbage-collected as long as the memoized function exists. For production use, a `WeakMap`-based trie for object keys would be preferable.

---

## Alternative: WeakMap + Map Hybrid (Production Grade)

```javascript
function memoize(fn) {
    // Use WeakMap for objects (GC-friendly), Map for primitives
    function makeNode() {
        return { objMap: new WeakMap(), primMap: new Map(), result: undefined, hasResult: false };
    }

    const root = makeNode();

    return function(...args) {
        let node = root;

        for (const arg of args) {
            const isObj = arg !== null && (typeof arg === 'object' || typeof arg === 'function');
            const map = isObj ? node.objMap : node.primMap;

            if (!map.has(arg)) {
                map.set(arg, makeNode());
            }
            node = map.get(arg);
        }

        if (!node.hasResult) {
            node.result = fn(...args);
            node.hasResult = true;
        }

        return node.result;
    };
}
```

This version uses `WeakMap` for object keys (allows GC when objects are no longer referenced) and `Map` for primitives, making it memory-safe for long-running applications.

---

## Key Takeaway

This is a **Trie-based cache** problem where each argument is a trie level:
- Map per node handles both primitive and reference equality via `===` semantics
- Traversing the trie per call gives O(k) lookup/store
- The sentinel key at leaf nodes stores the cached return value

**Pattern recognition:** "Cache function calls with arbitrary argument types and object identity" → Argument Trie with `Map` nodes. Never use string serialization for object arguments.

Classic problem that tests understanding of JavaScript equality semantics (`===` vs `==` vs `JSON.stringify`) and the difference between value identity and structural equality.
