# 2835. Minimum Operations to Form Subsequence With Target Sum

## O(N + log(Target)) — Bit Counting + Greedy Decomposition

### Problem Statement
You are given a 0-indexed array `nums` consisting of non-negative powers of 2, and an integer `target`.

In one operation, you must apply the following changes to the array:
* Choose any element of the array `nums[i]` such that `nums[i] > 1`.
* Remove `nums[i]` from the array.
* Add two occurrences of `nums[i] / 2` to the end of `nums`.

Return the minimum number of operations you need to perform so that `nums` contains a subsequence whose elements sum to `target`. If it is impossible to obtain such a subsequence, return `-1`.

---

### Core Idea — Bit-by-Bit Greedy Accumulation
Since every element in `nums` is a power of 2, this problem maps perfectly to binary representation. Dividing an element by 2 ($nums[i] \rightarrow nums[i]/2 + nums[i]/2$) is equivalent to shifting a bit to the right and duplicating it. 

The total sum of elements never changes during operations. Thus, if the total sum of `nums` is strictly less than `target`, it is mathematically impossible to form the target subsequence, and we can immediately return `-1`.

#### The Greedy Traversal Protocol
Instead of simulating the array expansion, we can count the frequencies of each power of 2 (from $2^0$ to $2^{30}$) present in `nums`. We then iterate through each bit position `i` from 0 up to 30:

1. **Resource Contribution:** Any available elements of size $2^i$ are added to our running resource pool: `runningSum += count[i] * (1 << i)`.
2. **Bit Target Evaluation:** We check if the $i$-th bit of `target` is active (i.e., `(target & (1 << i)) != 0`). If it is **not** active, we can skip to the next bit while preserving our `runningSum` pool (smaller powers can always combine to form larger ones).
3. **Fulfilling the Bit Requirement:** If the $i$-th bit is active, we attempt to satisfy it:
   * **Case A (Sufficient Pool):** If `runningSum >= (1 << i)`, our accumulated smaller elements are large enough to cover this bit. We deduct the required value from our pool: `runningSum -= (1 << i)`.
   * **Case B (Decomposition Required):** If `runningSum` is insufficient, we must find the nearest available larger power of 2 in our frequency array (find the smallest index `j > i` where `count[j] > 0`). We then break this element down step-by-step to the current level `i`. Each downward division step costs exactly $1$ operation.

### Full Optimal Implementation (C#)

```csharp
using System;
using System.Collections.Generic;

public class Solution {
    public int MinOperations(IList<int> nums, int target) {
        long totalSum = 0;
        int[] bitCounts = new int[32];
        
        // Step 1: Count frequencies of each power of 2 and compute total capacity
        foreach (int x in nums) {
            totalSum += x;
            int bitPosition = Log2(x);
            bitCounts[bitPosition]++;
        }
        
        // Base impossibility check
        if (totalSum < target) return -1;
        
        long runningSum = 0;
        int operations = 0;
        int i = 0;
        
        // Step 2: Iterate bit by bit from lowest to highest power
        while (i < 30) {
            // Accumulate available resources at the current bit level
            runningSum += (long)bitCounts[i] * (1 << i);
            
            // Check if the current bit is required by the target
            if ((target & (1 << i)) != 0) {
                if (runningSum >= (1 << i)) {
                    // We can satisfy this bit using our accumulated smaller elements
                    runningSum -= (1 << i);
                } else {
                    // We must find the closest larger element to decompose
                    int j = i + 1;
                    while (j < 32 && bitCounts[j] == 0) {
                        j++;
                    }
                    
                    // Decompose the element down to level i
                    while (j > i) {
                        bitCounts[j]--;
                        bitCounts[j - 1] += 2;
                        operations++;
                        j--;
                    }
                    
                    // Consume one of the freshly generated components
                    bitCounts[i]--;
                    runningSum += (1 << i);
                    runningSum -= (1 << i);
                }
            }
            
            i++;
        }
        
        return operations;
    }
    
    private int Log2(int value) {
        int bits = 0;
        while (value > 1) {
            value >>= 1;
            bits++;
        }
        return bits;
    }
}
```

### Complexity

| Metric | Value | Notes |
| :--- | :--- | :--- |
| **Time Complexity** | $O(N + \log(\text{Target}))$ | Collecting frequencies takes $O(N)$ time. The bit traversal loop runs up to 30 times. The inner decomposition loops process at most 30 steps total over the lifetime of the program, yielding an efficient linear execution path. |
| **Space Complexity** | $O(1)$ | Auxiliary space is bound to a fixed-size integer tracking array of length 32, which works independently of input size metrics. |

---

### Pitfalls & Edge Cases

* **Integer Capacity Limitations:** The `runningSum` tracker can scale beyond the boundaries of signed 32-bit `int` limits ($2 \times 10^9$) when multiplying counts by their large binary bit values. Maintain all aggregation states using 64-bit `long` keywords.
* **Skipping Empty Unneeded Bits:** If a bit is missing from `target`, do not initiate breakdowns. Simply allow lower values to bleed upward into `runningSum` via `runningSum += (long)bitCounts[i] * (1 << i)`, as they can be useful for higher-order bits.

---

### Key Takeaway

> When an allocation problem operates entirely within binary powers and allows splitting large fragments into identical halves, avoid search state graphs. Map frequencies into a bit array, process targets from lowest to highest order, and trigger greedy local lookaheads when resources deplete.

* **Pattern recognition:** "Minimum array division operations to hit a exact sum of binary powers" -> Bit Counting Bucket + Low-to-High Greedy Decompositions.
