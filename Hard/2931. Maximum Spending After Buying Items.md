# 2931. Maximum Spending After Buying Items
## O(mn log mn) — Rearrangement Inequality: Sort All Values, Pair Smallest with Earliest Day

---

## Problem Statement

m×n matrix `values`, non-increasing per row (`values[i][j] >= values[i][j+1]`). On day `d`, buy the **rightmost available** item from any shop, paying `values[i][j] * d`. Buy all m×n items over days 1…mn. Maximize total spending.

---

## Core Insight — The "Rightmost" Constraint Forces Non-Decreasing Value Order Per Shop

The rule "buy the rightmost available item" means that within shop `i`, items are purchased in the order:
```
values[i][n-1],  values[i][n-2],  …,  values[i][0]
```
Since the shop is **non-increasing** left-to-right, reading right-to-left gives a **non-decreasing** sequence:
```
values[i][n-1]  ≤  values[i][n-2]  ≤  …  ≤  values[i][0]
```
Each shop already provides its items in non-decreasing value order. The constraint that seemed restrictive is actually aligned with what we want.

---

## Rearrangement Inequality → Optimal Schedule

Total spending = Σ `values[i][j] * day[i][j]`. All m×n items are bought; only the day assignments vary. Days 1, 2, …, mn are fixed and increasing.

**Rearrangement inequality**: to maximize `Σ aₖ * bₖ` where `bₖ = k` is fixed and increasing, we maximize by also sorting `aₖ` in increasing order — pair the smallest value with day 1, next smallest with day 2, etc.

**Claim**: the optimal schedule is to globally sort all m×n item values in non-decreasing order and assign days 1, 2, …, mn in that order.

---

## Why the Per-Shop Constraint Is Automatically Satisfied

In the globally sorted schedule, consider any two items from the same shop `i` at positions `j` and `j'` with `j < j'` (so `j'` is to the right, values[i][j] ≥ values[i][j']`):
- `values[i][j'] ≤ values[i][j]` → `values[i][j']` appears no later in the global sort.
- So `j'` is assigned an earlier (or equal) day than `j`. ✓
- This is exactly the required order: rightmost item (j') bought before item (j). ✓

The globally sorted order never violates the "rightmost first" rule.

---

## Exchange Argument: Proof of Optimality

In any valid schedule, suppose item A (value `a`) is assigned day `dA` and item B (value `b > a`) is assigned day `dB < dA` (B gets an earlier day despite being more valuable):

**Case 1 — A and B are from different shops:** Swapping their days (A→dB, B→dA) is always valid across shops. New total: old total + `(b−a)(dA−dB) > 0`. This improves the total, so the original schedule was suboptimal.

**Case 2 — A and B are from the same shop:** Since `a < b`, item A has smaller value → A is further right in the shop (rows are non-increasing) → A must be bought **before** B → `dA < dB`. But we assumed `dA > dB` — contradiction! This case never occurs in a valid schedule.

Therefore: in any valid schedule, if `a < b` then `dA ≤ dB`. The globally sorted assignment is uniquely optimal (up to equal-value ties, which don't affect the total). ✓

---

## Algorithm

```
1. Flatten all values into a 1D array of length m*n.
2. Sort in non-decreasing order.
3. Return Σ sorted[k] * (k+1) for k = 0, 1, …, m*n−1.
```

That's it. The "Hard" problem reduces to four lines of code.

---

## Full Optimal Implementation (C#)

```csharp
public class Solution {
    public long MaxSpending(int[][] values) {
        int m = values.Length, n = values[0].Length;
        int[] flat = new int[m * n];
        int idx = 0;
        for (int i = 0; i < m; i++)
            for (int j = 0; j < n; j++)
                flat[idx++] = values[i][j];

        Array.Sort(flat);   // non-decreasing: smallest value → day 1

        long ans = 0;
        for (int k = 0; k < flat.Length; k++)
            ans += (long)flat[k] * (k + 1);   // day = k+1

        return ans;
    }
}
```

## Complexity

| Metric | Value | Notes |
|---|---|---|
| Time | O(mn log mn) | Flatten O(mn) + sort O(mn log mn) + sum O(mn) |
| Space | O(mn) | Flat array; in-place sort |

For m=10, n=10⁴: mn = 10⁵ elements. Sort: ≈ 10⁵ × 17 ≈ 1.7M ops. Trivially fast.

**Alternative**: min-heap initialized with the rightmost item from each shop, pop minimum per day. O(mn log m) — faster in theory (log m ≤ log 10 ≈ 3 vs log mn ≈ 17), but the flat-sort approach is simpler and fast enough.

### Why Suboptimal Schedules Lose

Suppose in Example 1 we swapped days 1 and 9 (buy value=9 first, value=1 last):
```
New total = 285 − (9×9 + 1×1) + (9×1 + 1×9)
          = 285 − (81 + 1) + (9 + 9)
          = 285 − 82 + 18 = 221
```
Loss = 64. Generalizes: any swap of items with values `a < b` and days `dA > dB` loses `(b−a)(dA−dB) > 0`.

## Pitfalls & Edge Cases

- **Integer overflow:** max single term = 10⁶ × (10×10⁴) = 10¹¹. Total ≤ 10¹¹ × 10⁵ = 10¹⁶. Cast to `long` before multiplying: `(long)flat[k] * (k+1)`. Accumulating in `int` silently wraps. ✓

- **Equal values across shops:** Two items with the same value can be assigned to days d₁ and d₂ in either order — the total (a·d₁ + a·d₂) = a·(d₁+d₂) is the same regardless. No special handling needed. ✓

- **Equal values within a shop (values[i][j] = values[i][j+1]):** Global sort may assign them in either order. Since they're equal, any order gives the same total. The per-shop constraint is automatically satisfied or irrelevant. ✓

- **Single shop (m=1):** Rows are non-increasing; rightmost to leftmost is non-decreasing; sort gives the same order. Works trivially. ✓

- **Single column (n=1):** Each shop has one item. All items can be bought in any order across shops. Sort and assign. ✓

- **The "pairwise different" note** in the problem means items are distinct objects (not values). Same column index j can be bought from multiple shops — this is already handled by our flat array. ✓

---

## Key Takeaway

**"Rightmost first" + non-increasing rows = items released in non-decreasing order per shop.** This constraint and the optimization objective (rearrangement inequality: pair large values with large days) point to the same greedy: sort all values globally and assign days 1, 2, …, mn. The "Hard" rating comes from noticing that the mandatory per-shop order is already the optimal global order — once seen, the solution is a 4-line sort.

**Pattern recognition:** "Assign increasing weights to items with ordering constraints, maximize weighted sum" → check if the ordering constraints align with a simple sort. If the constraint forces non-decreasing value order (as here), the exchange argument shows global sort is optimal and the constraint is automatically satisfied.
