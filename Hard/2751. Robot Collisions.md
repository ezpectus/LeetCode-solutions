# 2751. Robot Collisions
## O(n log n) — Sort by Position + Stack for R-Robots

---

## Problem Statement

- Robots move left (L) or right (R) on a line. Collisions happen when two robots meet.
- On collision: higher health survives with health-1; equal health → both removed.
- Return healths of survivors in **original order**.

---

## Core Idea — Process by Position, Stack for Rightward Robots

Only L-robot moving left can collide with R-robot moving right — specifically, an L-robot collides with the nearest R-robot to its left (the most recently seen unmatched R-robot).

**Algorithm**:
1. Sort robots by position (left to right).
2. Process each robot in position order:
   - If `R`: push onto stack (pending collision partner)
   - If `L`: fight the top of the stack (most recent unmatched R):
     - L wins → pop R, L health-1, continue fighting
     - R wins → L removed, R health-1
     - Tie → both removed, pop R
3. Remaining stack = R-robots that never met an L-robot → survivors.
4. Output in **original index order**.

---

## Full Optimal Implementation (C#)

```csharp
public class Solution {
    public IList<int> SurvivedRobotsHealths(int[] positions, int[] healths, string directions) {
        int n = positions.Length;
        int[] robots = Enumerable.Range(0, n).OrderBy(i => positions[i]).ToArray();
        var stack = new Stack<int>(); // original indices of R-robots
        var removed = new HashSet<int>();

        foreach (int i in robots) {
            if (directions[i] == 'R') {
                stack.Push(i);
            } else { // L robot
                while (stack.Count > 0) {
                    int top = stack.Peek();
                    if (healths[i] > healths[top]) {
                        removed.Add(top); stack.Pop();
                        healths[i]--;
                    } else if (healths[i] < healths[top]) {
                        removed.Add(i);
                        healths[top]--;
                        break;
                    } else { // equal
                        removed.Add(top); stack.Pop();
                        removed.Add(i);
                        break;
                    }
                }
            }
        }

        return Enumerable.Range(0, n)
            .Where(i => !removed.Contains(i))
            .Select(i => healths[i])
            .ToList();
    }
}
```

## Complexity

| **Metric** | **Value** | **Notes** |
|---|---|---|
| **Time** | **O(n log n)** | Sort + O(n) amortized stack ops (each robot pushed/popped once) |
| **Space** | **O(n)** | Stack + removed set |

## Why Stack Works

Robots sorted by position: an L-robot at position `p` can only collide with R-robots at positions < `p` (moving right toward it). The most recent unmatched R-robot (top of stack) is the first one the L-robot meets. If the L-robot survives, it then faces the next R-robot (new stack top).

This is the classic "asteroid collision" problem generalized — the stack efficiently maintains the active R-robots in positional order.

## Pitfalls & Edge Cases

- **Original order output**: sort to process, but output in original index order
- **L-robot surviving all R-robots in stack**: exits while loop naturally → not removed → survives
- **All same direction**: no collisions → all survive with original healths ✓

---

## Key Takeaway

"Robot collisions on a line" → **sort by position + stack for R-robots**:

```
R → push to stack
L → fight stack top repeatedly until L dies, R dies, or stack empty
```

Classic "asteroid collision" pattern. Stack maintains active R-robots in positional order; each collision is resolved in O(1) amortized.
