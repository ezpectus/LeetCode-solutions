## 2876. Count Visited Nodes in a Directed Graph

### Problem Statement
There is a functional directed graph consisting of $n$ nodes numbered from $0$ to $n - 1$ and exactly $n$ directed edges. You are given a 0-indexed array `edges` where `edges[i]` indicates a directed edge from node $i$ to node `edges[i]`.

If you start a traversal from node $i$ and follow the directed edges indefinitely, find the total number of **distinct** nodes visited before the traversal gets trapped in a cycle. Return an array containing this count for every node.

### Core Insight — Functional Graph Anatomy
Because each node has exactly **one** outgoing edge, this graph is classified as a **Functional Directed Graph** (or pseudo-tree forest). A crucial property of such graphs is that if you start from *any* node and walk along the edges, you will eventually enter a cycle. 

Every connected component in this graph consists of:
1. **A Cycle:** A group of nodes linked in a loop. Every node inside a cycle can reach all other nodes in that same cycle. Therefore, the answer for any node *inside* a cycle of length $L$ is exactly $L$.
2. **A Non-Cycle Path (Tree structures):** Nodes that branch into the cycle but are not part of it. If a node $u$ leads to a node $v$, and $v$ already has a known answer (either because it is in a cycle or its path was pre-calculated), the answer for $u$ is simply $\text{answer}[v] + 1$.

> **Architectural Constraint:** To solve this in linear time $\mathcal{O}(N)$, we can track our traversal using a step counter (timestamp) and a state tracker. This allows us to cleanly distinguish between discovering a brand-new cycle vs. hitting an already fully-computed path.

---

### Algorithm
1. Initialize an array `ans` of size $n$ filled with `0` to keep track of calculated visited counts.
2. Initialize an array `step` of size $n$ filled with `0` to act as local traversal timestamps.
3. Iterate through each node $i$ from $0$ to $n-1$:
   - If `ans[i] > 0`, this node is already fully resolved. Skip it.
   - If unresolved, start a traversal from $i$ using a temporary pointer `curr`, tracking nodes visited during this specific path in a list or stack.
   - Assign a monotonically increasing timestamp (`curr_step`) to `step[curr]`.
   - Move to the next node: `curr = edges[curr]`.
   - Loop until you hit a node `curr` where `ans[curr] > 0` **OR** `step[curr] > 0` (which implies a cycle closure within the current run).
4. **Determine State Closure:**
   - **Case 1: Hit a calculated node (`ans[curr] > 0`).** Every node collected in our temporary path receives an answer extending backward from `ans[curr]`.
   - **Case 2: Found a new cycle (`step[curr] > 0`).** The length of the cycle is calculated as `(current_timestamp) - step[curr] + 1`. Every node inside this cycle gets assigned this length. Any nodes trailing *before* the cycle get incremented sequentially backward from the cycle entry point.
5. Clear the localized `step` history for nodes processed in this path to prevent cross-contamination during subsequent loop cycles.

### Full Optimal Implementation (C#)
```csharp
using System;
using System.Collections.Generic;

public class Solution {
    public int[] CountVisitedNodes(IList<int> edges) {
        int n = edges.Count;
        int[] ans = new int[n];
        int[] step = new int[n]; // Tracks timestamps of the current traversal run

        for (int i = 0; i < n; i++) {
            if (ans[i] > 0) continue; // Already processed

            int curr = i;
            int currStep = 1;
            List<int> path = new List<int>();

            // Traverse until we hit an already calculated node or a node in the current path
            while (ans[curr] == 0 && step[curr] == 0) {
                step[curr] = currStep++;
                path.Add(curr);
                curr = edges[curr];
            }

            // Case 1: We hit a node that was already calculated in a previous run
            if (ans[curr] > 0) {
                int extra = ans[curr];
                // Fill backwards
                for (int j = path.Count - 1; j >= 0; j--) {
                    extra++;
                    ans[path[j]] = extra;
                }
            } 
            // Case 2: We detected a brand new cycle within the current run
            else if (step[curr] > 0) {
                int cycleLength = currStep - step[curr];
                
                // Find where the cycle starts in our recorded path list
                int cycleStartIdx = step[curr] - 1;

                // 1. Assign the cycle length to all nodes belonging to the loop
                for (int j = cycleStartIdx; j < path.Count; j++) {
                    ans[path[j]] = cycleLength;
                }

                // 2. Assign incrementing values to nodes leading up to the loop
                int extra = cycleLength;
                for (int j = cycleStartIdx - 1; j >= 0; j--) {
                    extra++;
                    ans[path[j]] = extra;
                }
            }

            // Reset timestamps for nodes processed in this run to avoid cross-contamination
            foreach (int node in path) {
                step[node] = 0;
            }
        }

        return ans;
    }
}
```
### Complexity
| Metric | Value | Notes |
|---|---|---|
| **Time** | $\mathcal{O}(N)$ | Each node is visited at most twice: once during the forward traversal loop, and once when unwinding the collection path array. |
| **Space** | $\mathcal{O}(N)$ | Storing the `ans` mapping, the `step` timestamps array, and the dynamic `path` lists requires memory that scales linearly with $N$. |

### Example Walkthrough
#### `edges = [1,2,0,0]`
* **$i = 0$:** Traversal starts.
  - `path = [0, 1, 2]`. Timestamps: `step[0]=1`, `step[1]=2`, `step[2]=3`.
  - Next node from 2 is `edges[2] = 0`. Since `step[0] = 1 > 0`, a cycle is found!
  - `cycleLength = 4 - 1 = 3`.
  - Cycle starts at index `1 - 1 = 0` in `path`. All nodes in `path[0...2]` get `ans = 3`.
  - `ans = [3, 3, 3, 0]`. Clean up `step`.
* **$i = 1, 2$:** Already have `ans > 0`. Skipped.
* **$i = 3$:** Traversal starts.
  - `path = [3]`. Timestamps: `step[3]=1`.
  - Next node from 3 is `edges[3] = 0`. `ans[0] = 3 > 0` (Hit calculated node).
  - Unwind backward: `ans[3] = ans[0] + 1 = 4`.
  - `ans = [3, 3, 3, 4]`. Clean up `step`.

**Total Output:** `[3, 3, 3, 4]`. ✓

### Pitfalls & Edge Cases
* **Timestamp Contamination:** Failing to reset the `step` tracker array back to `0` after completing a loop will corrupt subsequent traversal searches, mistaking old historical traces for an active loop collision.
* **Stack Overflow Risks:** Writing this logic using direct recursion can cause a stack overflow on large inputs ($N = 10^5$) with long linear tail streams. Implementing the walk via iterative loops with an explicit path collection list guarantees memory safety.

### Key Takeaway
Functional graphs break down cleanly into cycles and tails. Tracking traversal chains with simple local timestamps lets you calculate loop lengths instantly on collision, while unwinding paths backward handles tail lengths in complete constant-time operations.
