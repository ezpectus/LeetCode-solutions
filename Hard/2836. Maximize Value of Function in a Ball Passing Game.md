# 2836. Maximize Value of Function in a Ball Passing Game
## O(n log k) — Binary Lifting: Precompute 2^j Ancestors + Prefix Sums Along Jumps

---

## Problem Statement

Array `receiver` of length n (each player i passes to `receiver[i]`). Starting from any player i, make exactly k passes. The **score** is the sum of all player indices who touch the ball:
```
score(i, k) = i + receiver[i] + receiver²[i] + … + receiverᵏ[i]
```
That is, **k+1 terms** total (the starter plus k receivers). Return the maximum score over all starting players.

Constraints: n ≤ 10⁵, **k ≤ 10¹⁰**.

---

## Core Insight — Binary Lifting Splits Exponentially Large k into O(log k) Jumps

k ≤ 10¹⁰ makes direct simulation per starting player O(k) per query = O(n·k) total, which is ~10¹⁵ — far too slow.

**Binary lifting** precomputes tables for jump lengths that are powers of 2:
- `anc[j][i]` = the player reached after exactly **2ʲ passes** starting from player i.
- `val[j][i]` = the sum of the first **2ʲ player indices** along that path (i.e., indices of players 0, 1, …, 2ʲ−1 steps from i, **not** including the 2ʲ-th endpoint).

Any total of T = k+1 nodes can be decomposed into at most log₂T ≤ 34 power-of-2 blocks by reading its binary representation. Each block query takes O(1) after the O(n log k) build phase.

---

## Table Definitions (Precise)

**Base case (j = 0): one player visited**
```
anc[0][i] = receiver[i]    // node reached after 1 pass
val[0][i] = i              // sum of exactly the first node (i itself)
```

**Doubling recurrence (j → j+1):**
```
mid         = anc[j][i]             // midpoint: after 2^j passes
anc[j+1][i] = anc[j][mid]           // another 2^j passes from mid → total 2^(j+1) passes
val[j+1][i] = val[j][i] + val[j][mid]  // first half + second half sums
```

`val[j][i]` accumulates the first half (2ʲ nodes from i), then `val[j][mid]` the next 2ʲ nodes — together exactly 2^(j+1) nodes. ✓

---

## Query: Sum of k+1 Nodes From Starting Player i

Let `rem = k+1` (total players who touch the ball). Read rem's binary representation from highest to lowest bit:

```
cur = i; sum = 0
for j = LOG-1 downto 0:
  if bit j of rem is set:
    sum += val[j][cur]      // add the 2^j node-sum starting at cur
    cur  = anc[j][cur]      // advance cur past those 2^j nodes
```

After all bits, `sum` equals `score(i, k)` and `cur` is the player reached after k+1 passes (unused). ✓

---

## Algorithm

```
1. Set LOG = 34  (2^34 > 10^10 guarantees all k+1 ≤ 10^10+1 are covered).
2. Build anc[0..LOG-1] and val[0..LOG-1] via doubling.
3. Set rem = k+1.
4. For each starting player i = 0..n-1:
     query score(i, k) in O(LOG) using the bit-decomposition above.
5. Return max score.
```

---

## Full Optimal Implementation (C#)

```csharp
public class Solution {
    public long GetMaxFunctionValue(IList<int> receiver, long k) {
        int n = receiver.Count;
        const int LOG = 34;   // 2^34 > 10^10 ≥ k

        long[][] anc = new long[LOG][];
        long[][] val = new long[LOG][];
        for (int j = 0; j < LOG; j++) {
            anc[j] = new long[n];
            val[j] = new long[n];
        }

        // Base case: 1 pass / 1 node
        for (int i = 0; i < n; i++) {
            anc[0][i] = receiver[i];
            val[0][i] = i;              // contribution of node i alone
        }

        // Doubling
        for (int j = 1; j < LOG; j++) {
            for (int i = 0; i < n; i++) {
                long mid    = anc[j - 1][i];
                anc[j][i]   = anc[j - 1][mid];
                val[j][i]   = val[j - 1][i] + val[j - 1][mid];
            }
        }

        long ans = 0;
        long rem = k + 1;   // k+1 nodes total (starter + k receivers)

        for (int i = 0; i < n; i++) {
            long sum = 0, cur = i;
            for (int j = LOG - 1; j >= 0; j--) {
                if (((rem >> j) & 1L) == 1L) {
                    sum += val[j][cur];
                    cur  = anc[j][cur];
                }
            }
            ans = Math.Max(ans, sum);
        }
        return ans;
    }
}
```

## Interpretation of `val[j][i]` — Block Accumulation

`val[j][i]` stores the sum of nodes for a **block of exactly 2ʲ** consecutive players in the path starting at i. When we decompose rem=k+1 into binary and process bits high-to-low, we're partitioning the k+1-node path into non-overlapping, contiguous blocks of sizes 2^(j₁) > 2^(j₂) > … covering exactly all k+1 nodes.

```
Example: rem = 13 = 1101₂
Blocks: [2³=8 nodes] [2²=4 nodes] [2⁰=1 node]
          first 8      next 4       last 1
```

Each block's sum comes from one table lookup. Chain correctness: after block of size 2ʲ starting at `cur`, `anc[j][cur]` lands exactly where the next block starts. ✓

---

## Pitfalls & Edge Cases

- **LOG must accommodate rem = k+1, not just k:** For k=3, rem=4=2². We need bit 2 to be set in rem, which requires j=2 in the table — so LOG must be at least 3. Using `LOG=34` hardcoded covers all k ≤ 10¹⁰ since 2³⁴ > 10¹⁰. A common bug is setting LOG based on k alone and missing the final node. ✓

- **`val[0][i] = i` (not `receiver[i]`):** The base case covers "1 node visited = node i itself". The corresponding endpoint `anc[0][i] = receiver[i]` is where the *next* node starts. Mistakenly setting `val[0][i] = receiver[i]` would skip the starting player and count the first receiver twice. ✓

- **`cur` resets to `i` for each starting player:** The query loop for each `i` must start fresh with `cur = i`. Re-using `cur` across starting players produces garbage. ✓

- **`anc` values are `long`, not `int`:** Array indexing uses `long mid = anc[j-1][i]` as an index into `anc[j-1]`. In C#, array indexing with `long` casts cleanly. Storing as `int` is also fine (n ≤ 10⁵ fits in `int`), but consistency with `long` arrays avoids casting bugs.

- **Max possible sum:** (k+1) × (n−1) ≤ (10¹⁰+1) × (10⁵−1) ≈ 10¹⁵, comfortably within `long.MaxValue ≈ 9.2×10¹⁸`. ✓

- **Self-loops (`receiver[i] = i`):** Handled naturally. `anc[j][i] = i` for all j, and `val[j][i] = i × 2ʲ` (correct: the same player index repeated 2ʲ times). ✓

- **Cycles shorter than k:** Binary lifting works on any functional graph (exactly one outgoing edge per node), whether or not cycles exist. The doubling recurrence follows the actual path, cycling as needed. No cycle detection required. ✓

---

## Why Not Just Walk k Steps?

Naively simulating from each starting player: O(n) starts × O(k) steps = O(n·k) = O(10¹⁵). With k up to 10¹⁰, completely infeasible.

Binary lifting reduces each query to O(log k) ≈ 34 steps by "teleporting" across 2ʲ steps at once, exploiting the precomputed table. The O(n log k) build cost is amortized: build once, query n times.

---

## Key Takeaway

**Exponentially large traversal on a functional graph → binary lifting.** Precompute tables for powers-of-2 jump lengths, then decompose arbitrary jump length into binary and chain table lookups. The same technique is used for LCA (Lowest Common Ancestor) queries and other "jump k steps" problems.

**Pattern recognition:** "Walk exactly k steps on a graph from each possible start, sum/track quantities along the path, k is too large for simulation" → binary lifting with augmented tables (here: `val[j][i]` = sum of 2ʲ node indices). The only extra wrinkle beyond bare-minimum LCA lifting is the additive accumulator in `val`, built via the same doubling recurrence: `val[j][i] = val[j-1][i] + val[j-1][mid]`.
