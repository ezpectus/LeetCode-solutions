# 3298. Count Substrings That Can Be Rearranged to Contain a String II
## O(n + m) — Sliding Window with "Satisfied Characters" Counter

---

## Problem Statement

A substring of word1 is **valid** if it can be rearranged to have word2 as a prefix — equivalently, the substring must contain **at least as many** of each character as word2 does. Count all valid substrings.

---

## Core Insight #1 — Containment = Frequency Dominance

"Can be rearranged to have word2 as prefix" means: for every character c, `freq(c in substring) ≥ freq(c in word2)`. Character order doesn't matter — only counts.

This reduces to: count subarrays of word1 where the multiset of characters is a superset of word2's multiset.

---

## Core Insight #2 — Monotonicity Enables Sliding Window

If substring [l, r] is valid (contains all required characters), then [l-1, r] is also valid (adding one more character can't remove required characters). So for each fixed right endpoint r, the set of valid left endpoints is a **prefix** [0, threshold(r)], where threshold(r) = maximum left index such that [left, r] is still valid.

As r increases, threshold(r) is **non-decreasing** (the threshold can only move right). This enables a two-pointer approach.

**Count of valid substrings ending at r = threshold(r) + 1** (starting positions 0, 1, ..., threshold(r)).

---

## Core Insight #3 — Track Satisfaction with a Counter

Rather than checking all 26 characters every step:
- `need[c]` = frequency of character c in word2.
- `have[c]` = frequency of character c in current window.
- `satisfied` = number of distinct characters where `have[c] >= need[c]`.
- Window is valid iff `satisfied == totalNeeded` (where totalNeeded = number of distinct chars in word2).

When `have[c]` goes from `need[c]-1` to `need[c]`: `satisfied++`.
When `have[c]` drops from `need[c]` to `need[c]-1`: `satisfied--`.

---

## Sliding Window Algorithm

```
left = 0
For each right from 0 to n-1:
    Add word1[right] to window (update have[], maybe increment satisfied)
    
    // Shrink from left while window remains valid
    While satisfied == totalNeeded:
        If removing word1[left] would break validity: break
        Else: remove word1[left], advance left
    
    // If valid, positions 0..left are all valid starts for this right
    If satisfied == totalNeeded:
        count += left + 1
```

Each character enters and exits the window exactly once → O(n) total.

---

## Full Implementation (C#)

```csharp
public class Solution {
    public long ValidSubstringCount(string word1, string word2) {
        int[] need = new int[26];
        foreach (char c in word2) need[c - 'a']++;

        int totalNeeded = 0;
        for (int i = 0; i < 26; i++) if (need[i] > 0) totalNeeded++;

        int[] have = new int[26];
        int satisfied = 0, left = 0;
        long count = 0;

        for (int right = 0; right < word1.Length; right++) {
            int rc = word1[right] - 'a';
            have[rc]++;
            if (need[rc] > 0 && have[rc] == need[rc]) satisfied++;

            // Advance left as far as possible while window stays valid
            while (satisfied == totalNeeded) {
                int lc = word1[left] - 'a';
                // If removing word1[left] would drop have[lc] below need[lc]: stop
                if (need[lc] > 0 && have[lc] == need[lc]) break;
                have[lc]--;
                left++;
            }

            // All starting positions 0..left give valid substrings ending at right
            if (satisfied == totalNeeded)
                count += left + 1;
        }

        return count;
    }
}
```

## Complexity

| Metric | Value | Notes |
|---|---|---|
| Time | O(n + m) | n = len(word1), m = len(word2). Each char enters/exits window once. |
| Space | O(1) | Two 26-element arrays. No allocation proportional to n or m. |

This satisfies the problem's linear runtime and reduced memory requirements.

## Example Walkthrough

### Example 1 — `word1="bcca"`, `word2="abc"`

need: a=1, b=1, c=1. totalNeeded=3.

| r | char | have | satisfied | left | while action | count |
|---|---|---|---|---|---|---|
| 0 | b | b=1 | 1 | 0 | loop not entered | 0 |
| 1 | c | c=1 | 2 | 0 | not entered | 0 |
| 2 | c | c=2 | 2 | 0 | not entered | 0 |
| 3 | a | a=1 | 3 | 0 | left=0,lc=b: need[b]=1, have[b]=1 → **break** | 0+1=**1** |

**Output: 1** ✓

---

### Example 2 — `word1="abcabc"`, `word2="abc"`

need: a=1, b=1, c=1. totalNeeded=3.

| r | char | satisfied | while (shrink) | left after | count |
|---|---|---|---|---|---|
| 0 | a | 1 | — | 0 | 0 |
| 1 | b | 2 | — | 0 | 0 |
| 2 | c | 3 | lc=a: need=1,have=1 → break | 0 | +1=1 |
| 3 | a | 3 | lc=a: have[a]=2→remove→have[a]=1,left=1; lc=b: break | 1 | +2=3 |
| 4 | b | 3 | lc=b: have[b]=2→remove→have[b]=1,left=2; lc=c: break | 2 | +3=6 |
| 5 | c | 3 | lc=c: have[c]=2→remove→have[c]=1,left=3; lc=a: break | 3 | +4=10 |

**Output: 10** ✓

---

### Example 3 — `word1="abcabc"`, `word2="aaabc"`

need: a=3, b=1, c=1. totalNeeded=3.

word1 has only 2 a's total. `have[a]` never reaches `need[a]=3`, so `satisfied` never reaches 3. count remains 0.

**Output: 0** ✓

## Pitfalls & Edge Cases

- **`satisfied` is per-character, not per-count.** When `have[c]` goes from `need[c]-1` to `need[c]`, `satisfied++`. When `have[c]` exceeds `need[c]`, no further increment (we're already satisfied for c). This correctly handles characters with `need[c] > 1`.

- **`need[c] == 0` characters:** They don't contribute to `satisfied` and can always be freely added/removed from the window without affecting validity. In the while loop, `need[lc] == 0` means the break condition is false → we freely shrink.

- **Word2 has a character not in word1:** `satisfied` can never reach `totalNeeded`. count = 0. ✓

- **Entire word1 has just enough of each character:** threshold stays 0 for most rights, giving small count. Handled correctly by the break condition.

- **Memory:** Only two fixed-size `int[26]` arrays. No dynamic allocation. Satisfies reduced memory limit. ✓

---

## Key Takeaway

This is the **"minimum window superset count"** sliding window pattern. For each right endpoint, find the maximum left such that the window still satisfies all frequency requirements. The "satisfied characters" counter gives O(1) validity checks per left/right step.

**Pattern recognition:** "Count subarrays where frequency multiset dominates a target multiset" → sliding window with satisfied-count, shrink from left while valid, add `left+1` to count for each valid right.
