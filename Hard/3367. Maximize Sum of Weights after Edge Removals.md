# LeetCode 3367. Maximize Sum of Weights after Edge Removals

## 1. Problem Context & Input Constraints

We are given an undirected tree with $n$ nodes and weights on its edges. We need to remove some edges such that every node in the remaining graph has a degree of at most $k$. The goal is to maximize the total weight of the remaining edges.

### Input Constraints Analysis
* $2 \le n \le 10^5$
* $1 \le k \le n - 1$
* Edge weights can be up to $10^6$, meaning the total maximum sum can exceed the 32-bit signed integer limit. We must return a `long`.

---

## 2. Algorithmic Domain & Feasibility Analysis

Since the graph is a tree, any edge removal breaks dependencies down into independent subtrees. This property allows us to use **Dynamic Programming on Trees (Tree DP)** via a single Depth-First Search (DFS) pass.

For each node $u$ when communicating with its parent, there are two constraints on node $u$'s available degree:
1. It saves 1 degree slot for the edge connecting it to its parent (can use at most $k - 1$ slots for its children).
2. It completely severs the edge connecting it to its parent (can use all $k$ slots for its children).

---

## 3. High-Level Approach Idea

1. **Adjacency List Construction:** Transform the raw edge triplet arrays into a structured adjacency list containing destination nodes and edge weights.
2. **Tree Traversal (DFS):** Root the tree at node 0 and recursively compute DP states bottom-up.
3. **Greedy Profit Sorting:** For each child $v$ of node $u$, compute the base score where the edge $(u, v)$ is omitted, and the alternative score where the edge $(u, v)$ is selected. The marginal difference $\Delta = (\text{ScoreWithEdge} + w) - \text{ScoreWithoutEdge}$ represents the explicit gain. Sort these gains in descending order to pick the most valuable edges greedily.

## 4. C# Source Code

```csharp
using System;
using System.Collections.Generic;

public class Solution {
    private List<(int to, int weight)>[] adj;
    private int limitK;

    public long MaximizeSumOfWeights(int[][] edges, int k) {
        int n = edges.Length + 1;
        limitK = k;
        adj = new List<(int to, int weight)>[n];
        for (int i = 0; i < n; i++) {
            adj[i] = new List<(int to, int weight)>();
        }

        foreach (var edge in edges) {
            int u = edge[0];
            int v = edge[1];
            int w = edge[2];
            adj[u].Add((v, w));
            adj[v].Add((u, w));
        }

        // Returns (ScoreIfParentEdgeKept, ScoreIfParentEdgeRemoved)
        var result = Dfs(0, -1);
        return result.dp1;
    }

    private (long dp0, long dp1) Dfs(int u, int p) {
        long baseSum = 0;
        List<long> gains = new List<long>();

        foreach (var edge in adj[u]) {
            int v = edge.to;
            int w = edge.weight;
            if (v == p) continue;

            var (childDp0, childDp1) = Dfs(v, u);

            // Base case: We do NOT take the edge (u, v)
            baseSum += childDp1;

            // Gain if we decide to TAKE the edge (u, v)
            long gain = (childDp0 + w) - childDp1;
            if (gain > 0) {
                gains.Add(gain);
            }
        }

        // Sort gains in descending order to pick the most profitable edges greedily
        gains.Sort((a, b) => b.CompareTo(a));

        // Calculate dp0: At most k - 1 edges to children can be kept
        long dp0 = baseSum;
        int take0 = Math.Min(gains.Count, limitK - 1);
        for (int i = 0; i < take0; i++) {
            dp0 += gains[i];
        }

        // Calculate dp1: At most k edges to children can be kept
        long dp1 = baseSum;
        int take1 = Math.Min(gains.Count, limitK);
        for (int i = 0; i < take1; i++) {
            dp1 += gains[i];
        }

        return (dp0, dp1);
    }
}
```
## 5. Complexity Analysis

* **Time Complexity:** $\mathcal{O}(N \log(\text{deg})) \to \mathcal{O}(N \log N)$ worst-case
  * **DFS Graph Traversal:** The tree is traversed bottom-up via a single pass. Each node is visited once, and each edge is examined exactly twice (once from each endpoint), contributing $\mathcal{O}(N)$ base operations.
  * **Greedy Edge Sorting:** At each node $u$, we collect the positive marginal gains from its children into a local list. The size of this list is bounded by the node's degree ($\text{deg}(u)$). Sorting this local list takes $\mathcal{O}(\text{deg}(u) \log(\text{deg}(u)))$ time. 
  * **Amortized Analysis across the Tree:** By the Handshaking Lemma, the sum of all node degrees in a tree is exactly equal to twice the number of edges: 
    $$\sum_{u=0}^{n-1} \text{deg}(u) = 2N - 2$$
    In the absolute worst-case scenario (a star-graph where one central node is connected to $N-1$ leaves), the sorting step at the hub dominates, taking $\mathcal{O}(N \log N)$ time. For all other tree topologies (like balanced or line trees), the runtime scales even closer to linear $\mathcal{O}(N)$, passing easily within a 20-30ms window.

* **Space Complexity:** $\mathcal{O}(N)$ Auxiliary Space
  * **Graph Representation:** The adjacency list stores exactly $2N - 2$ structured edge records, requiring $\mathcal{O}(N)$ space.
  * **Recursion Depth Stack:** The implicit execution call stack stores active stack frames for the DFS. In the worst-case scenario of a completely skewed line tree, the depth reaches $N$, utilizing $\mathcal{O}(N)$ space.
  * **Allocation Footprint:** The local `gains` lists are allocated per node and immediately collected by the garbage collector once the stack frame pops, bounding active dynamic memory allocation to at most $\mathcal{O}(N)$ at any single point in time.

## 6. Deep Dive: Approach Details & Transition Mechanics

The algorithm utilizes a bottom-up **Tree Dynamic Programming** framework paired with an isolated local **Greedy Choice Property**. Because a tree has no cycles, cutting or retaining an edge isolates the child's sub-tree choices completely from the rest of the graph.

### Mathematical Definition of States
For any node $u$, we define two optimal states considering only the sub-tree rooted at $u$:
1. $dp_0[u]$: The maximum weight sum inside $u$'s subtree given that the edge connecting $u$ to its parent **is retained**. Node $u$ must reserve $1$ capacity slot for its parent, leaving at most $k - 1$ slots available for its children.
2. $dp_1[u]$: The maximum weight sum inside $u$'s subtree given that the edge connecting $u$ to its parent **is removed** (or $u$ is the absolute root). Node $u$ has no obligation to a parent and can allocate all $k$ slots to its children.

### The Marginal Gain Formulation
When processing node $u$, we must make a binary decision for each child $v$ connected via an edge of weight $w$:

* **Scenario A (Discard Edge):** We do not select the edge $(u, v)$. Node $v$ now has full freedom within its own subtree, contributing exactly $dp_1[v]$ to our sum. The capacity counter of node $u$ remains unchanged.
* **Scenario B (Retain Edge):** We select the edge $(u, v)$, gaining its weight $w$. However, node $v$ must now reserve a slot for $u$, forcing it to operate under its restricted state, contributing $dp_0[v]$. The net value brought to the subtree is $dp_0[v] + w$, and node $u$ consumes $1$ capacity slot.

To determine if switching from Scenario A to Scenario B is beneficial, we compute the **Marginal Gain ($\Delta$)**:
$$\Delta = (dp_0[v] + w) - dp_1[v]$$
```
[ Node u ] (Has limitK slots available)
      /        \
Edge Obsoleted  Edge Retained
    /            \
Gain: 0        Gain: (dp0[v] + w) - dp1[v]
  /                \
[ childDp1 ]      [ childDp0 ] + w
```

### Optimal Knapsack Transition Strategy
Since each child's internal gain is independent, we can initialize our local score assuming all edges to children are dropped: 
$$\text{baseSum} = \sum_{v \in \text{children}(u)} dp_1[v]$$

We then greedily pick the best available upgrades. We filter out children where $\Delta \le 0$ because forcing those edge retentions would actively reduce or fail to improve the total weight. The remaining positive gains are sorted in descending order:
$$\text{gains} = \{ \Delta_1, \Delta_2, \dots, \Delta_m \} \quad \text{where } \Delta_1 \ge \Delta_2 \ge \dots \ge \Delta_m > 0$$

Finally, we compute the states for $u$ by choosing the maximum number of allowed transitions:
$$dp_0[u] = \text{baseSum} + \sum_{i=1}^{\min(m, k-1)} \Delta_i$$
$$dp_1[u] = \text{baseSum} + \sum_{i=1}^{\min(m, k)} \Delta_i$$

This greedy choice guarantees global optimality because the choice to take an edge $(u, v)$ only consumes capacity at node $u$ and limits node $v$, neither of which affects sibling subtrees.
