# 2763. Sum of Imbalance Numbers of All Subarrays

## O(N²) — Incremental Subarray Tracking with Contribution Analysis

### Problem Statement
The **imbalance number** of a 0-indexed integer array `arr` is defined as the number of indices in `sarr = sorted(arr)` such that:
* `0 <= i < sarr.length - 1`
* `sarr[i+1] - sarr[i] > 1`

Given a 0-indexed integer array `nums`, return the sum of imbalance numbers of all its subarrays.

A **subarray** is a contiguous non-empty sequence of elements within an array.

---

### Core Idea — Step-by-Step Imbalance Contribution
Generating and sorting every single subarray independently would result in an inefficient $O(N^3 \log N)$ execution loop. Instead, we can fix a starting index `i` and expand the subarray rightward by moving an index `j`. By maintaining the state of the current subarray, we can compute the change in the imbalance number in $O(1)$ constant time per step.

An imbalance occurs whenever there is a "gap" larger than $1$ between two adjacent elements in the sorted representation of our active subset. When a new element `x = nums[j]` is introduced into a sorted set:

1. **Initial Contribution:** Adding `x` creates a new element in the subset. By default, unless it bridges an existing gap, it adds $+1$ to the total imbalance count because it creates a potential new gap with whatever element was previously above it.
2. **Left Neighbor Bridge (`x - 1`):** If `x - 1` is already present in the current subarray, the gap between `x - 1` and `x` is exactly $1$. Thus, the initial penalty is avoided (we subtract $1$).
3. **Right Neighbor Bridge (`x + 1`):** If `x + 1` is already present in the current subarray, `x` successfully plugs the gap that previously existed below `x + 1`. Thus, we subtract an additional $1$.

By tracking the presence of elements using a frequency array or a boolean lookup table (since values are bounded by $1 \le nums[i] \le N$), we can dynamically update a running `currentImbalance` counter for the current window `nums[i...j]`.

---

### Full Optimal Implementation (C#)

```csharp
using System;

public class Solution {
    public int SumImbalanceNumbers(int[] nums) {
        int n = nums.Length;
        int totalImbalanceSum = 0;
        
        // Since 1 <= nums[i] <= n, we can use a fast lookup array instead of a HashSet
        bool[] visited = new bool[n + 2];
        
        // Step 1: Fix the left bound of the subarray
        for (int i = 0; i < n; i++) {
            // Reset state for the new window tracking
            Array.Fill(visited, false);
            
            int currentImbalance = 0;
            visited[nums[i]] = true;
            
            // Step 2: Expand the right bound of the subarray
            for (int j = i + 1; j < n; j++) {
                int x = nums[j];
                
                if (!visited[x]) {
                    // Introduce the element: temporarily assume it introduces a new gap
                    currentImbalance++;
                    
                    // Check if it bridges a gap with an immediate smaller neighbor
                    if (visited[x - 1]) {
                        currentImbalance--;
                    }
                    
                    // Check if it bridges a gap with an immediate larger neighbor
                    if (visited[x + 1]) {
                        currentImbalance--;
                    }
                    
                    visited[x] = true;
                }
                
                // Step 3: Accumulate the running metrics into the absolute total
                totalImbalanceSum += currentImbalance;
            }
        }
        
        return totalImbalanceSum;
    }
}
```

### Complexity

| Metric | Value | Notes |
| :--- | :--- | :--- |
| **Time Complexity** | $O(N^2)$ | Driven by the nested dual loops over the array bounds. The inner block executes in absolute $O(1)$ constant time because element insertion and neighbor checks utilize raw array indexing. |
| **Space Complexity** | $O(N)$ | The boolean `visited` lookup tracker scales linearly up to $N + 2$ flags, which easily conforms to memory limitations. |

---

### Pitfalls & Edge Cases

* **Duplicate Values Within a Subarray:** If an element `x` is already present in the current sliding window (`visited[x] == true`), adding it again changes absolutely nothing in the sorted array structure (`sorted(arr)`). No new gaps are created, and no existing gaps are filled. Ensure the logic bypasses modifications when `visited[x]` is already active.
* **Out of Bounds Index Shifts:** When checking `x - 1` and `x + 1`, make sure the container tracking allocations can accommodate boundary values safely. Initializing the lookup space with a padding size of `n + 2` completely avoids index exceptions.

---

### Key Takeaway

> When tasked with aggregating a localized permutation attribute (like sorted interval disparities) across all possible nested windows, evaluate if expanding the tracking sequence dynamically allows you to offset the new element's structural cost by looking up its immediate numeric neighbors.

* **Pattern recognition:** "Calculate sorted array characteristics across all continuous ranges" -> Incremental Expansion + Local Neighbor Contribution Analysis.
