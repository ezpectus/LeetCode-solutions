# 3739. Count Subarrays With Majority Element II
## O(n log n) — ±1 Transform + Prefix Sums + Fenwick Tree Pair Count

---

## Problem Statement

Array `nums`, integer `target`. Count subarrays where `target` is the **majority element** — appears **strictly more than half** the time. Formally, for a subarray of length `L`, count of `target` must satisfy `count > L/2`.

---

## Core Insight — Majority ↔ Positive Prefix-Sum Subarray

**Transform every element:**
- `+1` if `nums[i] == target`
- `−1` otherwise

**Claim:** `target` is the majority of subarray `[l, r]`
↔ `count_target > count_other`
↔ `(transformed sum of [l,r]) > 0`
↔ `prefix[r+1] − prefix[l] > 0`
↔ **`prefix[l] < prefix[r+1]`**

So the answer = **number of pairs (i, j) with i < j and prefix[i] < prefix[j]**.

This is exactly counting "non-inversions" (pairs in strictly ascending order) in the prefix sum array — a classic Fenwick Tree problem.

---

## Why Majority ↔ Positive Sum (Detailed)

Let `c` = count of `target` in `[l, r]`, `k` = count of non-target. Subarray length = `c + k`.

Majority requires: `c > (c + k) / 2` → `2c > c + k` → `c > k` → `c − k > 0`.

The transformed sum equals exactly `c − k`, since each target contributes +1 and each non-target contributes −1. ✓

---

## From Positive Sum to Counting Prefix Pairs

Let `prefix[0] = 0`, `prefix[i] = prefix[i−1] + transform[i−1]`.

Transformed sum of `[l, r]` = `prefix[r+1] − prefix[l]`.

We need this > 0, so we need `prefix[l] < prefix[r+1]`, i.e., among all pairs `(i, j)` with `i < j` in the prefix array, count those where `prefix[i] < prefix[j]`.

**How to count efficiently:** Sweep `j` left to right. For each new `prefix[j]`, count how many previously inserted prefix values are **strictly less than** `prefix[j]` — i.e., `≤ prefix[j] − 1`. A Fenwick Tree (BIT) answers this in O(log n) per query.

---

## Algorithm

**Key bounds:** `prefix[i] ∈ [−n, n]` (changes by ±1 each step). Shift by `n` (offset) to map into `[0, 2n]` for the BIT.

1. Initialize BIT of size `2n + 1`, `prefix = 0`, `ans = 0`.
2. Insert `prefix[0] = 0` (shifted: `0 + n`).
3. For each element `nums[i]`:
   a. `prefix += (nums[i] == target) ? 1 : -1`
   b. Query BIT for count of values ≤ `prefix − 1 + n` (all previous prefixes strictly less than current). Add to `ans`.
   c. Insert `prefix + n` into BIT.
4. Return `ans`.

---

## Full Optimal Implementation (C++)

```cpp
class Solution {
public:
    long long countMajoritySubarrays(vector<int>& nums, int target) {
        int n = nums.size();
        int offset = n;            // prefix ∈ [-n, n] → shift to [0, 2n]
        int sz = 2 * n + 1;
        vector<int> bit(sz + 1, 0);

        // BIT: 1-indexed, supports point update and prefix-sum query
        auto update = [&](int v) {
            for (v++; v <= sz; v += v & -v) bit[v]++;
        };

        auto query = [&](int v) -> long long { // count of inserted values ≤ v
            long long res = 0;
            for (v++; v > 0; v -= v & -v) res += bit[v];
            return res;
        };

        long long ans = 0;
        int prefix = 0;
        update(prefix + offset);       // insert prefix[0] = 0

        for (int i = 0; i < n; i++) {
            prefix += (nums[i] == target) ? 1 : -1;

            // Count previous prefixes strictly less than current (i.e., ≤ prefix−1)
            int q = prefix - 1 + offset;
            if (q >= 0) ans += query(q);

            update(prefix + offset);   // insert prefix[i+1]
        }

        return ans;
    }
};
```

## Complexity

| Metric | Value | Notes |
|---|---|---|
| Time | O(n log n) | n+1 BIT updates + n BIT queries, each O(log n) |
| Space | O(n) | BIT of size 2n+1; prefix values ∈ [−n, n] |

For n = 10⁵: ~10⁵ × 17 ≈ 1.7M operations. Trivially within limits.

## Example Walkthroughs

### Example 1 — `nums = [1,2,2,3]`, `target = 2`, Output: 5

**Transform:** `[−1, +1, +1, −1]`
**Prefix array:** `[0, −1, 0, 1, 0]`
**Offset** = 4. Shifted prefix: `[4, 3, 4, 5, 4]`

**Pair verification** (i < j, prefix[i] < prefix[j]):

| Pair (i,j) | prefix[i] | prefix[j] | Valid? | Subarray |
|---|---|---|---|---|
| (0,3) | 0 | 1 | ✓ | [0..2] = [1,2,2] |
| (1,2) | −1 | 0 | ✓ | [1..1] = [2] |
| (1,3) | −1 | 1 | ✓ | [1..2] = [2,2] |
| (1,4) | −1 | 0 | ✓ | [1..3] = [2,2,3] |
| (2,3) | 0 | 1 | ✓ | [2..2] = [2] |

All other pairs have prefix[i] ≥ prefix[j]. Total = **5** ✓

**BIT trace:**

| Step | Action | ans |
|---|---|---|
| init | insert shifted 0+4=**4** | 0 |
| i=0, prefix=−1 | query(−1+4−1=**2**)=0; insert **3** | 0 |
| i=1, prefix=0 | query(0+4−1=**3**)=1; insert **4** | 1 |
| i=2, prefix=1 | query(1+4−1=**4**)=3; insert **5** | 4 |
| i=3, prefix=0 | query(0+4−1=**3**)=1; insert **4** | **5** |

At i=2, query(4) finds 3 values ≤ 4: `{3(→prefix −1), 4(→prefix 0), 4(→prefix 0)}` = the pairs (1,2), (0,2)→wait let me recheck.

At i=2 (prefix becomes 1): BIT currently contains shifted values `{4, 3, 4}` = original prefixes `{0, −1, 0}`. Query ≤ 4 returns 3. Those 3 left endpoints give subarrays `[0..2], [1..2], [2..2]`. ✓

### Example 2 — `nums = [1,1,1,1]`, `target = 1`, Output: 10

**Transform:** `[+1, +1, +1, +1]`
**Prefix array:** `[0, 1, 2, 3, 4]` — **strictly increasing**.

Every pair (i, j) with i < j satisfies prefix[i] < prefix[j].
Total pairs = C(5, 2) = **10** ✓

BIT accumulates: query answers are 1, 2, 3, 4 → total = 10. ✓

### Example 3 — `nums = [1,2,3]`, `target = 4`, Output: 0

**Transform:** `[−1, −1, −1]`
**Prefix array:** `[0, −1, −2, −3]` — **strictly decreasing**.

No pair (i, j) with i < j has prefix[i] < prefix[j]. Total = **0** ✓

## Pitfalls & Edge Cases

- **Strict majority, not ≥ majority:** `count > L/2` is strict. The prefix sum must be **strictly positive**, so we query for values **≤ prefix − 1**, not ≤ prefix. Off-by-one here gives wrong answers (would count subarrays where target appears exactly half the time, which are invalid for odd-length subarrays but harmless for even — still wrong).

- **Guard `q >= 0` before querying:** When `prefix = −n` (all elements so far are non-target), `q = prefix − 1 + offset = −n − 1 + n = −1`. The BIT has no index −1; skip the query. Without this guard: undefined behavior or wrong index.

- **Prefix array has n+1 elements (not n):** `prefix[0] = 0` must be inserted before the loop. Forgetting this misses all pairs where the left endpoint is the virtual empty prefix — i.e., all subarrays starting at index 0.

- **BIT is 1-indexed:** The `v++` inside update/query converts 0-indexed values to 1-indexed BIT positions. The array must be sized `sz + 1` = `2n + 2` to avoid out-of-bounds access at position `sz` (maximum shifted value = `n + n = 2n`, 1-indexed = `2n + 1 = sz`).

- **`target` never appears:** Prefix is strictly decreasing (all −1s). No pairs qualify, returns 0 correctly. ✓

- **All elements equal `target`:** Prefix is strictly increasing. All C(n+1, 2) pairs qualify. BIT correctly accumulates 0 + 1 + 2 + ... + n = n(n+1)/2. ✓

- **`long long` for the answer:** Maximum count is C(n+1, 2) ≈ (10⁵)²/2 = 5×10⁹, which overflows `int`. Return type and `ans` must be `long long`. Individual BIT queries can also accumulate up to n ≈ 10⁵ per query, so the query return type should be `long long` too.

## Connection to Classic Problems

| Problem | Transform | What to Count |
|---|---|---|
| Count subarrays sum = k | Raw values | Pairs prefix[i] = prefix[j] − k → hashmap |
| Count inversions | Indices | Pairs (i < j) with a[i] > a[j] → merge sort / BIT |
| **This problem** | ±1 | Pairs (i < j) with prefix[i] < prefix[j] → BIT |

This problem is **counting inversions in reverse** on the prefix array. Counting inversions gives pairs where a[i] > a[j]; here we want pairs where prefix[i] < prefix[j], i.e., **non-inversions** (excluding ties). Same O(n log n) BIT structure, different query direction.

---

## Key Takeaway

**"Majority element in subarrays"** → transform to ±1 → count subarrays with positive sum → count prefix pairs with prefix[i] < prefix[j] → BIT.

The ±1 transform converts a frequency dominance condition into a standard sum condition, reducing an element-counting problem to a pair-counting problem solvable in O(n log n).

**Pattern recognition:** Any problem of the form "element X appears more than element Y in subarrays" → ±1 transform (X→+1, Y→−1) → prefix sum pairs → BIT / merge sort.
