# 2509. Cycle Length Queries in a Tree
## O(m log n) — Climb to LCA via Parent Halving

---

## Problem Statement

- Complete binary tree (implicit, not built): node `val`'s parent is `val/2`, children are `2·val` and `2·val+1`.
- For each query `[a,b]`: adding edge (a,b) creates exactly one cycle (since the tree is connected and acyclic, one extra edge creates exactly one cycle). Return its length (number of edges).

---

## Core Idea — Cycle Length = Tree Distance + 1

Since the tree already connects `a` and `b` via a unique path (through their LCA), adding edge `(a,b)` creates a cycle consisting of that tree path PLUS the new edge. So:

```
cycleLength = treeDistance(a, b) + 1
```

**No need to build the tree or compute explicit depths/LCA via binary lifting** — since the parent of `val` is always `val / 2` (integer division), we can walk up by repeatedly halving.

**Algorithm per query:**
1. While `a != b`: if `a > b`, move `a` to its parent (`a /= 2`), else move `b` to its parent (`b /= 2`). Count each move as a step.
2. When `a == b`, all steps taken = the tree distance.
3. Answer = distance + 1.

This works because moving the larger value up always makes progress toward the LCA (since `val/2 < val`, and the node with the larger value is always at least as deep when values are compared this way — more precisely, repeatedly halving the larger of the two converges to their LCA).

---

## Full Optimal Implementation (C++)

```cpp
class Solution {
public:
    vector<int> cycleLengthQueries(int n, vector<vector<int>>& queries) {
        vector<int> ans;
        ans.reserve(queries.size());

        for (auto& q : queries) {
            long long a = q[0], b = q[1];
            int steps = 0;

            while (a != b) {
                if (a > b) a /= 2;
                else       b /= 2;
                steps++;
            }

            ans.push_back(steps + 1);
        }

        return ans;
    }
};
```

| Metric | Value | Notes |

| Space Complexity | O(m) | Output array |

m ≤ 10⁵, n ≤ 30 → ≤ 3×10⁶ total operations.

---

## Why "Move the Larger Value's Parent" Always Converges Correctly

In a binary heap-style numbering, a node's depth equals `floor(log2(val))`. If `a > b`, then either `a` is deeper than `b`, or they're at the same depth but `a` is to the right of `b` in the tree — in both cases, moving `a` to its parent (`a/2`) is a valid step toward the LCA without overshooting (since the LCA's value is always ≤ min(a,b) in this numbering... more precisely, the path to LCA only ever requires moving the "currently larger" node up, since the LCA is found by aligning depths first via the larger value's ancestors, then moving both up together once depths match, which is naturally handled since once `a==b` we stop — they've met at the LCA exactly).

## Example Walkthrough

**Example 1:** `n=3, queries=[[5,3],[4,7],[2,3]]`

**Query [5,3]:** a=5,b=3. 5>3→a=2,steps=1. Now a=2,b=3: 3>2→b=1,steps=2. a=2,b=1: 2>1→a=1,steps=3. a==b=1. distance=3. **answer=4** ✓

**Query [4,7]:** a=4,b=7. 7>4→b=3,steps=1. a=4,b=3: 4>3→a=2,steps=2. a=2,b=3: 3>2→b=1,steps=3. a=2,b=1: 2>1→a=1,steps=4. a==b=1. distance=4. **answer=5** ✓

**Query [2,3]:** a=2,b=3. 3>2→b=1,steps=1. a=2,b=1: 2>1→a=1,steps=2. a==b=1. distance=2. **answer=3** ✓

**Output:** [4,5,3] ✓ matches expected.

---

**Example 2:** `n=2, queries=[[1,2]]`

a=1,b=2. 2>1→b=1,steps=1. a==b=1. distance=1. **answer=2** ✓

## Pitfalls & Edge Cases

- **a or b could already be ancestor of the other** (e.g., a=1, b=4): loop still works — repeatedly halving the larger converges directly without unnecessary detours. ✓

- **Use of `long long` (or just `int`) for a,b** — values up to `2³⁰-1`, fits comfortably in `int`, but using a wider type is a safe habit. ✓

- **m up to 10⁵ queries, each O(n)=O(30) steps** — trivially fast, no need for binary lifting/LCA preprocessing despite this being a "Hard" problem; the implicit tree structure makes per-query O(log(maxVal)) trivial. ✓

---

## Key Takeaway

When the tree is an **implicit complete binary tree** (heap-indexed), there's no need for general LCA techniques (binary lifting, Euler tour, etc.) — the parent of any node is just `val/2`. Finding the LCA of two nodes reduces to the classic "two pointers converging by repeatedly halving the larger" trick, identical to finding the common ancestor in a binary heap array.

**Pattern recognition:** "Tree where parent(v) = v/2" → walk up via integer division; no tree construction or preprocessing needed.
