# LeetCode 3351. Sum of Good Subsequences

## Method Explanation

The problem asks us to find the sum of all "good" subsequences, where consecutive elements have an absolute difference of exactly 1. Single-element subsequences are also considered good.

Instead of a naive exponential sub-sequence generation, we can solve this efficiently using Dynamic Programming in a single pass. Since the values of $nums[i]$ are bounded by $10^5$, we can use linear direct-access arrays instead of hash maps for $\mathcal{O}(1)$ lookups.

### 1. DP State Tracking
We maintain two state tracking arrays updated incrementally:
* `count[v]`: The total number of valid good subsequences currently ending with the value $v$.
* `sum[v]`: The aggregate sum of all elements across all good subsequences currently ending with the value $v$.

### 2. Transition Logic
When processing an element $x = nums[i]$, it can either start a new subsequence of length 1 or append to any existing valid subsequences ending in $x-1$ or $x+1$.

The updates for the value $x$ are formulated as:
$$\Delta \text{count} = 1 + \text{count}[x-1] + \text{count}[x+1]$$
$$\Delta \text{sum} = x + (\text{sum}[x-1] + \text{count}[x-1] \cdot x) + (\text{sum}[x+1] + \text{count}[x+1] \cdot x)$$

We increment `count[x]` and `sum[x]` by these deltas, and concurrently add $\Delta \text{sum}$ directly to our global accumulation answer modulo $10^9 + 7$.

## C++ Source Code

```cpp
#include <vector>
#include <algorithm>

using namespace std;

class Solution {
public:
    int sumOfGoodSubsequences(vector<int>& nums) {
        const int MOD = 1e9 + 7;
        
        // Find maximum value to optimize memory allocation bounds
        int max_val = 0;
        for (int x : nums) {
            max_val = max(max_val, x);
        }
        
        // Storage for counts and sums of subsequences ending at a specific value
        vector<long long> count(max_val + 2, 0);
        vector<long long> sum(max_val + 2, 0);
        
        long long total_good_sum = 0;
        
        for (int x : nums) {
            // Base case: current item forms a valid single-element sequence
            long long current_count = 1;
            long long current_sum = x;
            
            // Append to sequences ending in x - 1
            if (x > 0) {
                current_count = (current_count + count[x - 1]) % MOD;
                current_sum = (current_sum + sum[x - 1] + count[x - 1] * x) % MOD;
            }
            
            // Append to sequences ending in x + 1
            if (x < max_val) {
                current_count = (current_count + count[x + 1]) % MOD;
                current_sum = (current_sum + sum[x + 1] + count[x + 1] * x) % MOD;
            }
            
            // Commit changes to state tables
            count[x] = (count[x] + current_count) % MOD;
            sum[x] = (sum[x] + current_sum) % MOD;
            
            // Accumulate into the final global sum
            total_good_sum = (total_good_sum + current_sum) % MOD;
        }
        
        return total_good_sum;
    }
};
```

## Complexity Analysis

* **Time Complexity:** $\mathcal{O}(N + M)$
  * **Pre-computation Max Scan:** We scan the entire input array `nums` of length $N$ once at the start to determine the maximum element $M$. This initial reduction operation scales strictly in linear time, $\mathcal{O}(N)$.
  * **Single-Pass DP Evaluation:** We loop through the elements of the input array exactly once. For each element $x$, the transitions depend entirely on looking up states at indices $x - 1$ and $x + 1$. Because we use flat, direct-access arrays instead of dynamic associative structures like `std::unordered_map`, every single read and write lookup takes guaranteed $\mathcal{O}(1)$ constant time.
  * **Arithmetic Overhead:** The arithmetic operations at each step involve basic additions and multiplications under a modulo constraint, which executes in constant CPU instructions.
  * **Total Time Bound:** With $N$ outer elements to process and a one-time allocation cost scaling up to $M$, the total computational time complexity is optimal at $\mathcal{O}(N + M)$. Given $N, M \le 10^5$, the code finishes within a fraction of a millisecond.

* **Space Complexity:** $\mathcal{O}(M)$
  * **Flat Array Allocation:** Instead of creating dynamic tables or storing a full historical tree of subsequences, we maintain the active counts and sums using two flat vectors: `count` and `sum`. The size of these tracking layers is bounded by the max element threshold ($M + 2$), which requires stable $\mathcal{O}(M)$ auxiliary space.
  * **Avoiding Map Overhead:** By using primitive array blocks over hash tables, we completely eliminate the memory footprint overhead associated with hash buckets, chaining nodes, and collision resolution mechanisms.
  * **Total Memory Footprint:** The total auxiliary space remains independent of the length of the sequence $N$ and stays locked within linear bounds of the element values, evaluating strictly to $\mathcal{O}(M)$.
