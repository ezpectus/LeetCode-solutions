# 1157. Online Majority Element In Subarray
## O(n log n) build, O(log n) per query — Segment Tree (Boyer-Moore) + Binary Search

---

## Problem Statement

- Given array `arr`. Answer queries: find element occurring ≥ `threshold` times in `arr[left..right]`, or -1.
- Guarantee: `2 * threshold > right - left + 1` (majority element is unique if it exists).
- Up to 10⁴ queries on array of length ≤ 2×10⁴.

---

## Core Idea — Two Phase Per Query

### Phase 1: Find the Majority Candidate (Boyer-Moore via Segment Tree)

**Boyer-Moore Majority Vote**: scan a sequence keeping `(candidate, count)`. When you see the same value, increment count; otherwise decrement. If count hits 0, reset. At the end, the candidate is the **only possible** majority element.

**Key insight**: Boyer-Moore is composable — two `(candidate, count)` pairs merge as:
```
merge((a, ca), (b, cb)):
    if a == b: return (a, ca + cb)
    if ca >= cb: return (a, ca - cb)
    else: return (b, cb - ca)
```

This allows a **segment tree** where each node stores the Boyer-Moore result for its range. Querying `[left, right]` takes O(log n) by merging O(log n) nodes.

### Phase 2: Verify the Candidate (Binary Search)

The candidate might not actually be majority (Boyer-Moore only guarantees "if a majority exists, this is it"). Verify by counting its occurrences in `[left, right]`.

**Precompute**: for each value, store sorted list of positions → `pos[v] = [i₀, i₁, ...]`.

**Count in range**: binary search for `left` and `right` in `pos[candidate]` → count = `upper_bound(right) - lower_bound(left)`.

If count ≥ threshold → return candidate. Else → return -1.

## Full Optimal Implementation (C#)

```csharp
public class MajorityChecker {
    int[] arr;
    (int cand, int cnt)[] tree;
    Dictionary<int, List<int>> pos = new();
    int n;

    public MajorityChecker(int[] arr) {
        this.arr = arr;
        n = arr.Length;
        tree = new (int,int)[4 * n];

        for (int i = 0; i < n; i++) {
            if (!pos.ContainsKey(arr[i])) pos[arr[i]] = new();
            pos[arr[i]].Add(i);
        }
        Build(1, 0, n - 1);
    }

    (int, int) Merge((int c, int k) a, (int c, int k) b) {
        if (a.c == b.c) return (a.c, a.k + b.k);
        return a.k >= b.k ? (a.c, a.k - b.k) : (b.c, b.k - a.k);
    }

    void Build(int node, int l, int r) {
        if (l == r) { tree[node] = (arr[l], 1); return; }
        int mid = (l + r) / 2;
        Build(2*node, l, mid);
        Build(2*node+1, mid+1, r);
        tree[node] = Merge(tree[2*node], tree[2*node+1]);
    }

    (int, int) QueryTree(int node, int l, int r, int ql, int qr) {
        if (ql <= l && r <= qr) return tree[node];
        int mid = (l + r) / 2;
        if (qr <= mid) return QueryTree(2*node, l, mid, ql, qr);
        if (ql > mid)  return QueryTree(2*node+1, mid+1, r, ql, qr);
        return Merge(QueryTree(2*node, l, mid, ql, qr),
                     QueryTree(2*node+1, mid+1, r, ql, qr));
    }

    public int Query(int left, int right, int threshold) {
        var (candidate, _) = QueryTree(1, 0, n-1, left, right);
        if (!pos.ContainsKey(candidate)) return -1;
        var positions = pos[candidate];
        int lo = LowerBound(positions, left);
        int hi = UpperBound(positions, right);
        return hi - lo >= threshold ? candidate : -1;
    }

    int LowerBound(List<int> list, int val) {
        int lo = 0, hi = list.Count;
        while (lo < hi) { int mid = (lo+hi)/2; if (list[mid] < val) lo=mid+1; else hi=mid; }
        return lo;
    }
    int UpperBound(List<int> list, int val) {
        int lo = 0, hi = list.Count;
        while (lo < hi) { int mid = (lo+hi)/2; if (list[mid] <= val) lo=mid+1; else hi=mid; }
        return lo;
    }
}
```

## Boyer-Moore Merge — Why It's Correct

Boyer-Moore works by "canceling" pairs of different elements. The merge operation applies this same cancellation principle across two subsegments:

- If both agree on candidate → combine counts (no cancellation needed)
- If they disagree → cancel `min(ca, cb)` pairs → winner has `|ca - cb|` votes remaining

**Key property**: if element `x` is majority in the full range, it must "win" the merge of any partition of that range. So the segment tree root gives the correct Boyer-Moore candidate for the full query range.

**But why verify?** Boyer-Moore only guarantees the candidate IF a majority exists. The problem's constraint `2*threshold > length` means at most one element can be majority, so a single candidate is sufficient to check.

## Example Walkthrough

**`arr=[1,1,2,2,1,1]`**

Segment tree nodes (some):
- `[0,5]`: merge of left half `[0,2]` and right half `[3,5]`
- `[0,2]`: Boyer-Moore of [1,1,2] → (1,1) (1 cancels one 2, net: one 1)
- `[3,5]`: Boyer-Moore of [2,1,1] → (1,1)
- `[0,5]`: merge((1,1),(1,1)) = (1,2) → candidate=1

**Query(0,5,4)**: candidate=1. Count in [0,5]: pos[1]=[0,1,4,5], all 4 in range → 4 ≥ 4 → **1** ✓

**Query(0,3,3)**: candidate from [1,1,2,2] → (1,1) vs (2,1) → merge → (1,0)... actually = (2,0) tie → (1,0) → candidate=1 or 2. Count 1 in [0,3]: pos[1]={0,1} → 2 < 3 → **-1** ✓

## Pitfalls & Edge Cases

- **Candidate from merge may have count 0**: still valid candidate to check (Boyer-Moore guarantee holds regardless of count)
- **Verification is mandatory**: the tree only finds a candidate, not the verified majority
- **`pos[candidate]` might not exist** if candidate value is invalid — guard with `ContainsKey`
- **Binary search implementation**: use `lower_bound(left)` and `upper_bound(right)` carefully; off-by-one gives wrong count

---

## Key Takeaway

"Online majority queries" → **Boyer-Moore segment tree + binary search verification**:

1. **Segment tree** stores Boyer-Moore `(candidate, count)` per node; merge in O(log n)
2. **Position lists** + binary search verify candidate frequency in O(log n)
3. Guarantee `2*threshold > length` ensures at most one majority element → one candidate check suffices

The Boyer-Moore merge is the elegant insight: majority vote is a monoid operation that composes across subranges.
