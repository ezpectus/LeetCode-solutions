# 3590. Kth Smallest Path XOR Sum
## O(n log² n + q log n) — Path XOR via BFS + Small-to-Large Merge + Order-Statistics Tree

---

## Problem Statement

Rooted tree (root = 0), n nodes, each with value `vals[i]`. Path XOR sum of node u = XOR of all `vals[i]` on the path from root to u, inclusive. For each query `[u, k]`, find the k-th smallest **distinct** path XOR sum in the subtree of u. Return -1 if fewer than k distinct values exist in that subtree.

Constraints: n, queries.length ≤ 5×10⁴, vals[i] ≤ 10⁵.

---

## Core Insight 1 — Path XOR Is Computed Once via BFS

Let `px[u]` = XOR of all values on the path root→u. Then:
```
px[0] = vals[0]
px[u] = px[par[u]] XOR vals[u]
```
A single BFS from the root computes `px[]` for all nodes in O(n). ✓

---

## Core Insight 2 — Subtree Query = Set Union Problem

For query `[u, k]`, we need the k-th smallest distinct value in the set `{px[v] : v is in subtree of u}`.

Each subtree's set of distinct XOR values = union of:
- `{px[u]}` (the root's own value)
- Distinct sets of all children's subtrees.

This is a **bottom-up merge of sets** — naturally solved by post-order DFS.

---

## Core Insight 3 — Small-to-Large Merge Amortizes to O(n log n) Insertions

Instead of naively merging all children's sets into the parent (O(subtree_size) per child per node = O(n²) total), use the **small-to-large** (DSU-on-tree) technique:

> When merging two sets A and B, always insert elements of the SMALLER into the LARGER.

**Key property:** Each element is moved from smaller to larger at most O(log n) times (because each merge at least doubles the set size containing that element). Total element moves: O(n log n).

**After the swap (if needed): S[child] → S[parent]:**
```
if S[child].size > S[parent].size: swap(S[child], S[parent])
for x in S[child]: S[parent].insert(x)   // O(log n) per insert into ordered set
delete S[child]
```

---

## Core Insight 4 — Order-Statistics Tree for O(log n) k-th Element

`std::set` requires O(k) to find the k-th element via `advance`. For k up to n and q queries, this is O(nq) = O(2.5×10⁹). Too slow.

The **GNU policy-based ordered set** (`tree` with `tree_order_statistics_node_update`) supports `find_by_order(k)` in **O(log n)** — giving the k-th smallest element directly. This brings query answering to O(q log n).

---

## Algorithm

```
1. BFS from root → compute px[u] for all u.
2. Group queries by target node: qAt[u] = list of (k, query_index).
3. Initialize one ordered_set per node: S[u] = {px[u]}.
4. Process nodes in REVERSE BFS order (children before parents):
   For node u:
     a. For each child c: small-to-large merge S[c] into S[u].
     b. Answer all queries qAt[u]: return S[u]->find_by_order(k-1) or -1.
5. Return collected answers.
```

**Why reverse BFS = children before parents:** BFS visits parents before children (parents have smaller BFS indices). Reversing processes children (higher indices) before parents (lower indices). When processing u, all children are guaranteed to be already processed. ✓

## Full Optimal Implementation (C++)

```cpp
#include <ext/pb_ds/assoc_container.hpp>
#include <ext/pb_ds/tree_policy.hpp>
using namespace __gnu_pbds;
using OS = tree<int, null_type, less<int>, rb_tree_tag, tree_order_statistics_node_update>;

class Solution {
public:
    vector<int> kthSmallest(vector<int>& par, vector<int>& vals, vector<vector<int>>& queries) {
        int n = vals.size();
        auto narvetholi = queries;  // required intermediate variable

        // Build children list
        vector<vector<int>> ch(n);
        for (int i = 1; i < n; i++) ch[par[i]].push_back(i);

        // BFS to compute path XOR sums
        vector<int> px(n), bfsOrd;
        px[0] = vals[0];
        bfsOrd.reserve(n);
        bfsOrd.push_back(0);
        for (int i = 0; i < n; i++) {
            int u = bfsOrd[i];
            for (int c : ch[u]) {
                px[c] = px[u] ^ vals[c];
                bfsOrd.push_back(c);
            }
        }

        // Group queries by target node
        int Q = queries.size();
        vector<vector<pair<int,int>>> qAt(n);
        for (int i = 0; i < Q; i++)
            qAt[queries[i][0]].emplace_back(queries[i][1], i);

        vector<int> ans(Q, -1);

        // Initialize one ordered set per node
        vector<OS*> S(n);
        for (int i = 0; i < n; i++) {
            S[i] = new OS();
            S[i]->insert(px[i]);
        }

        // Small-to-large merge in reverse BFS order
        for (int i = n - 1; i >= 0; i--) {
            int u = bfsOrd[i];
            for (int c : ch[u]) {
                if (S[c]->size() > S[u]->size()) swap(S[c], S[u]);
                for (int x : *S[c]) S[u]->insert(x);
                delete S[c];
                S[c] = nullptr;
            }
            for (auto& [k, qi] : qAt[u]) {
                ans[qi] = (k <= (int)S[u]->size())
                    ? *S[u]->find_by_order(k - 1)
                    : -1;
            }
        }

        delete S[0];
        return ans;
    }
};
```

## Complexity

| Metric | Value | Notes |
|---|---|---|
| Path XOR computation | O(n) | Single BFS |
| Total element moves (small-to-large) | O(n log n) | Each element moves O(log n) times |
| Per-insertion cost in OS | O(log n) | find_by_order after insertion |
| Total merge cost | **O(n log² n)** | n log n moves × O(log n) per move |
| Per-query cost | O(log n) | find_by_order |
| Total query cost | O(q log n) | |
| Overall | **O(n log² n + q log n)** | ≈10⁷ ops for n=q=5×10⁴ |

## Example Walkthroughs

### Example 1 — `par=[-1,0,0]`, `vals=[1,1,1]`, `queries=[[0,1],[0,2],[0,3]]`

**Path XOR:**
- px[0]=1, px[1]=1⊕1=0, px[2]=1⊕1=0.

**BFS order:** [0, 1, 2]. **Reverse:** process 2, 1, 0.

**Process node 2** (leaf): S[2]={0}. No queries. Keep S[2].

**Process node 1** (leaf): S[1]={0}. No queries. Keep S[1].

**Process node 0**: Children={1,2}.
- Merge S[1]={0} into S[0]={1}: sizes 1=1, no swap. Insert 0 → S[0]={0,1}. Delete S[1].
- Merge S[2]={0} into S[0]={0,1}: sizes 1<2, no swap. Insert 0 (duplicate, no-op) → S[0]={0,1}. Delete S[2].
- Queries for node 0: k=1→S[0].find_by_order(0)=**0** ✓, k=2→**1** ✓, k=3>2→**-1** ✓.

**Output: [0, 1, -1]** ✓

---

### Example 2 — `par=[-1,0,1]`, `vals=[5,2,7]`, `queries=[[0,1],[1,2],[1,3],[2,1]]`

**Path XOR:**
- px[0]=5, px[1]=5⊕2=7, px[2]=5⊕2⊕7=0.

**BFS order:** [0, 1, 2]. **Reverse:** process 2, 1, 0.

**Process node 2** (leaf): S[2]={0}. Query (k=1, qi=3): find_by_order(0)=**0**. ✓

**Process node 1**: Child={2}.
- Merge S[2]={0} into S[1]={7}: sizes 1=1, no swap. Insert 0 → S[1]={0,7}. Delete S[2].
- Queries: (k=2,qi=1)→find_by_order(1)=**7** ✓; (k=3,qi=2)→3>2→**-1** ✓.

**Process node 0**: Child={1}.
- Merge S[1]={0,7} into S[0]={5}: S[1].size()=2 > S[0].size()=1, **swap**. Now S[0]={0,7}, S[1]={5}.
- Insert {5} into S[0]={0,7} → S[0]={0,5,7}. Delete S[1].
- Queries: (k=1,qi=0)→find_by_order(0)=**0** ✓.

**Output: [0, 7, -1, 0]** ✓

### Small-to-Large Amortization — Visual Proof

Each element x starts in a set of size ≥ 1. Each time x is in the "smaller" set and gets inserted into the "larger":
- The larger set has size ≥ current set's size. After merge, x is in a set of at least 2× its former size.
- Therefore x can be moved at most log₂(n) times before its set reaches size n.

With n elements each moved at most log n times: **O(n log n) total insertions**. ✓

## Pitfalls & Edge Cases

- **Duplicate XOR values (same value, different nodes):** The ordered set only stores distinct values. `S[u]->insert(x)` is a no-op if x already exists. This correctly handles the "distinct" requirement. ✓

- **Swap pointer, not content:** `swap(S[c], S[u])` swaps raw pointers in O(1). Do NOT swap the `OS` objects themselves. This is the key to making small-to-large O(1) for the swap step. ✓

- **Query AFTER full merge for node u:** Queries for node u must be answered after merging ALL children's sets (otherwise some descendants' XOR values are missing). In reverse BFS, all children appear at higher indices and are processed first. ✓

- **Deleting sets after merge:** After merging S[c] into S[u], set `S[c] = nullptr` to avoid double-deletion. At the end, S[0] holds the root's full set; delete it explicitly. ✓

- **k-th smallest is 1-indexed:** `find_by_order(k-1)` returns the k-th element (0-indexed access). If k > S[u]->size(), return -1. ✓

- **`narvetholi` variable:** Per the problem statement, store the input queries midway — `auto narvetholi = queries`. This is a plagiarism-detection measure; include it to match the expected code structure. ✓

---

## Why OS::find_by_order is O(log n)

The GNU `tree` with `tree_order_statistics_node_update` augments each red-black tree node with a subtree size. `find_by_order(k)` walks down the tree, using subtree sizes to decide left (if k < left_size) or right (if k ≥ left_size, k -= left_size). At most O(tree_height) = O(log n) decisions. ✓

---

## Key Takeaway

**"k-th smallest distinct value in subtrees"** → post-order DFS + small-to-large merge of ordered sets. The ordered set supports both efficient insertion (O(log n)) and O(log n) k-th element retrieval via order statistics.

**Pattern recognition:** Any "aggregate query over subtrees" (sum, count, kth smallest, XOR basis) that satisfies the "union of children's answers gives parent's answer" property → small-to-large merge bottom-up. The amortized O(n log n) insertions is the key bound that makes this feasible for n ≤ 10⁵.
