# LeetCode 3250. Find the Count of Monotonic Pairs I

## Method Explanation

The problem asks us to count the number of pairs of arrays $(arr1, arr2)$ such that $arr1$ is monotonically non-decreasing, $arr2$ is monotonically non-increasing, and $arr1[i] + arr2[i] = nums[i]$ for all $i$.

### 1. State Definition
Since $arr2[i]$ is uniquely determined by $arr1[i]$ (i.e., $arr2[i] = nums[i] - arr1[i]$), we only need to track the choices made for $arr1$.
Let $dp[i][v]$ represent the number of valid monotonic pairs for the prefix of length $i+1$, where $arr1[i] = v$. The value of $v$ can range from $0$ to $nums[i]$.

### 2. State Transitions
To transition from index $i-1$ to index $i$, let $p$ be the value assigned to $arr1[i-1]$. The following conditions must hold:
1. $p \le v$ (to satisfy $arr1[i-1] \le arr1[i]$)
2. $nums[i-1] - p \ge nums[i] - v \implies p \le v - (nums[i] - nums[i-1])$ (to satisfy $arr2[i-1] \ge arr2[i]$)

Thus, for a fixed $v$, the valid values for $p$ lie in the range $[0, \min(v, v - nums[i] + nums[i-1])]$. The recurrence relation is:
$$dp[i][v] = \sum_{p=0}^{\min(v, v - nums[i] + nums[i-1])} dp[i-1][p]$$

### 3. Optimization using Prefix Sums
To make the transitions $\mathcal{O}(1)$ instead of scanning all valid $p$ values, we can compute the prefix sums of the previous DP row: `pref[p] = dp[i-1][0] + ... + dp[i-1][p]`. This optimizes the total time complexity and prepares the solution to easily scale for larger constraints.

## C++ Source Code

```cpp
#include <vector>
#include <algorithm>

using namespace std;

class Solution {
public:
    int countOfPairs(vector<int>& nums) {
        int n = nums.size();
        const int MOD = 1e9 + 7;
        
        // Find the maximum value in nums to bound the DP table size
        int max_val = 0;
        for (int x : nums) max_val = max(max_val, x);
        
        // dp[v] stores the number of combinations where arr1[i] == v
        vector<long long> dp(max_val + 1, 0);
        
        // Base case: initialization for the first element index 0
        for (int v = 0; v <= nums[0]; ++v) {
            dp[v] = 1;
        }
        
        // Dynamic programming transitions
        for (int i = 1; i < n; ++i) {
            vector<long long> next_dp(max_val + 1, 0);
            
            // Build prefix sums of the current DP array for O(1) range queries
            vector<long long> pref(max_val + 1, 0);
            pref[0] = dp[0];
            for (int d = 1; d <= max_val; ++d) {
                pref[d] = (pref[d - 1] + dp[d]) % MOD;
            }
            
            // Calculate transitions for the next element
            for (int v = 0; v <= nums[i]; ++v) {
                int max_p = min(v, v - nums[i] + nums[i - 1]);
                if (max_p >= 0) {
                    next_dp[v] = pref[max_p];
                }
            }
            dp = move(next_dp);
        }
        
        // Total sum of valid sequences at the final index
        long long ans = 0;
        for (int v = 0; v <= nums[n - 1]; ++v) {
            ans = (ans + dp[v]) % MOD;
        }
        
        return ans;
    }
};
```
## Complexity Analysis

* **Time Complexity:** $\mathcal{O}(N \times M)$
  * **Outer Array Traversal:** We iterate through the given array `nums` of length $N$ exactly $N - 1$ times.
  * **Prefix Sum Optimization:** For each element $i$, we construct a prefix sum array `pref` of size $M + 1$ (where $M = \max(nums[i]) \le 50$). Building this array takes linear time relative to the element range, which evaluates to $\mathcal{O}(M)$ per step.
  * **State Evaluation:** For each state $v$ from $0$ to $nums[i]$, computing the recurrence relation requires fetching a single value from the prefix sums (`pref[max_p]`). Thanks to this, the internal transition logic is optimized from a naive $\mathcal{O}(M)$ scan down to an $\mathcal{O}(1)$ step. 
  * **Total Time Bound:** With $N$ outer steps and $M$ inner states evaluated at $\mathcal{O}(1)$ apiece, the total computational bound scales strictly within $\mathcal{O}(N \times M)$. Given $N \le 2000$ and $M \le 50$, the maximum operations hover around $10^5$, executing near-instantaneously.

* **Space Complexity:** $\mathcal{O}(M)$
  * **Matrix Compression:** A naive implementation would maintain an explicit $N \times M$ matrix, demanding $\mathcal{O}(N \times M)$ auxiliary memory. However, since computing the states for index $i$ depends exclusively on the row results from index $i - 1$, we discard older layers.
  * **Auxiliary Buffers:** We restrict the active space footprint by holding only two linear arrays of size $M + 1$ simultaneously: the prior row state `dp` and the active buffer `next_dp`. The temporary prefix summation array `pref` also scales to $M + 1$.
  * **Total Memory Bound:** The total auxiliary overhead is independent of the input array length $N$ and remains entirely constrained by the maximum element upper bound, yielding a highly efficient $\mathcal{O}(M)$ space allocation.
