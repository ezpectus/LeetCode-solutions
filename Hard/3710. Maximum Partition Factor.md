# LeetCode 3710. Maximum Partition Factor

## 1. Constraints & Complexity Bottleneck Analysis

### Constraints:
- **Number of Points (N):** 2 <= points.length <= 500
  - The small constraint on N (N <= 500) indicates that an O(N^2) or O(N^2 * log(MAX_DIST)) approach is perfectly feasible. A cubic O(N^3) approach might be too slow if the constant factor is large.
- **Coordinates (xi, yi):** -10^8 <= xi, yi <= 10^8
  - The coordinates are large, meaning the maximum Manhattan distance can be up to 4 * 10^8.
  - This broad range of distances suggests that binary searching over the distance space is highly appropriate.

### The Complexity Bottleneck:
To maximize the partition factor D, we need to efficiently decide if there exists a valid partition of points into two non-empty groups such that no two points in the same group have a Manhattan distance strictly less than D. 
For a fixed target distance D, any pair of points (i, j) with Manhattan(i, j) < D cannot belong to the same group. This forces them to be in different groups, which is a classic bipartite matching / graph coloring constraint.
The bottleneck is:
1. Identifying the optimal partition factor D using binary search.
2. Validating if a candidate D is feasible by constructing a graph of "must-be-separated" constraints (edges where Manhattan(i, j) < D) and checking if this graph is 2-colorable (bipartite) while ensuring both partitions are non-empty.

## 2. Algorithm Design & Mathematical Invariants

### 2.1 Core Concept
We can binary search the maximum possible partition factor D. The search range for D is [0, Max_Manhattan_Distance]. 
For a candidate value D:
- We construct an undirected graph where each point is a vertex.
- We draw an edge between vertex i and vertex j if and only if the Manhattan distance between points[i] and points[j] is strictly less than D. This edge represents the constraint: "Point i and Point j must be in different groups."
- If the resulting graph is bipartite, we can color its vertices using 2 colors (representing the two groups) without violating any constraints.
- However, we must also satisfy the "non-empty" group constraint. If the graph is bipartite, we partition it into connected components. Each component can be 2-colored independently. We must verify if we can assign colors to these components such that both final groups are non-empty.

### 2.2 Key Invariants & Feasibility Checking
For a candidate D, let G be the graph defined by edges E = { (i, j) | Manhattan(i, j) < D }.

#### Bipartite Verification
For each connected component in G, we perform a Depth-First Search (DFS) or Breadth-First Search (BFS) to attempt 2-coloring:
- If we encounter an odd cycle (e.g., we try to color an already-colored neighbor with the same color), the graph is not bipartite. Candidate D is **invalid**.

#### Non-Empty Groups Invariant
If the graph is bipartite, we obtain a set of connected components. For each component `c`, we have two independent color sets of sizes `size1[c]` and `size2[c]` (where `size1[c] + size2[c] = component_size`).
We must combine these components to form two non-empty groups of sizes A and B (where A + B = N).
- If there is at least one component where both color classes are non-empty (`size1[c] > 0` and `size2[c] > 0`), then any bipartite coloring naturally puts elements into both groups. Thus, both groups are guaranteed to be non-empty.
- If ALL components have one of their color classes of size 0, it means all components are singletons (isolated vertices with no edges).
  - Since N >= 2, if all vertices are isolated, we have N independent components of size 1. We can trivially assign at least one vertex to group A and the rest to group B, ensuring both groups are non-empty. The only exception where we cannot form two non-empty groups is if N = 1 (which is prevented by the constraints).
  - Therefore, the non-empty condition is always satisfiable as long as N >= 2 and the graph is bipartite.

## 3. Full Optimal Implementation (C++)

```
// Binary Search + Depth-First Search + Bipartite Coloring
#include <vector>
#include <cmath>
#include <algorithm>

using namespace std;

class Solution {
private:
    // Helper to calculate Manhattan distance between two points
    long long getManhattanDistance(const vector<int>& p1, const vector<int>& p2) {
        return abs((long long)p1[0] - p2[0]) + abs((long long)p1[1] - p2[1]);
    }

    // DFS to check if the component is bipartite (2-colorable)
    bool dfs(int u, int c, const vector<vector<int>>& adj, vector<int>& color) {
        color[u] = c;
        for (int v : adj[u]) {
            if (color[v] == 0) {
                if (!dfs(v, 3 - c, adj, color)) {
                    return false;
                }
            } else if (color[v] == c) {
                return false; // Odd cycle detected
            }
        }
        return true;
    }

    // Check if a partition factor D is feasible
    bool isValid(long long D, int n, const vector<vector<int>>& points) {
        // Step 1: Build the adjacency list for constraints
        vector<vector<int>> adj(n);
        for (int i = 0; i < n; ++i) {
            for (int j = i + 1; j < n; ++j) {
                if (getManhattanDistance(points[i], points[j]) < D) {
                    adj[i].push_back(j);
                    adj[j].push_back(i);
                }
            }
        }

        // Step 2: Verify bipartite property
        vector<int> color(n, 0); // 0: uncolored, 1: color A, 2: color B
        for (int i = 0; i < n; ++i) {
            if (color[i] == 0) {
                if (!dfs(i, 1, adj, color)) {
                    return false;
                }
            }
        }

        return true;
    }

public:
    int maxPartitionFactor(vector<vector<int>>& points) {
        int n = points.size();
        if (n == 2) {
            return 0; // As defined by the problem note
        }

        // Find the maximum possible Manhattan distance to set binary search bounds
        long long low = 0;
        long long high = 0;
        for (int i = 0; i < n; ++i) {
            for (int j = i + 1; j < n; ++j) {
                high = max(high, getManhattanDistance(points[i], points[j]));
            }
        }

        long long ans = 0;
        while (low <= high) {
            long long mid = low + (high - low) / 2;
            if (isValid(mid, n, points)) {
                ans = mid;
                low = mid + 1; // Try to maximize the partition factor
            } else {
                high = mid - 1; // D is too large, reduce constraints
            }
        }

        return (int)ans;
    }
};
```

## 4. Complexity & Mathematical Analysis

### 4.1 Detailed Time Complexity Decomposition
The algorithm's time complexity is a product of the binary search steps and the graph validation step:

1. **Graph Construction & Coloring (Validation):**
   - For a given D, we check all pairs of points to build the adjacency list. There are N * (N - 1) / 2 pairs.
   - Vertices in the graph: V = N.
   - Edges in the graph: E <= N * (N - 1) / 2.
   - Building the graph takes O(N^2) time.
   - The DFS bipartite check visits each vertex and edge once, taking O(V + E) = O(N^2) time.
   - Therefore, the validation function `isValid` runs in `T_valid = O(N^2)` time.

2. **Binary Search Range:**
   - The search space range is from 0 to Max_Distance, where Max_Distance <= 4 * 10^8.
   - The binary search takes `log(Max_Distance)` iterations. Since 2^29 > 4 * 10^8, the binary search will run at most 30 times.

**Total Time Complexity:** O(N^2 * log(MAX_DIST))
For N = 500 and log(MAX_DIST) = 30, the total operations are approximately 30 * 250,000 = 7,500,000, which easily runs within 15-30ms in C++.

### 4.2 Detailed Space Complexity Decomposition
- **Adjacency List:** The adjacency list `adj` stores up to N vertices and up to O(N^2) edges. In the worst case, this requires O(N^2) memory.
- **Color Array & DFS Stack:** The recursion stack for DFS can go up to depth N, and the `color` tracking array requires O(N) memory.

**Total Space Complexity:** O(N^2) auxiliary memory to store graph constraints.

### 4.3 Complexity Summary
| Metric | Value | Notes |
|---|---|---|
| **Time Complexity** | O(N^2 * log(MAX_DIST)) | Dominated by pair-wise distance graph construction in each binary search step. |
| **Space Complexity** | O(N^2) | Required to store the constraint graph adjacency list. |

## 5. Mathematical Formulations & Foundational Algorithms

### Bipartite Graph Properties
A graph G is bipartite if and only if it contains no odd-length cycles. 
By translating the minimum intra-group distance requirement into a graph coloring constraint (where edges represent forced different-group assignments), we guarantee that if a 2-coloring exists, we can partition the vertices into two sets V1 and V2 such that no edge exists within V1 or V2. 
Since an edge exists if and only if Manhattan(i, j) < D, the absence of intra-set edges mathematically guarantees that:
`Manhattan(i, j) >= D` for all pairs within the same group.

---

## Pitfalls & Edge Cases
- **Strict vs Non-Strict Inequality:** Edges must be created when `Manhattan(i, j) < D`. If we used `<=`, we would incorrectly force points separated by exactly D to be in different groups, which unnecessarily constrains the partition.
- **N = 2 Base Case:** When N = 2, there are no intra-group pairs available to contribute to a partition factor. The problem specifically asks to return 0 in this case. Checking this early avoids unnecessary computation.
- **Integer Overflow:** Manhattan distance calculation can exceed the range of standard 32-bit signed integers if coordinates are near 10^8. Using `long long` for distance calculations avoids overflow.

---

## Key Takeaway
When tasked with maximizing a minimum threshold distance (a "bottleneck" maximization) where objects must be grouped under separation constraints, binary search on the threshold value combined with bipartite graph coloring is the gold-standard pattern.

**Pattern recognition:** "minimum distance among all pairs in the same group is maximized" -> Binary Search on Answer + Bipartite Graph Verification + DFS Coloring
