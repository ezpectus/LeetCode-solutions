# 2136. Earliest Possible Day of Full Bloom
## O(n log n) ‚Äî Greedy: Sort by Grow Time Descending, Plant Sequentially

---

## Problem Statement

`n` seeds, each with `plantTime[i]` (planting days needed) and `growTime[i]` (growing days after planting). One day of planting work per calendar day (can split non-consecutively). Minimize the day when all seeds have bloomed.

---

## Core Insight #1 ‚Äî Interleaving Never Helps

First, show that the optimal schedule plants seeds **sequentially** (fully finishing one before starting another).

For any two seeds A and B, whichever is scheduled to finish last has its finish day pinned at `(total planting time of seeds scheduled before it) + its own plantTime - 1`. Interleaving the two would push seed A's finish later (it can't finish until its `plantTime[A]` days are all done), which only hurts A's bloom time. The seed scheduled last is unaffected ‚Äî it still needs plantTime days from day 0 to its finish, but now it also waits longer.

**Formal exchange:** Interleaving seeds A and B in schedule slots `[d_1, d_2, ...]`:
- `fA ‚â• plantTime[A] ‚àí 1` (must use ‚â• plantTime[A] days for A)
- `fB ‚â• plantTime[A] + plantTime[B] ‚àí 1` (must use all A + B days)

Sequential A‚ÜíB achieves: `fA = plantTime[A]‚àí1`, `fB = plantTime[A]+plantTime[B]‚àí1`. Interleaving can only increase `fA` without decreasing `fB`. So sequential is optimal.

---

## Core Insight #2 ‚Äî Bloom Day Formula

If we plant seeds sequentially in order `œÉ[0], œÉ[1], ..., œÉ[n-1]`:

```
finish[œÉ[k]] = Œ£_{j=0}^{k} plantTime[œÉ[j]]  - 1
bloom[œÉ[k]]  = finish[œÉ[k]] + growTime[œÉ[k]] + 1
             = Œ£_{j=0}^{k} plantTime[œÉ[j]] + growTime[œÉ[k]]
```

We want to minimize `max_k (Œ£_{j‚â§k} plantTime[œÉ[j]] + growTime[œÉ[k]])`.

---

## Core Insight #3 ‚Äî Sort by growTime Descending (Exchange Argument)

For two adjacent seeds A (position k) and B (position k+1), let `P` = sum of plant times before them.

**A before B:**
```
max(P + plantTime[A] + growTime[A],
    P + plantTime[A] + plantTime[B] + growTime[B])
= P + plantTime[A] + max(growTime[A], plantTime[B] + growTime[B])
```

**B before A:**
```
= P + plantTime[B] + max(growTime[B], plantTime[A] + growTime[A])
```

Since `plantTime[A] ‚â• 1 > 0`, we have `plantTime[A] + growTime[A] > growTime[A] ‚â• growTime[B]` when `growTime[A] ‚â• growTime[B]`. So the RHS simplifies to `P + plantTime[A] + plantTime[B] + growTime[A]`.

**A before B is optimal when growTime[A] ‚â• growTime[B]:**

| Case | LHS (A‚ÜíB) | RHS (B‚ÜíA) | Winner |
|---|---|---|---|
| growTime[A] ‚â• plantTime[B]+growTime[B] | P+pA+gA | P+pA+pB+gA | A‚ÜíB ‚úì |
| growTime[A] < plantTime[B]+growTime[B] | P+pA+pB+gB | P+pA+pB+gA | A‚ÜíB ‚úì (gB ‚â§ gA) |

In both cases, putting the higher `growTime` seed first is optimal or tied.

**Conclusion: Sort seeds by `growTime` descending.**

## Full Implementation (C#)

```csharp
public class Solution {
    public int EarliestFullBloom(int[] plantTime, int[] growTime) {
        int n = plantTime.Length;
        int[] idx = Enumerable.Range(0, n).ToArray();

        // Sort seeds by growTime descending
        Array.Sort(idx, (a, b) => growTime[b].CompareTo(growTime[a]));

        int day = 0; // cumulative planting days used
        int ans = 0;

        foreach (int i in idx) {
            day += plantTime[i];          // finish planting seed i on day (day - 1)
            ans = Math.Max(ans, day + growTime[i]); // bloom = day + growTime[i]
        }

        return ans;
    }
}
```

**Compact math:** After `day += plantTime[i]`, the finish day is `day - 1` (0-indexed), so bloom = `(day - 1) + growTime[i] + 1 = day + growTime[i]`. ‚úì

## Why Ties in growTime Don't Matter

When `growTime[A] == growTime[B]`, the exchange argument gives equal cost either way:
```
A‚ÜíB cost: P + pA + max(gA, pB + gB)
B‚ÜíA cost: P + pB + max(gB, pA + gA) = P + pB + pA + gA  (since pA+gA > gA = gB)
```
For gA = gB: A‚ÜíB cost = P + pA + max(gA, pB+gA) = P + pA + pB + gA = B‚ÜíA cost. ‚úì

## Pitfalls & Edge Cases

- **Sequential (no interleaving) is required for correctness.** If you try to schedule greedily with interleaving (e.g., always work on the seed with the longest remaining grow time), the scheduling complexity increases and is unnecessary ‚Äî sequential is provably optimal.

- **Cumulative `day` gives the finish day + 1.** After `day += plantTime[i]`, `day - 1` is the last planting day. Bloom = `(day-1) + growTime[i] + 1 = day + growTime[i]`. No off-by-one if you remember this.

- **All seeds with the same growTime:** Sort is stable; order within a tie doesn't affect the answer. Any order gives the same `max bloom` for that group.

- **n=1:** One seed, planted starting day 0. Bloom = plantTime[0] + growTime[0]. The loop runs once and gives exactly this. ‚úì

---

## Key Takeaway

This is the **"minimize makespan with waiting costs"** greedy pattern. When tasks have a "suffix cost" (grow time), the optimal order is determined by sorting on that suffix cost (decreasing). The exchange argument shows that swapping any two adjacent out-of-order tasks never helps. Crucially, interleaving tasks never improves on sequential planting ‚Äî which makes this a pure ordering problem solvable in O(n log n).

**Pattern recognition:** "Sequential tasks with setup times and latency/growth after setup" ‚Üí sort by latency (growTime) descending. Same as "minimizing weighted completion time" in scheduling theory (but with equal weights and only the max matters, not sum).
