# 2234. Maximum Total Beauty of the Gardens
## O(n log n) — Sort + Enumerate Complete Gardens + Binary Search

---

## Problem Statement

- You have `n` gardens with `flowers[i]` flowers already planted (cannot be removed).
- You can plant at most `newFlowers` additional flowers total.
- A garden is **complete** if it has ≥ `target` flowers.
- **Total beauty** = `(# complete gardens) × full + (min flowers in incomplete gardens) × partial`
  - If no incomplete gardens → partial contribution = 0
- Return the **maximum total beauty**.

---

## Core Idea — Enumerate Complete Count + Binary Search for Min Height

**Observation**: After sorting `flowers`, the optimal strategy always completes the **rightmost** (largest) gardens first — they're cheapest to top up to `target`. We enumerate how many gardens we decide to complete (`j = 0..n`), then for the remaining incomplete gardens, maximize the **minimum height**.

**Two nested optimizations**:
1. **Outer loop**: enumerate `j` = number of complete gardens (0 to n), accumulating cost from the right
2. **Inner binary search**: given a budget `leftBudget` for the incomplete left side, find the maximum achievable minimum height `minH`

**Key formula — cost to raise all incomplete gardens to minH**:

Given sorted `flowers[0..leftCount-1]`, the cost to bring every garden up to `minH` is:

```
cost = minH × idx - prefSum[idx]
```

where `idx` = first index with `flowers[idx] >= minH` (found via binary search).  
Intuition: gardens below `minH` each need `minH - flowers[i]` flowers; summing gives `minH × idx - Σflowers[0..idx-1]`.

**Monotonicity**: as `minH` increases, cost increases → binary search is valid.

## Full Optimal Implementation (C#)

```csharp
public class Solution {
    public long MaximumBeauty(int[] flowers, long newFlowers, int target, int full, int partial) {
        Array.Sort(flowers);
        int n = flowers.Length;

        // Cap all gardens at target (already-complete cost 0 more)
        for (int i = 0; i < n; i++) flowers[i] = Math.Min(flowers[i], target);

        // Prefix sum for cost calculation
        long[] pref = new long[n + 1];
        for (int i = 0; i < n; i++) pref[i + 1] = pref[i] + flowers[i];

        // Cost to raise flowers[0..count-1] all to minH
        long CostToMin(int count, long minH) {
            int lo = 0, hi = count;
            while (lo < hi) { // binary search first index >= minH
                int mid = (lo + hi) / 2;
                if (flowers[mid] < minH) lo = mid + 1; else hi = mid;
            }
            return minH * lo - pref[lo];
        }

        long res = 0, completeCost = 0;

        for (int j = 0; j <= n; j++) {
            if (j > 0) completeCost += target - flowers[n - j];
            if (completeCost > newFlowers) break;

            long leftBudget = newFlowers - completeCost;
            int leftCount = n - j;
            long beauty = (long)j * full;

            if (leftCount == 0) { res = Math.Max(res, beauty); continue; }

            // If leftmost of incomplete side is already complete → no partial
            if (flowers[leftCount - 1] == target) { res = Math.Max(res, beauty); continue; }

            // Binary search: max minH in [flowers[0], target-1]
            long lo = flowers[0], hi = target - 1;
            while (lo < hi) {
                long mid = (lo + hi + 1) / 2;
                if (CostToMin(leftCount, mid) <= leftBudget) lo = mid;
                else hi = mid - 1;
            }

            if (CostToMin(leftCount, lo) <= leftBudget)
                beauty += lo * partial;

            res = Math.Max(res, beauty);
        }

        return res;
    }
}
```

| **Space Complexity** | **O(n)** | Prefix sum array |

## Why Completing from the Right is Optimal

After sorting, `flowers[n-1] ≥ flowers[n-2] ≥ ...`. Completing garden `i` costs `target - flowers[i]`. To minimize the cost of completing `j` gardens, always pick the `j` gardens with **highest existing counts** = the rightmost `j` in sorted order. Any other choice costs at least as much.

## Cost Formula — Math Breakdown

For sorted `flowers[0..count-1]` and target minimum height `minH`:

```
cost = Σ max(0, minH - flowers[i])  for i in [0, count)
     = Σ (minH - flowers[i])        for i in [0, idx)    where flowers[idx] >= minH
     = minH × idx - prefSum[idx]
```

`idx` found via binary search on sorted `flowers`. This is O(log n) per call.

**Monotonicity**: larger `minH` → `idx` ≥ previous `idx` and `minH × idx` grows → cost is non-decreasing → binary search on `minH` is valid.

## Pitfalls & Edge Cases

- **Cap `flowers[i]` at `target`** — already-complete gardens cost 0 to complete; treating them as `target` ensures `complete_cost += target - flowers[n-j]` correctly gives 0 for free completions
- **Check `flowers[leftCount-1] == target`** — if the rightmost incomplete garden is already at target, all "incomplete" gardens are actually complete → no partial contribution, skip binary search
- **`newFlowers` is `long` (up to 10¹⁰)** — `completeCost` and `leftBudget` must be `long`; `int` would overflow
- **Binary search floor at `flowers[0]`** — `minH` can't go below the smallest garden (no flowers can be removed)
- **Binary search ceiling at `target - 1`** — `minH = target` would make the garden complete, contradicting it being "incomplete"
- **`j = n` (all complete)** — `leftCount = 0`, no partial score; `beauty = n × full`

---

## Key Takeaway

Two-level optimization: **enumerate one dimension, binary search the other**:

- **Outer**: enumerate `j` complete gardens (right to left, cheapest first) in O(n)
- **Inner**: for each `j`, binary search the max `minH` for incomplete gardens in O(log n)
- **Cost formula**: `minH × idx - prefSum[idx]` computes fill cost in O(log n) via binary search on sorted prefix

Pattern: maximize `A × f(x) + B × g(x)` where one part is enumerated and the other optimized via binary search on a monotone function.
