# 2577. Minimum Time to Visit a Cell In a Grid
## O(m·n · log(m·n)) — Modified Dijkstra with Parity-Aware Waiting

---

## Problem Statement

You start at `(0,0)` at `t=0` on an `m×n` grid. Each step costs 1 second and moves you to an adjacent cell (4-directional). You may only enter cell `(r,c)` when the current time is `≥ grid[r][c]`. Return the minimum time to reach `(m-1, n-1)`, or `-1` if impossible.

---

## Core Idea — Dijkstra with a "Waiting" Trick

This is a shortest-path problem on a grid, but edge weights aren't fixed — the cost to enter a neighbor depends on **when you arrive**, not just on the distance.

**Key insight:** if you're at cell `A` at time `t` but neighbor `B` requires `grid[B] > t + 1`, you cannot wait in place. You must *bounce* between `A` and any other neighbor, wasting 2 seconds per round trip, until you are allowed to enter `B`. This means:

- Each bounce adds 2 to your time, so arrival time at `B` always has the same **parity** as `t + 1`.
- If `(grid[B] − t)` is **odd**: you can land at exactly `grid[B]` — no overshoot.
- If `(grid[B] − t)` is **even**: you must land at `grid[B] + 1` — one extra bounce required.

**Arrival time formula** for neighbor `(nr, nc)` from `(r, c)` at time `t`:

```
if t + 1 >= grid[nr][nc]:
    arrival = t + 1                                          (no waiting needed)
else:
    arrival = grid[nr][nc] + ((grid[nr][nc] - t) % 2 == 0 ? 1 : 0)
```

**Early impossibility check:** From `(0,0)` at `t=0`, we can only reach `(0,1)` or `(1,0)` at `t=1`.
If *both* have `grid > 1` we can never move at all → return `-1` immediately.

---

## Why Dijkstra and Not BFS?

Vanilla BFS assumes all edge weights equal 1. Here the "weight" to reach a neighbor can be 1 (direct move) or much higher (after bouncing). This creates a non-uniform cost landscape, making BFS incorrect. Dijkstra's min-heap always processes the cell reachable at the *smallest time first*, which is exactly what we want.

---

## Full Optimal Implementation (C++)

```cpp
class Solution {
public:
    int minimumTime(vector<vector<int>>& grid) {
        int m = grid.size(), n = grid[0].size();

        // If neither immediate neighbor is reachable at t=1, we're stuck forever
        if (grid[0][1] > 1 && grid[1][0] > 1) return -1;

        // dist[r][c] = minimum time to arrive at (r,c)
        vector<vector<int>> dist(m, vector<int>(n, INT_MAX));
        dist[0][0] = 0;

        // Min-heap: (arrival_time, row, col)
        using T3 = tuple<int,int,int>;
        priority_queue<T3, vector<T3>, greater<T3>> pq;
        pq.push({0, 0, 0});

        const int dirs[4][2] = {{0,1},{0,-1},{1,0},{-1,0}};

        while (!pq.empty()) {
            auto [t, r, c] = pq.top(); pq.pop();

            if (r == m-1 && c == n-1) return t;  // reached target
            if (t > dist[r][c]) continue;         // stale entry, skip

            for (auto& d : dirs) {
                int nr = r + d[0], nc = c + d[1];
                if (nr < 0 || nr >= m || nc < 0 || nc >= n) continue;

                int arrival = t + 1;

                // Need to wait? Bounce until parity allows entry
                if (arrival < grid[nr][nc]) {
                    int diff = grid[nr][nc] - t;
                    arrival = grid[nr][nc] + (diff % 2 == 0 ? 1 : 0);
                }

                if (arrival < dist[nr][nc]) {
                    dist[nr][nc] = arrival;
                    pq.push({arrival, nr, nc});
                }
            }
        }

        return -1;
    }
};
```

### The Fix — Why `using T3`?

The multi-line template in the broken version:
```cpp
priority_queue<
    tuple<int,int,int>,     // ← compiler may misread this as template specialization
    vector<tuple<int,int,int>>,
    greater<>
>
```
triggers a parser ambiguity in some compilers. Aliasing with `using T3 = tuple<int,int,int>` keeps everything on one line and is unambiguous. Also changed `greater<>` to `greater<T3>` for full explicitness — some older compilers reject the deduced form inside a `priority_queue` declaration.

## Parity — Full Derivation

You are at time `t`. You take 1 step (costs 1), then bounce `k` times (costs 2k each). Total time: `t + 1 + 2k`.

To enter a cell requiring threshold `T`, you need `t + 1 + 2k ≥ T`, so:

```
k ≥ (T - t - 1) / 2
k_min = ceil((T - t - 1) / 2)
```

Arrival time:

```
arrival = t + 1 + 2 * ceil((T - t - 1) / 2)
```

Two cases:

| `(T - t - 1) % 2` | `(T - t) % 2` | `k_min` | Arrival |
|---|---|---|---|
| even (0) | odd (1) | `(T-t-1)/2` | `T` exactly |
| odd (1) | even (0) | `(T-t)/2` | `T + 1` |

Collapsed into one expression:
```
arrival = T + ((T - t) % 2 == 0 ? 1 : 0)
```

---

## Pitfalls & Edge Cases

**Stale heap entries.** The same cell can be pushed multiple times with different arrival times. Always check `if (t > dist[r][c]) continue` — skipping stale pops is what keeps Dijkstra correct and efficient.

**Early -1 check is mandatory.** Without it, the algorithm finishes after full traversal and returns -1 anyway, but the check makes it O(1) instead of O(m·n·log(m·n)) for those inputs. More importantly, the logic is cleaner.

**Bouncing requires an adjacent free cell.** The trick assumes we can always bounce off some neighbor. From any interior cell this is trivially true. From `(0,0)` we need at least one of `(0,1)` or `(1,0)` to be reachable — that's exactly the early check.

**`grid[0][0] = 0` guaranteed.** No need to special-case the source. The initial push `{0, 0, 0}` is always valid.

**You cannot wait in place.** This is the core constraint distinguishing this problem from "wait until allowed." All waiting is encoded as bouncing, and the parity formula is the closed-form solution to "how many bounces do I need?"

**`greater<T3>` vs `greater<>`.** Some compilers (notably older Clang versions on LeetCode) reject `greater<>` (deduced comparator) when nested inside a multi-argument `priority_queue` template instantiation. Always provide the explicit type argument.

## Key Takeaway

"Grid traversal with per-cell time gates and no in-place waiting" → "Modified Dijkstra where edge weight = `max(t+1, grid[nb] + parity_fix)`."

**Pattern recognition:** Dijkstra with a non-trivial closed-form edge weight. The parity formula converts a "how many bounces?" question into O(1) arithmetic. Recognise this whenever you see: grid movement + per-cell unlock times + forced movement (no staying still).
