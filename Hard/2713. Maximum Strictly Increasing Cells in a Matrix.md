# 2713. Maximum Strictly Increasing Cells in a Matrix
## O(mn log mn) — Sort by Value + DP with Row/Column Max Arrays

---

## Problem Statement

- m × n matrix, 1 ≤ m × n ≤ 10⁵.
- From any cell, move to any cell in the same row or column with a strictly greater value.
- Return the maximum number of cells visitable starting from any cell.

---

## Core Idea — Process Cells in Value Order, Track Row/Column Bests

Key insight: This is a DAG problem (edges from smaller to larger values in same row/column). Process cells in increasing order.

Define `dp[r][c]` = longest path ending at cell (r, c).

For cell (r, c) with value v:
```
dp[r][c] = 1 + max(rowMax[r], colMax[c])
```
where `rowMax[r]` = max dp seen in row r for values < v, and similarly `colMax[c]`.

**Critical:** Cells with the **same value** cannot reach each other. Process all same-value cells together: compute all their dp values FIRST, then batch-update rowMax and colMax.

**Algorithm:**
1. Sort all cells by value.
2. Group cells with equal value.
3. For each group: compute dp values → update rowMax/colMax.
4. Track global max dp.

---

## Full Optimal Implementation (C#)

```csharp
public class Solution {
    public int MaxIncreasingCells(int[][] mat) {
        int m = mat.Length, n = mat[0].Length;
        int total = m * n;

        // Collect and sort all cells by value
        var cells = new (int val, int r, int c)[total];
        int idx = 0;
        for (int i = 0; i < m; i++)
            for (int j = 0; j < n; j++)
                cells[idx++] = (mat[i][j], i, j);

        Array.Sort(cells, (a, b) => a.val.CompareTo(b.val));

        int[] rowMax = new int[m];
        int[] colMax = new int[n];
        int res = 0;
        int start = 0;

        while (start < total) {
            // Find the extent of the current value group
            int end = start;
            while (end < total && cells[end].val == cells[start].val) end++;

            // Compute dp for all cells in group (before updating rowMax/colMax)
            int groupSize = end - start;
            int[] dp = new int[groupSize];
            for (int k = 0; k < groupSize; k++) {
                var (_, r, c) = cells[start + k];
                dp[k] = 1 + Math.Max(rowMax[r], colMax[c]);
                res = Math.Max(res, dp[k]);
            }

            // Batch update rowMax and colMax after computing all dp values
            for (int k = 0; k < groupSize; k++) {
                var (_, r, c) = cells[start + k];
                rowMax[r] = Math.Max(rowMax[r], dp[k]);
                colMax[c] = Math.Max(colMax[c], dp[k]);
            }

            start = end;
        }

        return res;
    }
}
```

## Complexity

| Metric | Value | Notes |
|---|---|---|
| Time Complexity | O(mn log mn) | Sort dominates; DP is O(mn) |
| Space Complexity | O(m + n + mn) | rowMax, colMax, cells array |

---

## Why Batch Update Is Critical

If two cells (r1, c1) and (r2, c2) have the same value and share row r1 = r2, they can't reach each other. If we updated rowMax after (r1,c1), then computing dp for (r2,c2) would see a falsely inflated rowMax[r1], allowing an illegal "same-value" transition.

**Batch update:** compute all dp values in the group first (using pre-group rowMax/colMax), then update. This ensures equal-valued cells never influence each other. ✓

## Example Walkthrough

**Example 3:** `mat = [[3,1,6],[-9,5,7]]`

Sorted cells: (−9, 1,0), (1, 0,1), (3, 0,0), (5, 1,1), (6, 0,2), (7, 1,2)

rowMax = [0, 0], colMax = [0, 0, 0]

| Value | Cell | rowMax[r] | colMax[c] | dp | rowMax→ | colMax→ |
|-------|------|-----------|-----------|----|---------|---------| 
| −9    | (1,0)| 0         | 0         | **1** | rowMax[1]=1 | colMax[0]=1 |
| 1     | (0,1)| 0         | 0         | **1** | rowMax[0]=1 | colMax[1]=1 |
| 3     | (0,0)| 1         | 1         | **2** | rowMax[0]=2 | colMax[0]=2 |
| 5     | (1,1)| 1         | 1         | **2** | rowMax[1]=2 | colMax[1]=2 |
| 6     | (0,2)| 2         | 0         | **3** | rowMax[0]=3 | colMax[2]=3 |
| 7     | (1,2)| 2         | 3         | **4** | rowMax[1]=4 | colMax[2]=4 |

**Output:** 4 ✓ (path: (1,0)→(1,1)→(0,2)→(1,2) = −9→5→6→7)

---

**Example 1:** `mat = [[3,1],[3,4]]`

Sorted: (1,0,1), then group (3,0,0),(3,1,0), then (4,1,1)

After value=1: rowMax=[1,0], colMax=[0,1]

Group value=3, cells (0,0) and (1,0):
- dp(0,0) = 1+max(rowMax[0]=1, colMax[0]=0) = **2**
- dp(1,0) = 1+max(rowMax[1]=0, colMax[0]=0) = **1**
Batch update: rowMax=[2,1], colMax=[2,1]

Value=4, cell (1,1): dp = 1+max(rowMax[1]=1, colMax[1]=1) = **2**

**Output:** 2 ✓

## Pitfalls & Edge Cases

- **Same-value cells in same row/column** — Must batch-update; otherwise same-value cells falsely "see" each other's dp contributions. The dp array stores temporary values for the current group. ✓

- **All equal values** (Example 2, all 1s) — Every group has all mn cells. dp = 1 for each (rowMax and colMax are all 0 during the single group). Answer = 1. ✓

- **Single cell matrix** — dp = 1, trivially returned. ✓

- **Matrix constraints** — m × n ≤ 10⁵ (not m ≤ 10⁵ AND n ≤ 10⁵). Total cells bounded, not individual dimensions.

---

## Key Takeaway

**"Maximum path length in a DAG where edges go from lower to higher values in same row/column"** → sort by value + DP propagating row/column bests.

The two key ideas:
1. **Row/column aggregation:** dp[r][c] only needs the maximum dp in its row and column from smaller values — no need to track individual cells.
2. **Batch update for equal values:** prevents illegal same-value transitions by updating after computing all dp values in a group.

**Pattern recognition:** Any "longest path in DAG defined by value ordering + spatial constraints" problem → sort by value, maintain aggregate (max/sum) per dimension, process equal values as a batch.
