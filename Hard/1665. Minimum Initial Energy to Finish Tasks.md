# 1665. Minimum Initial Energy to Finish Tasks
## O(n log n) — Greedy Sort by Slack + Backward Simulation

---

## Problem Statement

- Tasks require `minimum` energy to start and consume `actual` energy.
- Choose the execution order to minimize required initial energy.
- Return the minimum initial energy to complete all tasks.

---

## Core Idea — Sort by Slack, Simulate Backwards

**Key insight**: The optimal execution order sorts tasks by `minimum - actual` (**slack**) in **descending** order — tasks that "demand more buffer" are done first.

**Why?** Exchange argument: if task A and B are adjacent, doing A before B requires `max(minA, minB + actualA)`. Doing B before A requires `max(minB, minA + actualB)`. A should go before B when `minB + actualA < minA + actualB`, i.e., `minA - actualA > minB - actualB`. So sort by `minimum - actual` descending.

**Backward simulation**: Given the correct order, compute required initial energy by processing tasks from last to first:
```
energy = 0  (energy needed after all tasks)
for task in reverse(sorted_tasks):
    energy = max(energy + actual, minimum)
```
At each step: "to have `energy` after this task, I needed at least `energy + actual` before it, but also at least `minimum` to start it."

---

## Full Optimal Implementation (C#)

```csharp
public class Solution {
    public int MinimumEffort(int[][] tasks) {
        // Sort by slack (minimum - actual) descending = execution order
        Array.Sort(tasks, (a, b) => (b[1]-b[0]) - (a[1]-a[0]));

        // Backward simulation
        int energy = 0;
        for (int i = tasks.Length - 1; i >= 0; i--) {
            energy = Math.Max(energy + tasks[i][0], tasks[i][1]);
        }

        return energy;
    }
}
```

## Exchange Argument — Why This Sort is Optimal

For two tasks A `[a, mA]` and B `[b, mB]`:

| Order | Required energy |
|-------|----------------|
| A → B | `max(mA, mB + a)` |
| B → A | `max(mB, mA + b)` |

A before B is better when `max(mA, mB+a) < max(mB, mA+b)`.

The binding constraint is usually the larger term. Comparing: prefer A first when `mA - a > mB - b` → sort by `minimum - actual` descending.

This ensures tasks with high "overhead" (need much more than they cost) run first, while the task that barely needs a buffer runs last.

---

## Backward Simulation — Why It Works

Forward simulation: "given energy E, can I complete all tasks in order?"  
Backward simulation: "what energy do I need before task i, given I need `energy` after it?"

```
Before task [actual, minimum]:
  - Must have at least minimum to start
  - Must have at least energy + actual to have energy left after
  → required = max(energy + actual, minimum)
```

Processing from last to first accumulates the total required initial energy correctly.

## Example Walkthrough

**`tasks=[[1,2],[2,4],[4,8]]`**

Slacks: [2-1=1], [4-2=2], [8-4=4]. Sorted desc: `[[4,8],[2,4],[1,2]]`.

Backward simulation:
```
energy=0
i=2: [1,2] → max(0+1, 2) = 2
i=1: [2,4] → max(2+2, 4) = 4
i=0: [4,8] → max(4+4, 8) = 8
```

Answer: **8** ✓. Execution: [4,8]→[2,4]→[1,2] starting with E=8.

## Pitfalls & Edge Cases

- **Descending sort** (not ascending): tasks with larger slack run first
- **Backward iteration**: `for i in reverse` — last task in sorted order is processed first in the simulation
- **`max(energy + actual, minimum)`**: both constraints must hold — you need `minimum` to start AND `energy + actual` to have `energy` remaining after

---

## Key Takeaway

"Minimum initial energy for tasks with minimum requirements" → **sort by (minimum-actual) descending + backward simulation**:

```
Sort: high slack tasks first (they "demand" more buffer, so execute them first)
Simulate backwards: energy = max(energy + actual, minimum)
```

The exchange argument proves greedy optimality. The backward simulation elegantly computes required energy without forward-simulation guessing.
