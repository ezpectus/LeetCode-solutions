# LeetCode 3389. Minimum Operations to Make Character Frequencies Equal

## 1. Problem Context & Input Constraints

We are given a string `s` consisting only of lowercase English letters. A string `t` is defined as *good* if every unique character present in `t` appears exactly the same number of times (`target` frequency). 

We can modify `s` using three types of operations:
1. **Delete** a character from `s`.
2. **Insert** a character into `s`.
3. **Shift** a character to its immediate next letter in the alphabet (e.g., `'a' -> 'b'`). Crucially, `'z'` cannot be shifted to `'a'`.

Our goal is to find the minimum number of operations required to transform `s` into a *good* string.

### Input Constraints Analysis
* `3 <= s.length <= 2 * 10^4`
* `s` contains only lowercase English letters (`'a'` through `'z'`).

---

## 2. Algorithmic Domain & Feasibility Analysis

* **The Shift-as-Resource Modeling:** A shift operation from character $i$ to $i+1$ can be strategically broken down into a deletion of character $i$ and a simultaneous free insertion for character $i+1$. This means any operation spent deleting a character on the left inherently buffers a "free pool of resources" available for shifting into the current character position.
* **Top-Down DP with Leftover State Tracking:** Rather than evaluating multi-step look-aheads, we can process the alphabet sequentially from left to right. By carrying over the number of characters deleted from the immediate neighbor (`deleted`), we maintain a localized, exact state. The recursion safely processes the 26-letter alphabet without bloating into deep nested cycles.

---

## 3. High-Level Approach Idea

1. **Outer Loop (Target Selection):** We iterate through all possible target frequencies ranging from `0` up to `max_frequency` present in the string.
2. **Top-Down Recursive State:** We define a memoized function `DP(i, deleted, target)` representing the minimum cost to make the remaining alphabet suffix valid.
3. **Local Decision Matrix:** At each character index `i` with initial frequency `x = count[i]`:
   * **If $x \le \text{target}$:** We branch into two optimal choices. We can either completely erase the character (paying $x$ and passing $x$ deletions forward), or match the target frequency by pulling as many free characters as possible from `deleted` before paying for manual insertions.
   * **If $x > \text{target}$:** We are forced to trim down the surplus. We pay for $x - \text{target}$ deletions and forward this entire excess pool to the next character state.

## 4. C# Source Code

```csharp
using System;
using System.Collections.Generic;

public class Solution {
    private int[] count;
    private Dictionary<(int i, int deleted, int target), int> memo;
    
    public int MakeStringGood(string s) {
        count = new int[26];
        foreach (char c in s) {
            count[c - 'a']++;
        }
        
        memo = new Dictionary<(int i, int deleted, int target), int>();
        
        int maxCount = 0;
        foreach (int x in count) {
            maxCount = Math.Max(maxCount, x);
        }
        
        int minOperations = int.MaxValue;
        for (int target = 0; target <= maxCount; target++) {
            minOperations = Math.Min(minOperations, DP(0, 0, target));
        }
        
        return minOperations;
    }
    
    private int DP(int i, int deleted, int target) {
        // Base Case: All letters processed successfully
        if (i == 26) {
            return 0;
        }
        
        var key = (i, deleted, target);
        if (memo.ContainsKey(key)) {
            return memo[key];
        }
        
        int x = count[i];
        int ans;
        
        if (x <= target) {
            // Scenario 1: Wipe out the current character entirely (Target frequency = 0)
            int wipeOut = DP(i + 1, x, target) + x;
            
            // Scenario 2: Retain the current character at target frequency
            int increase = Math.Min(deleted, target - x);
            int fillToTarget = DP(i + 1, 0, target) + target - x - increase;
            
            ans = Math.Min(wipeOut, fillToTarget);
        } else {
            // Scenario 3: Forced trim down of the surplus, passing it forward to the right
            ans = DP(i + 1, x - target, target) + x - target;
        }
        
        memo[key] = ans;
        return ans;
    }
}
```

## 5. Complexity Analysis

* **Time Complexity:** $\mathcal{O}(N \cdot 26) \to \mathcal{O}(N)$
  * **Target Range Bound:** The outer loop checks target frequencies from `0` to `maxCount`, which scales up to $N$ in the worst-case scenario.
  * **State Space Limits:** For each uniform target check, the recursive subproblems visit states defined by `i` (0 to 25) and `deleted`. The value of `deleted` tracks accumulated cascading deletion operations bounded smoothly by the string configuration.
  * **Total Time Scale:** Transitions complete in $\mathcal{O}(1)$ time. The lookup tables remain compact, allowing the solution to comfortably process strings of length $2 \cdot 10^4$ well within milliseconds.

* **Space Complexity:** $\mathcal{O}(N)$ Auxiliary Space
  * **Memoization Storage:** The memory footprint is dominated by the `memo` hash map storing distinct evaluated `(i, deleted, target)` triplets.
  * **Call Stack:** The recursive execution stack matches the length of the alphabet chain, utilizing a constant $\mathcal{O}(26)$ stack depth frames.

## 6. Deep Dive: Approach Details & Transition Mechanics

The key to this refined approach is how it avoids tracking multi-step forward dependencies by focusing purely on immediate history. Instead of looking ahead to see if an empty character slot will need a character pushed to it, the state looks backward via the `deleted` variable.

### The Backward Transfer Window
When moving from letter $i-1$ to letter $i$, the number of characters deleted from $i-1$ is explicitly preserved. Because a deletion cost is identical to a shift cost, we treat these deleted elements as a credit balance that can be inherited by the adjacent character.

* **Wiping out a node:** When we decide to drop a character's frequency to zero, we pay for deleting its original components ($x$), and pass those $x$ elements forward. They become available as an active resource pool for the next character layer.
* **Building a node up:** If we choose to meet the target metric, we assess our deficit: $\text{target} - x$. We prioritize satisfying this deficit via the inheritance pool (`increase = Math.Min(deleted, target - x)`). Any residual deficit that cannot be satisfied by the inherited pool is compensated using explicit, paid insertion operations.

This perspective naturally processes deep cascading shifts across empty sections of the alphabet without explicit look-ahead structures or bloated multivariable matrices.
