# 3600. Maximize Spanning Tree Stability with Upgrades
## O(E log E · log(max_s)) — Binary Search + Kruskal Feasibility Check

---

## Problem Statement

- Graph of `n` nodes. Each edge has strength `s` and a `must` flag.
- **Must-edges** are forced into the spanning tree (can't be upgraded).
- **Optional edges** can be upgraded at most once (doubles strength), with at most `k` total upgrades.
- **Stability** = minimum edge strength in the spanning tree.
- Return the maximum achievable stability, or `-1` if no valid spanning tree exists.

---

## Core Idea — Binary Search on Stability + Greedy Feasibility

**Key insight**: "Is stability ≥ T achievable?" is a monotone predicate — if T is feasible, every T' < T is also feasible. This enables **binary search on the answer**.

**What values to binary search over?**
The stability equals the minimum edge strength in the final tree. That minimum must be one of the original strengths `s` or upgraded strengths `2s` of optional edges. So collect all such values as candidates and binary search over them.

**Feasibility check for threshold T**:
1. **Must-edges**: if any must-edge has `s < T` → impossible (can't upgrade them)
2. **Optional edges**:
   - `s ≥ T` → usable for free
   - `T/2 ≤ s < T` (i.e., `2s ≥ T`) → usable with 1 upgrade
   - `2s < T` → unusable
3. Add all must-edges to DSU. Then greedily add free optional edges (sorted by strength desc). Then add upgrade edges only when needed to connect remaining components, counting upgrades. If upgrades > k → infeasible.
4. Check if all nodes are connected.

**Why greedy within feasibility?** We want to minimize upgrade usage while achieving connectivity. Free edges are always preferable. Among upgrade edges, sorting by strength descending is fine — all that matters is connectivity, not which specific edges we pick (stability is already ≥ T by construction).

**Global -1 check**: before binary search, verify must-edges don't form a cycle (DSU check). If they do → impossible regardless of T.

## Full Optimal Implementation (C#)

```csharp
public class Solution {
    int[] parent, rank_;
    
    int Find(int x) {
        while (parent[x] != x) { parent[x] = parent[parent[x]]; x = parent[x]; }
        return x;
    }
    
    bool Union(int x, int y) {
        int px = Find(x), py = Find(y);
        if (px == py) return false;
        if (rank_[px] < rank_[py]) (px, py) = (py, px);
        parent[py] = px;
        if (rank_[px] == rank_[py]) rank_[px]++;
        return true;
    }
    
    public int MaxStability(int n, int[][] edges, int k) {
        var must     = edges.Where(e => e[3] == 1).Select(e => (e[0],e[1],e[2])).ToList();
        var optional = edges.Where(e => e[3] == 0).Select(e => (e[0],e[1],e[2])).ToList();
        
        // Check must-edges don't form a cycle
        parent = Enumerable.Range(0, n).ToArray();
        rank_  = new int[n];
        foreach (var (u,v,s) in must)
            if (!Union(u, v)) return -1;
        
        // Collect binary search candidates: all s and 2*s for optional
        var candidates = new SortedSet<int>();
        foreach (var e in edges) {
            candidates.Add(e[2]);
            if (e[3] == 0) candidates.Add(2 * e[2]);
        }
        var cands = candidates.ToArray();
        
        bool Feasible(int T) {
            foreach (var (u,v,s) in must)
                if (s < T) return false;
            
            parent = Enumerable.Range(0, n).ToArray();
            rank_  = new int[n];
            foreach (var (u,v,s) in must) Union(u, v);
            
            // Add free optional edges (s >= T)
            foreach (var (u,v,s) in optional.Where(e => e.Item3 >= T)
                                             .OrderByDescending(e => e.Item3))
                Union(u, v);
            
            // Add upgrade edges only when needed
            int upgrades = 0;
            foreach (var (u,v,s) in optional.Where(e => e.Item3 < T && 2*e.Item3 >= T)
                                             .OrderByDescending(e => 2*e.Item3)) {
                if (Find(u) != Find(v)) {
                    upgrades++;
                    if (upgrades > k) return false;
                    Union(u, v);
                }
            }
            
            int root = Find(0);
            return Enumerable.Range(0, n).All(i => Find(i) == root);
        }
        
        int lo = 0, hi = cands.Length - 1, res = -1;
        while (lo <= hi) {
            int mid = (lo + hi) / 2;
            if (Feasible(cands[mid])) { res = cands[mid]; lo = mid + 1; }
            else hi = mid - 1;
        }
        return res;
    }
}
```

## Complexity

| **Metric** | **Value** | **Notes** |
|---|---|---|
| **Time Complexity** | **O(E log E · log(max_s))** | Binary search over O(E) candidates × Kruskal O(E log E) per check |
| **Space Complexity** | **O(n + E)** | DSU + edge lists |

With `E, n ≤ 10⁵` and `max_s ≤ 10⁵`: ~2×10⁵ candidates × 10⁵ log(10⁵) ≈ manageable.

---

## Binary Search Correctness

**Monotonicity**: if stability ≥ T is feasible with upgrade budget k, then stability ≥ T-1 is also feasible (same spanning tree works, or an even better one). So the predicate `feasible(T)` is monotone non-increasing → binary search is valid.

**Candidate set**: the optimal stability equals the minimum edge in the best spanning tree. That minimum is always one of the `s` or `2s` values of edges in the tree. Checking all O(E) such values covers all possible answers.

---

## Feasibility — Why Greedy is Optimal

Within `Feasible(T)`:
- We want to connect all nodes using edges with effective strength ≥ T.
- Adding free edges first can only help — they never "waste" upgrade budget.
- Upgrade edges are only added when strictly needed for connectivity (Kruskal-style).
- Since all selected edges have effective strength ≥ T, the spanning tree's stability ≥ T.
- We count upgrades only for edges that actually enter the spanning tree.

## Example Walkthrough

**Example 2**: `n=3, edges=[[0,1,4,0],[1,2,3,0],[0,2,1,0]], k=2`

No must-edges. Candidates: `{1,2,3,4,6,8}`. Binary search checks descending:

**T=8**: free edges (s≥8): none. Upgrade edges (2s≥8): only [0,1] (2×4=8). Need 2 more edges but only 1 upgrade edge ≥ T → can't connect → ✗

**T=6**: free edges (s≥6): none. Upgrade edges (2s≥6): [0,1](2×4=8≥6), [1,2](2×3=6≥6). Need 2 edges, use 2 upgrades → connected with upgrades=2 ≤ k=2 → ✓

Result: `6` ✓

## Pitfalls & Edge Cases

- **Must-edges forming a cycle** — check before anything else with a clean DSU; return -1 immediately
- **Must-edge with `s < T`** — must be in tree but can't upgrade → infeasible; check first in `Feasible`
- **Disconnected graph** — even with upgrades, if not enough edges exist → `Find(0) != Find(i)` for some `i` → return -1
- **`k = 0`** — only free optional edges usable; upgrade edge loop exits immediately when `upgrades > 0`
- **Upgrade edges sorted by `2*s` descending** — consistent with "pick best available edges first" for Kruskal

---

## Key Takeaway

"Maximize minimum edge in spanning tree with optional upgrades" → **binary search on answer + Kruskal feasibility**:

```
Binary search: Is stability ≥ T achievable?
  Feasibility: 
    1. Must-edges all have s ≥ T?
    2. Build spanning tree: free optional edges first, upgrade edges only when needed
    3. Connected? Upgrades ≤ k?
```

Separating "what's the answer?" (binary search) from "is this answer possible?" (greedy Kruskal) is the key pattern for max-min spanning tree problems with constraints.
