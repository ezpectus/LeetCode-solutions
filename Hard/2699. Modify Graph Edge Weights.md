# 2699. Modify Graph Edge Weights
## O((V + E) log V) — Dual-Pass Dijkstra with Reverse Distance Target Matching

### Problem Statement
You are given an undirected weighted connected graph with $n$ nodes. Some edges have a positive weight, while others are assignable and marked with a weight of `-1`. 

Your goal is to modify all `-1` edges by assigning them a positive integer weight in the range $[1, 2 \times 10^9]$ such that the absolute shortest distance between a given `source` and `destination` becomes exactly equal to a given `target`. 

Return the modified edges array if a valid configuration exists; otherwise, return an empty array `[]`.

---

### Core Idea — The Reverse Distance Bound Strategy
We cannot greedily assign weights to `-1` edges without knowing how much each node still needs to travel to reach the `destination`. The most elegant way to solve this is to measure distances from both ends:

1. **Pass 1 (From Destination):** Run a reverse Dijkstra from the `destination` node to all other nodes. During this pass, we temporarily treat all modifiable (`-1`) edges as having their minimum possible valid weight, which is `1`. This gives us the absolute baseline minimum distance (`distToDest[i]`) that any node `i` requires to reach the destination under the most optimistic conditions.
2. **Pass 2 (From Source):** Run a forward Dijkstra starting from the `source` node. As we traverse and expand the shortest-path tree, we dynamically fix the weights of any `-1` edges we encounter. 

When standing at node `u` and looking at an unassigned `-1` edge leading to node `v`, we know:
* The distance from `source` to `u` is `distFromSource[u]`.
* The minimum possible distance from `v` to `destination` is `distToDest[v]`.

To force the shortest path going through this exact edge `(u, v)` to sum up to exactly `target`, the edge's weight must satisfy:
$$\text{Weight} = \text{target} - \text{distFromSource}[u] - \text{distToDest}[v]$$

If this calculated weight is less than 1, we clip it to `1` (the minimum allowed value) to keep the graph valid. If it is greater than 1, it perfectly absorbs the remaining gap required to hit the `target`.

---

### Algorithm Step-by-Step
1. **Build Adjacency List:** Map each node to its neighbors, storing the *index* of the edge rather than just the weight, so we can modify the original `edges` array directly.
2. **Reverse Pass:** Run Dijkstra from `destination` treating all `-1` edges as `1`. Save these distances into a `distToDest` array.
   * *Early Exit Check:* If `distToDest[source] > target`, it means that even if every single modifiable edge in the entire graph is set to its absolute minimum value of 1, the path is still too long. Hitting the target is physically impossible. Return `[]`.
3. **Forward Pass:** Run Dijkstra from `source`. When checking a neighbor `v` from current node `u` across an edge that is currently `-1`, update its weight in the master array to $\max(1, \text{target} - \text{distFromSource}[u] - \text{distToDest}[v])$.
4. **Final Validation:** If `distFromSource[destination]` does not equal `target` at the end of the second pass, it means a static path of positive edges already exists that is strictly shorter than `target`, making it impossible to fix. Return `[]`.
5. **Clean Up:** Fill any remaining untouched `-1` edges with a massive safe buffer value (like $2 \times 10^9$) to ensure they do not create unwanted shortcuts later.

### Full Optimal Implementation (C#)

```csharp
using System;
using System.Collections.Generic;

public class Solution {
    public int[][] ModifiedGraphEdges(int n, int[][] edges, int source, int destination, int target) {
        // Step 1: Construct the adjacency list using edge indices
        List<int>[] adj = new List<int>[n];
        for (int i = 0; i < n; i++) {
            adj[i] = new List<int>();
        }
        
        for (int i = 0; i < edges.Length; i++) {
            adj[edges[i][0]].Add(i);
            adj[edges[i][1]].Add(i);
        }

        // Step 2: First Dijkstra pass from destination (treating all -1 edges as 1)
        long[] distToDest = new long[n];
        Array.Fill(distToDest, long.MaxValue);
        PriorityQueue<int, long> pq = new PriorityQueue<int, long>();
        
        distToDest[destination] = 0;
        pq.Enqueue(destination, 0);
        
        while (pq.Count > 0) {
            pq.TryDequeue(out int u, out long d);
            if (d > distToDest[u]) continue;
            
            foreach (int edgeIdx in adj[u]) {
                int v = edges[edgeIdx][0] == u ? edges[edgeIdx][1] : edges[edgeIdx][0];
                long weight = edges[edgeIdx][2] == -1 ? 1 : edges[edgeIdx][2];
                
                if (distToDest[u] + weight < distToDest[v]) {
                    distToDest[v] = distToDest[u] + weight;
                    pq.Enqueue(v, distToDest[v]);
                }
            }
        }

        // Base structural check: If the absolute minimum possible path exceeds target, abort
        if (distToDest[source] > target) {
            return new int[0][];
        }

        // Step 3: Second Dijkstra pass from source (dynamically calculating -1 weights)
        long[] distFromSource = new long[n];
        Array.Fill(distFromSource, long.MaxValue);
        
        distFromSource[source] = 0;
        pq.Clear();
        pq.Enqueue(source, 0);
        
        while (pq.Count > 0) {
            pq.TryDequeue(out int u, out long d);
            if (d > distFromSource[u]) continue;
            
            foreach (int edgeIdx in adj[u]) {
                int v = edges[edgeIdx][0] == u ? edges[edgeIdx][1] : edges[edgeIdx][0];
                
                // If it's a modifiable edge, dynamically establish its target absorption weight
                if (edges[edgeIdx][2] == -1) {
                    long requiredWeight = target - distFromSource[u] - distToDest[v];
                    edges[edgeIdx][2] = (int)Math.Max(1L, requiredWeight);
                }
                
                long weight = edges[edgeIdx][2];
                if (distFromSource[u] + weight < distFromSource[v]) {
                    distFromSource[v] = distFromSource[u] + weight;
                    pq.Enqueue(v, distFromSource[v]);
                }
            }
        }

        // Step 4: Verification Check
        if (distFromSource[destination] != target) {
            return new int[0][];
        }

        // Step 5: Seal all remaining untouched -1 edges to a massive safe value
        for (int i = 0; i < edges.Length; i++) {
            if (edges[i][2] == -1) {
                edges[i][2] = 2000000000; // 2 * 10^9
            }
        }

        return edges;
    }
}
```

| **Time Complexity** | $O((V + E) \log V)$ | We execute exactly two isolated runs of standard Dijkstra's algorithm. For $n \le 100$, this executes almost instantaneously ($\approx 10^4$ operations max). |
| **Space Complexity** | $O(V + E)$ | Required to build the index-mapped adjacency list and maintain primitive distance state tracks across all $n$ nodes. |

---

### Pitfalls & Edge Cases
* **The Static Bypass Failure:** If a graph already contains a static path made entirely of fixed positive edges whose total weight is *smaller* than `target`, we cannot make the shortest distance equal to `target`. This is because we can only increase or assign weights to `-1` edges, which does nothing to change the existing shorter route. Our final verification `distFromSource[destination] != target` flags this instantly.
* **Type Headroom Safeguards:** The `target` value can scale up to $10^9$, and assignable weights can stretch up to $2 \times 10^9$. Performing arithmetic relaxation updates using standard 32-bit signed integers will cause immediate overflow wrap-arounds into negative bounds. Using a `long[]` array tracks state safety perfectly.
* **Dangling Modifiable Edges:** Any `-1` edges that do not lie along the active shortest path from `source` to `destination` are left untouched by the traversal. Leaving them as `-1` breaks LeetCode constraints. Setting them to a max ceiling of $2 \times 10^9$ ensures they are completely neutralized.

---

### Key Takeaway
> When tasked with tuning edge parameters to satisfy a global equilibrium path property, avoid blind updates. Run a reverse tracking pass from the destination point to establish baseline optimization bounds, and then use that lookahead context to calibrate parameters during the forward pass.

* **Pattern recognition:** "Modify negative or unassigned graph weights to hit exact target shortcut path" -> Dual-Pass Reverse Bound Dijkstra.
