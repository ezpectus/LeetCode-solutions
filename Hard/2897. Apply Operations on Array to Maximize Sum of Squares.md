# 2897. Apply Operations on Array to Maximize Sum of Squares
## O(n · B) — Bit-Count Invariant + Greedy Bit Packing

---

## Problem Statement

Array `nums`, operation: choose indices `i`, `j` and set `nums[i] = nums[i] AND nums[j]`, `nums[j] = nums[i] OR nums[j]`. Any number of operations allowed. Choose `k` elements from the final array. Return maximum sum of squares mod `10⁹ + 7`.

---

## Core Insight #1 — The AND/OR Operation Preserves Per-Bit Counts

For any bit position `b`, look at how many array elements have bit `b` set. Under the operation:

| Before (bit b) | After AND | After OR | Count change |
|---|---|---|---|
| i=1, j=1 | 1 | 1 | 2 → 2 (unchanged) |
| i=1, j=0 | 0 | 1 | 1 → 1 (unchanged) |
| i=0, j=1 | 0 | 1 | 1 → 1 (unchanged) |
| i=0, j=0 | 0 | 0 | 0 → 0 (unchanged) |

In every case, the count of elements with bit `b` set is **unchanged**. No matter how many operations we apply, `cnt[b]` = "number of elements with bit b set" is a **fixed invariant** of the array.

---

## Core Insight #2 — Bits Are Freely Redistributable Within Their Count Budget

Since AND/OR operations are transitive, we can apply a sequence of operations to move bits around arbitrarily. The **only** constraint on the final array state is: for each bit `b`, exactly `cnt[b]` elements have bit `b` set. Any assignment satisfying these counts is achievable.

Proof sketch: to move bit `b` from element `i` to element `j` (where `j` doesn't have it), apply `(i, j)` → `nums[j]` gets the OR (gains bit b), `nums[i]` gets AND (loses bit b). Repeat for any redistribution.

---

## Core Insight #3 — Greedy Packing Maximizes Sum of Squares

We want to pick `k` elements to maximize sum of squares. Since elements are under our control (up to bit budgets), we build the optimal `k` elements from scratch.

**Key observation:** For any set of bits, concentrating them into fewer elements always increases the sum of squares:

```
(a + b)² + 0  >  a² + b²    when a, b > 0
```

This means we should pack as many bits as possible into the same elements.

**Optimal assignment:** Label our k chosen elements 1, 2, ..., k (from largest to smallest). For each bit `b` with budget `cnt[b]`:
- Assign it to elements 1 through `min(cnt[b], k)` — the "richest" ones.

In other words:
```
element j gets bit b  ⟺  cnt[b] ≥ j
```

This is optimal because:
- Elements already large (many high bits set) gain the most square value from additional bits.
- Any reassignment moving a bit from element j to element j' > j (poorer) strictly reduces sum of squares.

---

## Algorithm

```
1. Count bits:
   for each x in nums:
       for each bit b in 0..29:
           if bit b is set in x: cnt[b]++

2. Build k optimal elements:
   for j = 1 to k:
       val_j = 0
       for each bit b in 0..29:
           if cnt[b] >= j: val_j |= (1 << b)
       ans += val_j²

3. Return ans mod (10⁹ + 7)
```

---

## Full Implementation (C#)

```csharp
public class Solution {
    public int MaxSum(IList<int> nums, int k) {
        const long MOD = 1_000_000_007;

        // Count how many elements have each bit set
        int[] cnt = new int[30]; // nums[i] <= 10^9 < 2^30
        foreach (int x in nums)
            for (int b = 0; b < 30; b++)
                if (((x >> b) & 1) == 1) cnt[b]++;

        // Build k optimal elements greedily
        long ans = 0;
        for (int j = 1; j <= k; j++) {
            long val = 0;
            for (int b = 0; b < 30; b++)
                if (cnt[b] >= j) val |= (1L << b);
            ans = (ans + val * val % MOD) % MOD;
        }

        return (int)ans;
    }
}
```

## Example Walkthrough

### Example 1 — `nums = [2,6,5,8]`, `k = 2`

Binary: `2=0010`, `6=0110`, `5=0101`, `8=1000`

| Bit | Elements with this bit | cnt[b] |
|---|---|---|
| 0 | 5 | 1 |
| 1 | 2, 6 | 2 |
| 2 | 6, 5 | 2 |
| 3 | 8 | 1 |

Build k=2 elements:

| j | Gets bits where cnt[b]≥j | val |
|---|---|---|
| 1 | bits 0,1,2,3 (all cnt≥1) | 1+2+4+8 = **15** |
| 2 | bits 1,2 (cnt≥2) | 2+4 = **6** |

Answer = 15² + 6² = 225 + 36 = **261** ✓

Verification: start from [2,6,5,8]:
- Op(0,3): nums[0]=2&8=0, nums[3]=2|8=10 → [0,6,5,10]
- Op(2,3): nums[2]=5&10=0, nums[3]=5|10=15 → [0,6,0,15]
- Pick 15 and 6. ✓

---

### Example 2 — `nums = [4,5,4,7]`, `k = 3`

Binary: `4=100`, `5=101`, `4=100`, `7=111`

| Bit | cnt |
|---|---|
| 0 | 2 (from 5,7) |
| 1 | 1 (from 7) |
| 2 | 4 (from all) |

Build k=3 elements:

| j | Gets bits where cnt[b]≥j | val |
|---|---|---|
| 1 | bits 0,1,2 | 1+2+4 = **7** |
| 2 | bits 0,2 | 1+4 = **5** |
| 3 | bit 2 | 4 = **4** |

Answer = 7² + 5² + 4² = 49 + 25 + 16 = **90** ✓

---

## Why This Beats Any Other Arrangement

For Example 1 with k=2, alternative arrangements:
- `{10, 7}`: 100 + 49 = 149. Worse.
- `{15, 5}`: 225 + 25 = 250. Worse.
- `{14, 7}`: 196 + 49 = 245. Worse.

The greedy bit-packing into `{15, 6}` wins because it concentrates high bits maximally into the first element before moving on.

## Pitfalls & Edge Cases

- **30 bits, not 32:** `nums[i] ≤ 10⁹ < 2³⁰`. Use `b in [0, 29]`. Using 32 accidentally counts bits 30 and 31 which are always 0 — harmless but wasteful.

- **Modular arithmetic on val²:** `val ≤ 2³⁰ − 1 ≈ 10⁹`. `val² ≤ 10¹⁸`, which fits in `long` (max ~9.2×10¹⁸). Apply `% MOD` immediately after `val * val` to prevent accumulation overflow in `ans`.

- **k = 1:** Only element 1 is built — it gets all bits with cnt≥1, i.e., the OR of all elements. Correct (the OR of all elements is the maximum achievable single value).

- **k = n:** All elements are chosen. The construction gives them all their bits greedily. Since we can redistribute freely, this still finds the optimal assignment.

- **All bits concentrated in one element:** If only one element has any bit set (all others 0), cnt[b] ≤ 1 for all b. Then only element j=1 gets any bits, and elements j=2..k are all 0. Answer = val₁² = original_max². Correct — we can't spread bits we don't have.

## Key Takeaway

This is a **bit-count invariant + greedy packing** pattern. When an operation preserves per-bit counts, the problem reduces to: "given these bit budgets, how do we optimally allocate them?" For maximizing sum of squares, the answer is always greedy packing — give each bit to the already-richest elements first. The invariant insight is the hard part; the greedy follows immediately from the convexity of x².

**Pattern recognition:** "AND/OR operations on arrays" → check if per-bit counts are invariant (they always are for AND/OR) → freely redistribute bits → optimize the target function under bit-count constraints. Appears in variants like "maximize XOR sum", "maximize largest element", etc.
