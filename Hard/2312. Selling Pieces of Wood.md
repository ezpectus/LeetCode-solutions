# 2312. Selling Pieces of Wood  
## O(m × n × (m + n)) — 2D Dynamic Programming with Full Cut Enumeration

---

## Problem Statement

You are given:

- A rectangular wooden board of size **m × n**.
- A list of sellable piece types, each described as:
  - height `h`
  - width `w`
  - price `p`

You may:

- Cut the board **horizontally** or **vertically** any number of times.
- Sell any resulting piece **only if** its exact `(h, w)` size appears in the price list.
- Otherwise, you may continue cutting it into smaller pieces.

Goal:  
**Maximize the total revenue** obtained from selling all resulting pieces.

Constraints:

- `1 ≤ m, n ≤ 200`
- Up to `10⁴` price entries
- Cuts are free
- Rotation is NOT allowed: `(h, w)` ≠ `(w, h)` unless explicitly listed

---

## Core Idea — 2D DP Over All Sub-Rectangles

Define:

```
dp[h][w] = maximum revenue obtainable from a piece of size h × w
```

This is a classic **2D generalization of rod cutting**.

### Initialization

If a piece `(h, w)` has a listed price, set:

```
dp[h][w] = price(h, w)
```

If not listed, default is `0` — meaning the piece has no direct selling value unless cut.

### Transitions

For each `(h, w)`:

#### 1. Horizontal cuts  
Try cutting at every row `cut`:

```
dp[h][w] = max(dp[h][w], dp[cut][w] + dp[h - cut][w])
```

#### 2. Vertical cuts  
Try cutting at every column `cut`:

```
dp[h][w] = max(dp[h][w], dp[h][cut] + dp[h][w - cut])
```

### Order of computation

We iterate:

```
for h = 1..m
  for w = 1..n
```

This ensures all smaller subproblems are solved before larger ones.

---

## Full Optimal Implementation (C#)

```csharp
public class Solution {
    public long SellingWood(int m, int n, int[][] prices) {
        long[,] dp = new long[m + 1, n + 1];

        // Initialize direct sell prices
        foreach (var p in prices)
            dp[p[0], p[1]] = p[2];

        // DP over all sub-rectangles
        for (int h = 1; h <= m; h++) {
            for (int w = 1; w <= n; w++) {

                // Horizontal cuts
                for (int cut = 1; cut < h; cut++)
                    dp[h, w] = Math.Max(dp[h, w], dp[cut, w] + dp[h - cut, w]);

                // Vertical cuts
                for (int cut = 1; cut < w; cut++)
                    dp[h, w] = Math.Max(dp[h, w], dp[h, cut] + dp[h, w - cut]);
            }
        }

        return dp[m, n];
    }
}
```

---

## Complexity

| Metric | Value | Notes |
|---|---|---|
| **Time Complexity** | **O(m × n × (m + n))** | For each (h,w) we try all horizontal + vertical cuts |
| **Space Complexity** | **O(m × n)** | DP table storing best revenue for each sub-rectangle |
| **Max operations** | ~16M | For m,n ≤ 200 — extremely fast |

## Why This Works — Example Walkthrough

### Example  
`m = 3, n = 5`  
`prices = [[1,4,2], [2,2,7], [2,1,3]]`

Initialize:

```
dp[1][4] = 2
dp[2][2] = 7
dp[2][1] = 3
```

Now compute larger rectangles.

### Example: dp[2][5]

Try all vertical cuts:

- cut at 1: dp[2][1] + dp[2][4] = 3 + ?
- cut at 2: dp[2][2] + dp[2][3] = 7 + ?
- cut at 3: dp[2][3] + dp[2][2] = ? + 7
- cut at 4: dp[2][4] + dp[2][1] = ? + 3

Eventually best = **10** (e.g., 7 + 3).

### Example: dp[3][5]

Try all horizontal + vertical cuts:

- Best decomposition uses:
  - two 2×2 pieces (7 each)
  - one 2×1 piece (3)
  - one 1×4 piece (2)

Total:

```
7 + 7 + 3 + 2 = 19
```

Thus:

```
dp[3][5] = 19
```

## Pitfalls & Edge Cases

- **Selling as-is must be considered**  
  Always initialize `dp[h][w]` with the direct price before cuts.

- **No rotation allowed**  
  `(h, w)` is not the same as `(w, h)` unless both appear in price list.

- **Zero-value pieces**  
  If a piece has no price and cannot be cut into valuable pieces → dp remains 0.

- **Symmetry optimization**  
  You can cut only up to `h/2` and `w/2`, but full enumeration is simpler and still fast.

- **Large prices**  
  Use `long` — prices can sum up to millions.

- **Order matters**  
  Must compute from small to large dimensions.

---

## Alternative Approaches (Not Needed but Useful to Know)

### 1. Memoized DFS  
Possible but slower due to recursion overhead and repeated state checks.

### 2. Knuth-like optimization  
Not applicable — no quadrangle inequality, cuts are independent.

### 3. DP with rotation  
If rotation were allowed, you'd need to check both `(h,w)` and `(w,h)`.

### 4. Sparse DP  
If price list is small, you still need full DP because cuts generate all sizes.

---

## Key Takeaway

This is a **2D generalization of rod cutting**:

- Each rectangle `(h, w)` is solved by combining optimal solutions of its sub-rectangles.
- DP ensures every possible cut is explored.
- Direct prices serve as base cases.
- The final answer is simply `dp[m][n]`.

A clean, deterministic, bottom-up DP that guarantees the optimal revenue.
