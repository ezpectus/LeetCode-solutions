# 3041. Maximize Consecutive Elements in an Array After Modification
## O(n log n) — Sort + Two-Option DP with Saved-State Trick to Prevent Double-Counting

---

## Problem Statement

Array `nums` of positive integers. Optionally increase each element by at most 1 (0 or +1, independently). Then select a subset of the resulting array that forms a consecutive integer sequence when sorted. Return the maximum subset size.

---

## Core Insight — Two Choices Per Element, One Consecutive Sequence

Each element `v = nums[i]` becomes either `v` (keep) or `v+1` (increment). After all choices, we want the longest consecutive run `[a, a+1, a+2, …, a+k]` that can be assembled using at most one element per target value.

**Key observation:** define `dp[t]` = length of the longest consecutive sequence ending at value `t` using elements processed so far. For a new element `v`, it can:

- **Option A — stay as `v`**: extends the sequence ending at `v−1`.
  `dp[v] = max(dp[v], dp[v-1] + 1)`
- **Option B — become `v+1`**: extends the sequence ending at `v`.
  `dp[v+1] = max(dp[v+1], dp[v] + 1)`

If we sort `nums` first and process elements in ascending order, all potential predecessors (values `v−1`, `v`) are already finalized when we process `v`.

---

## Critical Subtlety — Saving Old `dp[v]` Before Option A

**The bug:** naively applying both options in order (A then B) causes double-counting:

```
// WRONG for nums = [1]:
dp[1] = dp[0] + 1 = 1    // option A: element "becomes 1"
dp[2] = dp[1] + 1 = 2    // option B: uses UPDATED dp[1] → element counted TWICE
```

After Option A, `dp[1]` now represents a sequence that *includes this element* (as value 1). Using that for Option B would pretend this element is simultaneously value 1 (in the sequence ending at 1) and value 2 (extending that sequence). One element cannot fill two consecutive positions.

**The fix:** save `old_dp_v = dp[v]` **before** Option A modifies it, then use `old_dp_v` for Option B:

```
old_dp_v = dp[v]
dp[v]    = max(dp[v],    dp[v-1]    + 1)   // option A: keep as v
dp[v+1]  = max(dp[v+1], old_dp_v   + 1)   // option B: increase to v+1, using pre-A dp[v]
```

This ensures Option A and Option B each claim the element for a different value, never both at once.

---

## Why Sorting Is Required

Processing in sorted order guarantees that when we compute `dp[v] = dp[v−1] + 1` for some element, `dp[v−1]` is already fully computed — all elements that could contribute to sequences ending at `v−1` have been processed. Without sorting, we might read a stale `dp[v−1]` that hasn't yet accounted for elements equal to `v−1`.

**Multiple duplicate values are handled correctly:** if `k` elements all have value `v`, processing them in order:
- First element: `old_dp_v = 0` (no prior sequence ends at v). Option A sets `dp[v] = dp[v-1]+1`. Option B (using old=0) sets `dp[v+1] = 1`.
- Second element: `old_dp_v = dp[v-1]+1` (the sequence from the first element). Option A doesn't improve `dp[v]`. Option B uses `old_dp_v = dp[v-1]+1`, setting `dp[v+1] = dp[v-1]+2`.
- Each subsequent duplicate can extend Option B by one more (first element "fills" value v, second "fills" value v+1, etc.), correctly forming [… v, v+1, v+2 …] from k elements.

---

## Algorithm

```
1. Sort nums ascending.
2. dp = empty dictionary (default value 0).
3. ans = 1.
4. For each v in nums (sorted):
     old_dv = dp[v]                               // save before modification
     dp[v]   = max(dp[v],   dp.get(v-1, 0) + 1)  // option A: element stays as v
     dp[v+1] = max(dp[v+1], old_dv + 1)           // option B: element becomes v+1
     ans = max(ans, dp[v], dp[v+1])
5. Return ans.
```

---

## Full Optimal Implementation (C#)

```csharp
public class Solution {
    public int MaxSelectedElements(int[] nums) {
        Array.Sort(nums);
        var dp = new Dictionary<int, int>();
        int ans = 1;

        foreach (int v in nums) {
            int oldV = dp.GetValueOrDefault(v, 0);   // save before option A

            // Option A: element stays as v, extends sequence ending at v-1
            int newV = dp.GetValueOrDefault(v - 1, 0) + 1;
            if (newV > dp.GetValueOrDefault(v, 0))
                dp[v] = newV;

            // Option B: element becomes v+1, extends sequence ending at v (old state)
            int newV1 = oldV + 1;
            if (newV1 > dp.GetValueOrDefault(v + 1, 0))
                dp[v + 1] = newV1;

            ans = Math.Max(ans, Math.Max(dp[v], dp[v + 1]));
        }
        return ans;
    }
}
```

For n = 10⁵: sort ≈ 1.7M comparisons + 10⁵ DP steps. Trivially fast.

## Example Walkthroughs

### Example 1 — `nums = [2,1,5,1,1]`, Expected: 3

**Sorted:** `[1, 1, 1, 2, 5]`

| Step | v | old_dv | dp[v-1]+1 | dp[v] after A | dp[v+1] after B | ans |
|------|---|--------|-----------|---------------|-----------------|-----|
| i=0 | 1 | 0 | 0+1=1 | **1** | 0+1=1 → dp[2]=1 | 1 |
| i=1 | 1 | 1 | 0+1=1 | max(1,1)=1 | 1+1=2 → dp[2]=**2** | 2 |
| i=2 | 1 | 1 | 0+1=1 | max(1,1)=1 | 1+1=2 → dp[2]=max(2,2)=2 | 2 |
| i=3 | 2 | 2 | dp[1]+1=1+1=2 | max(2,2)=2 | 2+1=3 → dp[3]=**3** | **3** |
| i=4 | 5 | 0 | dp[4]+1=0+1=1 | dp[5]=1 | 0+1=1 → dp[6]=1 | 3 |

**Answer = 3** ✓

**Interpretation of `dp[3] = 3`:** built via sequence [1, 2, 3]:
- Element at index 3 (value 1) → stays 1. `dp[1]=1`.
- Element at index 4 (value 1) → increases to 2. `dp[2]=2` (extends [1]).
- Element at index 0 (value 2) → increases to 3. `dp[3]=3` (extends [1,2]).

*(In the original problem's 0-indexed terms: nums[3]=1 stays 1, nums[4]=1 becomes 2, nums[0]=2 becomes 3.)*

### Example 2 — `nums = [1,4,7,10]`, Expected: 1

**Sorted:** `[1, 4, 7, 10]`

| Step | v | old_dv | dp[v-1]+1 | dp[v] | dp[v+1] | ans |
|------|---|--------|-----------|-------|---------|-----|
| i=0 | 1 | 0 | 0+1=**1** | 1 | 0+1=1→dp[2]=1 | 1 |
| i=1 | 4 | 0 | 0+1=**1** | 1 | 0+1=1→dp[5]=1 | 1 |
| i=2 | 7 | 0 | 0+1=**1** | 1 | 0+1=1→dp[8]=1 | 1 |
| i=3 | 10 | 0 | 0+1=**1** | 1 | 0+1=1→dp[11]=1 | 1 |

**Answer = 1** ✓

No two elements (even after ±0/+1 adjustment) can become adjacent integers — gaps of 3 between consecutive elements are never bridgeable with at most +1. ✓

---

### Why the Bug Matters — `nums = [1]` Traced

**Without the fix:**
```
dp[1] = dp[0]+1 = 1   // option A
dp[2] = dp[1]+1 = 2   // option B uses UPDATED dp[1]=1
```
→ reports answer 2 from a single element. **Wrong.**

**With the fix:**
```
old_dp_1 = 0
dp[1] = dp[0]+1 = 1   // option A
dp[2] = old_dp_1+1 = 1  // option B uses old dp[1]=0
```
→ reports answer 1. **Correct.** (The element becomes either 1 or 2, contributing to a sequence of length 1.)

---

### Duplicate-Value Mechanics — `nums = [1, 1, 1]` Traced

| Step | v | old_dv | dp[0]+1 | dp[1] | dp[2] | Meaning |
|------|---|--------|---------|-------|-------|---------|
| i=0 | 1 | 0 | 1 | **1** | 1 | 1st element: stays→[1], or →2 |
| i=1 | 1 | 1 | 1 | max(1,1)=1 | max(1,**2**)=2 | 2nd element becomes 2, extends [1] |
| i=2 | 1 | 1 | 1 | max(1,1)=1 | max(2,2)=2 | 3rd element also becomes 2, no improvement |

Max = 2. Correct: three 1s can form at best [1, 2] — two of them (one stays 1, one becomes 2). The third is redundant since 2 is already filled.

---

### Reaching Beyond Gaps — `nums = [1, 1, 2]` Traced

| Step | v | old_dv | dp[v-1]+1 | dp[v] | dp[v+1] |
|------|---|--------|-----------|-------|---------|
| i=0 | 1 | 0 | 1 | 1 | dp[2]=1 |
| i=1 | 1 | 1 | 1 | 1 | dp[2]=max(1,**2**)=2 |
| i=2 | 2 | 2 | dp[1]+1=2 | max(2,2)=2 | dp[3]=**3** |

Max = 3. Correct: [1, 2, 3] using first 1 (stays 1), second 1 (→2), and 2 (→3). ✓

## Pitfalls & Edge Cases

- **`old_dv` must be captured before any dp[v] write.** Placing the save after Option A produces the same double-counting bug. In C# with a dictionary, ensure `oldV = dp.GetValueOrDefault(v, 0)` is the **first** line of the loop body. ✓

- **Option B uses `old_dv`, not the updated `dp[v]`.** Even though Option A might increase `dp[v]`, Option B uses the pre-A value. This is the core correctness constraint. ✓

- **Dictionary default is 0** (not −1 or similar). Sequences of length 0 represent "no sequence" — a valid base case, since any single element forms a sequence of length 1 via `0 + 1`. ✓

- **Answer initialized to 1, not 0.** At least one element can always be selected (the question asks for non-empty subsets, and any single element trivially forms a consecutive sequence of length 1). ✓

- **Max value reach:** `dp[v+1]` for `v = max(nums)` requires accessing key `max(nums)+1 ≤ 10⁶+1`. The dictionary handles this naturally (no fixed array bounds needed). ✓

- **Why not an array instead of dictionary?** Array of size 10⁶+2 (≈ 4 MB for int[]) is perfectly feasible and faster. Dictionary is used here for clarity; the array version replaces `dp.GetValueOrDefault(k, 0)` with `dp[k]` on a zeroed array. ✓

## Key Takeaway

**"Choose one of two values per element, maximize length of consecutive run"** → sort elements → one-pass DP where each element has two options (stay or +1). The state `dp[t]` = best run ending at t. Transition is O(1) per element; the only subtlety is preventing self-extension (using the pre-update dp[v] for Option B, not the post-Option-A value).

**Pattern recognition:** "Each element independently chooses from {v, v+1}" and we want a consecutive integer sequence → sort + DP over sorted elements with a two-option transition. The "save old state" trick prevents double-counting and applies whenever one element's two options could create a chain (A's output feeding B's input for the same element). This is a recurring pattern in knapsack and interval DP problems where forward-extension must not reuse the same item.
