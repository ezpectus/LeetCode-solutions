# 3850. Count Sequences to K

## 1. Idea & Structural Analysis

### 1.1 Core Idea & Mathematical Basis
Since all array elements fulfill `1 <= nums[i] <= 6`, their prime factorizations consist purely of the prime bases 2, 3, and 5:
- `1 = 2^0 * 3^0 * 5^0`
- `2 = 2^1 * 3^0 * 5^0`
- `3 = 2^0 * 3^1 * 5^0`
- `4 = 2^2 * 3^0 * 5^0`
- `5 = 2^0 * 3^0 * 5^1`
- `6 = 2^1 * 3^1 * 5^0`

The final rational value `val` can be represented entirely as `2^(p2) * 3^(p3) * 5^(p5)`, where exponents `p2`, `p3`, and `p5` are integers (positive for net multiplication, negative for net division).
Similarly, the target integer `k` must also be factorizable into prime powers of 2, 3, and 5. If `k` contains any prime factor other than 2, 3, or 5 (e.g., 7, 11, etc.), it is impossible to reach `k`, so the answer is immediately 0.

### 1.2 Meet-in-the-Middle Optimization
At each element, we have 3 choices:
1. Multiply by `nums[i]` -> add `nums[i]` prime exponents to `(p2, p3, p5)`.
2. Divide by `nums[i]` -> subtract `nums[i]` prime exponents from `(p2, p3, p5)`.
3. Leave unchanged -> add `(0, 0, 0)` to `(p2, p3, p5)`.

Brute force exploration gives 3^N states. For N = 19, 3^19 = 1,162,261,467, which causes Time Limit Exceeded (TLE).
By applying **Meet-in-the-Middle**:
- Divide `nums` into two halves: Left half of size `N1 = N / 2` (up to 9 elements) and Right half of size `N2 = N - N1` (up to 10 elements).
- Maximum operations for Left half: 3^9 = 19,683 states.
- Maximum operations for Right half: 3^10 = 59,049 states.

### 1.3 Target Matching Logic
Let `(target_p2, target_p3, target_p5)` be the prime exponent representation of target `k`.
For a state generated from the right half with exponents `(r2, r3, r5)`, the required state from the left half `(l2, l3, l5)` must satisfy:
- `l2 = target_p2 - r2`
- `l3 = target_p3 - r3`
- `l5 = target_p5 - r5`

We encode `(l2, l3, l5)` state tuples into an efficient 64-bit key or integer representation to perform fast hash map lookups.

## 2. Structural & Architectural Choice

### 2.1 Primary Choice: Meet-in-the-Middle + Hash Map State Compression
- **Why it fits constraints:** Reduces processing operations from ~1.16 * 10^9 down to ~80,000 recursive states.
- **State Compression:** Since exponents for N <= 19 satisfy bounds `-38 <= p2 <= 38`, `-19 <= p3 <= 19`, `-19 <= p5 <= 19`, each exponent offset can be shifted to positive integers and packed into a single 64-bit `long long` key for O(1) hash map operations.

### 2.2 Alternative Approaches Comparison
- **Brute Force Backtracking:** O(3^N) time complexity. Fails due to TLE for N >= 15.
- **Dynamic Programming / Memoization (Top-Down):** DP state `dp(index, p2, p3, p5)` can be used, but since exponent ranges are sparse and dynamically generated, memoization requires hash maps per index. Meet-in-the-middle has lower constant overhead and smaller memory footprint than maintaining dynamic maps for 19 layers.

## 3. Full Optimal Implementation (C++)

```cpp
// Meet-in-the-Middle + Prime Exponents State Compression
#include <vector>
#include <unordered_map>
#include <tuple>

class Solution {
private:
    // Decompose a number in range 1..6 into prime powers (p2, p3, p5)
    std::tuple<int, int, int> getPrimeExponents(int num) {
        int p2 = 0, p3 = 0, p5 = 0;
        if (num == 2) p2 = 1;
        else if (num == 3) p3 = 1;
        else if (num == 4) p2 = 2;
        else if (num == 5) p5 = 1;
        else if (num == 6) { p2 = 1; p3 = 1; }
        return {p2, p3, p5};
    }

    // Pack exponent tuple (p2, p3, p5) into a unique 64-bit integer
    long long encodeState(int p2, int p3, int p5) {
        // Offset exponents to ensure non-negative values
        long long u2 = p2 + 100;
        long long u3 = p3 + 100;
        long long u5 = p5 + 100;
        return (u2 << 32) | (u3 << 16) | u5;
    }

    void dfsLeft(int idx, int end_idx, const std::vector<std::tuple<int, int, int>>& primes, 
                 int cur2, int cur3, int cur5, std::unordered_map<long long, int>& left_counts) {
        if (idx == end_idx) {
            long long key = encodeState(cur2, cur3, cur5);
            left_counts[key]++;
            return;
        }

        auto [p2, p3, p5] = primes[idx];

        // Choice 1: Multiply by nums[idx]
        dfsLeft(idx + 1, end_idx, primes, cur2 + p2, cur3 + p3, cur5 + p5, left_counts);
        // Choice 2: Divide by nums[idx]
        dfsLeft(idx + 1, end_idx, primes, cur2 - p2, cur3 - p3, cur5 - p5, left_counts);
        // Choice 3: Leave unchanged
        dfsLeft(idx + 1, end_idx, primes, cur2, cur3, cur5, left_counts);
    }

    void dfsRight(int idx, int end_idx, const std::vector<std::tuple<int, int, int>>& primes,
                  int cur2, int cur3, int cur5, int target2, int target3, int target5,
                  const std::unordered_map<long long, int>& left_counts, long long& total_ways) {
        if (idx == end_idx) {
            int req2 = target2 - cur2;
            int req3 = target3 - cur3;
            int req5 = target5 - cur5;

            long long key = encodeState(req2, req3, req5);
            auto it = left_counts.find(key);
            if (it != left_counts.end()) {
                total_ways += it->second;
            }
            return;
        }

        auto [p2, p3, p5] = primes[idx];

        // Choice 1: Multiply by nums[idx]
        dfsRight(idx + 1, end_idx, primes, cur2 + p2, cur3 + p3, cur5 + p5, target2, target3, target5, left_counts, total_ways);
        // Choice 2: Divide by nums[idx]
        dfsRight(idx + 1, end_idx, primes, cur2 - p2, cur3 - p3, cur5 - p5, target2, target3, target5, left_counts, total_ways);
        // Choice 3: Leave unchanged
        dfsRight(idx + 1, end_idx, primes, cur2, cur3, cur5, target2, target3, target5, left_counts, total_ways);
    }

public:
    int countSequences(std::vector<int>& nums, long long k) {
        // Decompose target k into prime factors 2, 3, 5
        int target2 = 0, target3 = 0, target5 = 0;
        long long temp_k = k;

        while (temp_k % 2 == 0) { target2++; temp_k /= 2; }
        while (temp_k % 3 == 0) { target3++; temp_k /= 3; }
        while (temp_k % 5 == 0) { target5++; temp_k /= 5; }

        // If k has prime factors other than 2, 3, 5, reaching k is impossible
        if (temp_k != 1) {
            return 0;
        }

        int n = nums.size();
        std::vector<std::tuple<int, int, int>> primes(n);
        for (int i = 0; i < n; ++i) {
            primes[i] = getPrimeExponents(nums[i]);
        }

        int mid = n / 2;
        std::unordered_map<long long, int> left_counts;
        left_counts.reserve(60000);

        // Populate left half combinations
        dfsLeft(0, mid, primes, 0, 0, 0, left_counts);

        // Process right half combinations and match with left half
        long long total_ways = 0;
        dfsRight(mid, n, primes, 0, 0, 0, target2, target3, target5, left_counts, total_ways);

        return static_cast<int>(total_ways);
    }
};
```

## 4. Complexity & Mathematical Analysis

### 4.1 Detailed Time Complexity Decomposition
Let N be the size of array nums (N <= 19).
Divide nums into Left half of size N1 = floor(N / 2) and Right half of size N2 = ceil(N / 2).

1. **Prime Factorization Phase:**
   - Factorizing target k takes O(log k) operations.
   - Decomposing each element nums[i] takes O(1) since nums[i] <= 6.
   - Operations: T_factor = O(log k + N).

2. **Left Half Generation (DFS):**
   - Number of subsets generated = 3^N1 = 3^9 = 19,683 states.
   - Each state involves O(1) state encoding and map insertion.
   - Operations: T_left = O(3^(N/2)).

3. **Right Half Search & Matching (DFS):**
   - Number of subsets generated = 3^N2 = 3^10 = 59,049 states.
   - For each leaf state, required exponent tuple is calculated and looked up in left_counts in average O(1) time.
   - Operations: T_right = O(3^(N/2)).

**Total Time Complexity:**
T_total = T_factor + T_left + T_right = O(3^(N/2))

For N = 19, 3^9 + 3^10 = 19,683 + 59,049 = 78,732 total recursive steps, running in under 5 milliseconds.

### 4.2 Detailed Space Complexity Decomposition
- **Hash Map (left_counts):** Stores up to 3^(N1) unique compressed exponent states. For N1 = 9, at most 19,683 key-value entries (~ 600 KB memory).
- **Recursion Stack:** Call stack depth is bounded by max(N1, N2) <= 10 frames (~ 1 KB memory).

**Total Space Complexity:**
S_total = O(3^(N/2))

Auxiliary space is less than 1 MB, significantly below memory limits.

### 4.3 Complexity Summary

| Metric | Value | Notes |
|---|---|---|
| **Time Complexity** | O(3^(N/2)) | N <= 19. Meet-in-the-middle splits state exploration into 3^(N/2) calls. |
| **Space Complexity** | O(3^(N/2)) | Hash map storage for generated prime exponent tuples of left half. |

## 5. Mathematical Formulations & Foundational Algorithms

### Prime Power Transformation
Any non-zero rational number R generated by multiplications and divisions of elements from {1, 2, 3, 4, 5, 6} uniquely maps to an exponent 3-tuple (p2, p3, p5):
R = 2^(p2) * 3^(p3) * 5^(p5)

- **Multiplication Rule:** R * nums[i] -> (p2 + e2_i, p3 + e3_i, p5 + e5_i)
- **Division Rule:** R / nums[i] -> (p2 - e2_i, p3 - e3_i, p5 - e5_i)
- **No-op Rule:** R * 1 -> (p2, p3, p5)

This converts floating-point division and potential fraction precision loss into exact integer addition and subtraction operations on exponents.

---

## Pitfalls & Edge Cases

* **Invalid Prime Factors in Target k:** If k is divisible by primes other than 2, 3, or 5 (e.g., k = 7, 14, 22), it cannot be constructed. Handled by checking temp_k != 1 after stripping factors 2, 3, and 5, immediately returning 0.
* **Negative Exponents:** Division creates negative powers. Exponent tuple encoding offsets values by +100 to prevent negative bitwise shifting errors.
* **Precision Loss:** Doing actual float/double division (val / nums[i]) causes cumulative floating-point inaccuracy. Working strictly with prime exponent tuples guarantees exact precision.

---

## Key Takeaway

When operations involve fractions derived from small numbers with common prime bases, transform numbers into prime exponent vectors and reduce combinatorial choices using Meet-in-the-Middle.

**Pattern recognition:** "choose one of [multiply, divide, ignore] for each element to reach target k" -> Prime Exponents Representation + Meet-in-the-Middle
