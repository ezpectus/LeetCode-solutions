## 3139. Minimum Cost to Equalize Array

### Problem Statement
Given an integer array `nums` and two integers `cost1` and `cost2`, return the minimum cost to make all elements in the array equal. You can perform two types of operations any number of times:
1. Increase a single element `nums[i]` by 1 for a cost of `cost1`.
2. Increase two distinct elements `nums[i]` and `nums[j]` by 1 for a cost of `cost2`.

Since the total cost can be large, return the result modulo 10^9 + 7.

### Core Insight — Greedy Satiation & Target Peak Exploration
Let N be the length of the array `nums`. If N = 1 or N = 2, we are limited by tracking logic constraints, so we fallback to basic math. For general N > 2:

If 2 * cost1 <= cost2, using the single-increment operation is always more cost-efficient than pairing. We simply find the maximum value currently in the array (max_val) and calculate the sum of differences between max_val and every element, multiplying the entire total gap by `cost1`.

If pairing is cheaper (2 * cost1 > cost2), we want to use the two-increment operation as much as possible. Let the target value to equalize all elements be T. Initially, the minimum possible value for T is the current maximum element in the array (max_val). However, it is sometimes cheaper to raise T beyond max_val! 

Why? Increasing the target increases the total number of operations needed, but it may let us replace expensive single operations with highly discounted paired operations.

#### Math Behind Paired Increments
For a fixed target T:
* Compute the gap for each element: g_i = T - nums[i].
* Find the maximum individual gap: max_gap = T - min_val.
* Compute the total sum of all gaps: total_gap = sum(g_i).

When attempting to pair increments across elements:
1. **The Dominant Element Constraint:** If max_gap > total_gap - max_gap, the element with the largest gap is so deficient that it cannot be paired fully with all other elements combined. The number of completely paired operations we can perform is limited by the rest of the elements: (total_gap - max_gap). The remaining deficit for that single dominant element must be filled using single increments at `cost1`.
   
   Cost = (total_gap - max_gap) * cost2 + (2 * max_gap - total_gap) * cost1

2. **The Balanced Distribution Case:** If max_gap <= total_gap - max_gap, no single element dominates. We can successfully pair almost all increments together.
   * If total_gap is even, we can pair everything: (total_gap / 2) * cost2.
   * If total_gap is odd, one single increment will be left over: (total_gap / 2) * cost2 + cost1.

We search for the optimal target T starting from max_val up to 2 * max_val. Beyond 2 * max_val, the cost strictly increases because the extra paired operations can no longer offset the baseline cost of inflating the array size.

---

### Algorithm
1. Track the minimum element `min_val`, maximum element `max_val`, and the initial sum of elements.
2. If N <= 2 or 2 * cost1 <= cost2, compute the total gap using only `cost1` and return it.
3. Track a global minimum cost initialized to infinity (long.MaxValue).
4. Loop through potential target values T from `max_val` to 2 * max_val:
   * Calculate max_gap = T - min_val.
   * Calculate total_gap = (T * N) - initial_sum.
   * Apply the conditional cost formulas above based on whether `max_gap` dominates or stays balanced.
   * Update your global minimum cost with the lowest calculated amount.
5. Return the global minimum cost modulo 10^9 + 7.

---

### Full Optimal Implementation (C#)
```csharp
using System;

public class Solution {
    private const long MOD = 1_000_000_007;

    public int MinCostToEqualizeArray(int[] nums, int cost1, int cost2) {
        long n = nums.Length;
        long minVal = nums[0];
        long maxVal = nums[0];
        long sum = 0;

        foreach (int num in nums) {
            if (num < minVal) minVal = num;
            if (num > maxVal) maxVal = num;
            sum += num;
        }

        long initialTotalGap = maxVal * n - sum;

        // Base Case 1: If there's only 1 element, no operations are needed
        if (n == 1) {
            return 0;
        }

        // Base Case 2: If pairing operations offer no discount
        if (2 * (long)cost1 <= (long)cost2) {
            return (int)((initialTotalGap * cost1) % MOD);
        }

        // Base Case 3: For exactly 2 elements, we can only increase elements individually
        if (n == 2) {
            return (int)((initialTotalGap * cost1) % MOD);
        }

        long minTotalCost = long.MaxValue;
        long searchLimit = 2 * maxVal;

        // Step 4: Explore extended target peak thresholds
        for (long T = maxVal; T <= searchLimit; T++) {
            long maxGap = T - minVal;
            long totalGap = T * n - sum;
            long currentCost = 0;

            // Scenario A: Dominant gap element exists
            if (maxGap > totalGap - maxGap) {
                long pairs = totalGap - maxGap;
                long singles = maxGap - pairs;
                currentCost = pairs * cost2 + singles * cost1;
            } 
            // Scenario B: Balanced gaps spread evenly
            else {
                currentCost = (totalGap / 2) * cost2 + (totalGap % 2) * cost1;
            }

            if (currentCost < minTotalCost) {
                minTotalCost = currentCost;
            }
        }

        return (int)(minTotalCost % MOD);
    }
}
```

### Example Walkthrough
#### `nums = [2,3,3,3,5]`, `cost1 = 2`, `cost2 = 1`
* **Stats:** N = 5, min_val = 2, max_val = 5, sum = 16.
* Pairing is cheaper because 2 * cost1 = 4 > cost2 = 1.
* **Target T = 5:**
  - max_gap = 5 - 2 = 3.
  - total_gap = (5 * 5) - 16 = 9.
  - Condition check: max_gap = 3 <= (total_gap - max_gap) = 6. Balanced distribution!
  - Cost = (9 / 2) * 1 + (9 % 2) * 2 = 4 * 1 + 1 * 2 = 6.
* **Target T = 6:**
  - max_gap = 6 - 2 = 4.
  - total_gap = (6 * 5) - 16 = 14.
  - Balanced distribution: 4 <= 10.
  - Cost = (14 / 2) * 1 + 0 = 7.

**Total Output:** `6`. ✓

### Pitfalls & Edge Cases
* **Target Upper Bound Scopes:** It's a common mistake to only evaluate T = max_val. As shown in Example 2, artificially inflating the target can sometimes create the necessary parity room to let us match previously un-pairable elements, dropping the total cost significantly.
* **64-bit Integer Deficits:** Accumulating mathematical products and calculations like `T * n` or operations on arrays of size 10^5 can easily overshoot 32-bit limits. Perform all calculations inside 64-bit `long` spaces before downcasting for the return modulo block.

### Key Takeaway
When given the option to modify elements either singly or in pairs, analyze the problem as a gap-filling exercise. The element with the largest deficit sets the constraint. If it dominates the remaining space, you are forced to use single operations; otherwise, you can utilize the cheaper paired operations. Inflating the target artificially can eliminate a bottleneck, lowering the overall cost.
