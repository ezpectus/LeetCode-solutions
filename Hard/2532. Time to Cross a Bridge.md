# 2532. Time to Cross a Bridge
## O(n log k) — Event-Driven Simulation with Four Priority Queues

---

## Problem Statement

- k workers, n boxes to move right → left across a bridge (single-occupancy).
- Worker cycle: cross right → pick → cross left → put.
- **Priority rule:** when bridge is free, first let workers waiting on the **right** (least efficient) cross; then let workers from the **left** (least efficient) cross — but only if more boxes need fetching.
- Least efficient = highest `left + right` crossing time; tiebreak = highest index.
- Return the time the **last box** arrives on the left (i.e., when the last right→left crossing finishes).

---

## Core Idea — Four Queues, Event-Driven Time

Track four groups:
- `waitLeft` / `waitRight`: workers idle on left/right, **max-heap by inefficiency** (least efficient dequeues first)
- `workLeft` / `workRight`: workers putting/picking boxes, **min-heap by finish time**

At each step:
1. Move finished workers from `workLeft/workRight` → `waitLeft/waitRight`.
2. Pick who crosses the bridge (priority: right side first, then left side if boxes remain).
3. If no one can cross, jump time to the next finish event.

Terminate when all n boxes committed (`sent == n`) and no workers remain on the right side.

---

## Full Optimal Implementation (C#)

```csharp
public class Solution {
    public int FindCrossingTime(int n, int k, int[][] time) {
        // Max-heap: least efficient (highest left+right sum, tiebreak highest index) dequeues first
        var ineffCmp = Comparer<(int, int)>.Create((a, b) => {
            int c = b.Item1.CompareTo(a.Item1);
            return c != 0 ? c : b.Item2.CompareTo(a.Item2);
        });

        var waitLeft  = new PriorityQueue<int, (int, int)>(ineffCmp);
        var waitRight = new PriorityQueue<int, (int, int)>(ineffCmp);
        var workLeft  = new PriorityQueue<int, int>(); // min-heap by finish time
        var workRight = new PriorityQueue<int, int>();

        for (int i = 0; i < k; i++)
            waitLeft.Enqueue(i, (time[i][2] + time[i][0], i));

        int cur = 0, sent = 0, ans = 0;

        while (sent < n || workRight.Count > 0 || waitRight.Count > 0) {
            // 1. Release workers who finished their tasks
            while (workLeft.TryPeek(out int wl, out int fl) && fl <= cur) {
                workLeft.TryDequeue(out wl, out _);
                waitLeft.Enqueue(wl, (time[wl][2] + time[wl][0], wl));
            }
            while (workRight.TryPeek(out int wr, out int fr) && fr <= cur) {
                workRight.TryDequeue(out wr, out _);
                waitRight.Enqueue(wr, (time[wr][2] + time[wr][0], wr));
            }

            // 2. Bridge decision
            if (waitRight.Count > 0) {
                // Least efficient from right crosses left (priority)
                int w = waitRight.Dequeue();
                cur += time[w][2];
                ans = cur;
                workLeft.Enqueue(w, cur + time[w][3]);
            } else if (sent < n && waitLeft.Count > 0) {
                // Least efficient from left crosses right
                int w = waitLeft.Dequeue();
                cur += time[w][0];
                sent++;
                workRight.Enqueue(w, cur + time[w][1]);
            } else {
                // Bridge idle: jump to next event
                int next = int.MaxValue;
                if (workLeft.TryPeek(out _, out int tL)) next = Math.Min(next, tL);
                if (workRight.TryPeek(out _, out int tR)) next = Math.Min(next, tR);
                if (next < int.MaxValue) cur = next;
                else break;
            }
        }

        return ans;
    }
}
```

## Complexity

| Metric | Value | Notes |
|---|---|---|
| Time Complexity | O(n log k) | n bridge crossings, each queue op O(log k) |
| Space Complexity | O(k) | Four queues hold at most k workers total |

## Why `sent == n` Is the Right Termination Condition

"Enough dispatched" means n workers have started crossing right. Once `sent == n`, we never send anyone else from left. Workers already on the right side must still come back — the loop continues until `workRight` and `waitRight` are both empty.

**We don't wait for `workLeft` to empty** because `ans` is updated only when boxes ARRIVE on the left (right→left crossing ends), not when they're put down.

## Example Walkthrough

**Example 1:** `n=1, k=3, time=[[1,1,2,1],[1,1,3,1],[1,1,4,1]]`

Worker sums: 3, 4, 5 → worker 2 most inefficient.

| cur | Action | ans |
|-----|--------|-----|
| 0   | Send worker 2 (sum=5) right: cur=0+1=1, sent=1 | — |
| 1   | workRight: w2 finishes at 2 > 1. Advance → cur=2 | — |
| 2   | w2 finishes picking → waitRight. w2 crosses left: cur=2+4=**6**, ans=**6** | 6 |

Exit: sent==1==n, workRight empty, waitRight empty. **Output: 6** ✓

---

**Example 2:** `n=3, k=2, time=[[1,5,1,8],[10,10,10,10]]`

Sums: w0=2, w1=20.

| cur | Action | sent | ans |
|-----|--------|------|-----|
| 0   | Send w1 (sum=20) right: cur=10 | 1 | — |
| 10  | Send w0 (sum=2) right: cur=11 | 2 | — |
| 11  | Advance → cur=16 (w0 finishes picking) | 2 | — |
| 16  | w0→waitRight, crosses left: cur=17, ans=17 | 2 | 17 |
| 17  | Advance → cur=20 (w1 finishes picking) | 2 | — |
| 20  | w1→waitRight, crosses left: cur=30, ans=30 | 2 | 30 |
| 30  | w0 done putting→waitLeft. Send w0 right: cur=31 | 3 | — |
| 31  | Advance → cur=36 (w0 finishes picking) | 3 | — |
| 36  | w0→waitRight, crosses left: cur=37, ans=**37** | 3 | 37 |

Exit: sent==3==n, workRight empty, waitRight empty. **Output: 37** ✓

## Pitfalls & Edge Cases

- **Right side has priority over left** — even if only 1 box remains to be fetched and both sides have workers waiting, the right side crosses first. ✓

- **`ans` is updated on right→left crossing only** — the put-down time is irrelevant. ✓

- **Multiple workers finish at same cur** — both `while` loops drain everyone who finished. ✓

- **Bridge idle gap** — when no one is waiting (all working), the `else` branch jumps cur to the next finish event. This avoids busy-waiting. ✓

- **sent == n but workers still on right** — the loop condition `workRight.Count > 0 || waitRight.Count > 0` keeps running. ✓

- **Custom comparer direction** — `b.CompareTo(a)` makes the PQ dequeue the LARGEST first (max-heap behavior). ✓

---

## Key Takeaway

Complex simulations with priority rules → **multiple specialized priority queues**, each tracking one state transition.

The four-queue pattern:
- Two **max-heaps** (by inefficiency) for workers waiting to cross
- Two **min-heaps** (by finish time) for workers currently working

**Time jumping** (advance cur to next event) is the key efficiency trick — never loop through idle time.

**Pattern recognition:** "Schedule workers/jobs with priority + time constraints" → event-driven simulation with priority queues for each state. The bridge is just a resource with single-worker capacity.
