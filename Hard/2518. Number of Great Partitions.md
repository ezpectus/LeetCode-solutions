# 2518. Number of Great Partitions
## O(n · k) — Complementary Counting + Subset Sum DP

---

## Problem Statement

- Partition `nums` into two ordered groups (each element in exactly one group).
- A partition is **great** if BOTH groups have sum ≥ k.
- Return the number of great partitions mod 10⁹ + 7.

---

## Core Idea — Complement: Total − Bad Partitions

**Key observations:**

1. **Necessity check:** If both groups have sum ≥ k, then `totalSum = sum1 + sum2 ≥ 2k`. If `totalSum < 2k`, return 0 immediately.

2. **Complementary counting:**
   ```
   Great = 2^n − Bad
   ```
   Bad = partitions where group1 sum < k **OR** group2 sum < k.
   
   By inclusion-exclusion: `Bad = |A| + |B| − |A ∩ B|`
   where A = {group1 sum < k}, B = {group2 sum < k}.

3. **|A| = |B|** (symmetric: flipping groups maps A↔B).

4. **|A ∩ B| = 0 when totalSum ≥ 2k:** both sums < k → totalSum < 2k. Contradiction. ✓

5. So: `Great = 2^n − 2|A|`

**Computing |A| — number of assignments with group1 sum < k:**
- Any element with value ≥ k: putting it in group1 makes sum ≥ k. So for group1 sum < k, **all large elements must go to group2**.
- Only "small" elements (value < k) can form a group1 with sum < k.
- `|A|` = (ways to choose a subset of small elements with sum < k) × 1 (large elements forced to group2).

This is a classic 0-1 knapsack counting problem on elements with value < k.

## Full Optimal Implementation (C#)

```csharp
public class Solution {
    public int CountPartitions(int[] nums, int k) {
        const int MOD = 1_000_000_007;
        int n = nums.Length;
        long totalSum = 0;
        foreach (int x in nums) totalSum += x;

        if (totalSum < 2L * k) return 0; // impossible to have both groups >= k

        // dp[s] = number of subsets of small elements (value < k) with sum exactly s
        long[] dp = new long[k];
        dp[0] = 1;

        foreach (int x in nums) {
            if (x >= k) continue; // large elements can't contribute to sum < k
            for (int s = k - 1; s >= x; s--) {
                dp[s] = (dp[s] + dp[s - x]) % MOD;
            }
        }

        // |A| = number of subsets with sum in [0, k-1]
        long A = 0;
        for (int s = 0; s < k; s++) A = (A + dp[s]) % MOD;

        // 2^n mod MOD
        long pow2n = 1;
        for (int i = 0; i < n; i++) pow2n = pow2n * 2 % MOD;

        // Great = 2^n - 2|A| (|A ∩ B| = 0 since totalSum >= 2k)
        return (int)((pow2n - 2 * A % MOD + MOD) % MOD);
    }
}
```

## Why Large Elements Are Skipped in the DP

An element with `value ≥ k` cannot appear in a subset with sum `< k` (a single such element already has sum ≥ k). So we skip them entirely in the DP.

However, these large elements DO contribute to `|A|` — specifically by being forced to group2. Since every large element must go to group2 (not group1), there's exactly 1 valid placement for all large elements when group1 sum < k. The DP's `dp[0] = 1` base case already accounts for the configuration "no small elements in group1, all large go to group2."

## Example Walkthrough

**Example 1:** `nums = [1,2,3,4], k = 4`

totalSum = 10 ≥ 2k = 8 ✓. Small = {1,2,3}, large = {4}.

DP (sums 0..3):
- After 1: dp = [1,1,0,0]
- After 2: dp = [1,1,1,1]
- After 3: dp = [1,1,1,2] ← dp[3] += dp[0] = 1

|A| = 1+1+1+2 = 5, 2^4 = 16

Answer = 16 − 10 = **6** ✓

---

**Example 2:** `nums = [3,3,3], k = 4`

totalSum = 9 ≥ 8 ✓. Small = {3,3,3}, large = {}.

DP:
- After three 3s: dp = [1,0,0,3] (can pick 0 or 1 or 2 or 3 threes, but only sum<4 means 0 or 1 three)
- Actually: dp[0]=1 (empty), dp[3]=3 (pick any one of the three 3s). Other dp[s]=0.

|A| = 1+0+0+3 = 4, 2^3 = 8

Answer = 8 − 8 = **0** ✓

---

**Example 3:** `nums = [6,6], k = 2`

totalSum = 12 ≥ 4 ✓. Small = {} (both 6 ≥ 2), large = {6, 6}.

DP: dp = [1] (empty subset only). |A| = 1. 2^2 = 4.

Answer = 4 − 2 = **2** ✓

## Pitfalls & Edge Cases

- **totalSum < 2k → return 0** immediately (both groups ≥ k requires totalSum ≥ 2k). Covers cases where even one group can't reach k.

- **Modular arithmetic:** `2 * A % MOD` can exceed `pow2n`, making the result negative. Add MOD before taking mod: `(pow2n − 2*A%MOD + MOD) % MOD`. ✓

- **Large nums[i]:** values up to 10⁹ skip the DP loop (x ≥ k), so the DP only runs on values < k ≤ 1000. ✓

- **Empty group valid:** a group can have sum = 0 if all elements go to the other group. In this case, sum < k, so this is a "bad" partition and is correctly excluded.

---

## Key Takeaway

**Complementary counting** avoids the hard problem (count great directly) by counting bad partitions:

```
Great = 2^n − 2 × (subsets of small elements with sum < k)
```

The insight that `|A ∩ B| = 0` when `totalSum ≥ 2k` simplifies the inclusion-exclusion. The further insight that only "small" elements (value < k) can contribute to a group1 sum < k reduces the DP to k states with values < k.

**Pattern recognition:** "Count configurations satisfying two constraints" → complement + inclusion-exclusion, then observe one of the terms vanishes.
