# 2650. Design Cancellable Function
## O(N) — Coroutine Runner with Immediate Generator Interruption

### Problem Statement
Sometimes you have a long-running task, and you wish to cancel it before it completes. Write a function `cancellable` that accepts a generator object and returns an array of two values: a `cancel` function and a `promise`.

* The generator will only yield promises. Your function must pass resolved values back to the generator via `.next(val)`.
* If a yielded promise rejects, throw that error back into the generator via `.throw(err)`.
* If `cancel()` is invoked while waiting for a promise, throw the string `"Cancelled"` back into the generator. 
  * If the generator catches this error, the returned promise should resolve with the next value yielded or returned.
  * If uncaught, the promise should reject with `"Cancelled"`.
* When the generator finishes, the promise should resolve with the returned value.

---

### Core Idea — The Async-Spawn / Coroutine Pattern
An `async` function in JavaScript is internally managed as a state machine via a generator. To support cancellation, we need to implement a recursive step manager (`run`), which orchestrates feeding values back into the generator while monitoring a shutdown flag (`isClosed`).

```
[Initial Trigger] ──> generator.next() ──> Yields Promise ──┐
                                                           │
   ┌───────────────────────────────────────────────────────┘
   ▼
[Promise Settles?]
   ├───> Resolves ──> (if not cancelled) ──> generator.next(val) ──> Loop
   ├───> Rejects  ──> (if not cancelled) ──> generator.throw(err) ──> Loop
   │
   └─> [cancel() Interruption at any time]
              └───> isClosed = true ──> generator.throw("Cancelled") ──> Check Catch/Return
```

When `cancel()` is triggered, we must **immediately** force the generator to wake up and face a `.throw("Cancelled")`, completely ignoring whether the currently awaited promise resolves or rejects later.

---

### Algorithm
1. **Initialize State:** Maintain an `isClosed` flag to prevent duplicate cancellation handling or processing stale promise resolutions.
2. **Define Step Runner (`run`):** A recursive helper function that wraps the generator's state transitions (`.next()` or `.throw()`) inside a `try-catch` block.
3. **Evaluate States:**
   * If an execution step throws an unhandled exception, immediately reject the main promise.
   * If `step.done` is true, resolve the main promise with `step.value`.
   * If `step.done` is false, chain the yielded promise using `.then()`.
4. **Guard Microtasks:** Inside the `.then()` resolution and rejection callbacks, check `if (isClosed) return;`. This ensures that if the execution chain was canceled while this promise was pending, its delayed settlement won't interfere with the aborted state machine.

---

### Full Optimal Implementation (JavaScript)

```javascript
/**
 * @param {Generator} generator
 * @return {[Function, Promise]}
 * * Pattern: Asynchronous Coroutine Runner with Injected Exception Routing
 */
var cancellable = function(generator) {
    let cancel;
    
    const promise = new Promise((resolve, reject) => {
        let isClosed = false;

        // Injected cancellation callback
        cancel = () => {
            if (isClosed) return;
            isClosed = true;
            
            // Interrupt immediately by forcing the generator to throw
            run(() => generator.throw("Cancelled"));
        };

        function run(action) {
            let step;
            try {
                // Execute the transition (can be .next() or .throw())
                step = action();
            } catch (err) {
                // If the generator doesn't catch an error (or throws a new one)
                reject(err);
                isClosed = true;
                return;
            }

            // Base case: Generator reached a 'return' statement
            if (step.done) {
                resolve(step.value);
                isClosed = true;
                return;
            }

            // Recursive case: Await the yielded promise
            Promise.resolve(step.value).then(
                (val) => {
                    if (isClosed) return; // Stale resolution guard
                    run(() => generator.next(val));
                },
                (err) => {
                    if (isClosed) return; // Stale rejection guard
                    run(() => generator.throw(err));
                }
            );
        }

        // Bootstrap the coroutine execution loop
        run(() => generator.next());
    });

    return [cancel, promise];
};
```

### Complexity

| Metric | Value | Notes |
| :--- | :--- | :--- |
| **Time Complexity** | O(N) | Where $N$ is the number of total yield states/promises produced by the generator. Each step executes in $O(1)$ constant overhead processing time. |
| **Space Complexity** | O(N) | Due to the recursive call stack of the `run` loop (or call stack of the promise microtask queue) relative to the max structural depth of the generator's sequential yield count. |

---

### Pitfalls & Edge Cases
* **Immediate Interruption vs Delayed Settlement:** A naive solution might just reject the outer promise when `cancel()` is called. However, if the generator has an internal `try-catch` block (like in Example 5), it expects to catch `"Cancelled"`, clean up its internal variables, and **return** a valid fallback value. Calling `generator.throw()` inside `cancel()` satisfies this exactly. ✓
* **The Ghost Resolution Bug:** If `cancel()` is triggered, the generator handles the cancellation and returns early. However, the original async promise that was pending before the cancellation will still resolve a few milliseconds later. Without the `if (isClosed) return;` guard, that ghost promise would trigger a dead `.next()` call, corrupting the settled execution state. ✓
* **Non-Standard Yields:** Wrapping `step.value` in `Promise.resolve(step.value)` guarantees that even if the generator yields a raw primitive or synchronous value by accident, the engine converts it safely into a standard thenable. ✓

---

### Key Takeaway
> When building custom execution environments or asynchronous lifecycles (like cancellation tokens or pause/resume streams), decouple the lifecycle state management from the native async engine. Wrapping generator state jumps (`next`/`throw`) in isolated closure loops provides absolute control over thread-like synchronization in a single-threaded runtime.

* **Pattern recognition:** "Manage async functions with precise injection and intercept controls" -> Custom Coroutine Hook / Generator State Machine Runner.
