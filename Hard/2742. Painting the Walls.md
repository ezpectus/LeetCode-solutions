# 2742. Painting the Walls
## O(n²) — Minimum Cost Knapsack DP (Inverse Thinking)

### Problem Statement
You are given two arrays of size n: `cost` and `time`. 
* **Paid Painter:** Paints wall `i`, takes `time[i]` units of time, and costs `cost[i]`.
* **Free Painter:** Paints *any* wall, takes 1 unit of time, and costs 0. Can only work while the paid painter is occupied.

Return the minimum money required to paint all n walls.

---

### Core Idea — The Knapsack Transformation
If we try to simulate the process chronological step-by-step, we will get bogged down in combinations. Instead, let's look at what happens when we hire the paid painter for a specific wall $i$:
1. The paid painter occupies himself for `time[i]` units of time.
2. During this exact time, the free painter can paint up to `time[i]` walls for free.
3. Therefore, by paying for wall $i$, we effectively get **`time[i] + 1` walls painted** (1 by the paid painter + `time[i]` by the free painter).

The problem now mathematically transforms into the classic **0/1 Knapsack Problem**:
* We need to paint a target of **n walls**.
* Each wall $i$ behaves like an item with a **weight** of `time[i] + 1` and a **cost** of `cost[i]`.
* We want to find the minimum cost to accumulate a total weight of at least `n`.

---

### Algorithm
1. **Define State:** `dp[w]` represents the minimum cost to get *at least* `w` walls painted.
2. **Initialization:** Create a DP array of size `n + 1`. Initialize `dp[0] = 0` (0 walls cost 0 to paint) and all other states to infinity (a very large number like $10^9$).
3. **Transitions:** Iterate through each wall. For each wall, update the DP table backwards (from `n` down to 1) to ensure each wall is only bought once (standard 0/1 knapsack space optimization):
   * `dp[w] = min(dp[w], dp[max(0, w - weight)] + cost)`
   * *Note:* If `w - weight` drops below 0, it means this single wall over-satisfies our requirement, which maps directly to the base case `dp[0]`.

---

### Full Optimal Implementation (C#)

```csharp
using System;

public class Solution {
    public int PaintWalls(int[] cost, int[] time) {
        int n = cost.Length;
        int[] dp = new int[n + 1];
        
        // Initialize with a large value representing infinity.
        // 1e9 is safe because max total cost is 500 * 10^6 = 5 * 10^8.
        Array.Fill(dp, 1000000000); 
        dp[0] = 0;

        for (int i = 0; i < n; i++) {
            int c = cost[i];
            int wPushed = time[i] + 1; // Total walls covered by choosing this paid wall
            
            // Loop backwards to prevent using the same wall multiple times
            for (int w = n; w > 0; w--) {
                int remains = Math.Max(0, w - wPushed);
                dp[w] = Math.Min(dp[w], dp[remains] + c);
            }
        }
        
        return dp[n];
    }
}
```

---

### Complexity

| Metric | Value | Notes |
| :--- | :--- | :--- |
| **Time Complexity** | O(n²) | We use two nested loops: outer runs $n$ times (for each wall), inner runs $n$ times (updating the states). With $n \le 500$, $500 \times 500 = 250,000$ operations, which executes in a few milliseconds. |
| **Space Complexity** | O(n) | Optimizing the space allocation down to a 1D array of size $n + 1$ keeps our memory profile exceptionally lean. |

---

### Pitfalls & Edge Cases
* **Integer Overflow on Infinity:** If you use `int.MaxValue` as your infinity placeholder, adding `cost[i]` to it during the `Math.Min` calculation will wrap around into negative numbers, breaking the logic. Using `1,000,000,000` prevents this because the maximum total cost is $5 \times 10^8$, leaving plenty of headroom. ✓
* **The `Math.Max(0, ...)` Guard:** A single wall might have a time value of 500, which satisfies all $n$ walls immediately. If you don't cap the index floor at 0, your code will look for negative array indices and crash with an `IndexOutOfRangeException`. ✓

---

### Key Takeaway
> When an optimization problem tracks two distinct asynchronous workers (one free but constrained by a time resource generated by the paid worker), avoid tracking the timeline. Instead, convert time directly into a capacity multiplier and treat it as a resource-gathering Knapsack problem.

* **Pattern recognition:** "Paid actions unlock free parallel actions based on duration" -> Convert `time + 1` into a weight metric and solve via Minimum-Cost Subset Selection (Knapsack DP).
