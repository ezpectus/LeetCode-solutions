
# 3677. Count Binary Palindromic Numbers

## 1. Idea & Architectural Choice

### 1.1 Core Philosophy
The core requirement is to count all non-negative integers up to n whose binary representation forms a palindrome. Instead of iterating through all values or manually analyzing combinations bit-length by bit-length, we can leverage the foundational structural property of binary palindromes: **the first half (prefix) uniquely determines the entire sequence**. 

By assigning a continuous decimal index `mid` to represent the meaningful left half of a binary palindrome, we establish a strictly monotonic relationship between the value of the prefix and the decimal magnitude of the fully reconstructed palindrome. This strict monotonicity allows us to eliminate linear searching and instead use **Binary Search over the space of prefixes** to find the exact upper-bound constraint in logarithmic time.

### 1.2 Structural Decomposition of Binary Palindromes
Any positive binary palindrome must fall into one of three distinct architectural shapes depending on its length parity and center bit structure:
- **Case 0 (Even Length):** The binary string is formed by a core prefix appended to its direct mirror reflection (`str + rstr`). Example: Prefix "10" becomes "1001".
- **Case 1 (Odd Length with Center '0'):** The binary string embeds a fixed zero at its symmetry axis (`str + "0" + rstr`). Example: Prefix "10" becomes "10001".
- **Case 2 (Odd Length with Center '1'):** The binary string embeds a fixed one at its symmetry axis (`str + "1" + rstr`). Example: Prefix "10" becomes "10101".

By executing three isolated binary search calculations (one for each structural configuration), we find the maximum prefix index `mid` that produces a total value less than or equal to n. This configuration eliminates double counting because the underlying shapes are structurally orthogonal.

## 2. Algorithm Design & Mathematical Invariants

### 2.1 Variables and Predicate Logic
- `low` / `high`: The lower and upper boundary bounds of the binary search space for generating prefixes. `low` starts at 1 (the smallest valid positive binary prefix "1"), and `high` caps at 10^8 (since the maximum constraint n = 10^15 implies a maximum prefix bit-length of roughly 25 bits, which fits under 10^8).
- `mid`: The current prefix integer being tested as a search candidate.
- `op`: An operational discriminator code (0, 1, or 2) identifying the structural configuration.
- `ansT`: The local tracking variable preserving the maximum valid prefix index for a given case.
- Strict or Non-Strict Inequality: We evaluate the state validation condition using a **non-strict inequality** `reconstructed_decimal <= n`. This ensures that if the generated palindrome matches the target limit n exactly, it is successfully preserved in the safe set.

### 2.2 Mathematical Transition & Boundary Clamping Step-by-Step
The search space is split and handled by three calls to `getCount(op)`. Within each execution, the transition flow from a generic unbounded range to a boundary-truncated solution works via the following state verification sequence:

#### Step 1: Base Pruning
Before scanning, the base numbers 0 and 1 are added. If n = 0, the logic exits immediately returning 1 (the palindrome "0"). For n >= 1, we seed our core answer register with `ans = 2` to represent the pre-validated independent base elements: 0 ("0") and 1 ("1").

#### Step 2: Unbounded Search Convergence via Predicate Testing
For each specific structural option `op`:
1. We compute `mid = (low + high) / 2`.
2. Convert `mid` directly into its standard binary string sequence `str` and derive its mirrored string reflection `rstr`.
3. Assemble the full structural palindrome compound:
   - If op == 0: `temp = str + rstr`
   - If op == 1: `temp = str + "0" + rstr`
   - If op == 2: `temp = str + "1" + rstr`
4. Convert `temp` back into its 64-bit integer form `mxx = toDecimal(temp)`.

#### Step 3: Range Splitting Without Cycles
Instead of examining each outcome linearly, the dynamic boundary selection cuts the remaining prefix candidate pool in half at each checkpoint:
- **Case A (Valid Space Range): `mxx <= n`**
  This means the chosen prefix generates a binary palindrome that fits entirely within the allowed range. Because the mapping is monotonic, all prefixes smaller than `mid` are also guaranteed to be valid. Thus, we record the safe state `ansT = mid` and move our lower bound past the current point: `low = mid + 1`.
- **Case B (Overshoot Truncation Range): `mxx > n`**
  This implies the current prefix generates an oversized value that exceeds n. Due to monotonicity, any prefix larger than `mid` will also overshoot the target limit. We safely discard the entire upper half of the remaining choices by updating: `high = mid - 1`.

The total number of valid numbers for that structural category is exactly equal to the final converged value of `ansT`. The final total is computed via simple addition: `Result = 2 + getCount(0) + getCount(1) + getCount(2)`.

### 2.3 Architectural Choice: Prefix Binary Search vs Bit Combinatorics
An alternative approach to this problem involves manually iterating through all potential bit lengths from 1 up to 50, computing the counts of smaller lengths with combinatorial exponents (powers of 2), and processing the prefix boundary via individual bit extractions.

- **Why the alternative works:** It processes lengths independently and scales strictly by the bit count, avoiding conversions between strings and numbers.
- **Overheads of the alternative:** It requires complex and fragile conditional branching to manage boundary clamping for different lengths, making it prone to errors with bitwise arithmetic overflows and off-by-one index mistakes.
- **Why the Prefix Binary Search approach is preferred:** By treating prefixes as a continuous range of decimal integers, we can delegate the boundary logic to a clean binary search predicate. The overhead of converting numbers to binary strings is negligible because the binary search space is small (around 25-27 iterations per case). This makes the code exceptionally clean, readable, and highly maintainable while easily fitting within execution time limits.

## 3. Full Optimal Implementation (C++)
```
// Binary Search + String Manipulation + Bit Manipulation
#include <string>
#include <algorithm>

std::string toBinary(long long n) {
    if (n == 0) return "0";
    std::string res;
    while (n > 0) {
        res.push_back('0' + (n & 1));
        n >>= 1LL;
    }
    std::reverse(res.begin(), res.end());
    return res;
}

long long toDecimal(const std::string &bin) {
    long long res = 0;
    for (char c : bin) {
        res = res * 2 + (c - '0');
    }
    return res;
}

class Solution {
public:
    int countBinaryPalindromes(long long n) {
        if (n == 0) return 1;
        
        // Base count initialized to 2 to represent the numbers 0 ("0") and 1 ("1")
        int ans = 2;

        // Lambda routine encapsulating the monotonic binary search framework
        auto getCount = [&](int op) -> int {
            int low = 1;
            int high = 100000000; 
            int ansT = 0;
            
            std::string str, rstr, temp;

            while (low <= high) {
                int mid = (low + high) / 2;
                str = toBinary(mid);
                rstr = str; 
                std::reverse(rstr.begin(), rstr.end());
                
                // Construct structural variants according to operational case rules
                if (op == 0) {
                    temp = str + rstr;
                } else if (op == 1) {
                    temp = str + "0" + rstr;
                } else {
                    temp = str + "1" + rstr;
                }
                
                long long mxx = toDecimal(temp);
                
                // Evaluate boundary condition using non-strict inequality
                if (mxx <= n) {
                    ansT = mid;
                    low = mid + 1; // Expand right to find larger viable prefixes
                } else {
                    high = mid - 1; // Contract left to clear the overshooting zone
                }
            }
            return ansT;
        };

        // Combine base constants with structural branch counts
        ans += getCount(0) + getCount(1) + getCount(2);
        return ans;
    }
};
```

## 4. Complexity & Mathematical Analysis

### 4.1 Detailed Time Complexity Decomposition
The total execution runtime is broken down as follows:
T_total = 3 * (T_search_steps * (T_to_binary + T_reverse + T_to_decimal))

- **Binary Search Range Step Count (T_search_steps):** The search range spans from 1 up to 10^8. A standard binary search divides the search space in half at each step, taking log2(10^8) iterations. This requires roughly 27 iterations per case.
- **Binary Conversion Overheads (T_to_binary):** Converting the prefix variable mid to a binary string loops once for each bit. Since mid is bounded by 10^8, its binary representation takes at most 27 bits. This operation finishes in O(log2(mid)) steps.
- **String Reversal and Assembly (T_reverse):** Reversing a string of length B takes O(B) operations. With B bounded by 27 characters, this step is extremely fast and executes in constant time.
- **Decimal Conversion Conversion (T_to_decimal):** Converting the assembled palindrome back into a 64-bit long long integer reads the full length of the string. The max string length occurs when a 27-bit prefix is mirrored, resulting in at most 54 characters. This conversion runs in linear time relative to the string length.

Because the maximum bit length of the generated string is directly proportional to log2(n), each individual step inside the binary search loop takes O(log2(n)) operations. With 27 search steps per structural category, the total execution time is highly optimized.
T_total = O(log(MAX_PREFIX) * log(n)) = O(log(n) * log(n))

Given n <= 10^15, the loop body runs roughly 81 times across all three operations. The total execution completes in less than 3 microseconds, running well inside the 1-second limit.

### 4.2 Detailed Space Complexity Decomposition
- **Allocation Frame:** The program creates local 64-bit primitives and standard string storage buffers (str, rstr, temp) within the lambda's execution loop.
- **Dynamic Structural Overheads:** The temporary string buffers scale linearly with the bit length of the prefix, storing at most 55 characters. No persistent dynamic tables, multi-dimensional DP arrays, or heap-allocated structures are retained.

Thus:
Space Complexity = O(log(n)) auxiliary string workspace memory.

## 5. Mathematical Formulations & Foundational Algorithms
The solution leverages the Monotonicity Invariant of Symmetric Mirror Mappings.

**Formal Invariant Definition:** Let P_1 and P_2 be two distinct positive integers representing binary prefixes with no leading zeros. If P_1 < P_2, then their respective binary string representations satisfy the lexicographical order string(P_1) < string(P_2). Consequently, for any identical structural expansion operation (such as Case 0, 1, or 2), the reconstructed decimal values will satisfy: Value(Mirror(P_1)) < Value(Mirror(P_2)).

This mathematical invariant ensures that the generated values increase monotonically. As a result, the evaluation function changes from valid to invalid exactly once across the search range, satisfying the core requirement for a binary search.

By treating the prefix as an ordinary decimal integer index, the count of valid binary palindromes generated by prefixes smaller than or equal to a target prefix ansT is simply equal to ansT itself. This allows us to find the total count using a direct range subtraction mapping without needing to explicitly generate or store individual items.

## Pitfalls & Edge Cases
- **The Zero Input Trap:** If n = 0, the binary search parameters can miscalculate or overcount the base elements. The code avoids this by using an early return checkpoint that directly outputs 1 when n = 0.
- **Prefix Bit Limit Constraints:** Setting the high bound too low can truncate valid palindromes near the 10^15 limit. Setting it to 10^8 provides a safe buffer that covers all potential 25-bit prefixes.
- **Integer Multiplication Overflow:** Converting long binary strings back to decimal integers can exceed standard 32-bit limits. The code uses long long integers for all decimal accumulation steps to prevent overflow errors.
- **Center Bit Duplication:** Treating odd-length palindromes using even-length mirroring rules can accidentally double-count the center bit. Separating the logic into three distinct operations (op 0, 1, and 2) keeps the structural classifications completely independent.

## Key Takeaway
When counting numbers defined by symmetric structures up to a limit n, utilize the monotonicity of their generative halves. Convert the structural template into an integer prefix index and use binary search to determine the exact boundary limits in constant time.

**Pattern recognition:** "binary representation ... reads the same forward and backward ... count the number of integers k <= n" -> Binary Search + String Manipulation + Bit Manipulation
