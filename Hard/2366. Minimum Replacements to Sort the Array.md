# 2366. Minimum Replacements to Sort the Array

## O(n) — Greedy Right-to-Left Pass

---

## Problem Statement

- Given integer array `nums`.
- **Operation**: Replace element `x` with two elements `a` and `b` where `a + b = x`.
- Return minimum operations to make array **non-decreasing**.
- **Constraints**: 1 ≤ nums.length ≤ 10⁵, 1 ≤ nums[i] ≤ 10⁹

---

## Core Idea — Greedy from Right to Left

**Key observation**: We cannot increase any element value (only split). To achieve sorted order, work **backwards** from right to left.

**Greedy insight**: 
- Last element is free (no constraint from right)
- For element at position `i`, it must be ≤ element at position `i+1`
- If `nums[i] > nums[i+1]`, split `nums[i]` into parts ≤ `nums[i+1]`

**Splitting strategy**: 
- Split `nums[i]` into **k parts** where each part ≤ `nums[i+1]`
- Minimize operations: `k - 1` operations to create `k` parts
- Maximize the smallest part: distribute as evenly as possible

**Formula**: To split value `x` into parts ≤ `limit`:
```
k = ceil(x / limit) = (x + limit - 1) / limit
operations = k - 1
smallest_part = x / k (integer division)
```

**Why smallest part matters**: It becomes the new constraint for the next element to the left.

---

## Algorithm Breakdown

### Right-to-Left Greedy Pass

```
operations = 0
limit = nums[n-1] // Rightmost element sets initial constraint

For i from n-2 down to 0:
  if nums[i] <= limit:
    limit = nums[i] // No split needed, update constraint
  else:
    // Split nums[i] into k parts, each ≤ limit
    k = ceil(nums[i] / limit) = (nums[i] + limit - 1) / limit
    operations += (k - 1)
    
    // New constraint: smallest part after split
    limit = nums[i] / k

return operations
```

**Complexity**: O(n) single pass

---

## Full Optimal Implementation (C#)

```csharp
public class Solution {
    public long MinimumReplacement(int[] nums) {
        int n = nums.Length;
        long operations = 0;
        
        // Start from rightmost element
        int limit = nums[n - 1];
        
        // Process from right to left
        for (int i = n - 2; i >= 0; i--) {
            if (nums[i] <= limit) {
                // No split needed
                limit = nums[i];
            } else {
                // Split nums[i] into k parts, each ≤ limit
                int k = (nums[i] + limit - 1) / limit; // Ceiling division
                operations += k - 1;
                
                // New constraint: smallest part
                limit = nums[i] / k;
            }
        }
        
        return operations;
    }
}
```

With n = 10⁵: 10⁵ operations — extremely fast.

## Why This Works — Example Walkthrough

### Example 1: `nums = [3,9,3]`

**Initial state**: limit = 3 (rightmost)

**i = 1** (nums[1] = 9):
- 9 > 3 → need to split
- k = ceil(9 / 3) = 3 parts
- operations = 0 + (3 - 1) = 2
- Split 9 into [3, 3, 3]
- New limit = 9 / 3 = 3

**i = 0** (nums[0] = 3):
- 3 <= 3 → no split
- limit = 3

**Result**: 2 operations ✓

**Verification**:
- Original: [3, 9, 3]
- After 1st op: [3, 3, 6, 3] (split 9 → 3+6)
- After 2nd op: [3, 3, 3, 3, 3] (split 6 → 3+3)
- Sorted ✓

### Example 2: `nums = [1,2,3,4,5]`

**Initial state**: limit = 5

**i = 3**: 4 <= 5 → no split, limit = 4  
**i = 2**: 3 <= 4 → no split, limit = 3  
**i = 1**: 2 <= 3 → no split, limit = 2  
**i = 0**: 1 <= 2 → no split, limit = 1

**Result**: 0 operations ✓

## Mathematical Proof

### Why Split into k = ceil(x / limit) Parts?

**Goal**: Split `x` into parts ≤ `limit`.

**Minimum k**: If each part can be at most `limit`, we need at least `k = ceil(x / limit)` parts.

**Proof**:
- Sum of k parts = x
- Each part ≤ limit
- Therefore: x ≤ k × limit
- k ≥ x / limit
- Minimum integer k: k = ceil(x / limit)

### Why is Smallest Part = x / k?

**Even distribution**: Distribute `x` among `k` parts as evenly as possible.

**Math**:
- x = k × q + r (where q = x / k, r = x % k)
- r parts get (q + 1), remaining (k - r) parts get q
- Smallest part = q = x / k

**Example**: x = 9, limit = 3
- k = 3
- 9 / 3 = 3, 9 % 3 = 0
- Split: [3, 3, 3] → smallest = 3 ✓

**Example**: x = 10, limit = 3
- k = ceil(10/3) = 4
- 10 / 4 = 2, 10 % 4 = 2
- Split: [3, 3, 2, 2] → smallest = 2 ✓

## Edge Cases & Pitfalls

### Edge Case 1: Already Sorted
```
nums = [1, 2, 3]
```
No splits needed → 0 operations ✓

### Edge Case 2: Large Gap
```
nums = [1, 1000000000]
```
- i = 0: 1 <= 1000000000 → no split
- Result: 0 operations ✓

### Edge Case 3: Decreasing Array
```
nums = [5, 4, 3, 2, 1]
```
- limit = 1
- i = 3: 2 > 1 → k = 2, ops = 1, split [1, 1], limit = 1
- i = 2: 3 > 1 → k = 3, ops = 1+2=3, split [1, 1, 1], limit = 1
- i = 1: 4 > 1 → k = 4, ops = 3+3=6, split [1, 1, 1, 1], limit = 1
- i = 0: 5 > 1 → k = 5, ops = 6+4=10, split [1, 1, 1, 1, 1], limit = 1
- Result: 10 operations

### Pitfall: Integer Overflow
- Operations can reach 10⁵ × 10⁹ = 10¹⁴ in worst case
- Use `long` for operations counter

### Pitfall: Ceiling Division
```csharp
// WRONG: k = nums[i] / limit (floor division)
// RIGHT: k = (nums[i] + limit - 1) / limit (ceiling)
```

---

## Alternative Approaches (Suboptimal)

### Approach 1: Left-to-Right (Incorrect)

Processing left-to-right doesn't work because:
- Splitting early elements affects later elements unpredictably
- Cannot determine optimal split without knowing right constraints

### Approach 2: Dynamic Programming (Overkill)

Could use DP to track minimum operations for each subarray, but:
- O(n²) time complexity
- Unnecessary when greedy O(n) solution exists

---

## Key Takeaway

"Minimum operations to sort by splitting" → **Greedy right-to-left pass**. **Algorithm**:
1. **Start from right**: rightmost element sets initial constraint
2. **Process leftward**: for each element `nums[i]`
3. **Check constraint**: if `nums[i] > limit`, split into k parts
4. **Calculate k**: `k = ceil(nums[i] / limit)`
5. **Update**: operations += (k-1), limit = nums[i] / k

O(n) time, O(1) space. **Critical insight**: Work backwards because we can only decrease values (split), not increase. Smallest part after split becomes new constraint for left neighbor.
