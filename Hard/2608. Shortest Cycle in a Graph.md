# 2608. Shortest Cycle in a Graph
## O(n · (n + E)) — Multi-Source BFS: Cross-Edge Detection Formula

---

## Problem Statement

Undirected graph, `n` nodes, edges given as `[u, v]` pairs (no self-loops, no multi-edges). Return the length of the **shortest cycle** (girth), or `-1` if none exists.

A cycle uses each edge at most once and returns to the start node.

---

## Why Not DFS?

DFS finds cycles via **back-edges** (edges from a node to an ancestor in the DFS tree). The cycle length would be `depth[u] - depth[v] + 1` for a back-edge `u→v`. But the DFS tree path from `v` to `u` is **not guaranteed to be the shortest path** — it may snake through the graph unnecessarily. DFS can report a cycle length that's longer than the actual shortest cycle through those nodes.

**BFS uses shortest-path distances.** When BFS from source `s` detects a cross-edge `(u, v)` where both `dist[u]` and `dist[v]` are known, those distances are the **true shortest distances** from `s` to `u` and `v` in the graph. The resulting cycle length is provably tight.

---

## Core Insight #1 — BFS Cross-Edges Reveal Cycles

During BFS from source `s`, when we process node `u` (at distance `dist[u]`) and encounter a neighbor `v` that is **already visited** and **not `u`'s tree parent**:
- There is a cycle: `s → ... → u → v → ... → s`.
- Its length is `dist[u] + 1 + dist[v]`.

```
      s
     / \
dist[v]  dist[u]
    v — — u
     cycle!
```

The formula `dist[u] + dist[v] + 1` holds for both cases:
- **Cross-edge** (same BFS level, `dist[v] == dist[u]`): cycle length `2·dist[u] + 1` (odd).
- **Back-edge** (different levels, `dist[v] < dist[u]`, v ≠ parent[u]): cycle length `dist[u]+dist[v]+1` (could be even or odd).

---

## Core Insight #2 — The Parent Check Avoids False Positives

In an undirected graph, every tree edge `(u, parent[u])` appears in both directions. Without the parent check, we'd detect the tree edge itself as a "cycle of length 2," which is wrong.

Rule: when processing node `u` and seeing neighbor `v`:
- Skip if `dist[v] == -1` → tree edge (discover `v`).
- Skip if `v == parent[u]` → the edge we arrived on.
- Otherwise → genuine cross/back edge → cycle of length `dist[u] + dist[v] + 1`.

The parent check is sufficient because the problem guarantees no multi-edges (so u can have at most one edge to its parent).

---

## Core Insight #3 — Try Every Node as BFS Source

BFS from a single source `s` finds the shortest cycle **that passes through `s`**. Different cycles pass through different nodes. To guarantee finding the global shortest cycle, run BFS from every node `s ∈ [0, n-1]` and take the minimum.

**Why this is correct:** For the globally shortest cycle of length `g`, pick any node `s` on that cycle. BFS from `s` explores both "arms" of the cycle simultaneously. The arms meet at the node diametrically opposite `s` (at distance `⌊g/2⌋`), where the cross-edge gives:
```
dist[u] + dist[v] + 1 = ⌊g/2⌋ + ⌈g/2⌉ + 1
```
Wait: ⌊g/2⌋ + ⌈g/2⌉ = g-1. So cycle length = g-1+1 = g. ✓

## Full Implementation (C#)

```csharp
public class Solution {
    public int FindShortestCycle(int n, int[][] edges) {
        // Build adjacency list
        var adj = new List<int>[n];
        for (int i = 0; i < n; i++) adj[i] = new List<int>();
        foreach (var e in edges) {
            adj[e[0]].Add(e[1]);
            adj[e[1]].Add(e[0]);
        }

        int ans = int.MaxValue;

        for (int s = 0; s < n; s++) {
            int[] dist = new int[n];
            int[] par  = new int[n];
            Array.Fill(dist, -1);
            Array.Fill(par,  -1);

            dist[s] = 0;
            var q = new Queue<int>();
            q.Enqueue(s);

            while (q.Count > 0) {
                int u = q.Dequeue();
                foreach (int v in adj[u]) {
                    if (dist[v] == -1) {
                        // Unvisited: tree edge
                        dist[v] = dist[u] + 1;
                        par[v]  = u;
                        q.Enqueue(v);
                    } else if (v != par[u]) {
                        // Already visited, not our parent: cycle found
                        ans = Math.Min(ans, dist[u] + dist[v] + 1);
                    }
                }
            }
        }

        return ans == int.MaxValue ? -1 : ans;
    }
}
```

## Complexity

| Metric | Value | Notes |
|---|---|---|
| BFS runs | n | one per source node |
| Each BFS | O(n + E) | standard BFS cost |
| **Total Time** | **O(n(n + E))** | ≤ 1000 × 2000 = 2×10⁶ ops |
| Space | O(n + E) | adjacency list + dist/par arrays per BFS |

With `n ≤ 1000` and `E ≤ 1000`, this is well within limits.

## Example Walkthrough

### Example 1 — Two triangles: `n=7`, edges form 0–1–2–0 and 3–4–5–6–3

**BFS from s=0:**

| Step | Queue | Visiting | Neighbor | Action |
|---|---|---|---|---|
| 1 | [0] | 0 (d=0) | 1 | tree: dist[1]=1 |
| 1 | [0] | 0 (d=0) | 2 | tree: dist[2]=1 |
| 2 | [1,2] | 1 (d=1) | 0 | parent, skip |
| 2 | [1,2] | 1 (d=1) | 2 | visited, 2≠par[1]=0 → **cycle: 1+1+1=3** |

`ans = 3`. BFS continues but finds no shorter cycle.

**Output: 3** ✓

---

### Example 2 — `n=4`, edges `[[0,1],[0,2]]` — path graph

**BFS from every source:** Only tree edges are encountered; no cross/back edges (it's a tree). `ans` remains `int.MaxValue`.

**Output: -1** ✓

---

### Extended: 4-cycle 0–1–2–3–0

**BFS from s=0:**
- dist: [0, 1, -, 1] after processing 0. (Neighbors of 0 are 1 and 3.)
- Process 1 (d=1): see 0 (parent, skip), see 2 (unvisited): dist[2]=2.
- Process 3 (d=1): see 0 (parent, skip), see 2 (dist=2, ≠ parent[3]=0): **cycle = 1+2+1 = 4.** ✓

## Pitfalls & Edge Cases

- **Why BFS and not DFS:** DFS back-edge gives cycle length = `depth[u] - depth[ancestor] + 1` in the DFS tree, which may overcount (the DFS path is not a shortest path). A DFS back-edge from depth 10 to depth 2 gives "cycle of length 9" but there may be a direct edge cutting it short. BFS `dist[]` values are true shortest-path distances, so the formula is provably correct.

- **The parent check guards against the traversing-back-our-own-edge false positive**, but only works for simple graphs (no multi-edges). Since the problem guarantees no repeated edges, we're safe. If there were multi-edges, we'd need to track edge IDs instead of node IDs, and a multi-edge would give a valid 2-cycle.

- **Disconnected graph:** BFS from a node only explores its component. But running from every source covers every component. Components with no cycles contribute nothing to `ans`. ✓

- **No cycle exists:** Every BFS tree is a tree (forest), no cross-edges are ever found. `ans` stays `int.MaxValue`, return `-1`. ✓

- **Triangle is the minimum detectable cycle (length 3):** Two nodes at BFS distance 1 connected by an edge give `1+1+1=3`. No cycle of length 1 (self-loop) or 2 (multi-edge) can exist per constraints. ✓

- **Don't early-exit the BFS when finding a cycle.** We might find a cycle of length 5 early in the BFS from s, but a later cross-edge might give length 3. Complete each BFS to find the shortest cycle through that source. (Optimization: early-exit if `ans` already equals 3, the minimum possible.)

---

## Key Takeaway

This is the **"BFS for girth"** pattern: the shortest cycle is the **girth** of the graph, found by multi-source BFS. The core formula `dist[u] + dist[v] + 1` works because BFS distances are true shortest-path lengths — a property DFS doesn't have.

**Pattern recognition:** "Shortest cycle in an undirected unweighted graph with small n" → BFS from every node, detect cross/back edges, take minimum of `dist[u]+dist[v]+1`. For larger graphs (n ≤ 10⁵), specialized algorithms (BFS on each node with early termination, or even-cycle detection via bipartiteness) become necessary. Here O(n²) is fine.
