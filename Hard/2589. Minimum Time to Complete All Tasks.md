# 2589. Minimum Time to Complete All Tasks
## O(n log n + n·T) — Sort by End Time + Greedy Right-to-Left Filling

---

## Problem Statement

- Each task needs `duration[i]` seconds turned on within `[start[i], end[i]]` (inclusive, integer time slots).
- The computer can run unlimited tasks simultaneously when on.
- Return minimum total "on" time covering all tasks' duration requirements.

---

## Core Idea — Greedy: Sort by Deadline, Fill From the End

**Key insight:** Process tasks in order of increasing `end` time. For each task, count how many of its required slots are already "on" (from previously processed tasks, since shared on-time satisfies multiple tasks). If not enough, turn on additional slots, **preferring the latest possible slots** (closest to `end`) — this maximizes the chance these slots also help future tasks (which have later or equal deadlines).

**Why latest slots first?** Future tasks have `end ≥` current task's `end`. A slot turned on near the current task's `end` is more likely to fall within a future task's range than a slot turned on early in the current range.

---

## Full Optimal Implementation (C#)

```csharp
public class Solution {
    public int FindMinimumTime(int[][] tasks) {
        // Sort by end time ascending
        Array.Sort(tasks, (a, b) => a[1].CompareTo(b[1]));

        int maxEnd = 0;
        foreach (var t in tasks) maxEnd = Math.Max(maxEnd, t[1]);

        bool[] on = new bool[maxEnd + 2]; // 1-indexed time slots
        int total = 0;

        foreach (var task in tasks) {
            int start = task[0], end = task[1], duration = task[2];

            // Count already-on slots in [start, end]
            int already = 0;
            for (int t = start; t <= end; t++)
                if (on[t]) already++;

            int need = duration - already;
            if (need <= 0) continue;

            // Fill from end backwards
            for (int t = end; t >= start && need > 0; t--) {
                if (!on[t]) {
                    on[t] = true;
                    total++;
                    need--;
                }
            }
        }

        return total;
    }
}
```

n ≤ 2000, T ≤ 2000 → worst case ~4×10⁶ ops. Fast enough.

## Why Sorting by End Time (Not Start) Is Correct

If we processed by start time, a task with an early start but very late end could "waste" its on-slots early, not helping tasks with tighter (earlier) deadlines that come later in start-order but need to be satisfied sooner.

By sorting on **end time**, when we process task i, all previously processed tasks have `end ≤ task[i].end`. Filling near the end of task i's range maximizes overlap potential with all *remaining* tasks (whose end ≥ task[i].end), since their ranges could start anywhere but end later — slots near task i's end have the best chance of being in range `[future_start, future_end]`.

## Example Walkthrough

**Example 2:** `tasks = [[1,3,2],[2,5,3],[5,6,2]]`

Sorted by end: [1,3,2], [2,5,3], [5,6,2] (already sorted).

**Task [1,3,2]:** range [1,3], duration 2. already=0, need=2.
Fill from t=3 backward: on[3]=true, on[2]=true. total=2. on = {2,3}

**Task [2,5,3]:** range [2,5], duration 3. Check on[2..5]: on[2]=T, on[3]=T, on[4]=F, on[5]=F → already=2. need=1.
Fill from t=5 backward: on[5]=true. total=3. on = {2,3,5}

**Task [5,6,2]:** range [5,6], duration 2. Check on[5..6]: on[5]=T, on[6]=F → already=1. need=1.
Fill from t=6 backward: on[6]=true. total=4. on = {2,3,5,6}

**Output:** 4 ✓ (matches expected)

## Pitfalls & Edge Cases

- **Ties in end time:** order among same-end tasks doesn't matter — both will be filled within their (identical) range. ✓

- **Single time-slot tasks** (`start == end`): forces that exact slot on, which is correct. ✓

- **`already > duration`:** `need ≤ 0`, skip — already satisfied by shared slots. ✓

- **Array bounds:** `on` sized to `maxEnd + 2` avoids out-of-bounds; all tasks have `end ≤ maxEnd`. ✓

## Key Takeaway

This is the classic **"interval scheduling with deadline-greedy, fill backward"** pattern — same family as "minimum platforms" or "job scheduling with deadlines."

**General recipe:**
1. Sort by deadline (end time) ascending.
2. For each interval, count resources already allocated within range.
3. If insufficient, greedily allocate the *remaining slots closest to the deadline* — they have maximum future utility since all subsequent intervals have deadlines ≥ current.

**Pattern recognition:** "Minimize total resource usage across overlapping intervals with per-interval minimum requirements" → sort by end, greedily fill from the right.
