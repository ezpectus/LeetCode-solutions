# 2003. Smallest Missing Genetic Value in Each Subtree
## O(n) — Path from Node-1 to Root + Incremental Set DFS

---

## Problem Statement

- Tree rooted at 0. Each node has a **distinct** genetic value.
- For each node, find its **MEX** (minimum excludant = smallest positive integer missing from its subtree).
- Return array `ans` where `ans[i]` = MEX of subtree rooted at `i`.

---

## Core Idea — Only Ancestors of the Node-with-value-1 Matter

**Key observation**: if no node has value `1`, then every subtree is missing `1` → `ans[i] = 1` for all `i`.

If some node has value `1`, only its ancestors can have `ans > 1` — all other nodes don't contain `1` in their subtree, so their MEX is `1`.

**Algorithm**:
1. Find `node1` = the node where `nums[node1] = 1`.
2. Walk from `node1` up to the root — this is the **path**.
3. Process path nodes bottom-up. For each path node, DFS its subtree (skipping path nodes below it, which were already processed). Add new values to a growing `collected` set and advance `mex`.

**Efficient collection**: since path nodes are never re-entered during DFS, each tree node is visited exactly once across all iterations → O(n) total.

---

## Full Optimal Implementation (C#)

```csharp
public class Solution {
    public int[] SmallestMissingValueSubtree(int[] parents, int[] nums) {
        int n = nums.Length;
        int[] ans = new int[n];
        Array.Fill(ans, 1);

        int node1 = -1;
        for (int i = 0; i < n; i++) if (nums[i] == 1) { node1 = i; break; }
        if (node1 == -1) return ans;

        var children = new List<int>[n];
        for (int i = 0; i < n; i++) children[i] = new List<int>();
        for (int i = 1; i < n; i++) children[parents[i]].Add(i);

        // Build path from node1 up to root
        var path = new List<int>();
        int curr = node1;
        while (curr != -1) {
            path.Add(curr);
            curr = curr == 0 ? -1 : parents[curr];
        }

        var pathSet  = new HashSet<int>(path);
        var collected = new HashSet<int>();
        var visited   = new HashSet<int>(); // ensures each node is processed exactly once
        int mex = 1;

        foreach (int node in path) {
            var stack = new Stack<int>();
            stack.Push(node);
            while (stack.Count > 0) {
                int u = stack.Pop();
                if (!visited.Add(u)) continue;   // skip already-visited nodes
                collected.Add(nums[u]);
                foreach (int c in children[u])
                    if (!pathSet.Contains(c) && !visited.Contains(c))
                        stack.Push(c);
            }
            pathSet.Remove(node); // unblock this path node for future DFS passes
            while (collected.Contains(mex)) mex++;
            ans[node] = mex;
        }

        return ans;
    }
}
```

---

## Complexity

| **Metric** | **Value** | **Notes** |
|---|---|---|
| **Time** | **O(n)** | Each node added to `visited` exactly once across all DFS calls |
| **Space** | **O(n)** | `visited`, `collected`, `pathSet`, stack, path |

---

## Why This Works — Three Interlocking Pieces

**1. Why only the path from `node1` to root matters?**

MEX of a subtree is `1` unless the subtree contains `1`. The only nodes whose subtrees contain `node1` (and thus value `1`) are exactly its ancestors — i.e., the nodes on the path from `node1` to root. All other nodes can never have MEX > 1.

**2. Why `visited` is necessary (the TLE fix)?**

Without `visited`, every path node re-traverses all subtrees that previous path nodes already explored. For a path of length `k` with total subtree size `S`, that's `O(k × S)` = O(n²) in the worst case. With `visited`, each node is pushed and popped from the stack exactly once across all path iterations — O(n) total.

**3. Why `pathSet.Remove(node)` after processing?**

`pathSet` initially blocks all path nodes from being entered via DFS (so the DFS at each step stays local to that node's non-path subtrees). Once we move up to process the parent path node, the current node should be reachable as a regular tree node — we remove it from `pathSet` so the parent's DFS can include it (but `visited` will skip it anyway since it was already processed).

**Why MEX only increases along the path?**

We only ever *add* to `collected` — never remove. So `mex` is non-decreasing as we walk up. This means we never reset `mex` to 1 per node; we just continue advancing from where we left off.

## Example Walkthrough

**`parents=[-1,0,1,0,3,3], nums=[5,4,6,2,1,3]`**

Tree structure:
```
        0(5)
       / \
     1(4) 3(2)
      |   / \
     2(6)4(1) 5(3)
```

Node with value 1 = node 4. Path: `4 → 3 → 0`.

| Step | node | DFS explores | collected | mex | ans[node] |
|------|------|-------------|-----------|-----|-----------|
| 1 | 4 | node 4 only (no non-path children) | {1} | 2 | **2** |
| 2 | 3 | node 3, then node 5 (node 4 already visited) | {1,2,3} | 4 | **4** |
| 3 | 0 | node 0, then nodes 1, 2 | {1,2,3,4,5,6} | 7 | **7** |

Final: `ans = [7, 1, 1, 4, 2, 1]` ✓

---

## Pitfalls & Edge Cases

- **No node with value 1**: return all 1s immediately — no path to walk
- **`node1` is the root**: path = `[0]`, single step, DFS covers the entire tree
- **`visited` vs `pathSet` are different guards**:
  - `pathSet` prevents entering path nodes from a sibling DFS (they'll be processed in their own turn)
  - `visited` prevents re-processing any node already seen in a previous path iteration
- **`mex` is never reset**: it monotonically increases — always continue from the last value, never restart from 1

---

## Key Takeaway

"Subtree MEX for all nodes" → **only ancestors of `node1` matter; incremental DFS up the path with a visited set**:

```
path = node1 → parent → ... → root
visited = {}

for node in path:
    DFS from node, skip visited and unprocessed path nodes
    mark all newly reached nodes in visited
    advance mex while mex in collected
    ans[node] = mex
    remove node from pathSet
```

`visited` guarantees O(n) total work. `mex` monotonically increases. All non-path nodes keep `ans = 1`.
