# 3219. Minimum Cost for Cutting Cake II
## O(n log n + m log m) — Greedy: Most Expensive Cuts First with Multiplier

---

## Problem Statement

`m × n` cake, cut into `1 × 1` pieces. Cutting along horizontal line `i` costs `horizontalCut[i]`, vertical line `j` costs `verticalCut[j]`. Each cut applies to exactly the piece you choose; to cut the same line across multiple pieces, you pay the cost multiple times. Minimize total cost.

---

## Core Insight #1 — Every Line Must Be Cut Exactly Once Per Perpendicular Segment

When you cut horizontal line `i`, you pay `horizontalCut[i]` for the one piece you cut it on. But later, the same horizontal line `i` still exists in all adjacent vertical segments — you must cut it there too, each at cost `horizontalCut[i]`.

**Total cost of all cuts on horizontal line `i`** = `horizontalCut[i] × (number of vertical cuts already made + 1)`.

The same applies symmetrically: **total cost of all cuts on vertical line `j`** = `verticalCut[j] × (number of horizontal cuts already made + 1)`.

This is because each time a perpendicular cut is made, the pieces multiply, and the line must be re-cut in the new piece.

---

## Core Insight #2 — The Multiplier Grows Monotonically

When we make cuts in some order, the multiplier for any remaining horizontal cut is `(vertical cuts made so far + 1)`. This only increases as we make more vertical cuts. So:

- Making a horizontal cut **early** → low multiplier → low cost.
- Making a horizontal cut **late** (after many vertical cuts) → high multiplier → high cost.

To minimize the total cost we want the most **expensive** cuts to have the **smallest** multipliers. That means making expensive cuts first, before the perpendicular cuts multiply them.

---

## Core Insight #3 — Greedy Exchange Argument

**Claim:** In the optimal solution, cuts are made in **decreasing order of cost** (merging horizontal and vertical cuts into one sorted sequence).

**Proof (swap argument):** Suppose in some sequence, a cut of cost `A` appears before a cut of cost `B < A` (same direction ignored for now). The costs involve multipliers that are monotonically non-decreasing over time. Swapping them can only keep or reduce total cost. The formal calculation for two adjacent cuts:

*If we swap cut H (cost h_val) and cut V (cost v_val) where h_val < v_val:*
- Before swap: `h_val × (v + 1) + v_val × (h + 2)` (H first, then V)
- After swap: `v_val × (h + 1) + h_val × (v + 2)` (V first, then H)

Difference: `(before) − (after) = h_val×(v+1) + v_val×(h+2) − v_val×(h+1) − h_val×(v+2) = h_val − v_val < 0`.

"Before" is cheaper → we should always place the cheaper cut first, i.e., more expensive cuts go first. ✓

---

## Algorithm

```
Sort horizontalCut[] descending
Sort verticalCut[] descending
h_cuts = 0, v_cuts = 0   (cuts made so far of each type)
ans = 0

Use two pointers to merge-sort process both arrays:
  If next horizontal cut ≥ next vertical cut:
    ans += horizontalCut[i] × (v_cuts + 1)
    h_cuts++
  Else:
    ans += verticalCut[j] × (h_cuts + 1)
    v_cuts++

Return ans
```

---

## Full Implementation (C#)

```csharp
public class Solution {
    public long MinimumCost(int m, int n, int[] horizontalCut, int[] verticalCut) {
        Array.Sort(horizontalCut, (a, b) => b.CompareTo(a)); // descending
        Array.Sort(verticalCut,   (a, b) => b.CompareTo(a)); // descending

        long ans = 0;
        int h = 0, v = 0; // cuts made so far
        int i = 0, j = 0; // pointers into sorted arrays

        while (i < horizontalCut.Length || j < verticalCut.Length) {
            bool takeH = j >= verticalCut.Length ||
                         (i < horizontalCut.Length && horizontalCut[i] >= verticalCut[j]);

            if (takeH) {
                ans += (long)horizontalCut[i] * (v + 1);
                h++;
                i++;
            } else {
                ans += (long)verticalCut[j] * (h + 1);
                v++;
                j++;
            }
        }

        return ans;
    }
}
```

## Complexity

| Metric | Value | Notes |
|---|---|---|
| Sort horizontalCut | O(m log m) | m−1 cuts |
| Sort verticalCut | O(n log n) | n−1 cuts |
| Greedy merge pass | O(m + n) | one pointer per array |
| **Total** | **O(m log m + n log n)** | ≈ 10⁵ × 17 ≈ 1.7M ops |
| Space | O(1) extra | in-place sorts |

## Multiplier Growth — Illustrated

```
Initial state:   h_cuts=0, v_cuts=0
                 H-multiplier = v_cuts+1 = 1
                 V-multiplier = h_cuts+1 = 1

After 1 V cut:   v_cuts=1
                 H-multiplier = 2   ← H cuts now 2× more expensive!
                 V-multiplier = 1   (still cheap)

After 1 H cut:   h_cuts=1
                 V-multiplier = 2
                 H-multiplier = 2 (if v_cuts=1)
```

Making expensive cuts early "locks in" low multipliers for them.

## Example Walkthrough

### Example 1 — `m=3, n=2, horizontalCut=[1,3], verticalCut=[5]`

Sorted: H = [3, 1], V = [5]

| Step | Cut | Cost | Multiplier | Running Total | h | v |
|---|---|---|---|---|---|---|
| 1 | V=5 | 5 | h+1 = 0+1 = 1 | 5 | 0 | 1 |
| 2 | H=3 | 3 | v+1 = 1+1 = 2 | 5+6=11 | 1 | 1 |
| 3 | H=1 | 1 | v+1 = 1+1 = 2 | 11+2=**13** | 2 | 1 |

**Output: 13** ✓

Explanation matches: V=5 (1 piece width), H=3 applied twice (2 vertical segments), H=1 applied twice.

---

### Example 2 — `m=2, n=2, horizontalCut=[7], verticalCut=[4]`

Sorted: H = [7], V = [4]

| Step | Cut | Cost | Multiplier | Total |
|---|---|---|---|---|
| 1 | H=7 | 7 | v+1 = 1 | 7 |
| 2 | V=4 | 4 | h+1 = 2 | 7+8=**15** |

**Output: 15** ✓

If we had cut V first (wrong order):
- V=4 × (0+1) = 4
- H=7 × (1+1) = 14 → Total = 18. More expensive!

---

### Tie-Breaking When Costs Are Equal

If `horizontalCut[i] == verticalCut[j]`, the code takes H first. Does order matter?

For equal costs `c = H = V` at state `(h, v)`:
- H then V: `c×(v+1) + c×(h+2) = c×(v+h+3)`
- V then H: `c×(h+1) + c×(v+2) = c×(v+h+3)`

**Identical cost.** Tie-breaking is irrelevant. ✓

## Pitfalls & Edge Cases

- **Overflow:** `horizontalCut[i] ≤ 10³` and multiplier ≤ max(m,n) ≤ 10⁵. Product ≤ 10⁸. Total sum ≤ (m+n) × 10⁸ ≈ 2×10¹³. Exceeds `int` range → use `long`. Cast before multiplication: `(long)horizontalCut[i] * (v + 1)`.

- **Only horizontal cuts:** If `verticalCut` is empty, `j >= verticalCut.Length` is always true, and `takeH = true` always. The multiplier for horizontal cuts = `v_cuts + 1 = 1` always (no vertical cuts). Each horizontal cut costs its face value. Correct: with no vertical divisions, each H line needs to be cut exactly once. ✓

- **One row or one column:** If m=1, `horizontalCut` is empty; all costs come from vertical cuts × 1 (h_cuts stays 0). If n=1, `verticalCut` is empty. Both handled naturally.

- **Descending sort is essential.** Using ascending sort would give the wrong answer — it would schedule cheap cuts first, leaving expensive cuts with large multipliers.

---

## Key Takeaway

This is the **"greedy with multiplicative costs"** pattern. When making a cut "earlier" keeps its cost multiplier small (and doesn't affect other cuts' multipliers), the optimal strategy is always: **process the most expensive items first**. This is proven by the exchange argument: swapping any two adjacent cuts (cheaper before more expensive) never increases cost.

**Pattern recognition:** "Each operation costs a fixed amount × current width/height" → the cost of each line is `base_cost × (perpendicular_cuts_so_far + 1)`. Sort all operations descending, apply greedily. Same pattern appears in the "Minimum Cost to Connect Sticks" and similar problems.
