# 3244. Shortest Distance After Road Addition Queries II
## O(n α(n) + q) — DSU "Skip Path" with Non-Crossing Interval Property

---

## Problem Statement

Graph with cities 0..n-1 and edges i→i+1. After each query [u, v] (adding edge u→v), return shortest path length from 0 to n-1. The critical constraint: **no two queries cross** (queries[i][0] < queries[j][0] < queries[i][1] < queries[j][1] is forbidden).

---

## Core Insight #1 — Non-Crossing Means Containment

The non-crossing constraint means all added intervals are either **nested** or **disjoint** — they form a **laminar family**. This has a crucial consequence:

**When we add edge [u, v], every shortcut starting from any node w ∈ (u, v) must also end ≤ v (inside [u, v]).**

Proof: If some w ∈ (u, v) had a shortcut to z > v, the intervals [u,v] and [w,z] would satisfy u < w < v < z — a crossing. Forbidden!

Therefore, **the shortcut u→v completely dominates the segment (u, v)**: no shortcut from within (u, v) can reach beyond v, so the entire sub-path from u to v is replaced by the single edge u→v.

---

## Core Insight #2 — Shortest Path = Active Node Count − 1

Define the set of **active nodes** = nodes currently visited by the shortest path. Initially, all nodes {0, 1, ..., n-1} are active, and the path is 0→1→...→n-1 with length n-1.

**Path length = |active nodes| − 1** (since we traverse from the first to the last active node).

When we add edge [u, v]:
- All active nodes strictly between u and v are **removed** from the path.
- Path length decreases by the count of removed nodes.

---

## Core Insight #3 — DSU to Find "Next Active Node"

Use Union-Find where `find(i)` returns the **first active node ≥ i**:
- When node `w` is active: `par[w] = w`.
- When node `w` is deactivated: `par[w] = w+1`. Path compression makes future queries O(α(n)).

**Processing query [u, v]:**
```
w = find(u+1)           // first active node after u
while w < v:
    pathLen--            // remove w from path
    par[w] = w+1         // deactivate w
    w = find(w+1)        // next active node
```

Each node is deactivated **at most once** across all queries → total work = O(n α(n) + q).

---

## Full Implementation (C++)

```cpp
class Solution {
    vector<int> par;

    int find(int x) {
        while (par[x] != x) {
            par[x] = par[par[x]]; // path halving for O(α(n))
            x = par[x];
        }
        return x;
    }

public:
    vector<int> shortestDistanceAfterQueries(int n, vector<vector<int>>& queries) {
        par.resize(n);
        iota(par.begin(), par.end(), 0); // par[i] = i (every node starts active)

        int pathLen = n - 1; // initial path: 0→1→...→n-1
        vector<int> ans;

        for (auto& q : queries) {
            int u = q[0], v = q[1];
            // Remove all active nodes strictly between u and v
            for (int w = find(u + 1); w < v; w = find(w + 1)) {
                pathLen--;
                par[w] = w + 1; // deactivate w, point toward w+1
            }
            ans.push_back(pathLen);
        }

        return ans;
    }
};
```

---

## Complexity

| Metric | Value | Notes |
|---|---|---|
| Total deactivations | O(n) | Each node deactivated ≤ once |
| Per deactivation | O(α(n)) amortized | Path compression in DSU |
| Per query overhead | O(α(n)) | find(u+1) call |
| **Total time** | **O(n α(n) + q)** | ≈ linear for practical inputs |
| Space | O(n + q) | DSU array + output |

## Example Walkthrough

### `n=5`, `queries=[[2,4],[0,2],[0,4]]`

**Initial:** par=[0,1,2,3,4], pathLen=4. Active: {0,1,2,3,4}.

**Query [2,4]:**
```
w = find(3) = 3.  3 < 4 → pathLen=3, par[3]=4.
w = find(4) = 4.  4 ≮ 4 → stop.
```
Active: {0,1,2,**~~3~~**,4}. Path: 0→1→2→4. ans=[3] ✓

**Query [0,2]:**
```
w = find(1) = 1.  1 < 2 → pathLen=2, par[1]=2.
w = find(2) = 2.  2 ≮ 2 → stop.
```
Active: {0,**~~1~~**,2,4}. Path: 0→2→4. ans=[3,2] ✓

**Query [0,4]:**
```
w = find(1) = find(par[1]) = find(2) = 2.  2 < 4 → pathLen=1, par[2]=3.
w = find(3) = find(par[3]) = find(4) = 4.  4 ≮ 4 → stop.
```
Active: {0,**~~2~~**,4}. Path: 0→4. ans=[3,2,1] ✓

### `n=4`, `queries=[[0,3],[0,2]]`

**Initial:** pathLen=3.

**Query [0,3]:**
```
w=find(1)=1 < 3 → pathLen=2, par[1]=2.
w=find(2)=2 < 3 → pathLen=1, par[2]=3.
w=find(3)=3 ≮ 3 → stop.
```
ans=[1] ✓

**Query [0,2]:**
```
w=find(1)=find(2)=find(3)=3 ≮ 2 → stop. (nothing removed)
```
ans=[1,1] ✓

## Why the Non-Crossing Constraint Is Essential

**Without non-crossing:** Adding [0,4] and [2,6] (which cross) means that from node 3 ∈ (0,4), there's a shortcut 3→7 that extends BEYOND 4. The shortcut [0,4] would NOT absorb node 3 (since the optimal path from 3 to n-1 goes 3→7, not 3→4→...→n-1). So removing 3 from the active set would be WRONG.

**With non-crossing (Part II guarantee):** This situation is impossible. Every shortcut from within (u, v) is guaranteed to stay within [u, v]. So removing all active nodes from (u, v) is always safe.

---

## Pitfalls & Edge Cases

- **query [u, v] where v − u = 1 (adjacent):** The loop starts at `find(u+1) = find(v) = v` which is NOT `< v`, so the loop doesn't execute. pathLen stays the same. Correct (no intermediate nodes to remove).

  *But the problem states `queries[i][1] - queries[i][0] > 1`, so this can't happen.* Still handled correctly.

- **Redundant query (nodes already removed):** If all nodes in (u, v) were already removed by previous queries, `find(u+1)` immediately returns something ≥ v. Loop doesn't execute. pathLen unchanged. ✓

- **Path compression after deactivation:** After `par[w] = w+1`, future calls to `find(w)` skip directly to the next active node, not just w+1 (because w+1 might also be deactivated). Path compression propagates this lazily. ✓

---

## Key Takeaway

This is the **"interval absorption with non-crossing guarantee"** pattern. The non-crossing property ensures each added edge [u,v] completely absorbs its interior, allowing a simple DSU to track active nodes. Without this property (Part I), the intervals can cross and we'd need BFS/Dijkstra per query.

**Pattern recognition:** "Range of nodes becomes redundant when edge bypasses them, with laminar family structure" → DSU where `find(i)` = next un-bypassed node. Each node enters the DSU once and is compressed lazily, giving near-linear total complexity.
