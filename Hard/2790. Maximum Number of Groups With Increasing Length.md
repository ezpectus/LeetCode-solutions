# 2790. Maximum Number of Groups With Increasing Length

## O(N log N) — Greedy Resource Accumulation via Math Math Prefix Bounds

### Problem Statement
You are given a 0-indexed array `usageLimits` of length $n$. Your task is to create groups using numbers from $0$ to $n - 1$, ensuring that each number `i` is used no more than `usageLimits[i]` times in total across all groups. 

You must also satisfy the following conditions:
1. Each group must consist of **distinct** numbers.
2. Each group (except the first one) must have a length **strictly greater** than the previous group.

Return an integer denoting the maximum number of groups you can create.

---

### Core Idea — Triangular Math Bounds and Resource Accumulation
If we want to create $k$ groups of strictly increasing lengths, the most efficient way is to make their lengths exactly $1, 2, 3, \dots, k$. The total number of elements required to form these $k$ groups is the triangular number:
$$\text{Required Elements} = \sum_{i=1}^{k} i = \frac{k(k + 1)}{2}$$

However, because of the strict constraint that **each group must contain distinct numbers**, we cannot just blindly use any elements. To maximize our chance of filling these slots, we must sort `usageLimits` in ascending order. 

Why sort ascending? Because smaller limits are highly restrictive. By accumulating our available resources from smallest to largest, we can check if the current prefix sum of allocations is structurally sufficient to support a targeted number of groups.

Let `totalAvailable` be the running sum of elements we can safely pull from. As we iterate through the sorted `usageLimits`:
1. We add the current limit to our pool: `totalAvailable += limit`.
2. We evaluate if this pool is big enough to sustain creating the *next* sequential group size. If we currently have successfully validated `count` groups, the next group will need a length of `count + 1`.
3. The total elements needed for `count + 1` groups is $\frac{(\text{count} + 1)(\text{count} + 2)}{2}$. 
4. If `totalAvailable` is greater than or equal to this threshold, it means we have enough mathematical headroom to safely form this next group. We then increment `count++`.

Any excess elements from larger limits naturally spill over and help satisfy the distinctness criteria for subsequent groups.

### Full Optimal Implementation (C#)

```csharp
using System;
using System.Collections.Generic;
using System.Linq;

public class Solution {
    public int MaxIncreasingGroups(IList<int> usageLimits) {
        // Step 1: Sort the limits ascending to handle the most restrictive elements first
        List<int> sortedLimits = usageLimits.ToList();
        sortedLimits.Sort();
        
        long totalAvailable = 0;
        int count = 0;
        
        // Step 2: Greedily accumulate resource limits
        foreach (int limit in sortedLimits) {
            totalAvailable += limit;
            
            // Step 3: Check if total elements can satisfy a triangle of size (count + 1)
            // Total required elements for (count + 1) groups = (count + 1) * (count + 2) / 2
            long nextGroupRequirement = ((long)(count + 1) * (count + 2)) / 2;
            
            if (totalAvailable >= nextGroupRequirement) {
                count++;
            }
        }
        
        return count;
    }
}
```
### Complexity

| Metric | Value | Notes |
| :--- | :--- | :--- |
| **Time Complexity** | $O(N \log N)$ | Dominated entirely by sorting the `usageLimits` collection. The subsequent iteration processes each element exactly once in $O(1)$ scalar steps, leading to an extremely fast runtime that easily handles $N = 10^5$. |
| **Space Complexity** | $O(N)$ | Required to create a mutable `List<int>` copy of the input `IList<int>` interface to trigger the quicksort routing. |

### Pitfalls & Edge Cases

* **Integer Arithmetic Overflow:** The cumulative sum of elements (`totalAvailable`) and the mathematical formulas checking group size bounds can quickly exceed the standard 32-bit `int` container max capacity ($2 \times 10^9$) due to $10^5 \times 10^9$ max scale. Always explicitly cast calculation segments to 64-bit `long`.
* **The "Distinct" Misconception:** It feels like sorting and tracking index positions is required to enforce the "distinct elements per group" rule. However, sorting the array ascending guarantees that any over-abundant resource processed later can dynamically fill the base/widest slots of our virtual inverted pyramid without violating uniqueness rules.

---

### Key Takeaway

> When a combinatorial group-building constraint requires strictly increasing partition sizes combined with element frequency caps, bypass physical matrix construction. Sort the frequency array to prioritize restricted items, and validate feasibility using sequential prefix math sum thresholds.

* **Pattern recognition:** "Create max number of subsets with strictly increasing lengths under usage limits" -> Sort Ascending + Running Prefix Allocation Tracker.
