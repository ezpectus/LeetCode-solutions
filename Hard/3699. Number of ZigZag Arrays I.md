# 3699. Number of ZigZag Arrays I
## O(n·m) — Direction-Tagged DP with Prefix Sum Transitions

---

## Problem Statement

Count arrays of length `n` where:
1. Every element ∈ `[l, r]`.
2. No two adjacent elements are equal.
3. No three consecutive elements are strictly increasing or strictly decreasing.

Return the count mod `10⁹ + 7`.

---

## Core Insight #1 — Condition 3 Forces Strict Alternation

Combine conditions 2 and 3:
- `a ≠ b`, `b ≠ c`.
- NOT `(a < b < c)` and NOT `(a > b > c)`.

If the last step was **up** (`prev < cur`), then `next > cur` would make three strictly increasing — forbidden. And `next == cur` is forbidden by condition 2. So `next` **must** be `< cur`.

Symmetrically, if the last step was **down**, the next step must be **up**.

Conclusion: every valid array strictly **alternates** between upward and downward steps — a true zigzag. The direction of each step fully determines the legal range for the next value.

---

## Core Insight #2 — State = (Last Value, Last Direction)

Define:
```
up[v]   = count of valid prefixes ending at value v where the last step was upward
down[v] = count of valid prefixes ending at value v where the last step was downward
```

**Transitions** (adding one more element at value `w`):

| Previous state | Constraint on `w` | Updates |
|---|---|---|
| `up[v]` (arrived going up at v) | `w < v` (must go down next) | `new_down[w] += up[v]` for all `w < v` |
| `down[v]` (arrived going down at v) | `w > v` (must go up next) | `new_up[w] += down[v]` for all `w > v` |

---

## Core Insight #3 — Prefix Sums Turn O(m²) Into O(m) Per Step

Naive transitions iterate over all pairs `(v, w)` — O(m²) per step, O(n·m²) total. Too slow for m=2000, n=2000.

Observe:
```
new_up[w]   = Σ down[v] for v ∈ [l, w-1]   = prefix sum of down up to w-1
new_down[w] = Σ up[v]   for v ∈ [w+1, r]   = suffix sum of up from w+1
```

With prefix/suffix arrays precomputed in O(m), every `new_up[w]` and `new_down[w]` is a single O(1) lookup. Total: O(m) per step, O(n·m) overall.

---

## Initialization (After Position 2)

For each value `w ∈ [l, r]`, the sequences of length 2 ending at `w` are:
- Up arrivals: any `prev ∈ [l, w-1]` → `up[w] = w - l`
- Down arrivals: any `prev ∈ [w+1, r]` → `down[w] = r - w`

---

## Prefix Sum Transition (Illustrated)

```
down[] = [d_l, d_{l+1}, ..., d_r]
                 ↓ prefix sum
prefixDown[w] = d_l + d_{l+1} + ... + d_{w-1}

new_up[w] = prefixDown[w]   (all down-values strictly below w)

up[] = [u_l, u_{l+1}, ..., u_r]
                 ↓ suffix sum
suffixUp[w] = u_{w+1} + ... + u_r

new_down[w] = suffixUp[w]   (all up-values strictly above w)
```

## Full Implementation (C#)

```csharp
public class Solution {
    public int ZigZagArrays(int n, int l, int r) {
        const long MOD = 1_000_000_007;
        int m = r - l + 1;  // number of distinct values

        // 0-indexed: index i corresponds to value (l + i)
        long[] up   = new long[m];  // up[i] = prefix ending at value l+i, last step up
        long[] down = new long[m];  // down[i] = prefix ending at value l+i, last step down

        // Initialize for prefixes of length 2
        for (int i = 0; i < m; i++) {
            up[i]   = i;         // i values strictly below (l+i) in [l, r]
            down[i] = m - 1 - i; // (m-1-i) values strictly above (l+i) in [l, r]
        }

        // Extend from length 2 to length n, one step at a time
        for (int step = 2; step < n; step++) {
            // Prefix sums of down[]: prefixDown[i] = down[0] + ... + down[i-1]
            long[] prefixDown = new long[m + 1];
            for (int i = 0; i < m; i++)
                prefixDown[i + 1] = (prefixDown[i] + down[i]) % MOD;

            // Suffix sums of up[]: suffixUp[i] = up[i] + ... + up[m-1]
            long[] suffixUp = new long[m + 1];
            for (int i = m - 1; i >= 0; i--)
                suffixUp[i] = (suffixUp[i + 1] + up[i]) % MOD;

            long[] newUp   = new long[m];
            long[] newDown = new long[m];
            for (int w = 0; w < m; w++) {
                newUp[w]   = prefixDown[w];      // sum of down[0..w-1]
                newDown[w] = suffixUp[w + 1];    // sum of up[w+1..m-1]
            }

            up   = newUp;
            down = newDown;
        }

        long ans = 0;
        for (int i = 0; i < m; i++)
            ans = (ans + up[i] + down[i]) % MOD;

        return (int)ans;
    }
}
```

| Metric | Value | Notes |
|---|---|---|
| Time Complexity | O(n · m) | n−2 steps × O(m) prefix scan |
| Space Complexity | O(m) | `up`, `down`, two sum arrays |

`n, m ≤ 2000` → ≤ 4×10⁶ iterations — well within limits.

## Example Walkthrough

### Example 1 — `n=3, l=4, r=5` → `m=2`

**Init (length 2):**
| index | value | up | down |
|---|---|---|---|
| 0 | 4 | 0 | 1 |
| 1 | 5 | 1 | 0 |

**Step to length 3:**
```
prefixDown = [0, 1, 1]
suffixUp   = [1, 1, 0]

newUp[0]   = prefixDown[0] = 0    (no value below 4)
newUp[1]   = prefixDown[1] = 1    (down[4] = 1)
newDown[0] = suffixUp[1]   = 1    (up[5] = 1)
newDown[1] = suffixUp[2]   = 0    (no value above 5)
```

**Answer = 0+1+1+0 = 2** ✓ `[4,5,4]` and `[5,4,5]`

---

### Example 2 — `n=3, l=1, r=3` → `m=3`

**Init (length 2):**
| index | value | up | down |
|---|---|---|---|
| 0 | 1 | 0 | 2 |
| 1 | 2 | 1 | 1 |
| 2 | 3 | 2 | 0 |

**Step to length 3:**
```
prefixDown = [0, 2, 3, 3]
suffixUp   = [3, 3, 2, 0]

newUp[0] = 0, newUp[1] = 2, newUp[2] = 3
newDown[0] = 3, newDown[1] = 2, newDown[2] = 0
```

**Answer = 0+2+3+3+2+0 = 10** ✓

Manual check: 10 sequences verified in problem statement.

## Growth Pattern for Fixed n=3

The answer formula for n=3 is actually algebraic. Let `m = r - l + 1`.

```
answer(n=3) = Σ_w [ (w-l) + (r-w) ] * 1 transitions
```

Actually it's the sum of `new_up[w] + new_down[w]`:
- `new_up[w]` = #{down-arrivals at values < w} = sum of `(r - v)` for `v < w`
- `new_down[w]` = #{up-arrivals at values > w} = sum of `(v - l)` for `v > w`

For the curious: total = `m(m-1)(m+1)/3` when l=1 (shifts don't change counts). But the DP is cleaner and generalizes to any n.

---

## Pitfalls & Edge Cases

- **`n = 3` and `m = 1`:** Only one value exists — any two adjacent elements would be equal (violating condition 2). After init, `up[0] = 0`, `down[0] = 0`. Step gives all zeros. Answer = 0. ✓ (Actually `l < r` is guaranteed by constraints, so `m ≥ 2` always.)

- **Large n, small m:** The DP converges to a steady state once `up[]` and `down[]` stabilize. No special handling needed — the loop just runs.

- **Mod during prefix sums:** Apply `% MOD` at each step of the prefix/suffix accumulation to prevent overflow before the values are combined.

- **0-indexed offset:** The arrays are shifted so index `i` = value `l + i`. The formula `up[i] = i` means "number of values in `[l, l+i-1]`" = i values. `down[i] = m-1-i` means "number of values in `[l+i+1, r]`" = `m-1-i` values. Check: `up[0]=0` (no values below `l`), `down[m-1]=0` (no values above `r`). ✓

## Key Takeaway

This is a **direction-tagged counting DP** with **range-update transitions** compressed via prefix sums. The zigzag constraint elegantly collapses to: "if you went up, you must go down next; if you went down, you must go up" — making the transition purely one-directional and thus summable as prefix/suffix over the value range.

**Pattern recognition:** "Count sequences with alternating monotonicity constraints + value range" → DP with `(value, direction)` state; transitions are prefix/suffix sums over the value dimension → O(n·m) with O(m) space.
