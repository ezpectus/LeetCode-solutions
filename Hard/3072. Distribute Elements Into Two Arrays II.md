# 3072. Distribute Elements Into Two Arrays II
## O(n log n) — Coordinate Compression + Dual Fenwick Trees for Live greaterCount Queries

---

## Problem Statement

1-indexed array `nums`. Distribute all elements into `arr1` and `arr2` via n rules:
- Operation 1: `arr1 ← nums[1]`.
- Operation 2: `arr2 ← nums[2]`.
- Operation i ≥ 3: compare `greaterCount(arr1, nums[i])` vs `greaterCount(arr2, nums[i])`:
  - Larger greaterCount wins the element.
  - Tie on greaterCount → smaller array wins.
  - Tie on both → `arr1` wins.

`greaterCount(arr, val)` = count of elements in `arr` strictly greater than `val`.

Return `arr1 ++ arr2`.

---

## Core Insight — greaterCount Is a Dynamic Order-Statistic Query

At each step, we need to count elements **strictly greater than `val`** in a dynamically growing array. This is a classic **rank query** problem:

```
greaterCount(arr, val) = |arr| − #{elements in arr ≤ val}
                       = size − prefixCount(val)
```

where `prefixCount(val)` = count of elements ≤ val inserted so far. A **Fenwick Tree (BIT)** answers this as a prefix sum in O(log n).

**Why not a sorted structure (SortedList, etc.):** binary search on insertion gives O(log n) but BIT has a smaller constant and the prefix-sum structure integrates cleanly with the rank query. Both are O(n log n) total; BIT is typically faster in practice.

---

## The Coordinate Compression Step

`nums[i] ≤ 10⁹` — BIT indices must be small (≤ n). Map each distinct value to a rank 1…m (m = number of distinct values ≤ n):

```
Sorted distinct values: v₁ < v₂ < … < vₘ
rank[vᵢ] = i
```

After compression, BIT indices ∈ [1, m], update/query both run in O(log m) = O(log n).

**Critical correctness point:** ranks preserve the strict ordering of original values — if `a < b` then `rank[a] < rank[b]`. So `greaterCount(arr, val) = size − bit.Query(rank[val])` is always correct. ✓

---

## State per Step

Maintain two BITs (`bit1`, `bit2`) and sizes (`s1`, `s2`):

```
greaterCount1(val) = s1 − bit1.Query(rank[val])
greaterCount2(val) = s2 − bit2.Query(rank[val])
```

After deciding which array gets `nums[i]`:
```
if arr1: bit1.Update(rank[nums[i]], +1);  s1++
else:    bit2.Update(rank[nums[i]], +1);  s2++
```

---

## Algorithm

```
1. Coordinate-compress nums: sort distinct values, build rank map.
2. bit1, bit2: Fenwick Trees of size m (distinct count).
3. s1 = s2 = 0.
4. Append nums[0] to arr1: bit1.Update(rank[0], 1); s1++.
5. Append nums[1] to arr2: bit2.Update(rank[1], 1); s2++.
6. For i = 2 to n-1:
     r = rank[nums[i]]
     gc1 = s1 - bit1.Query(r)
     gc2 = s2 - bit2.Query(r)
     if gc1 > gc2 or (gc1 == gc2 and s1 <= s2):
         arr1 ← nums[i]; bit1.Update(r, 1); s1++
     else:
         arr2 ← nums[i]; bit2.Update(r, 1); s2++
7. Return arr1 ++ arr2.
```

**Tiebreaking refinement:** the conditions collapse neatly:
- `gc1 > gc2` → arr1.
- `gc1 < gc2` → arr2.
- `gc1 == gc2` and `s1 <= s2` → arr1 (fewer or equal elements, and tie on equal also goes to arr1).
- `gc1 == gc2` and `s1 > s2` → arr2.

So the single compound check `gc1 > gc2 || (gc1 == gc2 && s1 <= s2)` correctly handles all cases. ✓

## Full Optimal Implementation (C#)

```csharp
public class Solution {
    public int[] ResultArray(int[] nums) {
        int n = nums.Length;

        // --- Coordinate compression ---
        int[] sorted = nums.Distinct().OrderBy(x => x).ToArray();
        var rank = new Dictionary<int, int>(sorted.Length);
        for (int i = 0; i < sorted.Length; i++) rank[sorted[i]] = i + 1;  // 1-indexed
        int m = sorted.Length;

        // --- Fenwick Trees ---
        int[] bit1 = new int[m + 1];
        int[] bit2 = new int[m + 1];

        void Update(int[] bit, int i) {
            for (; i <= m; i += i & -i) bit[i]++;
        }
        int Query(int[] bit, int i) {
            int s = 0;
            for (; i > 0; i -= i & -i) s += bit[i];
            return s;
        }

        // --- Distribution ---
        var arr1 = new List<int> { nums[0] };
        var arr2 = new List<int> { nums[1] };
        Update(bit1, rank[nums[0]]);
        Update(bit2, rank[nums[1]]);
        int s1 = 1, s2 = 1;

        for (int i = 2; i < n; i++) {
            int r  = rank[nums[i]];
            int gc1 = s1 - Query(bit1, r);
            int gc2 = s2 - Query(bit2, r);

            if (gc1 > gc2 || (gc1 == gc2 && s1 <= s2)) {
                arr1.Add(nums[i]);
                Update(bit1, r);
                s1++;
            } else {
                arr2.Add(nums[i]);
                Update(bit2, r);
                s2++;
            }
        }

        arr1.AddRange(arr2);
        return arr1.ToArray();
    }
}
```

## Complexity

| Metric | Value | Notes |
|---|---|---|
| Coordinate compression | O(n log n) | Sort + distinct pass |
| Each BIT update/query | O(log n) | m ≤ n |
| Total | **O(n log n)** | n operations × O(log n) each |
| Space | O(n) | Two BITs of size m + two lists |

For n = 10⁵: ≈ 10⁵ × 17 ≈ 1.7M BIT ops. Trivially fast.

## Example Walkthroughs

### Example 1 — `nums = [2,1,3,3]`, Expected: `[2,3,1,3]`

**Distinct sorted:** `[1,2,3]`. Ranks: `{1→1, 2→2, 3→3}`. m=3.

**Init:** arr1=[2], arr2=[1], s1=s2=1. bit1 has 1 at rank 2. bit2 has 1 at rank 1.

| i | nums[i] | rank r | gc1 = s1−Q1(r) | gc2 = s2−Q2(r) | Decision |
|---|---------|--------|----------------|----------------|----------|
| 2 | 3 | 3 | 1−Q1(3)=1−1=**0** | 1−Q2(3)=1−1=**0** | tie; s1=s2=1 → arr1 |
| 3 | 3 | 3 | 2−Q1(3)=2−2=**0** | 1−Q2(3)=1−1=**0** | tie; s1=2 > s2=1 → arr2 |

After: arr1=[2,3], arr2=[1,3]. Result = **[2,3,1,3]** ✓

---

### Example 2 — `nums = [5,14,3,1,2]`, Expected: `[5,3,1,2,14]`

**Distinct sorted:** `[1,2,3,5,14]`. Ranks: `{1→1, 2→2, 3→3, 5→4, 14→5}`. m=5.

**Init:** arr1=[5], arr2=[14], s1=s2=1. bit1 has 1 at rank 4. bit2 has 1 at rank 5.

| i | nums[i] | rank r | gc1 | gc2 | Decision |
|---|---------|--------|-----|-----|----------|
| 2 | 3 | 3 | 1−Q1(3)=1−0=**1** | 1−Q2(3)=1−0=**1** | tie; s1=s2=1 → arr1 |
| 3 | 1 | 1 | 2−Q1(1)=2−0=**2** | 1−Q2(1)=1−0=**1** | gc1>gc2 → arr1 |
| 4 | 2 | 2 | 3−Q1(2)=3−0=**3** | 1−Q2(2)=1−0=**1** | gc1>gc2 → arr1 |

After: arr1=[5,3,1,2], arr2=[14]. Result = **[5,3,1,2,14]** ✓

*At i=2 (val=3): arr1 has {5} with 1 element > 3; arr2 has {14} with 1 element > 3. Equal counts, equal sizes → arr1.*
*At i=3 (val=1): arr1 has {5,3} with 2 elements > 1; arr2 has {14} with 1 element > 1. arr1 wins.*

**Ranks:** `{3→1}`. m=1.

**Init:** arr1=[3], arr2=[3], s1=s2=1. Both BITs have 1 at rank 1.

|---|---------|--------|--------|--------|----------|
| 2 | 3 | 1 | 1−1=**0** | 1−1=**0** | tie; s1=s2=1 → arr1 |

After: arr1=[3,3], arr2=[3,3]. Result = **[3,3,3,3]** ✓

## Manual BIT Trace — Example 1, i=2

After init: `bit1 = [0,0,1,0]` (index 2 set for value 2).

`Query(bit1, 3)`: sum positions 1..3.
- i=3: bit1[3]=0, i→3−(3&−3)=3−1=2
- i=2: bit1[2]=1, i→2−(2&−2)=2−2=0
- Result = 1.

`gc1 = s1 − 1 = 1 − 1 = 0`. (No element > 3 in arr1 = {2}.) ✓

After append to arr1: `Update(bit1, 3)`:
- i=3: bit1[3]++; i→3+(3&−3)=3+1=4>m=3, stop.
- Now bit1[3]=1.

## Pitfalls & Edge Cases

- **Ranks must be 1-indexed for the BIT:** The BIT's inner loop `i -= i & -i` terminates when `i = 0`. A 0-indexed rank would cause `Query(0)` to immediately return 0 before entering the loop — correct by accident, but `Update(0)` would loop forever (`0 & -0 = 0`, no forward progress). Always use 1-indexed ranks. ✓

- **`Distinct().OrderBy()` for compression, not just `Array.Sort(nums)`:** Sorting the original `nums` in-place would destroy the input order needed for the distribution loop. Compress on a separate copy. ✓

- **Tiebreaker is `s1 <= s2` (not `<`):** When lengths are equal, the problem says "append to arr1" (last tiebreaker always favors arr1). The `<=` covers both the "s1 < s2" (arr1 is smaller → arr1) and `s1 == s2` (equal length → arr1) cases. Using strict `<` would miss the equal-length tie case. ✓

- **Do not reuse the same BIT array across iterations without reset:** The BIT is grown monotonically (only incremented), never decremented. This is correct because elements are never removed from arr1 or arr2. ✓

- **n = 3 (minimum):** Operations 0 and 1 always go to arr1 and arr2 unconditionally; only operation 2 uses the comparison. The loop `for i = 2 to n-1` runs exactly once. ✓

- **All values identical (Example 3):** All ranks = 1. `Query(bit, 1)` always returns the current size, giving `gc = 0` always. Distribution alternates via size tiebreaker. ✓

- **Large values up to 10⁹:** Without coordinate compression, a BIT of size 10⁹ would be infeasible (OOM). Compression to size m ≤ n solves this. ✓

- **`greaterCount` query uses `Query(bit, rank[val])`:** This counts elements ≤ val (i.e., with rank ≤ rank[val]). Subtracting from size gives elements > val. Make sure to query the rank of val itself (prefix up to and including val's rank), not rank−1. ✓

---

## Key Takeaway

**"Count elements strictly greater than val in a growing array"** = live order-statistic query = Fenwick Tree on compressed ranks. The structural insight:

```
greaterCount(arr, val) = |arr| − prefixCount(rank[val])
```

turns a hard "count greater elements" query into a simple BIT prefix sum that updates in O(log n) per insertion.

**Pattern recognition:** Any problem requiring repeated "how many inserted elements are < / ≤ / > / ≥ x" queries on a dynamically growing set → coordinate-compress the value space → Fenwick Tree (or segment tree) for O(log n) point updates and range queries. This replaces O(n) per naive scan (O(n²) total) with O(log n) per operation (O(n log n) total).
