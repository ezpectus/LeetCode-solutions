# 1803. Count Pairs With XOR in a Range
## O(n × 15) — Binary Trie with Frequency Counts

---

## Problem Statement

- Count pairs `(i,j)` where `i < j` and `low ≤ nums[i] XOR nums[j] ≤ high`.
- `nums[i] ≤ 2×10⁴ < 2^15` → 15-bit numbers.

---

## Core Idea — Trie + Inclusion-Exclusion

**Key formula**:
```
count(low ≤ XOR ≤ high) = countAtMost(high) - countAtMost(low - 1)
```

Reduces the range problem to "count pairs with XOR ≤ limit".

**`countAtMost(num, limit)`**: given a number `num` and a Trie containing previously inserted numbers, count how many of them XOR with `num` to give a value ≤ `limit`. Walk the Trie bit-by-bit from MSB:

```
For each bit position (MSB → LSB):
  nb = bit of num
  lb = bit of limit

  if lb == 1:
    - Going the SAME direction as nb gives XOR bit = 0 < 1 → all those pairs are ≤ limit → add their count
    - Continue down the OPPOSITE direction (XOR bit = 1, equal to limit bit so far)
  
  if lb == 0:
    - Must go the SAME direction as nb (XOR bit = 0 = lb) → continue
    - Going opposite gives XOR bit = 1 > 0 → all those pairs exceed limit → skip them
```

After exhausting all bits, the remaining node represents numbers with XOR **exactly equal** to limit → add its count too.

**Online processing**: insert numbers one by one. For each new `num`, query the Trie (containing only previous numbers) → counts only pairs `(i,j)` with `i < j`. Then insert `num`.

---

## Trie Node Structure

Each node stores:
- `children[0]`, `children[1]` — for bit 0 and bit 1
- `cnt` — how many numbers pass through this node

`cnt` at any node = number of inserted numbers whose bit-prefix leads through this node.

---

## Full Optimal Implementation (C#)

```csharp
public class Solution {
    class TrieNode {
        public TrieNode[] ch = new TrieNode[2];
        public int cnt;
    }

    TrieNode root = new TrieNode();

    void Insert(int num) {
        var node = root;
        for (int bit = 14; bit >= 0; bit--) {
            int b = (num >> bit) & 1;
            node.ch[b] ??= new TrieNode();
            node = node.ch[b];
            node.cnt++;
        }
    }

    int CountAtMost(int num, int limit) {
        var node = root;
        int res = 0;
        for (int bit = 14; bit >= 0; bit--) {
            int nb = (num >> bit) & 1;
            int lb = (limit >> bit) & 1;
            if (lb == 1) {
                // same-direction path gives XOR bit=0 < 1 → all count
                if (node.ch[nb] != null) res += node.ch[nb].cnt;
                // continue on opposite direction (XOR bit = 1 = lb)
                node = node.ch[1 - nb];
            } else {
                // must go same direction (XOR bit = 0 = lb)
                node = node.ch[nb];
            }
            if (node == null) return res;
        }
        res += node.cnt; // exact match: XOR == limit
        return res;
    }

    public int CountPairs(int[] nums, int low, int high) {
        int res = 0;
        foreach (int num in nums) {
            res += CountAtMost(num, high) - CountAtMost(num, low - 1);
            Insert(num);
        }
        return res;
    }
}
```

## Complexity

| **Metric** | **Value** | **Notes** |
|---|---|---|
| **Time Complexity** | **O(n × 15)** | n numbers × 15-bit Trie traversal per query+insert |
| **Space Complexity** | **O(n × 15)** | Trie nodes: at most n×15 nodes |

Brute force O(n²) with n=2×10⁴ → 4×10⁸ operations. Trie gives 3×10⁵ — 1000× faster.

---

## Bit-by-Bit Decision Logic

```
num  = ...  n_14  n_13  ...  n_b  ...
limit= ...  l_14  l_13  ...  l_b  ...

At bit b, if all higher bits gave XOR = limit bits so far:
  lb=1:
    take nb path → XOR bit = 0 < lb=1 → guaranteed ≤ limit regardless of lower bits
                    → add cnt of nb-child, then go to (1-nb)-child to continue matching
    take (1-nb) path → XOR bit = 1 = lb → still "tied" with limit, continue
  lb=0:
    take nb path → XOR bit = 0 = lb → still "tied", continue
    take (1-nb) path → XOR bit = 1 > lb=0 → guaranteed > limit → skip
```

This is the standard "count numbers with value ≤ limit in Trie" technique applied to XOR queries.

## Example Walkthrough

**`nums=[1,4,2,7], low=2, high=6`**

Process num=1: Trie empty → query gives 0, insert 1.
Process num=4: query Trie({1}) for XOR≤6: 1 XOR 4=5≤6 → 1; for XOR≤1: 5>1 → 0. Add 1.
Process num=2: query Trie({1,4}): 2 XOR 1=3≤6✓, 2 XOR 4=6≤6✓ → 2; XOR≤1: 3>1,6>1 → 0. Add 2.
Process num=7: query Trie({1,4,2}): 6≤6✓, 3≤6✓, 5≤6✓ → 3; XOR≤1: all>1 → 0. Add 3.

Total: 0+1+2+3 = 6 ✓

## Pitfalls & Edge Cases

- **15 bits enough**: `nums[i] ≤ 2×10⁴ < 2^15 = 32768` → use bits 14..0
- **`low - 1` can be 0**: `countAtMost(num, 0)` counts pairs with XOR=0 (identical numbers) — works correctly since XOR=0 path always follows same bits as num
- **`node == null` early return**: if a required child doesn't exist, no numbers went that way → return accumulated `res` immediately
- **Insert AFTER query**: ensures we only count pairs `(i,j)` with `i < j` (previous numbers only in Trie when querying current number)

---

## Key Takeaway

"Count pairs with XOR in range [low, high]" → **Trie + inclusion-exclusion**:

```
answer = countAtMost(high) - countAtMost(low-1)
```

`countAtMost(limit)` uses bit-by-bit Trie traversal: when limit bit=1, collect the "guaranteed smaller" subtree and continue on the "equal so far" path; when limit bit=0, can only go the equal path.

Pattern: any XOR range query on a set → Binary Trie with frequency counts.
