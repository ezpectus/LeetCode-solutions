## 3193. Count the Number of Inversions

### Problem Statement
You are given an integer `n` and a 2D array `requirements`, where `requirements[i] = [endi, cnti]` represents the end index and the inversion count of each requirement.

A pair of indices `(i, j)` from an integer array `nums` is called an inversion if `i < j` and `nums[i] > nums[j]`.

Return the number of permutations `perm` of `[0, 1, 2, ..., n - 1]` such that for all `requirements[i]`, the prefix `perm[0..endi]` has exactly `cnti` inversions.

Since the answer may be very large, return it **modulo 10^9 + 7**.

---

### Core Insight — Dynamic Programming with Requirements Filtering
When building a permutation incrementally by placing the element at index `i` (0-indexed), the new element can create anywhere from `0` to `i` new inversions depending on its relative position among the previously placed `i` elements.

We can define our state as:
* `dp[i][j]`: The number of valid permutations of the prefix up to index `i` that contain exactly `j` inversions.

**State Transition:**
To achieve `j` total inversions at index `i`, we can transition from any state at index `i - 1` that had `j - k` inversions, where `k` is the number of new inversions introduced by the `i`-th element (`0 <= k <= i`):
`dp[i][j] = sum(dp[i - 1][j - k])` for `0 <= k <= min(i, j)`

**Enforcing Requirements:**
If an index `i` has a constraint `cnt` specified in `requirements`, then after computing the transitions for index `i`, any state `dp[i][j]` where `j != cnt` becomes invalid and must be explicitly set to `0`. If a requirement specifies `cnt` for index `0` and it is not `0`, the total valid permutations immediately drop to `0`.

---

### Algorithm
1. Map all `requirements` into a lookup array or hash map `req` where `req[i]` gives the required inversion count for prefix `i` (initialize with `-1` to represent no requirement).
2. If `req[0] > 0`, return `0` immediately (prefix of length 1 can only have 0 inversions).
3. Initialize a 2D table `dp` of size `n x 401` with zeros, and set the base case `dp[0][0] = 1`.
4. Loop `i` from 1 to `n - 1`:
   * For each possible inversion count `j` from 0 to 400:
     * Accumulate the rolling sum of `dp[i - 1][j - k]` for `0 <= k <= i`.
   * If `req[i] != -1`:
     * For all `j != req[i]`, invalidate the state by setting `dp[i][j] = 0`.
5. The final answer will be stored in `dp[n - 1][req[n - 1]]`.

---

### Full Optimal Implementation (C++)
```cpp
#include <vector>
#include <algorithm>

using namespace std;

class Solution {
public:
    int numberOfPermutations(int n, vector<vector<int>>& requirements) {
        const int MOD = 1e9 + 7;
        const int MAX_INV = 400;
        
        // Map constraints to their respective ending prefixes
        vector<int> req(n, -1);
        for (const auto& r : requirements) {
            req[r[0]] = r[1];
        }
        
        // Base case check: index 0 must have 0 inversions if restricted
        if (req[0] != -1 && req[0] != 0) {
            return 0;
        }
        
        // dp[i][j] stores the number of ways to have j inversions using prefix up to index i
        vector<vector<int>> dp(n, vector<int>(MAX_INV + 1, 0));
        dp[0][0] = 1;
        
        for (int i = 1; i < n; i++) {
            for (int j = 0; j <= MAX_INV; j++) {
                long long total_ways = 0;
                // Element at position i can introduce up to i new inversions
                for (int k = 0; k <= i; k++) {
                    if (j - k >= 0) {
                        total_ways = (total_ways + dp[i - 1][j - k]) % MOD;
                    }
                }
                dp[i][j] = total_ways;
            }
            
            // If there's a strict requirement for the current prefix, wipe out non-matching states
            if (req[i] != -1) {
                int target = req[i];
                for (int j = 0; j <= MAX_INV; j++) {
                    if (j != target) {
                        dp[i][j] = 0;
                    }
                }
            }
        }
        
        return dp[n - 1][req[n - 1]];
    }
};
```
### Complexity
| Metric | Value | Notes |
|---|---|---|
| **Time** | O(N * MAX_INV * N) | The constraints guarantee `N <= 300` and `MAX_INV <= 400`. The nested loops run bounded transformations well within execution limits. It can be optimized to O(N * MAX_INV) using prefix sums, but the current implementation easily passes within the given limits. |
| **Space** | O(N * MAX_INV) | Accommodated via a static 2D grid storing precalculated combinations up to 300 x 401 elements. |

### Pitfalls & Edge Cases
* **Invalid Base Anchors:** A single element (prefix at index 0) cannot contain any elements to its left, meaning its inversion count must be 0. If `req[0]` asks for anything higher, the answer is structurally 0.
* **Modulo Operations:** Ensure the accumulation variable handles standard integer updates safely using modulo addition rules on every loop step to prevent precision overflows.

### Key Takeaway
When tasked with computing valid sequences under conditional checkpoints, start with the standard unconstrained recurrence relation. By applying an aggressive filter to clear out any states that violate active checkpoints at the end of each step, the DP space naturally reduces to track only legally compliant pathways.
