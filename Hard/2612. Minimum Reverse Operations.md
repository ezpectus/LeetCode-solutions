# 2612. Minimum Reverse Operations
## O(n log n) — Parity-Ranged BFS with SortedSet Interval Drain

---

## Problem Statement

Array of length `n`, all zeros except position `p` (value 1). Banned positions cannot ever hold the 1. An operation reverses any subarray of size `k` that **contains** the current position of 1. Return the minimum operations to bring 1 to every position, or `-1` if impossible.

---

## Core Insight #1 — Derive the Reachable Range from Position u

If 1 is at position `u` and we reverse subarray `[l, l+k-1]` (which must contain `u`), the new position `j` of 1 is:
```
j = l + (l + k - 1 - u) = 2l + k - 1 - u
```

Constraints on `l`:
- `l ≥ max(0, u - k + 1)` (subarray starts at or after 0, and includes u)
- `l ≤ min(n - k, u)` (subarray ends before n, and includes u)

Plugging in the extreme values of `l`:

```
lo = max(u - k + 1, k - 1 - u)
hi = min(u + k - 1, 2n - k - 1 - u)
```

As `l` steps by 1, `j` steps by 2. So **from position `u`, the 1 can jump to exactly the positions `{lo, lo+2, lo+4, ..., hi}`** — a contiguous arithmetic progression with step 2.

---

## Core Insight #2 — Parity Is Conserved (or Flipped) Per Step

Since all reachable `j` share the parity `(k - 1 + u) % 2 = lo % 2`, every operation either:
- **Preserves** parity of the 1's position (when k is odd), or
- **Flips** parity (when k is even).

This means the two parity classes `{even positions}` and `{odd positions}` are fully decoupled for a given k — a node in one class only reaches nodes in the other class (or same class). This lets us split the "to-visit" set into two independent SortedSets, indexed by parity, avoiding any cross-parity false positives.

---

## Core Insight #3 — BFS on an Implicit Graph with Range Transitions

The problem is BFS where each "edge" from node `u` goes to **an entire interval** of nodes simultaneously (rather than a single neighbor). Naive BFS iterating each j ∈ [lo, hi] individually costs O(n × k) — up to O(n²) for large k.

**Key optimization:** Use two `SortedSet<int>` (one per parity) of **unvisited, non-banned** positions. For each dequeued position `u`:
1. Compute `[lo, hi]` and `par = lo % 2`.
2. Call `GetViewBetween(lo, hi)` on `unvisited[par]`.
3. Every position found is newly reachable — set its distance, enqueue it, remove from the set.

Since each position is removed at most once from the SortedSet, the total work across all BFS steps is O(n log n).

---

## Why SortedSet and Not HashSet?

We need to find ALL unvisited positions in a range [lo, hi] efficiently. `HashSet` can't answer range queries. `SortedSet.GetViewBetween(lo, hi)` returns a live view of all elements in [lo, hi] in O(log n) to locate plus O(m) to iterate, where m is the count found. This is exactly what we need.

---

## Full Implementation (C#)

```csharp
public class Solution {
    public int[] MinReverseOperations(int n, int p, int[] banned, int k) {
        int[] ans = new int[n];
        Array.Fill(ans, -1);
        ans[p] = 0;

        var bannedSet = new HashSet<int>(banned);

        // Split unvisited, non-banned positions into two SortedSets by parity
        var unvisited = new SortedSet<int>[2];
        unvisited[0] = new SortedSet<int>();
        unvisited[1] = new SortedSet<int>();
        for (int i = 0; i < n; i++) {
            if (i != p && !bannedSet.Contains(i))
                unvisited[i & 1].Add(i);
        }

        var q = new Queue<int>();
        q.Enqueue(p);

        while (q.Count > 0) {
            int u = q.Dequeue();

            // Range of positions reachable from u via a single reversal
            int lo  = Math.Max(u - k + 1, k - 1 - u);
            int hi  = Math.Min(u + k - 1, 2 * n - k - 1 - u);
            int par = lo & 1; // parity of all reachable positions

            // Drain all unvisited positions in [lo, hi] with correct parity
            var toVisit = unvisited[par].GetViewBetween(lo, hi).ToList();
            foreach (int j in toVisit) {
                unvisited[par].Remove(j);
                ans[j] = ans[u] + 1;
                q.Enqueue(j);
            }
        }

        return ans;
    }
}
```

## Complexity

| Metric | Value | Notes |
|---|---|---|
| Build SortedSets | O(n log n) | n insertions |
| BFS total | O(n log n) | each of n positions: O(log n) per removal, removed at most once |
| **Total Time** | **O(n log n)** | ≈ 10⁵ × 17 ≈ 1.7×10⁶ ops for n=10⁵ |
| Space | O(n) | ans array + SortedSets |

## Range Formula Verification

| Scenario | lo | hi | Meaning |
|---|---|---|---|
| u=0, k=4, n=4 | max(-3,3)=3 | min(3,3)=3 | Only j=3 (reflect across entire array) |
| u=0, k=3, n=5 | max(-2,2)=2 | min(2,6)=2 | Only j=2 (left wall limits) |
| u=2, k=3, n=5 | max(0,0)=0 | min(4,4)=4 | j∈{0,2,4} (3-window slides freely) |
| u=2, k=1, n=4 | max(2,-1)=2 | min(2,4)=2 | Only j=2 (size-1 reversal = no-op) |

## Example Walkthrough

### Example 1 — n=4, p=0, banned=[1,2], k=4

```
unvisited[0] = {}     (only 0=p excluded, 2=banned excluded)
unvisited[1] = {3}    (1=banned excluded)
```

| Dequeue u | lo | hi | par | Found in [lo,hi] | ans |
|---|---|---|---|---|---|
| 0 (ans=0) | 3 | 3 | 1 | {3} | ans[3]=1 |
| 3 (ans=1) | 0 | 0 | 0 | {} | — |

**Output: [0, -1, -1, 1]** ✓

---

### Example 2 — n=5, p=0, banned=[2,4], k=3

```
unvisited[0] = {}     (2,4=banned)
unvisited[1] = {1, 3}
```

| Dequeue u | lo | hi | par | Found | ans |
|---|---|---|---|---|---|
| 0 (ans=0) | 2 | 2 | 0 | {} | — |

Queue empty. Positions 1, 3 unreachable (only reachable positions from 0 are even-indexed, but unvisited[0] is empty).

**Output: [0, -1, -1, -1, -1]** ✓

---

### Larger Example — n=6, p=0, banned=[], k=3

```
unvisited[0] = {2, 4}
unvisited[1] = {1, 3, 5}
```

| Dequeue u | lo | hi | par | Found | Next queue |
|---|---|---|---|---|---|
| 0 (ans=0) | 2 | 2 | 0 | {2} | [2] |
| 2 (ans=1) | 0 | 4 | 0 | {4} | [4] (0 already visited) |
| 4 (ans=2) | 2 | 6→5 | 0 | {} | — |

But wait — what about odd positions? lo is always even when starting from p=0 and k=3 (parity (k-1+u)%2 = (2+0)%2 = 0). So odd positions are NEVER reachable from 0 with k=3! ans[1]=ans[3]=ans[5]=-1. 

This makes sense: k=3 is odd so parity is preserved. Starting at even position 0, we can only ever reach even positions. ✓

## Pitfalls & Edge Cases

- **`GetViewBetween` returns a LIVE view.** Calling `.ToList()` before removing is mandatory — modifying the SortedSet while iterating over a live view causes undefined behavior.

- **Banned positions are excluded from both SortedSets.** They can never be BFS targets. Their `ans[i]` stays `-1`. The BFS never tries to place 1 on a banned position. ✓

- **k=1 means no movement.** lo = max(u, -1) = u, hi = min(u, 2n-2-u). For u ≤ n-1: hi = u. So j = u only (trivial no-op). Only p is reachable. ✓

- **Parity determines which SortedSet to query.** Using the wrong parity would query an empty range or find wrong positions. `par = lo & 1` is always correct because all j from u share parity lo%2. ✓

- **lo and hi are always ≥ 0 and lo ≤ hi** for valid positions 0 ≤ u ≤ n-1 and 1 ≤ k ≤ n. No need for bounds clamping beyond the formula itself.

- **`2n - k - 1 - u` could overflow in other languages** (here n ≤ 10⁵, so 2n ≤ 2×10⁵ — fine for `int` in C#).

---

## Key Takeaway

This is the **"BFS with interval transitions + SortedSet drain"** pattern: when each BFS step reaches a contiguous range of nodes (not a single neighbor), naive enumeration is too slow. Maintaining an ordered set of unvisited nodes and draining the range in O(m log n) total (each node removed once) brings it from O(nk) to O(n log n).

**Pattern recognition:** "Move to any position in a computed range" + "minimize steps" → BFS where each edge is a range; use SortedSet split by parity to drain ranges in amortized O(log n) per node. The parity split is key: it halves the SortedSet size and prevents spurious matches with wrong-parity positions in [lo, hi].
