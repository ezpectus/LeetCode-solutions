# 3045. Count Prefix and Suffix Pairs II
## O(L) — Paired-Character Trie on zip(s) = [(s[i], s[n−1−i])] Encoding

---

## Problem Statement

String array `words`. Count pairs `(i, j)` with `i < j` where `words[i]` is simultaneously a **prefix** and **suffix** of `words[j]`.

Constraints: `words.length ≤ 10⁵`, each length ≤ 10⁵, **sum of all lengths L ≤ 5×10⁵**.

---

## Core Insight — Zip Encoding Reduces Prefix-And-Suffix to Prefix Only

For `w` (length m) to be both prefix and suffix of `s` (length n, n ≥ m):
```
Prefix condition:  s[0..m-1]   = w[0..m-1]
Suffix condition:  s[n-m..n-1] = w[0..m-1]
```

Define the **zip sequence** of a string `t` of length n:
```
zip(t) = [(t[0], t[n-1]), (t[1], t[n-2]), …, (t[n-1], t[0])]
```
Each character is paired with its mirror from the other end.

**Claim:** `w` is both a prefix and suffix of `s` if and only if `zip(w)` is a prefix of `zip(s)`.

**Proof:** For position `i ∈ [0, m-1)`:
```
zip(s)[i] = (s[i], s[n-1-i])
zip(w)[i] = (w[i], w[m-1-i])
```
Matching `zip(s)[i] = zip(w)[i]` requires:
- `s[i] = w[i]`              → prefix condition at position i ✓
- `s[n-1-i] = w[m-1-i]`     → let `j = m-1-i`: `s[n-m+j] = w[j]` → suffix condition at position j ✓

Both conditions are simultaneously encoded in a single character-pair match. Therefore `w` is prefix+suffix of `s` ↔ the first m pairs of `zip(s)` equal `zip(w)` ↔ **`zip(w)` is a prefix of `zip(s)`**. ✓

---

## Reduction to Trie Count of Prefix Matches

Build a **Trie on pair-sequences** where each edge is labeled by a pair `(a, b)` of characters (676 possible edges per node, encoded as `a × 26 + b`).

**Process words left to right. For each word `s` at index `j`:**
1. **Query**: traverse `zip(s)` in the trie, accumulating `counts[node]` at every visited node (each node stores how many previously inserted words ended there). Each such word has its full zip sequence as a prefix of `zip(s)` — i.e., it is both prefix and suffix of `s`.
2. **Insert**: traverse/create the path for `zip(s)` in the trie, marking the final node with `+1`.

Since `zip(w)` being a prefix of `zip(s)` requires `len(w) ≤ len(s)` (zip(s) has exactly `len(s)` pairs), longer words in the trie can never be counted when querying a shorter word — this constraint is handled naturally by the trie traversal breaking once no edge exists.

---

## Algorithm

```
1. Initialize trie with root node (id=0, children={}, count=0).
2. ans = 0
3. For each word s:
     n = s.Length
     // Query phase
     node = root
     for i = 0 to n-1:
         key = (s[i]-'a')*26 + (s[n-1-i]-'a')
         if key not in children[node]: break
         node = children[node][key]
         ans += counts[node]          // all words ending here are prefix+suffix of s
     // Insert phase
     node = root
     for i = 0 to n-1:
         key = (s[i]-'a')*26 + (s[n-1-i]-'a')
         if key not in children[node]: create new node
         node = children[node][key]
     counts[node]++                   // mark this word ends here
4. Return ans.
```

## Full Optimal Implementation (C#)

```csharp
public class Solution {
    public long CountPrefixSuffixPairs(string[] words) {
        // Trie: nodes stored as parallel lists
        var children = new List<Dictionary<int, int>>();
        var counts   = new List<int>();
        children.Add(new Dictionary<int, int>());   // root
        counts.Add(0);

        long ans = 0;

        foreach (string s in words) {
            int n = s.Length;

            // --- Query: count words already inserted whose zip is prefix of zip(s) ---
            int node = 0;
            for (int i = 0; i < n; i++) {
                int key = (s[i] - 'a') * 26 + (s[n - 1 - i] - 'a');
                if (!children[node].TryGetValue(key, out int next)) break;
                node = next;
                ans += counts[node];
            }

            // --- Insert: add zip(s) to trie ---
            node = 0;
            for (int i = 0; i < n; i++) {
                int key = (s[i] - 'a') * 26 + (s[n - 1 - i] - 'a');
                if (!children[node].TryGetValue(key, out int next)) {
                    next = children.Count;
                    children.Add(new Dictionary<int, int>());
                    counts.Add(0);
                    children[node][key] = next;
                }
                node = next;
            }
            counts[node]++;
        }

        return ans;
    }
}
```

## Complexity

| Metric | Value | Notes |
|---|---|---|
| Time | O(L) | Each character processed exactly twice (query + insert); dictionary ops O(1) amortized |
| Space | O(L) | At most L trie nodes total (one per character across all insertions) |

For L = 5×10⁵: ≈ 10⁶ dictionary lookups. Trivially fast vs O(n² × L) brute force ≈ 5×10¹⁵ ops.

## Example Walkthroughs

### Example 1 — `words = ["a","aba","ababa","aa"]`, Expected: 4

**zip sequences:**

| word | n | zip(word) pairs |
|------|---|-----------------|
| "a" | 1 | [(a,a)] |
| "aba" | 3 | [(a,a),(b,b),(a,a)] |
| "ababa" | 5 | [(a,a),(b,b),(a,a),(b,b),(a,a)] |
| "aa" | 2 | [(a,a),(a,a)] |

*(Encoding: a→0, b→1. Pair (a,b) → 0×26+1=1, etc.)*

---

**Process "a" (j=0):**
- Query: trie empty → ans += 0.
- Insert: root →^{(a,a)} node₁. `counts[node₁]++` = 1.

Trie: `root → (aa) → node₁[count=1]`

---

**Process "aba" (j=1):**
- Query: traverse zip("aba") = [(a,a),(b,b),(a,a)]
  - i=0: edge (a,a) → node₁. `ans += counts[node₁] = 1`. (**pair (0,1)** found)
  - i=1: edge (b,b) from node₁ → not in trie. **Break**.
- ans = 1.
- Insert: root→^{aa} node₁→^{bb} node₂→^{aa} node₃. `counts[node₃]++` = 1.

---

**Process "ababa" (j=2):**
- Query: traverse zip("ababa") = [(a,a),(b,b),(a,a),(b,b),(a,a)]
  - i=0: (a,a) → node₁. `ans += 1`. (**pair (0,2)**)
  - i=1: (b,b) → node₂. `ans += counts[node₂] = 0`.
  - i=2: (a,a) → node₃. `ans += counts[node₃] = 1`. (**pair (1,2)**)
  - i=3: (b,b) from node₃ → not in trie. **Break**.
- ans = 1 + 0 + 1 + (previous 1) = 3.
- Insert: extend path with two more nodes. `counts[node₅]++` = 1.

---

**Process "aa" (j=3):**
- Query: traverse zip("aa") = [(a,a),(a,a)]
  - i=0: (a,a) → node₁. `ans += 1`. (**pair (0,3)**)
  - i=1: (a,a) from node₁ → not in trie (node₁ has only (b,b)). **Break**.
- ans = 3 + 1 = **4** ✓

*(Only "a" qualifies as prefix+suffix of "aa": zip("a")=[(a,a)] is a prefix of zip("aa")=[(a,a),(a,a)]. "aba" has zip length 3 > 2, so it can never be found in the first 2 steps of "aa"'s traversal — handled automatically.)*

---

### Example 2 — `words = ["pa","papa","ma","mama"]`, Expected: 2

**zip sequences:**

| word | zip |
|------|-----|
| "pa" | [(p,a),(a,p)] |
| "papa" | [(p,a),(a,p),(p,a),(a,p)] |
| "ma" | [(m,a),(a,m)] |
| "mama" | [(m,a),(a,m),(m,a),(a,m)] |

- "pa" inserted, count=1 at depth 2.
- "papa" queried: (p,a)→(a,p)→node at depth 2, `ans += 1`. Then (p,a) not extended. ans=1. (**pair (0,1)**)
- "ma" inserted, count=1 at depth 2 (different branch from "pa").
- "mama" queried: (m,a)→(a,m)→node at depth 2, `ans += 1`. ans=2. (**pair (2,3)**)

**Answer = 2** ✓

---

### Example 3 — `words = ["abab","ab"]`, Expected: 0

zip("abab") = [(a,b),(b,a),(a,b),(b,a)], zip("ab") = [(a,b),(b,a)].

- "abab" inserted at depth 4.
- "ab" queried: follows [(a,b),(b,a)] — no words at depth 1 or 2 yet. ans = 0.
- **Answer = 0** ✓

Note: "ab" is shorter than "abab", so "abab" can never be prefix+suffix of "ab". The trie traversal for "ab" only reaches depth 2, never seeing the depth-4 node where "abab" was inserted. ✓

## Why Zip Encoding Works for the Middle Character (Odd Length)

For odd-length string `s` of length n, the middle position i = n/2:
```
zip(s)[n/2] = (s[n/2], s[n-1-n/2]) = (s[n/2], s[n/2])
```
The middle character pairs with itself. For word `w` to be prefix+suffix of `s`, if `len(w)` reaches the middle of `s`, then both the prefix-position and suffix-position point to the same character — the match condition `(s[n/2], s[n/2]) = (w[i], w[m-1-i])` reduces to two copies of the same character constraint, which is automatically symmetric. No special-casing needed. ✓

---

## Pitfalls & Edge Cases

- **Query before insert:** for index pair `(i, j)` with `i < j`, word `j` must query the trie *before* inserting itself. Querying and then inserting in the same iteration correctly enforces `i < j`. (If you insert first, `(i, i)` self-pairs would be counted.) ✓

- **Same word appearing multiple times:** e.g., `words = ["a","a"]`. zip("a")=[(a,a)]. At j=1, query finds node₁ with count=1 → ans=1 (pair (0,1)). Correct — "a" is both prefix and suffix of "a" (trivially). ✓

- **words[i] longer than words[j]:** word `j`'s zip has length `len(j)`. Traversing it in the trie stops after `len(j)` steps, never reaching nodes for words longer than `len(j)`. Automatically excluded. ✓

- **676 possible pair-keys per node:** keys are `a*26+b ∈ [0, 675]`. Using `Dictionary<int,int>` for children is O(1) average per lookup and allocates only as many entries as actually used — much more memory-efficient than a fixed `int[676]` child array per node (which would use 676 × 4 bytes = 2.7 KB per node × 5×10⁵ nodes = 1.35 GB). ✓

- **Counting at each depth, not just at the end of traversal:** when we query, we add `counts[node]` at every node visited (every depth), not only the final depth. This correctly counts all words of any length m ≤ n whose zip is a prefix of zip(s). Accumulating only at the final node would miss shorter words that are prefix+suffix of `s`. ✓

- **`long` for answer:** max pairs = C(10⁵, 2) ≈ 5×10⁹, overflows `int`. Use `long ans`. ✓

---

## Why Not KMP / Z-Function per Pair?

For each pair (i, j), checking whether `words[i]` is prefix+suffix of `words[j]` can be done in O(m) with KMP or Z-function. But this is O(n² × m_avg) overall — for n=10⁵ pairs with m_avg=5, this is 5×10¹⁰ ops: too slow.

The zip-trie approach folds all `n` queries into a single trie traversal per word, giving O(L) total independent of n — a genuinely different structural insight, not just a constant-factor improvement.

---

## Key Takeaway

**"Two simultaneous string alignment conditions"** (prefix AND suffix) → encode both into one sequence using **zip pairing** `(front[i], back[i])` → reduce "both conditions" to one standard trie-prefix query. One trie traversal per word, O(L) total.

**Pattern recognition:** Any problem of the form "string w satisfies two independent positional conditions relative to string s simultaneously, each a character-by-character match" → check if a zip/interleaving encoding reduces the two-condition match to a single-sequence prefix/substring match, then use a Trie or string hashing on the zipped sequence.
