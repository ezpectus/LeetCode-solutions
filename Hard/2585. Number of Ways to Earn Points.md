# 2585. Number of Ways to Earn Points
## O(n ¬∑ target ¬∑ count) ‚Äî Bounded Knapsack DP

---

## Problem Statement

- n question types, each type i has `count` questions worth `marks` each.
- Count distinct ways to choose questions earning exactly `target` total points.
- Questions of the same type are indistinguishable ‚Äî only the count taken matters.
- Return result mod 10‚Åπ + 7.

---

## Core Idea ‚Äî Bounded Knapsack DP

Classic 1D knapsack where each item type i can be used 0..count[i] times with weight = marks[i].

`dp[j]` = number of ways to reach exactly `j` points considering the question types processed so far.

For each type (count, marks), update `dp` in **reverse order** (as in 0-1 knapsack) but allow up to `count` uses. This is the **bounded knapsack** variant:

For each target score `j` from high to low:
- We can take 0, 1, 2, ..., min(count, j/marks) questions of this type
- Sum all those ways into dp[j]

**Two approaches:**
1. Naively: for each j, loop over k = 0..min(count, j/marks): O(n √ó target √ó count/marks) ‚âà O(n √ó target √ó count)
2. For this constraint (target ‚â§ 1000, count ‚â§ 50, marks ‚â§ 50): the naive is fast enough.

## Full Optimal Implementation (C#)

```csharp
public class Solution {
    public int WaysToReachTarget(int target, int[][] types) {
        const int MOD = 1_000_000_007;
        long[] dp = new long[target + 1];
        dp[0] = 1;

        foreach (var type in types) {
            int cnt = type[0], marks = type[1];

            // Traverse in reverse to avoid reusing same type within one pass
            for (int j = target; j >= 0; j--) {
                if (dp[j] == 0) continue;

                // Take k questions of this type (k >= 1)
                for (int k = 1; k <= cnt && j + k * marks <= target; k++) {
                    dp[j + k * marks] = (dp[j + k * marks] + dp[j]) % MOD;
                }
            }
        }

        return (int)dp[target];
    }
}
```

## Why This Works ‚Äî Example Walkthrough

**Example 1:** `target = 6, types = [[6,1],[3,2],[2,3]]`

**Initial:** dp = [1,0,0,0,0,0,0]

**Type [6,1] (up to 6 questions worth 1 each):**

After pass: dp = [1,1,1,1,1,1,1]
(dp[k] = 1 for all k = 0..6, since there's exactly 1 way to pick exactly k ones)

**Type [3,2] (up to 3 questions worth 2 each):**

Process j from high to low, adding k=1,2,3 twos:
- j=6: dp[6]+=dp[4]+dp[2]+dp[0] = 1+1+1 = 3 ‚Üí dp[6]=4
- j=5: dp[5]+=dp[3]+dp[1] = 1+1 = 2 ‚Üí dp[5]=3
- j=4: dp[4]+=dp[2]+dp[0] = 1+1 = 2 ‚Üí dp[4]=3
- j=3: dp[3]+=dp[1] = 1 ‚Üí dp[3]=2
- j=2: dp[2]+=dp[0] = 1 ‚Üí dp[2]=2

dp = [1,1,2,2,3,3,4]

**Type [2,3] (up to 2 questions worth 3 each):**

- j=6: dp[6]+=dp[3]+dp[0] = 2+1 = 3 ‚Üí dp[6]=7
- j=5: dp[5]+=dp[2] = 2 ‚Üí dp[5]=5
- j=4: dp[4]+=dp[1] = 1 ‚Üí dp[4]=4
- j=3: dp[3]+=dp[0] = 1 ‚Üí dp[3]=3

**dp[6] = 7** ‚úì

---

**Example 2:** `target = 5, types = [[50,1],[50,2],[50,5]]`

**Type [50,1]:** dp[k]=1 for k=0..5 (only up to target)
**Type [50,2]:**
- dp[5]+=dp[3]+dp[1]=1+1=2 ‚Üí dp[5]=3
- dp[4]+=dp[2]+dp[0]=1+1=2 ‚Üí dp[4]=3
- dp[3]+=dp[1]=1 ‚Üí dp[3]=2
- dp[2]+=dp[0]=1 ‚Üí dp[2]=2

dp = [1,1,2,2,3,3]

**Type [50,5]:**
- j=0: dp[5]+=dp[0]=1 ‚Üí dp[5]=4

**dp[5] = 4** ‚úì

## Pitfalls & Edge Cases

- **Reverse iteration is mandatory** ‚Äî Processing j from high to low ensures each type is counted at most `cnt` times per position. If we went low to high, a single type could be used unboundedly within the same pass.

- **skip dp[j] == 0 optimization** ‚Äî Avoids inner loop when no base contribution exists. Small speedup in practice.

- **count can exceed target/marks** ‚Äî `j + k * marks <= target` is the natural cap, not just `k <= cnt`.

- **Overflow** ‚Äî dp values can be up to MOD-1 ‚âà 10‚Åπ. Sum of 50 such values ‚âà 5√ó10¬π‚Å∞ ‚Üí exceeds int. Use `long` and apply MOD. ‚úì

- **All types have marks > target** ‚Äî dp[0] stays 1, everything else stays 0. dp[target] = 0 correctly returned.

---

## Key Takeaway

Classic **bounded knapsack** (each item usable 0..k times):
- `dp[j]` = ways to reach sum j
- Outer loop: item types
- Inner loop (reverse): current sum
- Innermost loop: how many of this type to use

The reverse-order trick from 0-1 knapsack extends naturally: iterating j from high to low ensures each item type contributes at most `cnt` times to any single target sum.

**Pattern recognition:** "Count ways to reach exact sum, each item usable limited times" ‚Üí bounded knapsack. The bounded variant is the same as 0-1 knapsack with an extra inner loop for count, keeping space at O(target).
