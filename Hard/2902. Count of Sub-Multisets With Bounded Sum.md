# 2902. Count of Sub-Multisets With Bounded Sum
## O(S^{3/2}) — Grouped Sliding-Window Bounded Knapsack + Distinct-Value Count Bound

---

## Problem Statement

Array `nums` of non-negative integers, bounds `l` and `r`. A **sub-multiset** picks element `x` between 0 and `freq[x]` times. Count sub-multisets whose element sum falls in `[l, r]`. Return count mod 10⁹+7.

Key constraints: `nums.length ≤ 2×10⁴`, all `nums[i] ≤ 2×10⁴`, **sum of all nums ≤ S = 2×10⁴**.

---

## Core Insight — Sub-multiset Counting = Bounded Knapsack DP

Group elements by value. For each distinct value `v` with count `c = freq[v]`, we independently choose how many copies to take: 0, 1, …, c. The number of sub-multisets with sum exactly `s` is:

```
dp[s] = number of sub-multisets with element sum = s
```

**Recurrence:** For each value `v` with count `c`, update dp so that:
```
new_dp[s] = Σ_{k=0}^{min(c, s/v)} dp[s − k·v]
           = dp[s] + dp[s−v] + dp[s−2v] + … + dp[s−c·v]
```

This is a **sliding window sum** of dp values spaced `v` apart — the heart of the algorithm.

The **answer** is `(Σ dp[l..r]) × (freq[0] + 1) mod MOD` (zeros handled separately; see below).

---

## The Distinct-Value Bound: d ≤ O(√S)

Each distinct non-zero value `v` appears at least once (by definition). Their contributions to the total sum are at least `v₁ + v₂ + … + v_d`. The minimum sum for `d` distinct positive integers is `1 + 2 + … + d = d(d+1)/2`.

Since the sum of all elements ≤ S:

```
d(d+1)/2 ≤ S   →   d ≤ ⌊(−1 + √(1 + 8S)) / 2⌋ = O(√S)
```

For S = 2×10⁴: **d ≤ 199 distinct non-zero values**.

This bound — invisible without the sum constraint — is what makes the O(d·S) algorithm run in O(S^{3/2}) ≈ 2.8×10⁶ operations instead of O(S²).

---

## Sliding Window for Bounded Knapsack

For value `v` and count `c`, split the dp array into `v` independent **chains** by remainder mod `v`:

```
Chain[rem]:  dp[rem],  dp[rem+v],  dp[rem+2v],  …  (elements spaced v apart)
```

The update for each chain position `i` (where `j = rem + i·v`) is:
```
new_dp[j] = Chain[i] + Chain[i−1] + … + Chain[i−c]   (up to c+1 terms)
           = prefix[i+1] − prefix[max(0, i−c)]
```

where `prefix[k]` = `Chain[0] + Chain[1] + … + Chain[k−1]`.

This turns an O(S·c) per-value update into O(S) via prefix sums. All `v` chains together cover all S+1 dp indices exactly once.

**Two passes per chain:**
1. Build `prefix[]` from the **current** (old) dp values.
2. Overwrite dp from prefix. 

The two-pass structure is critical: reading old dp while writing new values would corrupt mid-chain results.

---

## Special Case: Zeros

Value 0 has no contribution to sums, so including or excluding zeros doesn't change which sums are reachable. If `freq[0] = z`, every existing sub-multiset can be extended by 0, 1, …, z zeros, independently. This multiplies every dp entry by `(z+1)`.

**Handle zeros by** removing them from the main DP and multiplying the final answer by `(z+1)`. Including `v=0` in the DP loop would cause a divide-by-zero (the chain structure requires `v ≥ 1`).

---

## Algorithm

```
1. Count freq[v] for all v.
2. Compute S = sum(nums). Clamp r = min(r, S). If l > r: return 0.
3. z = freq[0]. Remove 0 from freq.
4. dp[0] = 1; dp[1..S] = 0.
5. For each distinct (val, cnt) in freq:
     For rem = 0 to val-1:
       Build prefix[] from chain dp[rem], dp[rem+val], dp[rem+2*val], …
       Update chain using sliding window of size cnt+1.
6. ans = sum(dp[l..r]) * (z+1) mod MOD.
```

---

## Full Optimal Implementation (C#)

```csharp
public class Solution {
    const int MOD = 1_000_000_007;

    public int CountSubMultisets(IList<int> nums, int l, int r) {
        // Count frequencies and total sum
        var freq = new Dictionary<int, int>();
        int S = 0;
        foreach (int x in nums) {
            freq[x] = freq.GetValueOrDefault(x, 0) + 1;
            S += x;
        }

        r = Math.Min(r, S);
        if (l > r) return 0;

        // Zeros multiply every answer by (z+1); skip them in DP
        int zeros = freq.GetValueOrDefault(0, 0);
        freq.Remove(0);

        long[] dp     = new long[S + 1];
        long[] prefix = new long[S + 2];  // reused across all chains
        dp[0] = 1;

        foreach (var (val, cnt) in freq) {
            // val chains, each spanning dp[rem], dp[rem+val], ... ≤ S
            for (int rem = 0; rem < val; rem++) {
                // Pass 1: prefix sums from old dp values
                prefix[0] = 0;
                int k = 0;
                for (int j = rem; j <= S; j += val) {
                    prefix[k + 1] = (prefix[k] + dp[j]) % MOD;
                    k++;
                }
                // Pass 2: sliding window update (window = cnt+1 chain positions)
                k = 0;
                for (int j = rem; j <= S; j += val) {
                    dp[j] = (prefix[k + 1] - prefix[Math.Max(0, k - cnt)] + MOD) % MOD;
                    k++;
                }
            }
        }

        long ans = 0;
        for (int s = l; s <= r; s++) ans = (ans + dp[s]) % MOD;
        return (int)(ans * (zeros + 1) % MOD);
    }
}
```

## Complexity

| Metric | Value | Notes |
|---|---|---|
| Distinct non-zero values `d` | O(√S) ≤ 200 | From d(d+1)/2 ≤ S bound |
| Work per distinct value | O(S) | `val` chains × O(S/val) each = O(S) total |
| Total DP work | O(d·S) = **O(S^{3/2})** | ≈ 2.8×10⁶ ops for S=2×10⁴ |
| Space | O(S) | dp and prefix arrays; O(1) per freq entry |

The O(S^{3/2}) bound is non-obvious: it follows from d ≤ O(√S) which in turn requires the sum constraint `Σnums ≤ 2×10⁴`. Without this constraint, d could be O(n) and the algorithm would be O(n·S).

## Why Two Passes per Chain Are Required

The update `new_dp[s] = dp[s] + dp[s−v] + … + dp[s−c·v]` uses OLD dp values only. If we update dp[rem+i·v] in place, it corrupts dp[rem+(i+1)·v]'s computation (which needs old dp[rem+i·v]).

**Pass 1** captures the old chain in `prefix[]` before any dp modifications.  
**Pass 2** overwrites dp safely using only `prefix[]`.

This is the same reason the 0-1 knapsack iterates in reverse — here, the prefix-sum structure cleanly separates the two roles. ✓

---

## Pitfalls & Edge Cases

- **Value 0 in DP loop:** Would cause `for (rem = 0; rem < 0; ...)` — never executes, leaving dp unchanged as if no copies of 0 were considered. This silently gives wrong answers. Always remove 0 from freq before the DP and multiply by `(freq[0]+1)` at the end. ✓

- **Clamp `r = min(r, S)`:** dp is only defined up to index S. If `r > S`, dp[S+1..r] are implicitly 0 and needn't be summed. The clamp also enables the early exit `l > r → 0`.

- **Mod subtraction:** `prefix[k+1] - prefix[lo]` can be negative after mod operations. Always add MOD before taking mod: `(prefix[k+1] - prefix[lo] + MOD) % MOD`. ✓

- **prefix[] is reused across all chains.** It must be large enough for the longest chain (≤ S/1 + 1 = S+1 elements). Allocate once with size S+2 outside all loops. ✓

- **l = 0, r = 0 (empty sub-multiset):** dp[0] = 1 (the empty multiset) is always 1 after all updates, since no combination of non-zero items sums to 0. Answer = (zeros+1) if l=0, otherwise 0. ✓

- **All elements are 0:** S=0, freq has only 0. After removing 0, freq is empty; dp stays [1]. Sum dp[l..r] = dp[0] if l=0 else 0. Answer = (n+1) if l=0 else 0. ✓

- **Large cnt with small val:** For val=1, cnt can be up to n=2×10⁴. The sliding window handles this in O(S) just like any other cnt. ✓

---

## Key Takeaway

**Bounded knapsack counting with a grouped sliding window:** instead of adding one item copy at a time (O(n·S) total), group by value and use prefix sums over each "chain" (positions spaced v apart). The update for a window of size c+1 is `O(chain length)`, and summing all chains gives O(S) per distinct value.

**Why O(S^{3/2}) and not O(n·S):** the "sum of all nums ≤ S" constraint forces `d ≤ O(√S)` distinct non-zero values. Iterating over values (not items) gives O(d·S) = O(S^{3/2}).

**Pattern recognition:** "Count subsets/sub-multisets with bounded sum" where elements can repeat → bounded knapsack DP. For tight sum bounds, group by value + sliding window + exploit d = O(√S) to achieve O(S^{3/2}). The two-pass prefix-sum structure (read old → write new) is the standard trick for all bounded knapsack variants.
