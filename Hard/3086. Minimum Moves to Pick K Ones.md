## 3086. Minimum Moves to Pick K Ones

### Problem Statement
Given a binary array `nums` of length $n$, a positive integer `k`, and a non-negative integer `maxChanges`. Alice stands at a chosen index `aliceIndex`. If `nums[aliceIndex] == 1`, she picks it up for free. Otherwise, she can perform two types of moves:
1. Set an index $j \neq \text{aliceIndex}$ where `nums[j] == 0` to `1` (at most `maxChanges` times). Cost: **1 move**.
2. Swap a `1` with an adjacent `0`. If a `1` reaches `aliceIndex`, she picks it up. Cost: **1 move** per step.

Return the minimum total moves required to pick exactly `k` ones.

### Core Insight — Distance-Based Cost & Windowing
Alice can gather ones using four strategic priorities based on their structural distance and operation costs:
1. **Distance 0:** Existing `1` at `aliceIndex`. Cost = **0 moves** (Max: 1).
2. **Distance 1:** Existing `1`s at `aliceIndex` $\pm 1$. Cost = **1 move** per unit (Max: 2).
3. **`maxChanges` Operation:** Synthesizing a `1` on an adjacent cell and immediately picking it up. Cost = **2 moves** per unit ($1 \text{ move to mutate} + 1 \text{ move to swap/collect}$).
4. **Distance $\ge 2$:** Shifting real ones from remote indices. Cost = **$\Delta$ moves**, where $\Delta = |pos[i] - \text{aliceIndex}|$.

> **Architectural Constraint:** Since the mutation strategy via `maxChanges` generates a one for exactly 2 moves, executing remote swaps for ones at a distance $\ge 2$ is viable **only** when the `maxChanges` budget is completely exhausted. Any existing one at distance $\ge 2$ costs $\ge 2$ moves. Mutation is strictly more cost-effective or equivalent.

Therefore, the optimization branches into two distinct execution strategies:
* **Local Greedy Evaluation:** If $k \le \text{local ones} + \text{maxChanges}$ (where $\text{local ones} \le 3$), we can satisfy $k$ purely in a radius of $\le 1$ around a chosen index.
* **Global Median Windowing:** If `maxChanges` is fully utilized and we still need more ones, we must collect exactly $rem = k - \text{maxChanges}$ real ones from the array. To minimize the aggregate travel distance, these $rem$ ones must form a **contiguous subsequence** within the indices of all available ones. The optimal anchor (`aliceIndex`) for such a subsegment is its **median**.

---

### From Distance Summation to O(1) Prefix Scans
Let `pos` be a compressed array containing the absolute indices where `nums[i] == 1`. For a continuous window of ones spanning from index `l` to `r` (size $rem = r - l + 1$), the optimal point of convergence is the median index `mid = (l + r) / 2`.

The total travel distance from all elements in the window to `pos[mid]` is given by:
$$\sum_{i=l}^{r} |pos[i] - pos[mid]|$$

Splitting the absolute value summation around the median anchor yields:
$$\sum_{i=mid+1}^{r} pos[i] - \sum_{i=l}^{mid-1} pos[i] - pos[mid] \cdot ((mid - l) - (r - mid))$$

By precomputing a prefix sum array `pref` over `pos`, we can evaluate this distance for any window configuration in $\mathcal{O}(1)$ time.

---

### Algorithm
1. Iterate through all indices $i \in [0, n-1]$ to evaluate local states. Calculate the cost to collect up to $k$ ones using only `nums[i]`, its immediate neighbors, and `maxChanges`. Track the minimum.
2. Filter and extract the indices of all `1`s into a coordinate array `pos`.
3. If $rem = k - \text{maxChanges} \le 0$, return the minimum local cost calculated in Step 1.
4. If the total number of available ones is less than $rem$, return the local cost (or fallback).
5. Build a prefix sum array `pref` over `pos`.
6. Slide a window of fixed size $rem$ across `pos` from left to right. For each window $[l, r]$:
   - Compute `mid = l + (r - l) / 2`.
   - Resolve the distance cost in $\mathcal{O}(1)$ using `pref`.
   - Add the fixed cost of mutations: $2 \times \text{maxChanges}$.
   - Update the global minimum cost.
7. Return the overall minimum moves.

---

### Full Optimal Implementation (C++)
```cpp
class Solution {
public:
    long long minimumMoves(vector<int>& nums, int k, int maxChanges) {
        const int n = nums.size();
        long long min_total_moves = -1;

        // Step 1: Local greedy check for each index as potential aliceIndex
        for (int i = 0; i < n; ++i) {
            int in_place = nums[i];
            int adjacent = 0;
            if (i > 0) adjacent += nums[i - 1];
            if (i + 1 < n) adjacent += nums[i + 1];

            if (in_place + adjacent + maxChanges >= k) {
                long long current_local_cost = 0;
                if (k <= in_place) {
                    current_local_cost = 0;
                } else if (k <= in_place + adjacent) {
                    current_local_cost = k - in_place;
                } else {
                    current_local_cost = adjacent + 2LL * (k - in_place - adjacent);
                }
                
                if (min_total_moves == -1 || current_local_cost < min_total_moves) {
                    min_total_moves = current_local_cost;
                }
            }
        }

        // Step 2: Global window tracking if maxChanges budget is exhausted
        const int rem = k - maxChanges;
        if (rem <= 0) {
            return min_total_moves; 
        }

        // Memory optimization: compress coordinate space to cache lines
        vector<int> pos;
        pos.reserve(n); 
        for (int i = 0; i < n; ++i) {
            if (nums[i] == 1) {
                pos.push_back(i);
            }
        }

        const int m = pos.size();
        if (m < rem) {
            return min_total_moves; 
        }

        // Step 3: Compute Prefix Sums for O(1) interval queries
        vector<long long> pref(m + 1, 0);
        for (int i = 0; i < m; ++i) {
            pref[i + 1] = pref[i] + pos[i];
        }

        // Step 4: Sliding window evaluation over coordinates
        for (int l = 0; l <= m - rem; ++l) {
            const int r = l + rem - 1;
            const int mid = l + (r - l) / 2;
            
            long long left_sum = pref[mid] - pref[l];
            long long right_sum = pref[r + 1] - pref[mid + 1];
            
            long long distance_cost = right_sum - left_sum + (long long)pos[mid] * ((mid - l) - (r - mid));
            long long current_global_cost = distance_cost + 2LL * maxChanges;
            
            if (min_total_moves == -1 || current_global_cost < min_total_moves) {
                min_total_moves = current_global_cost;
            }
        }

        return min_total_moves;
    }
};
```

### Complexity
| Metric | Value | Notes |
|---|---|---|
| **Time** | $\mathcal{O}(N)$ | One pass for local validation, one for coordinate extraction, and one sliding window pass. |
| **Space** | $\mathcal{O}(N)$ | Coordinates stored in `pos` and range sums stored in `pref`. Allocation overhead minimized via `reserve()`. |

### Example Walkthrough
#### `nums = [1,1,0,0,0,1,1,0,0,1]`, `k = 3`, `maxChanges = 1`
* **Local Evaluation Check:** At $i = 1$, `in_place = 1`, `adjacent = 1` (from index 0). Total local ones = 2. Combined with `maxChanges = 1`, we can reach $2 + 1 = 3 \ge k$.
* **Cost Calculation:** $k$ is greater than `in_place + adjacent`, so we collect `adjacent` for 1 move, and use 1 mutation for 2 moves. Total Cost = $1 + 2(3 - 1 - 1) = 3$. 
* Global minimum updated to `3`.
* **Global Segment Assessment:** $rem = 3 - 1 = 2$.
  - `pos = [0, 1, 5, 6, 9]`
  - `pref = [0, 1, 6, 12, 21]` *(Note: pref[0]=0, pref[1]=0, pref[2]=1, pref[3]=6, pref[4]=12, pref[5]=21)*
* Checking windows of size 2 ($rem=2$):
  - Window $[0, 1]$ (`pos` elements 0, 1): `mid = 0`, cost resolves to $1 + 2 \times 1 = 3$.
  - Window $[1, 2]$ (`pos` elements 1, 5): `mid = 1`, cost resolves to $4 + 2 \times 1 = 6$.
* Minimum remains `3`. Returns `3`. ✓

### Pitfalls & Edge Cases
* **`maxChanges` Over-allocation:** If `maxChanges` is larger than $k$, $rem$ becomes negative. Failing to return early at `rem <= 0` can lead to out-of-bounds calculations when checking window intervals.
* **Cache Re-allocations:** Filtering the sparse array into `pos` can trigger repeated memory re-allocations in heap memory. Calling `pos.reserve(n)` forces a single contiguous block assignment, avoiding capacity doublings.
* **Integer Overflow:** The total moves can sum up to indices roughly around $10^5 \times 10^5$, pushing values past the maximum capacity of standard 32-bit signed integers (`int`). Intermediate calculations and accumulators must be typed as `long long`.

### Key Takeaway
Cost optimization with a cheaper local alternative (mutations at cost 2) implies that distant data components (elements at distance $\ge 2$) should only be leveraged when the alternative budget is completely drained. Once forced to use distant points, minimizing linear distances across a 1D vector is mathematically equivalent to grouping elements around their localized window median. This converts a complex spatial search into an $\mathcal{O}(1)$ interval summary.
