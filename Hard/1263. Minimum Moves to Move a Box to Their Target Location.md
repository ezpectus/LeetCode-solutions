# 1263. Minimum Moves to Move a Box to Their Target Location
## O(m³n³) — BFS on (Box, Player) State with Inner Reachability Check

---

## Problem Statement

Given an `m × n` grid with:
- `'B'` — the box (exactly one)
- `'S'` — the player (exactly one)
- `'T'` — the target (exactly one)
- `'#'` — walls (impassable)
- `'.'` — free floor

The player can walk freely (4 directions) unless blocked by a wall **or the box**. To push the box one cell in direction `d`, the player must occupy the cell directly opposite to `d` from the box, then step into the box's cell. Return the **minimum number of pushes** to move `'B'` to `'T'`, or `-1` if impossible.

---

## Core Insight #1 — Box Position Alone Is Not Enough State

The naive instinct is BFS purely on box position. This fails because **whether a push is legal depends on where the player is**, and the player's position is constrained by the current box location (the box blocks paths). Two situations where the box is at the same cell but the player is in different quadrants may require very different push sequences to continue — one might be solvable, the other not.

The correct state is:

```
state = (box_row, box_col, player_row, player_col)
```

Total distinct states: `O(m²n²)` = at most `20⁴ = 160,000` for the given constraints. Tractable.

---

## Core Insight #2 — Pushes vs. Player Steps Have Different "Costs"

We want to **minimize pushes only**, not total steps. Player movement between pushes is free bookkeeping. This immediately suggests:

- **Outer BFS** over states `(box, player)`, where each BFS level = **one push**.
- **Inner BFS** to check whether the player can navigate from their current cell to the cell required to make a given push, with the box acting as a dynamic wall during that walk.

Because every transition in the outer BFS increments the push count by exactly 1, standard BFS (FIFO queue) correctly returns the minimum number of pushes at the first time the box reaches `'T'` — no Dijkstra or 0-1 BFS needed.

---

## Core Insight #3 — For Each Push Direction, the Required Player Cell Is Fixed

To push the box at `(br, bc)` in direction `(dr, dc)`:
- The new box cell is `(br+dr, bc+dc)` — must be in-bounds and not a wall.
- The player must stand at `(br-dr, bc-dc)` before the push — the cell directly **opposite** to the push direction.
- After the push, the player ends up at `(br, bc)` (where the box was).

So for each of 4 directions, we have a fully determined "required player pre-position." The only question is: **can the player walk there from their current position, treating the box as a wall?** That's a standard BFS/DFS on the grid.

```
For each direction d ∈ {up, down, left, right}:
  reqCell = boxPos - d        // player must stand here
  newBox  = boxPos + d        // box lands here

  if newBox is out-of-bounds or '#': skip
  if reqCell is out-of-bounds or '#': skip
  if player cannot BFS-reach reqCell (box = wall): skip

  enqueue (newBox, boxPos, pushes+1)   // player ends up at old boxPos
```

## Full Optimal Implementation (C++)

```cpp
class Solution {
public:
    int minPushBox(vector<vector<char>>& grid) {
        int m = grid.size(), n = grid[0].size();
        int bx, by, px, py, tx, ty;

        for (int i = 0; i < m; i++)
            for (int j = 0; j < n; j++) {
                if (grid[i][j] == 'B') { bx = i; by = j; }
                if (grid[i][j] == 'S') { px = i; py = j; }
                if (grid[i][j] == 'T') { tx = i; ty = j; }
            }

        const int dirs[4][2] = {{0,1},{0,-1},{1,0},{-1,0}};

        // Inner BFS: can player reach (tr,tc) from (sr,sc) with box blocking (boxR,boxC)?
        auto canReach = [&](int sr, int sc, int tr, int tc, int boxR, int boxC) -> bool {
            if (sr == tr && sc == tc) return true;
            vector<vector<bool>> vis(m, vector<bool>(n, false));
            queue<pair<int,int>> q;
            q.push({sr, sc});
            vis[sr][sc] = true;
            while (!q.empty()) {
                auto [r, c] = q.front(); q.pop();
                for (auto& d : dirs) {
                    int nr = r + d[0], nc = c + d[1];
                    if (nr < 0 || nr >= m || nc < 0 || nc >= n) continue;
                    if (vis[nr][nc] || grid[nr][nc] == '#') continue;
                    if (nr == boxR && nc == boxC) continue; // box = wall
                    vis[nr][nc] = true;
                    if (nr == tr && nc == tc) return true;
                    q.push({nr, nc});
                }
            }
            return false;
        };

        // Outer BFS: state = (boxR, boxC, playerR, playerC)
        // vis[bx][by][px][py] to avoid reprocessing
        vector<vector<vector<vector<bool>>>> vis(
            m, vector<vector<vector<bool>>>(
            n, vector<vector<bool>>(
            m, vector<bool>(n, false))));

        // Queue stores: {boxR, boxC, playerR, playerC, pushes}
        queue<tuple<int,int,int,int,int>> bfsQ;
        bfsQ.push({bx, by, px, py, 0});
        vis[bx][by][px][py] = true;

        while (!bfsQ.empty()) {
            auto [bRow, bCol, pRow, pCol, pushes] = bfsQ.front();
            bfsQ.pop();

            if (bRow == tx && bCol == ty) return pushes;

            for (auto& d : dirs) {
                int newBR = bRow + d[0], newBC = bCol + d[1]; // where box lands
                int reqPR = bRow - d[0], reqPC = bCol - d[1]; // where player must stand

                if (newBR < 0 || newBR >= m || newBC < 0 || newBC >= n) continue;
                if (grid[newBR][newBC] == '#') continue;
                if (reqPR < 0 || reqPR >= m || reqPC < 0 || reqPC >= n) continue;
                if (grid[reqPR][reqPC] == '#') continue;

                // Player walks to reqPos (box = obstacle); after push, player is at old box cell
                if (!canReach(pRow, pCol, reqPR, reqPC, bRow, bCol)) continue;

                if (!vis[newBR][newBC][bRow][bCol]) {
                    vis[newBR][newBC][bRow][bCol] = true;
                    bfsQ.push({newBR, newBC, bRow, bCol, pushes + 1});
                }
            }
        }

        return -1;
    }
};
```

## State Machine Diagram

```
State: (B_pos, P_pos)
         │
         ▼
  Try 4 push directions
         │
    ┌────┴────┐
    │  newBox │  in-bounds & not '#'?  ──── NO ──► skip
    └────┬────┘
         │ YES
    ┌────▼──────┐
    │  reqCell  │  in-bounds & not '#'?  ── NO ──► skip
    └────┬──────┘
         │ YES
    ┌────▼──────────────────────────┐
    │  Inner BFS: P_pos → reqCell   │
    │  (box blocks one cell)        │  unreachable ──► skip
    └────┬──────────────────────────┘
         │ reachable
         ▼
  Enqueue (newBox, oldBox, pushes + 1)
```

## Complexity

| Metric | Value | Notes |
|---|---|---|
| Outer BFS states | O(m²n²) | box × player positions |
| Per state work | O(mn) | inner BFS on m×n grid |
| **Total Time** | **O(m³n³)** | ≤ 20⁶ = 64M, fine in practice |
| Space | O(m²n²) | vis array + queue |

For `m = n = 20`: outer states = 160,000; inner BFS = 400 cells each → ~64M ops, comfortably within time limits.

## Example Walkthrough

**Example 1:**
```
# # # # # #
# T # # # #
# . . B . #
# . # # . #
# . . . S #
# # # # # #
```

Box at (2,3), Player at (4,4), Target at (1,1).

- **Push 1 — Left:** player walks from (4,4) to (2,4) [right of box]. Box moves (2,3)→(2,2). Player at (2,3).
- **Push 2 — Left:** player already at (2,3) [right of box]. Box moves (2,2)→(2,1). Player at (2,2).
- **Push 3 — Up:** player walks from (2,2) to (3,1) [below box]. Box moves (2,1)→(1,1) = **T**. ✓

**Output: 3** ✓

---

**Example 2:**
```
# # # # # #
# T # # # #
# . . B . #
# # # # . #   ← row 3 left side walled off
# . . . S #
# # # # # #
```

Box at (2,3). To push it left the player needs to reach (2,4) — reachable. But to then push it left again, player needs (2,3) while box is at (2,2), but row 3 is fully blocked left of column 4 — player can **never** reach the left side of the box to continue pushing. All paths to target are blocked.

**Output: -1** ✓

## Pitfalls & Edge Cases

- **Box blocks player paths:** Always pass the current box position as the "wall" to the inner BFS. Forgetting this makes the player appear to walk through the box — produces wrong answers on any grid where the box cuts off corridors.

- **Visited state must include both box AND player position:** If you only track `(box_pos)`, you may mark a box cell as visited at push count `k` when reached with the player in a bad quadrant, blocking a later arrival with player in the correct quadrant at the same push count but from which more pushes are possible.

- **Outer BFS is FIFO, not priority queue:** Every transition costs exactly 1 push. FIFO BFS is both correct and faster than Dijkstra here.

- **Player's post-push position is the old box cell:** After pushing the box in direction `d`, the player steps into where the box was. `(bRow, bCol)` — not `reqCell` — is the player's new position in the enqueued state.

- **Target already reached at state load time:** Check `bRow == tx && bCol == ty` when popping, not when enqueuing. Both work, but checking on pop is cleaner and handles the rare edge case where the box starts on the target (then the answer is 0 and the initial pop immediately returns).

- **Grid borders as walls:** The outer border is always `'#'` per the examples, but the bound checks `nr < 0 || nr >= m` in the inner BFS handle grids without border walls too.

---

## Key Takeaway

This is a **"two-level BFS"** pattern: an outer BFS minimizes the primary cost metric (pushes), and an inner BFS handles the auxiliary feasibility check (player reachability) that gates each outer transition. The state space must capture **all information that affects future decisions** — here, both the box and player positions, not just the box.

**Pattern recognition:** "Minimize some event count; each event requires a subproblem to be feasible first" → outer BFS on (primary state + auxiliary state), inner BFS/DFS for feasibility. Classic in Sokoban-type puzzles, gate-and-key problems, and robot-with-obstacles navigation.
