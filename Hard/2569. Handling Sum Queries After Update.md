# 2569. Handling Sum Queries After Update
## O((n + q) log n) — Range Flip Tracking via Lazy Segment Tree

### Problem Statement
You are given two 0-indexed arrays `nums1` (containing only 0s and 1s) and `nums2` of length n, along with a 2D array `queries`. There are three types of queries:
* **Type 1 (`[1, l, r]`):** Flip all values from index l to r in `nums1` (0 becomes 1, 1 becomes 0).
* **Type 2 (`[2, p, 0]`):** For every index i, update `nums2[i] = nums2[i] + nums1[i] * p`.
* **Type 3 (`[3, 0, 0]`):** Calculate and return the total sum of all elements in `nums2`.

Return an array containing the results for all Type 3 queries.

---

### Core Idea — Global Sum Isolation & Lazy Propagation
If we try to update every individual element in `nums2` during a Type 2 query, it will take O(n) time per query, leading to an unusable O(q * n) time complexity. 

Instead, notice that a Type 2 query modifies the **global sum** of `nums2` by a predictable amount:
* Every index where `nums1[i] == 1` contributes exactly `p` to the total sum.
* Therefore, the total sum increases by: `(total number of 1s in nums1) * p`.

Because Type 3 only asks for the global sum, we do not need to maintain the individual elements of `nums2` at all! We can keep a single 64-bit variable tracking `sumNums2`. The problem then simplifies to efficiently counting the total number of 1s in `nums1` after performing range flip operations. This is a classic textbook use case for a **Segment Tree with Lazy Propagation**.

---

### Algorithm
1. **Initialize Global Sum:** Compute the initial sum of `nums2` into a 64-bit integer (`sumNums2`).
2. **Build the Segment Tree:** Construct a segment tree over `nums1`. Each node stores the number of 1s in its respective range.
3. **Handle Queries:**
   * **Type 1:** Perform a range update on the segment tree from `l` to `r`. When flipping a range of length `len` that currently has `k` ones, the new count of ones becomes `len - k`. Use a lazy array to defer updates to child nodes.
   * **Type 2:** Retrieve the total number of 1s in `nums1` (which is always stored at the root of the segment tree, `tree[1]`). Update `sumNums2 += (long)total_ones * p`.
   * **Type 3:** Append the current value of `sumNums2` to the results array.

---

### Full Optimal Implementation (C#)

```csharp
using System;
using System.Collections.Generic;

public class Solution {
    private int[] tree;
    private bool[] lazy;

    private void Build(int node, int start, int end, int[] nums1) {
        if (start == end) {
            tree[node] = nums1[start];
            return;
        }
        int mid = start + (end - start) / 2;
        Build(2 * node, start, mid, nums1);
        Build(2 * node + 1, mid + 1, end, nums1);
        tree[node] = tree[2 * node] + tree[2 * node + 1];
    }

    private void Push(int node, int start, int end) {
        if (lazy[node]) {
            int mid = start + (end - start) / 2;
            
            // Push down to left child
            lazy[2 * node] = !lazy[2 * node];
            tree[2 * node] = (mid - start + 1) - tree[2 * node];
            
            // Push down to right child
            lazy[2 * node + 1] = !lazy[2 * node + 1];
            tree[2 * node + 1] = (end - mid) - tree[2 * node + 1];
            
            lazy[node] = false;
        }
    }

    private void Update(int node, int start, int end, int l, int r) {
        if (l <= start && end <= r) {
            lazy[node] = !lazy[node];
            tree[node] = (end - start + 1) - tree[node];
            return;
        }
        Push(node, start, end);
        int mid = start + (end - start) / 2;
        if (l <= mid) {
            Update(2 * node, start, mid, l, r);
        }
        if (r > mid) {
            Update(2 * node + 1, mid + 1, end, l, r);
        }
        tree[node] = tree[2 * node] + tree[2 * node + 1];
    }

    public long[] HandleQuery(int[] nums1, int[] nums2, int[][] queries) {
        int n = nums1.Length;
        tree = new int[4 * n];
        lazy = new bool[4 * n];
        
        Build(1, 0, n - 1, nums1);
        
        long sumNums2 = 0;
        foreach (int val in nums2) {
            sumNums2 += val;
        }
        
        List<long> resultList = new List<long>();
        
        foreach (var q in queries) {
            int type = q[0];
            if (type == 1) {
                int l = q[1];
                int r = q[2];
                Update(1, 0, n - 1, l, r);
            } else if (type == 2) {
                long p = q[1];
                long totalOnes = tree[1];
                sumNums2 += totalOnes * p;
            } else if (type == 3) {
                resultList.Add(sumNums2);
            }
        }
        
        return resultList.ToArray();
    }
}
```

### Complexity

| Metric | Value | Notes |
| :--- | :--- | :--- |
| **Time Complexity** | O((n + q) log n) | Tree initialization takes O(n). Each range flip takes O(log n) via lazy propagation. Type 2 and Type 3 queries run in O(1) time. |
| **Space Complexity** | O(n) | The segment tree tracking arrays require 4 * n space, which easily fits within memory limit benchmarks. |

---

### Why "One Universal Sink" Always Suffices
It is never optimal to create complex cyclic shifting networks (e.g., Device A gives to B, B gives to C, C gives to A) to avoid dropping ratings. In any group of devices that mutually donate, the absolute smallest unit among them will inevitably dictate the upper bound of the next device in the chain.

By dedicating exactly one device to absorb the lowest values, we isolate the negative constraint to a single spot. Any secondary tracking becomes redundant because a single sink can accept an infinite number of incoming units without changing its minimum rating boundary.

---

### Pitfalls & Edge Cases
* **Integer Overflow:** Elements of `nums2` can be up to 10^9, and n can be 10^5. The initial sum can be up to 10^14. Additionally, multiplying `totalOnes * p` can yield up to 10^11 per operation. Using standard 32-bit signed integers will guarantee overflow. The tracking variables must use `long`. ✓
* **Size of Tree Array:** Array-based binary segment trees require a size boundary of up to 4 * n to handle leaf distribution without throwing out-of-bounds index exceptions during execution. ✓
* **Toggling Lazy Flags:** When a node already has a pending lazy operation, a second flip cancels it out. Use `lazy[node] = !lazy[node]` to accurately handle overlapping toggles instead of hard-setting it to true. ✓

---

### Key Takeaway
> When query patterns ask for a global aggregation property (like a grand sum) but request linear updates predicated on another array, avoid updating individual values. Decouple the aggregation target, translate the transaction to a range problem, and capture the structural variance efficiently.

* **Pattern recognition:** "Range flip operations on binary tracking array + scalar collection accumulation" -> Lazy-propagated tracking on a Segment Tree tracking unit presence boundaries.
