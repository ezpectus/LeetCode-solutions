# 2736. Maximum Sum Queries
## O((N + Q) log (N + Q)) — Offline Queries + Monotonic Stack with Binary Search

### Problem Statement
You are given two 0-indexed integer arrays `nums1` and `nums2`, both of length $n$, and a 1-indexed 2D array `queries` where `queries[i] = [xi, yi]`.

For each query $i$, find the maximum value of `nums1[j] + nums2[j]` among all indices $j$ such that `nums1[j] >= xi` and `nums2[j] >= yi`. If no such index exists, return `-1`. Return an array containing the answers for all queries.

---

### Core Idea — Eliminating a Dimension via Offline Processing
This problem asks us to evaluate a 2D range query optimization condition: $\max(nums1[j] + nums2[j])$ subject to $nums1[j] \ge x_i$ and $nums2[j] \ge y_i$. 

Because the queries don't depend on each other, we can process them **offline** (reordering them to our advantage and filling the results back into their original positional slots). 

1. **Sort Data Pairs:** Combine `nums1[j]` and `nums2[j]` into objects tracking `(nums1, nums2, sum)`, and sort them in **descending order** of `nums1`.
2. **Sort Queries:** Store the queries along with their original indices, and sort them in **descending order** of $x_i$.

By iterating through the sorted queries, we can dynamically insert data pairs into a custom data structure as long as `pair.nums1 >= query.x`. Because subsequent queries have a smaller or equal $x_i$, any element already added is guaranteed to remain valid for all future queries. This completely eliminates the first constraint dimension ($nums1[j] \ge x_i$).

#### Dynamic Monotonic Stack Dominance
For the second constraint ($nums2[j] \ge y_i$), we need to find an added pair that maximizes the `sum`. Suppose we maintain a stack of valid pairs. A pair $A$ completely dominates a pair $B$ if:
$$A.nums2 \ge B.nums2 \quad \text{and} \quad A.sum \ge B.sum$$

Since we process pairs in descending order of `nums1`, any new pair `curr` has a smaller or equal `nums1` than the elements already in the stack (`top`). 
If `curr.sum >= top.sum`, then:
$$curr.nums1 + curr.nums2 \ge top.nums1 + top.nums2$$
Because $curr.nums1 \le top.nums1$, it mathematically forces $curr.nums2 \ge top.nums2$. 

Therefore, whenever a new pair has a `sum` greater than or equal to the top of the stack, **it completely dominates the top element** on both metrics and we can safely pop it. This keeps our stack ordered such that from bottom to top:
* `nums2` is strictly increasing.
* `sum` is strictly decreasing.

To resolve a query for a given $y_i$, we can binary search the stack to locate the first element from the bottom that satisfies `nums2 >= y_i`. Since `sum` decreases from bottom to top, this first valid element is guaranteed to yield the absolute maximum sum.

### Full Optimal Implementation (C#)

```csharp
using System;
using System.Collections.Generic;

public class Solution {
    public int[] MaximumSumQueries(int[] nums1, int[] nums2, int[][] queries) {
        int n = nums1.Length;
        int q = queries.Length;
        
        // Step 1: Pair up corresponding array coordinates and sort descending by nums1
        List<(int n1, int n2, int sum)> pairs = new List<(int, int, int)>(n);
        for (int i = 0; i < n; i++) {
            pairs.Add((nums1[i], nums2[i], nums1[i] + nums2[i]));
        }
        pairs.Sort((a, b) => b.n1.CompareTo(a.n1));
        
        // Step 2: Bind queries with their initial position index and sort descending by x
        List<(int x, int y, int id)> sortedQueries = new List<(int, int, int)>(q);
        for (int i = 0; i < q; i++) {
            sortedQueries.Add((queries[i][0], queries[i][1], i));
        }
        sortedQueries.Sort((a, b) => b.x.CompareTo(a.x));
        
        int[] result = new int[q];
        // Stack stores elements as (nums2, sum)
        List<(int n2, int sum)> stack = new List<(int, int)>();
        int pairIdx = 0;
        
        // Step 3: Process the sorted queries offline
        for (int i = 0; i < q; i++) {
            var query = sortedQueries[i];
            
            // Push all available data pairs that satisfy the current x boundary threshold
            while (pairIdx < n && pairs[pairIdx].n1 >= query.x) {
                var curr = pairs[pairIdx];
                
                // If the current pair has a higher or equal sum, it dominates the top element
                while (stack.Count > 0 && stack[stack.Count - 1].sum <= curr.sum) {
                    stack.RemoveAt(stack.Count - 1);
                }
                
                // Only insert if it provides a higher value range than the current monotonic floor
                if (stack.Count == 0 || stack[stack.Count - 1].n2 < curr.n2) {
                    stack.Add((curr.n2, curr.sum));
                }
                
                pairIdx++;
            }
            
            // Step 4: Binary search inside the monotonic stack to satisfy the y boundary condition
            int low = 0, high = stack.Count - 1;
            int maxSum = -1;
            
            while (low <= high) {
                int mid = low + (high - low) / 2;
                if (stack[mid].n2 >= query.y) {
                    maxSum = stack[mid].sum; // Cache this valid option
                    high = mid - 1;          // Continue searching left (closer to bottom = higher sum)
                } else {
                    low = mid + 1;
                }
            }
            
            result[query.id] = maxSum;
        }
        
        return result;
    }
}
```

### Complexity

| Metric | Value | Notes |
| :--- | :--- | :--- |
| **Time Complexity** | $O((N + Q) \log (N + Q))$ | Driven primarily by sorting the data pairs and queries. The scan pushes each pair into the stack at most once ($O(N)$ amortized), and performs a binary search of length $\le N$ for each query ($O(Q \log N)$). |
| **Space Complexity** | $O(N + Q)$ | Required to preserve intermediate structural allocations (the sorted pairs collection, the query mapping wrappers, and the monotonic stack array tracker). |

---

### Pitfalls & Edge Cases

* **The Monotonic Fallacy:** A common mistake is to sort elements by `nums2` and use a Segment Tree. While completely correct, a coordinate-compressed Segment Tree requires significant boilerplate allocation overhead. Capitalizing on the algebraic trick `curr.sum >= top.sum => curr.nums2 >= top.nums2` allows us to handle the query spaces using a lean linear `List` stack.
* **Preserving Original Position Identifiers:** Because the output array must map perfectly back to the original order of the `queries` argument, always bundle the initial element indexing position (`i`) prior to triggering any sorting operations.

---

### Key Takeaway

> When dealing with multi-dimensional interval properties ($\ge x$ and $\ge y$), sort the query space to neutralize one of the dimensions completely. If processing the remaining parameters sequentially reveals an algebraic dominance correlation between parameters, replace heavy range-query trees with a lightweight binary-searched monotonic stack.

* **Pattern recognition:** "Maximize combination metric across two arrays subject to dual lower-bound query bounds" -> Offline Descending Reordering + Monotonic Dominance Stack.
