# LeetCode 3382. Maximum Area Rectangle With Point Constraints II

## 1. Constraints & Complexity Bottleneck Analysis

* **Number of points ($N$):** $1 \le N \le 2 \times 10^5$. Due to this constraint, any naive $\mathcal{O}(N^2)$ exhaustive search for pairs will trigger a **Time Limit Exceeded (TLE)** status. The algorithm must scale tightly within $\mathcal{O}(N \log N)$ execution boundaries.
* **Coordinate Bounds:** $0 \le xCoord[i], yCoord[i] \le 8 \times 10^7$. The potential space is highly sparse. Storing points directly inside a 2D dense grid matrix is blocked by physical memory restrictions. Thus, **Coordinate Compression** on the $Y$-coordinates is required to translate large sparse numbers into a continuous index space $[0, M-1]$, where $M \le N$.
* **The Vertical Overlap Bottleneck:** Dense edge-cases feature numerous points sharing the exact same $X$-coordinate (e.g., thousands of coordinates resting on a single line like $X = 0$). To prevent performance degradation, the algorithm avoids redundant structural searches by pre-caching coordinate indexes per active column block and checking active segments sequentially.

---

## 2. Algorithm Design & Mathematical Invariants

### The Core Concept: Why Sorting by X Solves the Problem
The brute-force approach to finding the maximum valid rectangle requires evaluating every possible combination of four points. With $2 \times 10^5$ nodes, any $\mathcal{O}(N^4)$ or even $\mathcal{O}(N^2)$ exhaustive search will instantly trigger a Time Limit Exceeded (TLE) status. The spatial grid must be calculated in a single coordinated execution pass.

Sorting the points primarily by their $X$-coordinate ascending (left-to-right sweep) and secondarily by their $Y$-coordinate ascending (bottom-to-top column traversal) transforms a chaotic 2D spatial plane into a structured timeline. This forms the foundation of the **Sweep-Line Algorithm**.

Imagine a vertical line sweeping across the plane from left to right. Every time it encounters a set of points sharing the exact same $X$-coordinate, it processes them simultaneously as a unified column block. Because the elements inside this column are strictly sorted by $Y$, any two consecutive points form a natural vertical edge.

* **Identifying the Boundaries:** If two consecutive points in the active column share the same $X$-coordinate at heights $y_1$ and $y_2$, they form a potential vertical edge. To turn this segment into a rectangle, the algorithm looks back into a hash map (`std::map`) to see if an identical vertical height interval $[y_1, y_2]$ was processed at an earlier coordinate, $x_{\text{prev}}$.
* **Calculating the Surface Area:** The moment a matching pair of vertical intervals is verified, a candidate rectangle is formed. Its area is calculated using a constant-time execution step:

$$\text{Area} = (x_{\text{current}} - x_{\text{prev}}) \times (y_2 - y_1)$$

---

### Concrete Execution Example
Consider five coordinates plotted across a plane:
* **Column 1:** $A(1, 1)$, $B(1, 3)$
* **Column 2:** $C(2, 2)$
* **Column 3:** $D(4, 1)$, $E(4, 3)$

Sorting by $X$, then $Y$, guarantees they are processed in this exact sequential order: $[A, B] \to [C] \to [D, E]$.

1. **The sweep-line stops at $X = 1$:** It captures $A(1, 1)$ and $B(1, 3)$. They are consecutive on the active line, forming a vertical wall over the height interval $[1, 3]$. The algorithm caches this state: the height signature $[1, 3]$ was last seen at $X = 1$.
2. **The line moves to $X = 2$:** It encounters $C(2, 2)$. It is an isolated point in this column, meaning it cannot form a vertical wall by itself.
3. **The line reaches $X = 4$:** It identifies $D(4, 1)$ and $E(4, 3)$. They form a vertical wall over the height interval $[1, 3]$. Checking the tracking cache reveals that the height signature $[1, 3]$ was previously established at $X = 1$. This maps a structural candidate rectangle bounded by $X \in [1, 4]$ and $Y \in [1, 3]$.

---

### Bounding Box Obstacle Verification via Segment Tree
The core constraint of the problem requires that no other points exist inside the rectangle or on its perimeter lines. In the example above, the candidate rectangle spans from $X = 1$ to $X = 4$, and from $Y = 1$ to $Y = 3$.

However, coordinate $C(2, 2)$ sits directly inside this spatial boundary because $1 < 2 < 4$ and $1 < 2 < 3$. The presence of $C$ completely invalidates the rectangle formed by $A, B, D, E$. Scanning every previously processed point manually to find $C$ would degrade performance back to $\mathcal{O}(N^2)$. To solve this, a **Segment Tree** is integrated as a dynamic boundary validator.

* **The Inverted Index Structure:** The Segment Tree acts as an inverted lookup index. The leaves of the tree represent the unique, compressed $Y$-coordinates (heights) present in the input. Each leaf node dynamically stores the maximum (most recent) $X$-coordinate where a point was processed at that specific height.
* **Dynamic Range Queries:** As the sweep-line advances, every point it passes updates the tree state. When evaluating a candidate rectangle spanning from $y_1$ to $y_2$, the algorithm executes a Range Maximum Query (RMQ): *"What is the maximum X-coordinate registered across the entire height corridor from $y_1$ to $y_2$?"*
* **The Validation Invariant:** Let the maximum value returned by the Segment Tree range query be $\mathcal{M}$. The geometric safety of the rectangle is determined by a strict mathematical condition:
  * When processing the wall at $X = 4$ for the interval $[1, 3]$, the tree inspects all leaves within that height range.
  * Because point $C(2, 2)$ was processed at $X = 2$, the leaf tracking height $Y = 2$ stores the value $X = 2$.
  * The Range Maximum Query over $[1, 3]$ bubbles up the value $2$ (caused by the obstruction at $C$).
  * The algorithm compares this internal maximum against the left boundary of the rectangle ($X = 1$).
  * Since $\mathcal{M} > x_{\text{prev}}$ ($2 > 1$), it proves mathematically that an obstacle point breached the height corridor after the left boundary was established but before the right boundary was closed. The rectangle is instantly marked invalid.

---

### Architectural Choice: Why Fenwick Trees (BIT) Are Rejected
While a Fenwick Tree (Binary Indexed Tree) features lower memory overhead and simpler code implementation, it is structurally unsuited for this specific algorithm due to the mathematical nature of the queries.

* **The Prefix Restriction:** A Fenwick Tree is fundamentally designed to evaluate prefix ranges—meaning it calculates values aggregated from the beginning of an array up to a target index $i$ (ranges like $[0, i]$).
* **The Irreversibility of Maximum Queries:** To extract a random, isolated range query over an arbitrary interval $[l, r]$ using a Fenwick Tree, the underlying operation must be invertible. For example, a range sum query over $[l, r]$ is easily resolved via subtraction: 

$$\text{PrefixSum}(r) - \text{PrefixSum}(l-1)$$

  However, the $\max(a, b)$ operation is non-invertible. Knowing the maximum value in range $[0, r]$ and range $[0, l-1]$ does not allow you to deduce the maximum value strictly inside the isolated middle window $[l, r]$.
* **The Segment Tree Advantage:** A Segment Tree handles arbitrary range queries without requiring operational invertibility. Its explicit, symmetrical node-splitting architecture allows it to isolate and query any nested sub-interval $[l, r]$ with equal efficiency. It guarantees clean $\mathcal{O}(\log M)$ execution boundaries for Range Maximum Queries, making it the mathematically correct choice for this optimization engine.

## 3. High-Performance C++ Implementation

```cpp
#include <vector>
#include <algorithm>
#include <map>

using namespace std;

// High-Performance Point-Update Range-Maximum Segment Tree
class SegmentTree {
private:
    vector<int> tree;
    int n;

public:
    SegmentTree(int size) {
        n = size;
        tree.assign(4 * n, -1); // -1 signifies no point has been processed at this height index yet
    }

    void update(int node, int start, int end, int idx, int val) {
        if (start == end) {
            tree[node] = val; // Store the latest X coordinate at this specific Y index
            return;
        }
        int mid = start + (end - start) / 2;
        if (idx <= mid) {
            update(2 * node, start, mid, idx, val);
        } else {
            update(2 * node + 1, mid + 1, end, idx, val);
        }
        tree[node] = max(tree[2 * node], tree[2 * node + 1]);
    }

    int query(int node, int start, int end, int l, int r) {
        if (r < start || end < l) {
            return -1;
        }
        if (l <= start && end <= r) {
            return tree[node];
        }
        int mid = start + (end - start) / 2;
        int leftQuery = query(2 * node, start, mid, l, r);
        int rightQuery = query(2 * node + 1, mid + 1, end, l, r);
        return max(leftQuery, rightQuery);
    }
};

class Solution {
private:
    struct Point {
        int x, y;
    };

public:
    long long maxRectangleArea(vector<int>& xCoord, vector<int>& yCoord) {
        int n = xCoord.size();
        vector<Point> points(n);
        vector<int> rawY(n);

        for (int i = 0; i < n; i++) {
            points[i] = {xCoord[i], yCoord[i]};
            rawY[i] = yCoord[i];
        }

        // STEP 1: HIGH-SPEED SWEEP-LINE SORTING
        // Sort primarily by X ascending to move left-to-right.
        // Sort secondarily by Y ascending to process bottom-to-top on the same X line.
        sort(points.begin(), points.end(), [](const Point& a, const Point& b) {
            if (a.x != b.x) return a.x < b.x;
            return a.y < b.y;
        });

        // STEP 2: FAST COORDINATE COMPRESSION (Y-AXIS)
        // Shrink sparse Y bounds (up to 8*10^7) to dense indices array space [0, M-1].
        sort(rawY.begin(), rawY.end());
        rawY.erase(unique(rawY.begin(), rawY.end()), rawY.end());
        int m = rawY.size();

        SegmentTree tree(m);
        
        // Maps a packed pair of Y-coordinate indices to the most recent X-coordinate where this segment was seen.
        // Key: Bit-packed 64-bit long representation of compressed Y indices.
        map<long long, int> lastXForPair;
        long long maxArea = -1;

        // STEP 3: OPTIMIZED SWEEP-LINE ENGINE
        int idx = 0;
        while (idx < n) {
            int nextIdx = idx;
            // Scan and group all coordinates resting on the current X column
            while (nextIdx < n && points[nextIdx].x == points[idx].x) {
                nextIdx++;
            }

            int count = nextIdx - idx;
            if (count > 1) {
                // Pre-calculate compressed Y indices for all points on this X line to avoid repeated lookups
                vector<int> currentYIdx(count);
                for (int i = 0; i < count; i++) {
                    currentYIdx[i] = lower_bound(rawY.begin(), rawY.end(), points[idx + i].y) - rawY.begin();
                }

                // Evaluate consecutive vertical point segments on this line
                for (int i = 0; i < count - 1; i++) {
                    int yIdx1 = currentYIdx[i];
                    int yIdx2 = currentYIdx[i + 1];

                    // Pack two 32-bit integer indices into one 64-bit long key for O(1) map operations
                    long long pairKey = ((long long)yIdx1 << 32) | (unsigned int)yIdx2;

                    if (lastXForPair.count(pairKey)) {
                        int prevX = lastXForPair[pairKey];

                        // SEGMENT TREE RANGE QUERY:
                        // Find the maximum X-coordinate of ANY point within the compressed Y-range [yIdx1, yIdx2].
                        int maxInsideX = tree.query(1, 0, m - 1, yIdx1, yIdx2);

                        // POINT CONSTRAINT VALIDATION:
                        // If no obstacle points exist strictly between prevX and current X on this Y-interval,
                        // then the rectangle is completely empty and valid.
                        if (maxInsideX <= prevX) {
                            long long area = (long long)(points[idx].x - prevX) * (long long)(points[idx + i + 1].y - points[idx + i].y);
                            if (area > maxArea) {
                                maxArea = area;
                            }
                        }
                    }

                    // Cache the current X-coordinate as the latest boundary for this specific height interval
                    lastXForPair[pairKey] = points[idx].x;
                }
            }

            // SEGMENT TREE STATE UPDATE:
            // Push all points processed on this X line into the Segment Tree state.
            // This registers these points as potential boundary breaches for subsequent queries.
            for (int i = idx; i < nextIdx; i++) {
                int yIdx = lower_bound(rawY.begin(), rawY.end(), points[i].y) - rawY.begin();
                tree.update(1, 0, m - 1, yIdx, points[i].x);
            }

            idx = nextIdx;
        }

        return maxArea;
    }
};
```

## 4. Comprehensive Complexity & Mathematical Analysis

### Detailed Time Complexity Decomposition

The execution pipeline consists of four isolated stages. The total time complexity is derived from the summation of these processes:

$$\mathcal{T}_{\text{total}} = \mathcal{T}_{\text{sorting}} + \mathcal{T}_{\text{compression}} + \mathcal{T}_{\text{sweep}} + \mathcal{T}_{\text{tree}}$$

1. **High-Speed Sweep-Line Sorting ($\mathcal{T}_{\text{sorting}}$):**
   Packing spatial coordinates into a contiguous vector block of structures and invoking `std::sort` relies on an Introsort variant. It requires structural coordinate sorting partitioned by a strict weak ordering:
   $$\mathcal{T}_{\text{sorting}} = \mathcal{O}(N \log N)$$

2. **Y-Coordinate Space Compression ($\mathcal{T}_{\text{compression}}$):**
   Allocating the distinct coordinate heights requires a full sorting pass ($\mathcal{O}(N \log N)$), followed by a linear duplication extraction pass via `std::unique` operating in $\mathcal{O}(N)$. Thus:
   $$\mathcal{T}_{\text{compression}} = \mathcal{O}(N \log N)$$

3. **Optimized Sweep-Line Traversal Engine ($\mathcal{T}_{\text{sweep}}$):**
   The primary pointer engine advances columns linearly. Inside a column block containing $K$ elements, we run a localized cache compilation. Invoking `std::lower_bound` executing a binary search over the dense bounds takes $\mathcal{O}(\log M)$ per element. For all points across the plane:
   $$\sum \mathcal{O}(K \log M) = \mathcal{O}(N \log M)$$
   Evaluating individual consecutive segments inside a column performs exactly $K - 1$ operations. Hash map lookups via `std::map::count` take $\mathcal{O}(\log U)$, where $U$ is the number of distinct vertical pairs cached. Since $U \le N$:
   $$\mathcal{T}_{\text{lookup}} = \mathcal{O}(N \log N)$$

4. **Segment Tree Dynamic Verification Engine ($\mathcal{T}_{\text{tree}}$):**
   For every column block containing $K$ points, we execute at most $K-1$ range queries to test empty interior properties, followed by exactly $K$ point updates to update the tracking state. Both `query` and `update` methods recursively traverse a tree of height bounded by $\lceil \log_2 M \rceil$. Summing over all points across the sweep line execution yield:
   $$\mathcal{T}_{\text{tree}} = \mathcal{O}(N \log M)$$

Given that the unique coordinate scale satisfies $M \le N$, the overall asymptotic time complexity is tightly bounded by:

$$\mathcal{T}_{\text{total}} = \mathcal{O}(N \log N) + \mathcal{O}(N \log N) + \mathcal{O}(N \log M) = \mathcal{O}(N \log N)$$

### Detailed Space Complexity Decomposition

The structural memory ceiling is determined by three allocation structures:

1. **Continuous Vector Allocations:**
   * `points` vector tracking structures: $N \times \text{sizeof(Point)} = N \times 8 \text{ bytes} \implies \mathcal{O}(N)$
   * `rawY` transformation vector: $N \times 4 \text{ bytes} \implies \mathcal{O}(N)$
   * Internal column buffers `currentYIdx`: Maximum allocation space dynamically scales to $\mathcal{O}(K)$ per column block, safely reclamation within linear spatial rules.

2. **Segment Tree Memory Layout:**
   The leaf level maps exactly $M$ keys. The structural array size allocation relies on a standard $4M$ allocation block layout to prevent memory out-of-bounds tracking under full structural allocations:
   $$\mathcal{S}_{\text{tree}} = 4 \times M \times 4 \text{ bytes} \implies \mathcal{O}(M) \subseteq \mathcal{O}(N)$$

3. **Dynamic Pair Tracking Hash Space:**
   The `lastXForPair` map records distinct height transitions. In the absolute mathematical worst-case layout, every pair is unique, expanding memory allocation linearly:
   $$\mathcal{S}_{\text{map}} = \mathcal{O}(U) \subseteq \mathcal{O}(N)$$

Combining all structural parts, the total space complexity remains completely linear:

$$\mathcal{S}_{\text{total}} = \mathcal{O}(N)$$

## 5. Mathematical Formulations & Foundational Algorithms

### The Sweep-Line Optimization Equation
The algorithm reduces a two-dimensional geometric problem into a sequence of one-dimensional range queries. Instead of checking every combination of four points, sorting by $X$ means a vertical line $x = X_k$ scans the space. 

Let a pair of points on the line $x = X_k$ be $P_i = (X_k, y_1)$ and $P_j = (X_k, y_2)$ where $y_1 < y_2$. This segment forms a valid right vertical boundary if and only if there exists a cached entry in `lastXForPair` such that:

$$\exists \, x_{\text{prev}} = \text{lastXForPair}[(y_1, y_2)]$$

The area of this candidate rectangle is immediately calculated using the constant-time equation:

$$\text{Area} = (X_k - x_{\text{prev}}) \times (y_2 - y_1)$$

### The Segment Tree Obstacle Checking Equation
To prove that a candidate rectangle is empty, we must verify that no other point $P^*(x^*, y^*)$ satisfies the inclusion constraints:

$$\text{Inclusion} \iff (x_{\text{prev}} \le x^* \le X_k) \land (y_1 \le y^* \le y_2)$$

Since the unique point constraint rules out duplicate points, the points forming the vertices themselves are allowed on the corners. However, any other point inside or on the border invalidates the rectangle. 

By utilizing the Range Maximum Query (RMQ) feature of the segment tree, we check the maximum $X$-coordinate recorded across the compressed height range $[y_{\text{idx1}}, y_{\text{idx2}}]$:

$$\mathcal{M} = \max_{y \in [y_{\text{idx1}}, y_{\text{idx2}}]} \{ \text{Tree}[y] \}$$

The geometric validity of the candidate rectangle is determined by the following mathematical boolean invariant:

$$\text{Validity} = \begin{cases} \text{True}, & \text{if } \mathcal{M} \le x_{\text{prev}} \\ \text{False}, & \text{if } \mathcal{M} > x_{\text{prev}} \end{cases}$$

If $\mathcal{M} > x_{\text{prev}}$, it means an obstacle point $P^*$ exists at some coordinate $x^* = \mathcal{M}$. Because of the sweep-line order, this point is guaranteed to be to the left of or on the current line ($x^* \le X_k$), and the condition $\mathcal{M} > x_{\text{prev}}$ proves it lies to the right of the previous boundary, invalidating the rectangle.
