# 3806. Maximum Bitwise AND After Increment Operations
## O(30¬≤ ¬∑ n + 30 ¬∑ n log n) ‚Äî Greedy Bit Construction + Min-Cost Feasibility Check

---

## Problem Statement

Array `nums`, integers `k` (max operations) and `m` (subset size). Each operation increments one element by 1. Find the maximum possible bitwise AND of any m-element subset after at most k operations.

Constraints: n ‚â§ 5√ó10‚Å¥, nums[i] ‚â§ 10‚Åπ, k ‚â§ 10‚Åπ, m ‚â§ n.

---

## Core Insight 1 ‚Äî Greedy Bit-by-Bit Construction From High to Low

Build the answer `res` one bit at a time from bit 29 down to bit 0. For each bit b:
1. Tentatively set bit b: `candidate = res | (1 << b)`.
2. Check **feasibility**: can we choose m elements and increment them (within budget k) so that every chosen element has all bits of `candidate` set?
3. If feasible: keep bit b in `res`. Otherwise: discard it.

This greedy works because bits of AND are independent: setting a higher bit never hurts lower bits (the feasibility only becomes stricter, not inconsistent).

---

## Core Insight 2 ‚Äî Feasibility Check: Minimum Cost Per Element

For element `v` and candidate mask `res`, the minimum increment to make `v` have all bits of `res` set is computed by a **bit-setting scan from high to low**:

For each bit b in `res` (highest ‚Üí lowest):
- If bit b of `v` is already 1: no change needed.
- If bit b of `v` is 0: round up `v` to the next integer with bit b set:
  ```
  v = ((v >> b) + 1) √ó 2^b
  ```
  This clears bits 0..b-1 and sets bit b.

**Why processing high‚Üílow is correct and safe:**

When bit b of `v` is 0, `v >> b` is **even** (bit 0 of `v >> b` = bit b of `v` = 0). Adding 1 to an even number flips only bit 0 ‚Äî **no carry propagates** into higher bits. Therefore, all bits above b in `v` are preserved exactly. Lower bits (0..b-1) are cleared, but those will be re-set in subsequent iterations (when we process lower bits of `res`).

**Cost for element v with mask res = minimum x ‚â• v with (x & res) = res, then x - v.**

---

## Feasibility Decision

After computing `cost[i]` for all i = 0..n-1:
- Sort costs ascending.
- Sum the m smallest: `total = cost[0] + cost[1] + ‚Ä¶ + cost[m-1]`.
- If `total ‚â§ k`: the candidate is achievable. Set `res = candidate`.

The m smallest costs correspond to choosing the m cheapest elements to "convert" to the candidate. ‚úì

---

## Algorithm

```
res = 0
for b = 29 downto 0:
  candidate = res | (1 << b)
  for each v in nums:
    x = v
    for bit = 29 downto 0:           // process bits of candidate high‚Üílow
      if bit set in candidate and bit NOT set in x:
        x = ((x >> bit) + 1) * (1 << bit)
    cost[i] = x - v
  sort cost[]
  if cost[0] + cost[1] + ... + cost[m-1] <= k:
    res = candidate
return res
```

## Full Optimal Implementation (C++)

```csharp
class Solution {
    int getOps(int x, int t){
        if((x & t) == t) return 0;
        int cur = 0;
        for(int i = 30; i >= 0; i--)  {
            int t1 = t & (1LL << i) , x1 = x & (1LL << i);
            if(t1 && !x1) {
                cur |= (1LL << i);
                cur |= (t & ((1LL << i) -1));
                return cur -x;
            }
            if(x1) cur |= (1LL << i);
        }
        return cur - x;
    }
public:
    int maximumAND(vector<int>& nums, int k, int m) {
        int cur = 0;
        for(int i = 30; i >= 0; i--) {
            int t = cur  | (1 << i);
            vector<int> cost;
            for(int x: nums) 
                cost.push_back(getOps(x, t));
            sort(begin(cost), end(cost));

            int need = 0;
            for(int i = 0; i < m; i++){
                need += cost[i];
                if(need > k) break;
            }
            if(need <=k) cur = t;
        }
        return (int) cur;
    }
};
```

### The Bit-Setting Algorithm ‚Äî Proof That High‚ÜíLow Order Is Correct

**Key lemma:** When bit b of x is 0, `x >> b` is even. Adding 1 to an even number flips only bit 0 ‚Äî **no carry propagates above bit 0 of `x >> b`**, which corresponds to bit b+1 of x. All bits of x above b are unchanged.

**Consequence:** When we set bit b (by rounding up x), the bits of x above b remain exactly as they were. These are bits of `res` set in earlier iterations (higher bits), so they remain set. ‚úì

**Example:** res=0b110, v=1.
- Process bit 2: `1>>2=0` (even). `(0+1)√ó4=4`. x=4=0b100. bit 1 of 4 is 0 (lower bits cleared ‚Äî will be set next).
- Process bit 1: `4>>1=2` (even). `(2+1)√ó2=6`. x=6=0b110. **bit 2 of 6 = 1** ‚úì (preserved).

Cost = 6-1=5. Minimum x ‚â• 1 with bits 1,2 set = 6. ‚úì

---

### Why OR (v | candidate) Is NOT Sufficient

A common mistake: use `(v | candidate) - v` as the cost. This gives:
- v=5 (0b101), candidate=0b110: `5|6=7`, cost=7-5=2. But minimum x=6, cost=1. ‚ùå

The OR approach sets both bits at once but may overshoot. Our bit-by-bit rounding always finds the minimum.

## Pitfalls & Edge Cases

- **High‚Üílow bit processing is mandatory:** Processing low‚Üíhigh fails. Example: v=1, res=0b110. If we first set bit 1 (x=2), then set bit 2 (x=4), bit 1 of 4 is cleared. Result is wrong. High‚Üílow ensures carry from bit b setting only affects bits ‚â§ b (which are re-set in subsequent steps). ‚úì

- **Cost overflow:** Maximum cost per element ‚âà 2^30 ‚âà 10^9. With m=5√ó10^4 elements: total ‚â§ 5√ó10^13. Use `long` for costs and total. ‚úì

- **x can temporarily exceed 32-bit range:** During intermediate computations, `((x >> bit) + 1L) * (1L << bit)` uses `long` arithmetic. Maximum x ‚â§ 2^30 - 1 ‚âà 10^9. ‚úì

- **Element with cost=0:** If v already has all bits of candidate, it contributes cost 0 and is always chosen first (cheapest). ‚úì

- **Bits 30-31 (values > 10^9):** nums[i] ‚â§ 10^9 < 2^30. Starting at bit 29 covers all possible values. ‚úì

- **Sorting needed per candidate:** Costs change entirely when candidate changes (the new bit adds a new rounding step for each element). Re-sort for each bit. ‚úì

## Key Takeaway

**"Maximize bitwise AND of subset with limited increments"** ‚Üí greedy bit construction from high to low + feasibility check per candidate.

The feasibility check reduces to: "can we make m elements have all bits of candidate set with total cost ‚â§ k?" The minimum per-element cost is found by the **high-to-low bit-setting scan**, which works because `(even) + 1` never carries ‚Äî preserving already-set higher bits while setting the current bit.

**Pattern recognition:** "Maximize bitwise value under budget" ‚Üí always try bits from high to low, check if each bit is "affordable" using minimum total cost across the m cheapest elements. The independence of the cost check (each element's cost only depends on its value and the current candidate) makes the greedy valid.
