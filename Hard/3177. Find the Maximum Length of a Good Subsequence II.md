## 3177. Find the Maximum Length of a Good Subsequence II

### Problem Statement
An integer array `nums` and a non-negative integer `k` are given. A sequence of integers `seq` is called **good** if there are at most `k` indices $i$ in the range $[0, \text{seq.length} - 2]$ such that $\text{seq}[i] \neq \text{seq}[i + 1]$.

Return the maximum possible length of a good subsequence of `nums`.

### Core Insight — Dynamic Programming with Transition Optimization
The problem requires finding a maximum length subsequence where value transitions are restricted. Let $dp[i][j]$ represent the maximum length of a good subsequence ending at index $i$ containing exactly $j$ transitions where adjacent elements are not equal ($\text{seq}[m] \neq \text{seq}[m+1]$).

For the current element `nums[i]`, we can look at any previous element `nums[p]` (where $p < i$):
1. **If `nums[p] == nums[i]`:** No new mismatch is introduced. The number of changes remains $j$.
   $$dp[i][j] = \max(dp[i][j], dp[p][j] + 1)$$
2. **If `nums[p] != nums[i]`:** A new mismatch is introduced. The number of changes increases from $j-1$ to $j$.
   $$dp[i][j] = \max(dp[i][j], dp[p][j-1] + 1)$$

> **Architectural Constraint:** Evaluating this directly results in an $\mathcal{O}(N^2 \cdot K)$ runtime, which triggers a TLE given $N = 5000$ and $K = 50$. We must optimize both lookups to $\mathcal{O}(1)$ time.

#### Optimization Strategies
* **Optimizing Same-Element Transitions:** Map directly from values to lengths using a hash map tracking maximum lengths for specific values: `same_element_max[value][j]`. This drops the matching value search down to $\mathcal{O}(1)$.
* **Optimizing Different-Element Transitions:** To compute the max value from *any* previous element where `nums[p] != nums[i]` with $j-1$ changes, we do not need to check every distinct value. We maintain a running array `global_max[j]` representing the maximum subsequence length found so far using exactly $j$ changes across any historical element.

---

### From Cubic to Quadratic Execution Loop
By tracking states dynamically, when we process an element $x = \text{nums[i]}$:
- For a fixed number of mismatches $j$, the best choice if the previous element was also $x$ is `same_element_max[x][j] + 1`.
- The best choice if the previous element was different from $x$ is `global_max[j-1] + 1` (if $j > 0$).
- We pick the maximum of these two values to determine the new sequence length for state $j$.
- We then update both `same_element_max[x][j]` and `global_max[j]` for subsequent iterations.

---

### Algorithm
1. Create a hash map `same_element_max` where keys are values from `nums` and values are arrays of size $k + 1$ initialized to 0.
2. Create an array `global_max` of size $k + 1$ initialized to 0.
3. Iterate through every element $x \in \text{nums}$:
   - Compute temporary states for $j$ from $0$ up to $k$ to find the maximum possible length for each change threshold:
     - Initialize `current_length` as `same_element_max[x][j] + 1` (extending a matching value).
     - If $j > 0$, try to extend a non-matching value: `current_length = max(current_length, global_max[j-1] + 1)`.
     - Record this computed length in a temporary tracker to avoid contaminating dependencies during the loop.
   - After computing all tiers for the current element, update `same_element_max[x][j]` and `global_max[j]` with the maximum of their existing values and the newly calculated lengths.
4. Return `global_max[k]`.

---

### Full Optimal Implementation (C++)
```cpp
#include <vector>
#include <unordered_map>
#include <algorithm>

using namespace std;

class Solution {
public:
    int maximumLength(vector<int>& nums, int k) {
        // Map to track maximum length for each value given 'j' changes
        unordered_map<int, vector<int>> same_element_max;
        // Array to track global maximum length for 'j' changes across any value
        vector<int> global_max(k + 1, 0);

        for (int x : nums) {
            if (same_element_max.find(x) == same_element_max.end()) {
                same_element_max[x] = vector<int>(k + 1, 0);
            }

            // Temp storage to avoid updating global_max dependencies during execution
            vector<int> next_lengths(k + 1, 0);

            for (int j = 0; j <= k; ++j) {
                // Option 1: Append to an identical element (0 cost)
                int length = same_element_max[x][j] + 1;

                // Option 2: Append to a different element (1 cost, from j-1 state)
                if (j > 0) {
                    length = max(length, global_max[j - 1] + 1);
                }

                next_lengths[j] = length;
            }

            // Flush temporary calculations to state tracking structures
            for (int j = 0; j <= k; ++j) {
                same_element_max[x][j] = max(same_element_max[x][j], next_lengths[j]);
                global_max[j] = max(global_max[j], next_lengths[j]);
            }
        }

        return global_max[k];
    }
};
```

### Complexity
| Metric | Value | Notes |
|---|---|---|
| **Time** | $\mathcal{O}(N \cdot K)$ | For each of the $N$ elements, we run a localized transition loop up to $K$ times. Hash map queries take $\mathcal{O}(1)$ average time. |
| **Space** | $\mathcal{O}(N \cdot K)$ | The `same_element_max` map stores up to $N$ unique element profiles, each containing an array of size $K + 1$. |

### Example Walkthrough
#### `nums = [1,2,1,1,3]`, `k = 2`
* **$x = 1$:**
  - $j=0: \text{len} = \max(0+1) = 1$
  - Updates: `same_element_max[1] = [1,1,1]`, `global_max = [1,1,1]`
* **$x = 2$:**
  - $j=0: \text{len} = \max(0+1) = 1$
  - $j=1: \text{len} = \max(0+1, \text{global\_max}[0]+1) = \max(1, 1+1) = 2$
  - $j=2: \text{len} = \max(0+1, \text{global\_max}[1]+1) = \max(1, 1+1) = 2$
  - Updates: `same_element_max[2] = [1,2,2]`, `global_max = [1,2,2]`
* **$x = 1$:**
  - $j=0: \text{len} = \text{same\_element\_max}[1][0]+1 = 1+1 = 2$
  - $j=1: \text{len} = \max(\text{same\_element\_max}[1][1]+1, \text{global\_max}[0]+1) = \max(1+1, 1+1) = 2$
  - $j=2: \text{len} = \max(\text{same\_element\_max}[1][2]+1, \text{global\_max}[1]+1) = \max(1+1, 2+1) = 3$
  - Updates: `same_element_max[1] = [2,2,3]`, `global_max = [2,2,3]`
* **$x = 1$:**
  - $j=0: \text{len} = \text{same\_element\_max}[1][0]+1 = 2+1 = 3$
  - $j=1: \text{len} = \max(2+1, 2+1) = 3$
  - $j=2: \text{len} = \max(3+1, 2+1) = 4$
  - Updates: `same_element_max[1] = [3,3,4]`, `global_max = [3,3,4]`
* **$x = 3$:**
  - $j=2: \text{len} = \max(0+1, \text{global\_max}[1]+1) = \max(1, 3+1) = 4$
  - Updates: `global_max[2]` remains 4.

**Total Output:** `4`. ✓

### Pitfalls & Edge Cases
* **Premature State Contamination:** Updating `global_max[j-1]` while actively calculating states for the *same* element can let a single element trigger multiple mismatch transitions illegally. Using a temporary buffer (`next_lengths`) guarantees that transitions use records solely from previous indices.
* **Hash Map Overhead:** While `unordered_map` provides $\mathcal{O}(1)$ average lookup, high bucket collision rates could degrade performance. If constraints were tighter, coordinate compression mapping values directly to indices $0 \dots N-1$ would allow a flat `vector<vector<int>>` structure for absolute performance optimization.
* **$k=0$ Boundary Condition:** When $k=0$, the loop only processes $j=0$. The logic cleanly reduces to finding the maximum frequency of any single element in the array, resolving correctly.

### Key Takeaway
When a dynamic programming transition depends on checking all historical records, look for structural patterns in the lookups. Splitting your lookup into two optimized tracks—one tracking matching values and another tracking a running global maximum—turns a slow linear scan into a constant-time $\mathcal{O}(1)$ step.
