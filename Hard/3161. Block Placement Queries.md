# 3161. Block Placement Queries
## O(Q log Q) — Segment Tree + Sorted Set for Predecessor/Successor

---

## Problem Statement

- Infinite number line starting at 0.
- **Type 1** query: place obstacle at position `x`.
- **Type 2** query: check if a block of size `sz` fits entirely in `[0, x]` (can touch obstacles but not intersect them).
- Return a boolean array for all type 2 queries.

---

## Core Idea — Store Max Gap Per Obstacle in a Segment Tree

Key observations:
- Origin 0 is an implicit obstacle (block must stay in `[0, x]`).
- The number line is divided by obstacles into gaps. A block of size `sz` fits in `[0, x]` iff any gap within `[0, x]` has length ≥ `sz`.
- Two types of gaps for a query at x:
  - **Internal gaps**: between consecutive obstacles both ≤ x.
  - **Tail gap**: from the largest obstacle ≤ x to x itself.

**Segment tree approach:** For each obstacle at position `p`, store `p − prev(p)` (gap from its predecessor) in a segment tree at index `p`. A range-max query on `[1, x]` gives the largest internal gap. Add the tail gap and we have the answer.

When adding obstacle at `p`:
- Update `seg[p] = p − prev(p)`
- Update `seg[next(p)] = next(p) − p` (its gap shrinks since `p` is now between them)

## Full Optimal Implementation (C#)

```csharp
public class Solution {
    private const int MAXN = 50001;
    private int[] seg = new int[4 * MAXN];

    private void Update(int node, int l, int r, int pos, int val) {
        if (l == r) { seg[node] = val; return; }
        int mid = (l + r) >> 1;
        if (pos <= mid) Update(node << 1, l, mid, pos, val);
        else Update((node << 1) | 1, mid + 1, r, pos, val);
        seg[node] = Math.Max(seg[node << 1], seg[(node << 1) | 1]);
    }

    private int Query(int node, int l, int r, int ql, int qr) {
        if (l > qr || r < ql) return 0;
        if (ql <= l && r <= qr) return seg[node];
        int mid = (l + r) >> 1;
        return Math.Max(
            Query(node << 1, l, mid, ql, qr),
            Query((node << 1) | 1, mid + 1, r, ql, qr)
        );
    }

    public IList<bool> GetResults(int[][] queries) {
        var obs = new SortedSet<int> { 0 }; // 0 is the implicit left boundary
        var res = new List<bool>();

        foreach (var q in queries) {
            int x = q[1];

            if (q[0] == 1) {
                // Find predecessor (always exists since 0 is in obs)
                int prev = obs.GetViewBetween(0, x).Max;

                // Find successor (O(log n) via enumeration, avoids O(n) Count)
                int next = -1;
                foreach (int v in obs.GetViewBetween(x + 1, MAXN)) {
                    next = v;
                    break;
                }

                obs.Add(x);
                Update(1, 0, MAXN - 1, x, x - prev);
                if (next != -1) Update(1, 0, MAXN - 1, next, next - x);

            } else {
                int sz = q[2];
                // Tail gap: from last obstacle ≤ x to x
                int prevObs = obs.GetViewBetween(0, x).Max;
                int tailGap = x - prevObs;
                // Internal max gap: max stored gap for obstacles in [1, x]
                int maxGap = Query(1, 0, MAXN - 1, 1, x);
                res.Add(Math.Max(tailGap, maxGap) >= sz);
            }
        }

        return res;
    }
}
```

## Complexity

| Metric | Value | Notes |
|---|---|---|
| Time Complexity | O(Q log Q) | Each query: O(log n) SortedSet + O(log MAXN) seg tree |
| Space Complexity | O(MAXN + Q) | Seg tree of size 4·MAXN + sorted obstacle set |

MAXN = 50000, so log MAXN ≈ 16 — effectively constant per query.

## Why This Works — Example Walkthrough

**Example 2:** `queries = [[1,7],[2,7,6],[1,2],[2,7,5],[2,7,6]]`

Initial state: obs = {0}, seg = all zeros.

**[1,7]:** prev=0, next=−1 → Update seg[7]=7. obs={0,7}.

```
seg: pos 7 → 7
```

**[2,7,6]:** prevObs=7, tailGap=0, maxGap=Query(1,7)=7 → max(0,7)=7 ≥ 6 → **true** ✓

**[1,2]:** prev=0, next=7 → Update seg[2]=2, Update seg[7]=5. obs={0,2,7}.

```
seg: pos 2 → 2, pos 7 → 5
```

**[2,7,5]:** prevObs=7, tailGap=0, maxGap=Query(1,7)=max(2,5)=5 → max(0,5)=5 ≥ 5 → **true** ✓

**[2,7,6]:** Same as above → max(0,5)=5 ≥ 6 → **false** ✓

**Output:** [true, true, false] ✓

---

**Example 1:** `queries = [[1,2],[2,3,3],[2,3,1],[2,2,2]]`

**[1,2]:** Update seg[2]=2. obs={0,2}.

**[2,3,3]:** prevObs=2, tailGap=1, maxGap=2 → max(1,2)=2 ≥ 3 → **false** ✓

**[2,3,1]:** Same → 2 ≥ 1 → **true** ✓

**[2,2,2]:** prevObs=2, tailGap=0, maxGap=Query(1,2)=2 → max(0,2)=2 ≥ 2 → **true** ✓

**Output:** [false, true, true] ✓

## Pitfalls & Edge Cases

- **SortedSet.Count on view is O(n)** — `GetViewBetween(...).Count > 0` would be O(n), defeating the purpose. Use `foreach` with a `break` to get just the first element in O(log n) via tree traversal.

- **0 always in obs** — Guarantees `GetViewBetween(0, x).Max` is always valid (no empty-set exception). Also correctly handles queries before any type 1 query: block of size sz fits in [0, x] iff x ≥ sz.

- **Tail gap is critical** — When the last obstacle before x is not at x itself, the "free space" after it up to x must be checked separately. The segment tree only stores gaps ending at obstacles, not the trailing portion.

- **Two-step update on type 1** — Adding obstacle at p splits the gap [prev, next] into [prev, p] and [p, next]. Both updates are needed: seg[p] = p − prev, seg[next] = next − p.

- **x ≤ 50000** — Constraint guarantees x ≤ min(5·10⁴, 3·Q) ≤ 50000, so MAXN = 50001 covers all positions.

---

## Key Takeaway

**"Maximum gap in prefix [0, x] under dynamic insertions"** → Segment tree storing gap-per-obstacle, with a sorted set for predecessor/successor lookups.

The insight is that the "max gap" decomposes into:
1. Max internal gap (between two obstacles ≤ x) → range-max on segment tree
2. Tail gap (last obstacle to x) → single sorted set lookup

Both queries are O(log n), and updates require only two point-updates to the segment tree when an obstacle is inserted.

**Pattern recognition:** Any time you need "max gap in a dynamic prefix" combine a segment tree (for range max over gaps) with a BST/sorted set (for predecessor/successor in O(log n)). The two structures complement each other perfectly.
