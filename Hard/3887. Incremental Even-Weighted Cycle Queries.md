# 3887. Incremental Even-Weighted Cycle Queries
## O(E · α(n)) — Parity-Weighted DSU: All-Even-Cycle Condition = 2-Colorability

---

## Problem Statement

Undirected graph, n nodes. Process edges `[u, v, w]` (w ∈ {0,1}) in order; add each edge only if after adding it, every cycle in the resulting graph has **even total weight**. Count edges added.

---

## Core Insight 1 — Even-Cycle Condition = Consistent Node Parity Assignment

**Claim:** A weighted graph (weights 0 or 1) has all cycles with even total weight if and only if there exists a parity assignment `p[v] ∈ {0,1}` for each node v such that for every edge (u, v, w):
```
p[u] XOR p[v] = w
```

**Proof (sketch):**
- **⟸:** If such an assignment exists, for any cycle u₀→u₁→...→uₖ→u₀ with edges of weights w₀,...,wₖ:
  `Σwᵢ mod 2 = ⊕wᵢ = ⊕(p[uᵢ]⊕p[uᵢ₊₁]) = 0` (telescoping XOR). ✓
- **⟹:** If all cycles are even, assign p[v] = XOR-distance from an arbitrary root in each component. This is well-defined (same value regardless of path taken, since all cycles are even) and satisfies the constraint. ✓

This is exactly the condition for a **balanced signed graph** — or equivalently, the graph being "XOR-2-colorable."

---

## Core Insight 2 — DSU with Parity Tracking

Use a **weighted DSU** where each node stores its parity relative to its root. `parity[v]` = XOR of weights along the path from v to its root.

**Find with path compression:**
Returns `(root, parity_of_x_relative_to_root)`.
```
find(x):
  if parent[x] == x: return (x, 0)
  (root, p) = find(parent[x])
  parity[x] ^= p          // compose parities: x→parent→root
  parent[x] = root        // path compression
  return (root, parity[x])
```

**Adding edge (u, v, w):**
- Let `(ru, pu) = find(u)` and `(rv, pv) = find(v)`.
- **Same component (ru = rv):** Edge creates a new cycle. The cycle's parity = `pu XOR pv XOR w`. For the cycle to be even (= 0): need `pu XOR pv = w`. If so, accept (consistent); otherwise reject.
- **Different components:** No cycle created. Always merge. Set `parent[rv] = ru` and `parity[rv] = pu XOR pv XOR w` (encodes the constraint `pu XOR parity[rv] XOR pv = w`). ✓

**Why accepted same-component edges don't require DSU update:** Adding a consistent redundant edge doesn't change the XOR-distance between any pair of nodes (since the path parity through the new edge equals the existing path parity). All future parity checks remain valid. ✓

---

## Algorithm

```
Initialize DSU: parent[i] = i, parity[i] = 0 for all i.
ans = 0
for each edge (u, v, w):
  (ru, pu) = find(u)
  (rv, pv) = find(v)
  if ru == rv:
    if pu XOR pv == w: ans++   // consistent: even cycle created
  else:
    parent[rv] = ru
    parity[rv] = pu XOR pv XOR w
    ans++                       // no cycle: always add
return ans
```

---

## Full Optimal Implementation (C++)

```cpp
class Solution {
    vector<int> par, parity;

    pair<int,int> find(int x) {
        if (par[x] == x) return {x, 0};
        auto [root, p] = find(par[x]);
        parity[x] ^= p;   // path-compress parity: x→root directly
        par[x] = root;
        return {root, parity[x]};
    }

public:
    int numberOfEdgesAdded(int n, vector<vector<int>>& edges) {
        par.resize(n);
        parity.resize(n, 0);
        iota(par.begin(), par.end(), 0);

        int ans = 0;
        for (auto& e : edges) {
            auto [ru, pu] = find(e[0]);
            auto [rv, pv] = find(e[1]);
            int w = e[2];

            if (ru == rv) {
                ans += (pu ^ pv) == w;  // accept iff cycle parity = 0
            } else {
                par[rv] = ru;
                parity[rv] = pu ^ pv ^ w;  // encode constraint
                ans++;
            }
        }
        return ans;
    }
};
```

## Example Walkthroughs

### Example 1 — `n=3`, `edges=[[0,1,1],[1,2,1],[0,2,1]]`, Expected: 2

**State:** par=[0,1,2], parity=[0,0,0].

**Edge [0,1,1]:** find(0)=(0,0), find(1)=(1,0). Different roots.
→ par[1]=0, parity[1]=0⊕0⊕1=**1**. ans=1.
→ par=[0,0,2], parity=[0,1,0].

**Edge [1,2,1]:** find(1): par[1]=0→find(0)=(0,0). parity[1]⊕=0→1. Returns (0,**1**).
find(2)=(2,0). Different roots.
→ par[2]=0, parity[2]=1⊕0⊕1=**0**. ans=2.
→ par=[0,0,0], parity=[0,1,0].

**Edge [0,2,1]:** find(0)=(0,0). pu=0. find(2): par[2]=0→(0,0). parity[2]⊕=0→0. Returns (0,**0**). pv=0.
Same root. pu⊕pv=0⊕0=**0** ≠ 1=w. **Reject.**

**ans=2** ✓

*Meaning: path 0→1→2 has parity 1⊕1=0. Adding edge (0,2,1) would create a cycle 0→1→2→0 with weight 1+1+1=3 (odd). pu⊕pv=0≠1 correctly detects this.*

---

### Example 2 — `n=3`, `edges=[[0,1,1],[1,2,1],[0,2,0]]`, Expected: 3

After first two edges (same as above): par=[0,0,0], parity=[0,1,0].

**Edge [0,2,0]:** find(0)=(0,0). find(2)=(0,0). Same root.
pu⊕pv=0⊕0=**0** == 0=w. **Accept.** ans=3.

**ans=3** ✓

*Path 0→1→2 has parity 0 (1⊕1 mod 2). Adding edge (0,2,0): cycle 0→1→2→0 has weight 1+1+0=2 (even). Consistent.*

---

### Parity Derivation for the Merge Step

After processing [0,1,1] and [1,2,1]:
- parity[1] = 1 means: XOR-distance from node 1 to root 0 = 1. This encodes that node 1 has "color" (parity) 1 relative to node 0's color 0.
- parity[2] = 0: XOR-distance from node 2 to root 0 = 0. Node 2 has color 0 (same as node 0).

Checking edge [0,2,1]: pu (color of 0 relative to root) = 0. pv (color of 2 relative to root) = 0. We need 0 XOR 0 = 0 = w? But w=1. **Contradiction** → reject. ✓

Checking edge [0,2,0]: pu=0, pv=0. Need 0 XOR 0 = 0 = w=0. **Consistent** → accept. ✓

---

### Why Merge Parity Formula `pu XOR pv XOR w` Is Correct

We set `par[rv] = ru` and need `parity[rv]` such that for any future query:
```
parity(u relative to ru) = pu
parity(v relative to ru) = parity[rv] XOR pv
```

Constraint for edge (u,v,w): `pu XOR (parity[rv] XOR pv) = w`
→ `parity[rv] = pu XOR pv XOR w` ✓

**Example:** After adding edge [0,1,1]: parity[1] = pu(0) ⊕ pv(1) ⊕ w = 0⊕0⊕1 = 1. This means traversing from node 1 to root 0 flips parity by 1, encoding that edge (0,1) has weight 1. ✓

## Pitfalls & Edge Cases

- **Path compression updates parity correctly:** When compressing x→parent→root to x→root, we do `parity[x] ^= p` (where p is the parity from parent to root). This correctly sets parity[x] = parity from x to root. ✓

- **Accepting redundant even-cycle edges:** When (u,v,w) is consistent in the same component, we accept it (count it) but don't update the DSU. The parity structure remains valid — adding an even-cycle edge doesn't change XOR-distances between any nodes. ✓

- **Zero-weight edges:** Edge with w=0 requires u and v to have the same parity (same color). If they're in the same component: accept iff pu⊕pv=0 (same parity); else: merge with parity[rv] = pu⊕pv⊕0 = pu⊕pv (difference of their current parities). ✓

- **Self-loops (u=v):** Problem states 0 ≤ u < v < n, so no self-loops. ✓

- **Cycle parity check subsumes multi-cycle cases:** If we add edge (u,v,w) in the same component, it creates exactly one new fundamental cycle. All other new cycles are XOR combinations of this cycle with existing cycles. Since existing cycles are all even, and the new fundamental cycle is also even (we checked), all new cycles are even. ✓

---

## Key Takeaway

**"All cycles have even total weight"** ↔ **"The graph is XOR-2-colorable"** ↔ **"Node parities satisfying `p[u] XOR p[v] = w` for all edges exist"**. The DSU efficiently tracks these parity constraints with O(α(n)) per edge.

**Pattern recognition:** Any problem of the form "check consistency of XOR-constraints between pairs" (graph 2-colorability, balanced sign graphs, parity propagation) → **weighted/parity DSU**. The key invariant: `parity[v]` = XOR distance from v to its component root, maintained under path compression by `parity[x] ^= parity_from_parent_to_root`.
