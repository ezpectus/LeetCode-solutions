# LeetCode 3430. Maximum and Minimum Sums of at Most Size K Subarrays

## 1. Constraints & Complexity Bottleneck Analysis

**Number of Elements (N):** `1 <= N <= 80000`. Due to this constraint, any standard `O(N * k)` or `O(N^2)` brute-force iteration over all possible sub-segments will trigger a Time Limit Exceeded (TLE) status. The entire sum computation must run inside strict `O(N)` linear time boundaries.

**Subarray Length Limit (k):** `1 <= k <= N`. The size of the window is dynamically bounded. When analyzing the contribution of each element as a maximum or minimum, the number of valid left and right extensions must be tightly constrained by `k`, preventing naive two-pointer structural expansions.

**The Combinatorial Boundary Bottleneck:** An element `nums[i]` acts as the minimum (or maximum) for a set of contiguous subarrays. In an unbounded problem, the number of subarrays is simply `(left_choices * right_choices)`. However, introducing the limit `k` breaks this simple product structure. The algorithm must use a combinatorial boundary invariant to calculate the exact count of valid subarrays of length at most `k` in `O(1)` time per element.

## 2. Algorithm Design & Mathematical Invariants

### The Core Concept: Monotonic Stack Contribution Analysis

Instead of evaluating the minimum and maximum elements for every individual subarray, we invert the problem: we calculate exactly how many valid subarrays (of size at most `k`) each element `nums[i]` dominates as the absolute maximum or minimum.

To find these boundaries efficiently, we utilize a **Monotonic Stack**. For each element `nums[i]`, we find:

- **L:** The number of contiguous elements to the left that are strictly greater (for minimums) or strictly smaller (for maximums).
- **R:** The number of contiguous elements to the right that are greater/equal (for minimums) or smaller/equal (for maximums).

> **Note:** To handle duplicate elements correctly without double-counting subarrays, we use a strict inequality on the left boundary and a non-strict inequality on the right boundary.

Once `L` and `R` are determined, `nums[i]` can be extended at most `L` steps to the left and `R` steps to the right.

### The Combinatorial Invariant with Bounded Size K

Let the current element be at index `i`, with a valid left extension capacity of `L` elements and a right extension capacity of `R` elements. The total number of elements chosen from the left is `x` (where `0 <= x <= L`), and from the right is `y` (where `0 <= y <= R`). The resulting subarray contains the element itself, so its length is `x + y + 1`.

The constraint states that the length cannot exceed `k`:

```
x + y + 1 <= k  ==>  x + y <= k - 1
```

We must calculate the number of valid pairs `(x, y)` satisfying these boundaries. This is solved using a flat, constant-time mathematical function:

**`CountSubarrays(L, R, k)`**

**Case 1: No Length Restriction.** If `L + R + 1 <= k`, then every combination is valid. The count is simply:

```
(L + 1) * (R + 1)
```

**Case 2: Bounded Complement Calculation.** If the total possible span exceeds `k`, we evaluate the valid grid points. For each possible left choice `x` from `0` up to `min(L, k - 1)`, the number of valid right choices `y` is bounded by `min(R, k - 1 - x)`. This forms a geometric trapezoid on a discrete coordinate grid, which can be computed in `O(1)` time using arithmetic progression sums rather than loops.

### Architectural Choice: Why Dual Monotonic Passes Are Preferred Over Online Trees

While a Segment Tree or Fenwick Tree can compute range boundaries dynamically in `O(N log N)` time, they are structurally rejected for this high-performance engine due to memory overhead and constant-factor latency.

- **The Tree Overhead:** A Segment Tree requires an array of size `4 * N` and introduces logarithmic pointer-chasing overhead on every update and query pass.
- **The Monotonic Stack Advantage:** A monotonic stack maintains elements in a clean, linear vector. Each index is pushed onto the stack exactly once and popped at most once. This guarantees true `O(N)` execution time with optimal CPU cache locality. By running separate passes for the minimum and maximum states, the code remains highly readable, perfectly optimized, and completely branch-predictor friendly.

## 3. High-Performance C++ Implementation

```cpp
#include <vector>
#include <stack>
#include <algorithm>

using namespace std;

class Solution {
private:
    // O(1) Combinatorial Grid Counter for Subarrays of Length <= K
    long long countSubarrays(long long L, long long R, long long k) {
        if (k <= 0) return 0;
        
        long long total = 0;
        long long max_x = min(L, k - 1);
        
        // Optimize using case divisions to maintain strict O(1) execution cost
        if (max_x + R <= k - 1) {
            // Whole trapezoid/rectangle is valid
            total = (max_x + 1) * (R + 1);
        } else {
            // Calculate using arithmetic progression partitions
            // For each x from 0 to max_x, we sum: min(R, k - 1 - x) + 1
            // Let boundary split happen where R <= k - 1 - x  =>  x <= k - 1 - R
            long long split_x = k - 1 - R;
            
            if (split_x >= 0) {
                long long switch_point = min(max_x, split_x);
                // First part: x <= split_x => min is always R
                total += (switch_point + 1) * (R + 1);
                
                // Second part: x > split_x => min is k - 1 - x
                if (max_x > switch_point) {
                    long long count = max_x - switch_point;
                    long long first_val = k - 1 - (switch_point + 1) + 1;
                    long long last_val = k - 1 - max_x + 1;
                    total += count * (first_val + last_val) / 2;
                }
            } else {
                // R is always greater than k - 1 - x for all x >= 0
                // Thus min is always k - 1 - x + 1
                long long first_val = k - 1 - 0 + 1;
                long long last_val = k - 1 - max_x + 1;
                total += (max_x + 1) * (first_val + last_val) / 2;
            }
        }
        return total;
    }

    // High-Speed Contiguous Boundary Solver
    long long computeContribution(const vector<int>& nums, int k, bool find_max) {
        int n = nums.size();
        vector<int> left_bound(n), right_bound(n);
        stack<int> st;

        // Pass 1: Resolve Left Strict Boundaries
        for (int i = 0; i < n; ++i) {
            while (!st.empty()) {
                bool condition = find_max ? (nums[st.top()] < nums[i]) : (nums[st.top()] > nums[i]);
                if (condition) st.pop();
                else break;
            }
            left_bound[i] = st.empty() ? -1 : st.top();
            st.push(i);
        }

        while (!st.empty()) st.pop();

        // Pass 2: Resolve Right Non-Strict Boundaries (Prevents Duplicate Double Counting)
        for (int i = n - 1; i >= 0; --i) {
            while (!st.empty()) {
                bool condition = find_max ? (nums[st.top()] <= nums[i]) : (nums[st.top()] >= nums[i]);
                if (condition) st.pop();
                else break;
            }
            right_bound[i] = st.empty() ? n : st.top();
            st.push(i);
        }

        // Pass 3: Accumulate Linear Combinatorial Sums
        long long total_sum = 0;
        for (int i = 0; i < n; ++i) {
            long long L = i - left_bound[i] - 1;
            long long R = right_bound[i] - i - 1;
            long long sub_count = countSubarrays(L, R, k);
            total_sum += sub_count * nums[i];
        }

        return total_sum;
    }

public:
    long long minMaxSubarraySum(vector<int>& nums, int k) {
        long long min_contribution = computeContribution(nums, k, false);
        long long max_contribution = computeContribution(nums, k, true);
        return min_contribution + max_contribution;
    }
};
```

## 4. Comprehensive Complexity & Mathematical Analysis

### Detailed Time Complexity Decomposition

The computational pipeline of this optimization engine is split into isolated processing steps. The total time complexity is derived from the summation of these steps:

```
T_total = T_min_pass + T_max_pass + T_accumulation
```

**Minimum State Boundary Passes (T_min_pass):**

The left monotonic search loops through `N` elements. Inside the loop, elements are pushed onto the stack once and popped at most once across the entire run. This yields a strict amortized operation bound: `O(N)`.

The right non-strict monotonic search mirrors this behavior from right to left, processing the array elements in `O(N)`.

```
T_min_pass = O(N)
```

**Maximum State Boundary Passes (T_max_pass):**

The logic duplicates the minimum pass layout but flips the comparison inequalities. It processes the vector coordinates via linear stack entries, ensuring an exact runtime match:

```
T_max_pass = O(N)
```

**O(1) Algebraic Sum Accumulation (T_accumulation):**

The final calculation loop iterates through the `N` array items exactly once. At each coordinate, it calculates the valid subarray counts using arithmetic progression logic instead of nested loops. This runs in true deterministic `O(1)` clock cycles per element:

```
T_accumulation = O(N)
```

Combining these stages, the total asymptotic runtime complexity is tightly bounded by:

```
T_total = O(N) + O(N) + O(N) = O(N)
```

### Detailed Space Complexity Decomposition

The structural memory overhead is determined by the following internal allocation blocks:

**Boundary Vector Layouts:**

- `left_bound` vector caching left boundary indices: `N * 4 bytes` => `O(N)`
- `right_bound` vector caching right boundary indices: `N * 4 bytes` => `O(N)`

These temporary arrays are allocated inside the execution frames and recycled instantly.

**Monotonic Stack Overhead:**

The `st` integer stack tracks active array indices. In the absolute worst-case scenario (a perfectly sorted array layout), the stack holds at most `N` elements before a clearing pass:

```
S_stack = N * 4 bytes => O(N)
```

**Total Auxiliary Footprint:**

No multi-dimensional grids, maps, or auxiliary heap spaces are used. The implementation operates inside linear space limits, making it completely safe from stack overflows or high memory usage:

```
S_total = O(N)
```

## 5. Mathematical Formulations & Foundational Algorithms

### The Boundary Domination Invariant

The algorithm maps every individual index `i` into an isolated domain of extreme values. For a given element `nums[i]`, we define its left and right boundaries `left_bound[i]` and `right_bound[i]` such that:

- For all indices `j` in the range `left_bound[i] < j < i`, `nums[j] > nums[i]` (when finding minimums)
- For all indices `j` in the range `i < j < right_bound[i]`, `nums[j] >= nums[i]` (when finding minimums)

This mathematical definition guarantees that `nums[i]` is the absolute minimum element for any subarray starting at index `start` and ending at index `end` if and only if:

```
left_bound[i] < start <= i   AND   i <= end < right_bound[i]
```

### The Discrete Arithmetic Progression Equation

Let `L = i - left_bound[i] - 1` and `R = right_bound[i] - i - 1`. The total valid choices for the left extension size `x` and right extension size `y` are bounded by:

```
0 <= x <= L,   0 <= y <= R,   x + y <= k - 1
```

When the constraint `x + y <= k - 1` cuts through the available rectangular choice space `(L + 1) * (R + 1)`, the number of valid pairs in the overlapping region is calculated using discrete summation.

For the diagonal cutting line where the right boundary becomes restricted (`y = k - 1 - x`), the remaining options form a linear sequence where each step drops by exactly `1`. The total points under this sloped line are accumulated instantly via the classic arithmetic progression formula:

```
Sum = Count * (First_Term + Last_Term) / 2
```

This algebraic reduction converts a slow `O(k)` linear loop into a single constant-time `O(1)` step. This allows the engine to compute the exact subarray counts for any input array size instantly.
