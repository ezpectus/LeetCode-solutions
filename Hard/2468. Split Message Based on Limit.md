# 2468. Split Message Based on Limit  
## $O(n^2)$ worst‑case search on parts, $O(n)$ construction

---

## Problem Statement

You are given:
* A string `message`
* An integer `limit`

You must split `message` into one or more parts, each part having a suffix of the form:
`"<a/b>"`

where:
* `b` = total number of parts
* `a` = index of the part (1‑based)

### Constraints:
* Each part’s total length (content + suffix) must be:
  * Exactly `limit` for all parts except possibly the last
  * The last part’s length must be $\le$ `limit`
* Concatenating all parts’ contents (without suffixes) in order must equal `message`.
* The number of parts `b` must be as small as possible.
* If impossible, return an empty array.

---

## Core Idea — Guess Number of Parts and Check Capacity

For a fixed number of parts `b`, we know that the suffix for part `a` is `"<a/b>"`. Its length is:
`lenSuffix(a, b) = 3 + digits(a) + digits(b)`  *(where `"<"`, `"/"`, and `">"` contribute 3 characters)*

For part `a`, the maximum number of message characters we can put is:
`capacity(a, b) = limit - lenSuffix(a, b)`

This must be $> 0$ for every part we actually use. The total capacity over all `b` parts is:
`totalCapacity(b) = Σ (limit - lenSuffix(a, b))` for $a = 1..b$

We need to find the smallest `b` such that:
`totalCapacity(b) >= message.Length` and all `capacity(a, b) > 0`.

---

## Algorithm

1. **Quick Impossibility Check** The smallest possible suffix is `"<1/1>"` (length = $3 + 1 + 1 = 5$). If `limit < 5`, there is no room even for a single character + suffix → return `[]`.

2. **Try all possible `b` from 1 up to an upper bound** In the worst case, each part carries at least 1 character, so $b \le \text{message.Length}$.  
   We can iterate `for b in [1 .. message.Length]`:
   * Compute `digitsB = digits(b)`.
   * Initialize `totalCapacity = 0`.
   * For `a` from 1 to `b`:
     * `lenSuffix = 3 + digits(a) + digitsB`.
     * If `lenSuffix >= limit` → this part has no room for content, so this `b` is invalid.
     * `totalCapacity += limit - lenSuffix`.
     * If `totalCapacity >= message.Length` → `b` is feasible; stop and accept this `b`.
   * We pick the first `b` that is feasible. If none is feasible → return `[]`.

3. **Construct the actual parts** Once we know the minimal `b`:
   * Track `pos = 0` (current index in message).
   * For `a` from 1 to `b`:
     * `lenSuffix = 3 + digits(a) + digits(b)`.
     * `cap = limit - lenSuffix`.
     * `take = min(cap, message.Length - pos)`.
     * `content = message.Substring(pos, take)`.
     * `suffix = "<" + a + "/" + b + ">"`.
     * Add `content + suffix` to the result, and advance `pos += take`.

## Full Implementation (C#)

```csharp
public class Solution {
    public string[] SplitMessage(string message, int limit) {
        int n = message.Length;

        // Minimal suffix length is "<1/1>" = 5
        if (limit < 5)
            return Array.Empty<string>();

        int bestB = -1;

        // Try all possible number of parts
        for (int b = 1; b <= n; b++) {
            int digitsB = Digits(b);
            long totalCapacity = 0;
            bool ok = true;

            for (int a = 1; a <= b; a++) {
                int lenSuffix = 3 + Digits(a) + digitsB;
                if (lenSuffix >= limit) {
                    ok = false;
                    break;
                }
                totalCapacity += (limit - lenSuffix);
                if (totalCapacity >= n)
                    break;
            }

            if (!ok) continue;
            if (totalCapacity >= n) {
                bestB = b;
                break;
            }
        }

        if (bestB == -1)
            return Array.Empty<string>();

        // Build the actual parts
        var res = new List<string>();
        int pos = 0;
        int digitsBestB = Digits(bestB);

        for (int a = 1; a <= bestB; a++) {
            int lenSuffix = 3 + Digits(a) + digitsBestB;
            int cap = limit - lenSuffix;

            int remaining = n - pos;
            int take = Math.Min(cap, remaining);
            if (take < 0) take = 0; // safety

            string content = message.Substring(pos, take);
            string suffix = "<" + a.ToString() + "/" + bestB.ToString() + ">";
            res.Add(content + suffix);

            pos += take;
        }

        if (pos != n)
            return Array.Empty<string>();

        return res.ToArray();
    }

    private int Digits(int x) {
        int d = 0;
        while (x > 0) {
            d++;
            x /= 10;
        }
        return d;
    }
}
```

## Complexity

| Metric | Value |
| :--- | :--- |
| **Time Complexity** | $O(n^2)$ worst case (`b` loop $\times$ `a` loop) |
| **Space Complexity** | $O(n)$ for result and substrings |

*Given $n \le 10^4$, an $O(n^2)$ worst-case capacity search is highly efficient and acceptable in practice.*

---

## Pitfalls & Edge Cases

* **Dynamic Suffix Length:** Suffix length depends on both `a` and `b`: `3 + digits(a) + digits(b)`.
* **Zero Capacity:** If `lenSuffix >= limit` for any part, that `b` is invalid.
* **Exact Check:** You must ensure total capacity $\ge$ `message.Length` before accepting `b`.
* **The Last Part:** The last part is allowed to be shorter than `limit`, but never longer.
* **Hard Limit:** `limit < 5` is immediately impossible (no room for even 1 char + minimal suffix).

## Example Trace

### Example 2
`message = "short message"`, `limit = 15`

We try increasing `b` until it becomes feasible:
* **Try `b = 2`:**
  * **Part 1 ($a = 1$):** `suffix = "<1/2>"`, `lenSuffix = 5` $\rightarrow$ `cap = 15 - 5 = 10`.  
    Takes `"short mess"`.
  * **Part 2 ($a = 2$):** `suffix = "<2/2>"`, `lenSuffix = 5` $\rightarrow$ `cap = 15 - 5 = 10`.  
    Remaining content: `"age"` (3 chars). Part becomes `"age<2/2>"` (length $8 \le 15$).

**Result:** `["short mess<1/2>", "age<2/2>"]`

---

## Key Takeaway

The problem is entirely about **capacity accounting**:
For each candidate number of parts `b`, compute how many message characters can fit once you reserve space for all suffixes. 
Once you find the smallest feasible `b`, constructing the parts is straightforward by slicing the message using per‑part capacities.
