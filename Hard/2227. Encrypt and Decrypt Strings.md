# 2227. Encrypt and Decrypt Strings
## O(D·L) constructor, O(L) encrypt, O(1) decrypt — Pre-Encrypt the Dictionary

---

## Problem Statement

`Encrypt` maps each character to a fixed 2-char string. `Decrypt(word2)` asks: how many words in `dictionary` produce `word2` when encrypted? Multiple decryptions of word2 may be valid (one encrypted chunk can map to multiple original chars), so naively enumerating all decryptions is exponential. The key is to flip the perspective.

---

## Core Insight — Decrypt by Counting Forward Encryptions

**Observation:** A dictionary word `d` is a valid decryption of `word2` if and only if `Encrypt(d) == word2`.

Instead of:
```
word2 → enumerate all possible decryptions → check each against dictionary
```
Do:
```
dictionary[i] → Encrypt(dictionary[i]) → store in frequency map
```
Then `Decrypt(word2)` = `freqMap[word2]` — a single hash map lookup.

---

## Why This Is Better

Naive decryption: each 2-char chunk may map to 0–26 original characters. For a word2 of length 2L, there are up to 26^L possible decryptions — exponential. For L=100, completely intractable.

Pre-encryption of dictionary: `|dictionary| ≤ 100`, each word ≤ 100 chars. Total preprocessing: O(D × L) = O(10,000) string operations — trivial.

---

## Full Implementation (C#)

```csharp
public class Encrypter {
    private readonly Dictionary<char, string> encMap = new();
    private readonly Dictionary<string, int> encryptedCount = new();

    public Encrypter(char[] keys, string[] values, string[] dictionary) {
        // Build char → 2-char encryption map
        for (int i = 0; i < keys.Length; i++)
            encMap[keys[i]] = values[i];

        // Pre-encrypt every dictionary word and count by encrypted form
        foreach (string word in dictionary) {
            string enc = Encrypt(word);
            if (enc.Length == 0) continue; // word has unencryptable chars (impossible per constraints, but safe)
            encryptedCount[enc] = encryptedCount.GetValueOrDefault(enc) + 1;
        }
    }

    public string Encrypt(string word1) {
        var sb = new System.Text.StringBuilder(word1.Length * 2);
        foreach (char c in word1) {
            if (!encMap.TryGetValue(c, out string? val)) return "";
            sb.Append(val);
        }
        return sb.ToString();
    }

    public int Decrypt(string word2) {
        encryptedCount.TryGetValue(word2, out int count);
        return count;
    }
}
```

---

## Complexity

| Operation | Time | Notes |
|---|---|---|
| Constructor | O(D · L) | D ≤ 100 words, L ≤ 100 chars each |
| Encrypt | O(L) | L = word length, StringBuilder avoids quadratic concat |
| Decrypt | O(1) | Hash map lookup |
| Space | O(D · L + K) | encrypted strings + enc map |

With ≤ 200 total calls at ≤ 2000 char inputs: at most 200 × 2000 × 2 = 800K char operations.

## Example Walkthrough

**Setup:** keys=`['a','b','c','d']`, values=`["ei","zf","ei","am"]`

**Encryption map:** a→"ei", b→"zf", c→"ei", d→"am"

**Pre-encrypting dictionary:**
| Word | Encrypted |
|---|---|
| "abcd" | "ei"+"zf"+"ei"+"am" = **"eizfeiam"** |
| "acbd" | "ei"+"ei"+"zf"+"am" = "eieizfam" |
| "adbc" | "ei"+"am"+"zf"+"ei" = "eiamzfei" |
| "badc" | "zf"+"ei"+"am"+"ei" = "zfeiamei" |
| "dacb" | "am"+"ei"+"ei"+"zf" = "ameieizf" |
| "cadb" | "ei"+"ei"+"am"+"zf" = "eieiamzf" |
| "cbda" | "ei"+"zf"+"am"+"ei" = "eizfamei" |
| "abad" | "ei"+"zf"+"ei"+"am" = **"eizfeiam"** |

`encryptedCount["eizfeiam"] = 2` (from "abcd" and "abad").

**`Encrypt("abcd")`:** a→"ei", b→"zf", c→"ei", d→"am" → `"eizfeiam"` ✓

**`Decrypt("eizfeiam")`:** `encryptedCount["eizfeiam"] = 2` ✓

The 2 valid decryptions are "abcd" and "abad". The other two candidates from the chunk ambiguity ("cbad" and "cbcd") are not in the dictionary.

## Pitfalls & Edge Cases

- **Many-to-one encryption:** Multiple chars can map to the same 2-char string (here both 'a' and 'c' map to "ei"). This is exactly why naive decryption is exponential — but irrelevant to our forward-encryption approach.

- **Empty string from Encrypt:** If a word has a char not in keys, Encrypt returns "". We skip such dictionary words during preprocessing (`if (enc.Length == 0) continue`). Per constraints, all dictionary chars appear in keys, so this is defensive only.

- **Decrypt words not in dictionary:** If word2 doesn't match any pre-encrypted form, `encryptedCount.TryGetValue` returns 0 — correct.

- **StringBuilder vs string concatenation:** In the Encrypt loop, naive `result += val` builds intermediate strings, giving O(L²) total. `StringBuilder` with pre-allocated capacity `word.Length * 2` gives O(L).

- **Multiple calls to Decrypt:** Since we pre-built the map in the constructor, each Decrypt call is O(1) regardless of call count — critical when up to 200 calls are made.

---

## Key Takeaway

This is the **"invert the query direction"** pattern. When a "reverse lookup" is exponential but the "forward lookup" is cheap, precompute all forward lookups from the known set (dictionary) and answer reverse queries with direct lookups. The constructor does O(D·L) work once; each Decrypt query costs O(1) forever after.

**Pattern recognition:** "Count how many items in a set produce a given output under some function" → precompute `function(item)` for all items, store in frequency map, answer queries with map lookups. Avoids enumerating all inverse function values (which can be exponential).
